**What would you like to be added**:

Graduate SchedulingEquivalenceHashing to Beta. 

That was originally the plan for the bugfix in https://github.com/kubernetes-sigs/kueue/pull/10001. but we encountered some complications with :
1. MultiKueue preemption gates mechanism requiring adjustment
2. Performance regression tests requiring adjustment (lowering the requeue interval to 100ms helps)
3. many unit tests failing requiring adjustments

**Why is this needed**:

This is a very useful performance bugfix for large scale systems, where currently Kueue scheduler wastes
a lot of cycles processing identical (from the scheduling perspective) workloads.

### Comment by [@kshalot](https://github.com/kshalot) — 2026-03-19T17:39:47Z

> Another approach would be to just always add something unique to the hash for workloads with preemptionGates. For example Workload UUID, when computing the hash.

I like that direction, the only reservation I have is the fact that it makes the hash a little bit harder to reason about. Just in principle, because it's not a practical difference. And potentially missing the fact that you need to add new handling to the hash if we have a new scenario where the equivalent workloads shouldn't be skipped (e.g. "borrowing gates").

> 1. add preemptionGate to the hash if the condition is already set
> 2. add UUID to the has if the condition is not set yet

You mean either 1 or 2 or both? 2 on its own seems sufficient, because if the condition is set I think we can skip the workload regardless of whether it has the gate opened or closed:
1. Opened - we won't fit anyway.
2. Closed - we already signaled anyway. We are currently overwriting the transition time of the condition on each pass, but I think this is an oversight, we should not overwrite the timestamp.

### Comment by [@sohankunkerkar](https://github.com/sohankunkerkar) — 2026-03-31T20:54:58Z

We've been seeing this as well where one architecture flavor (e.g., s390x) is stuck at full capacity, high-priority workloads        targeting it head-of-line block schedulable workloads for healthy architectures. 

I think the graduation path should be narrowing the mechanism to the safe cases first, not re-enabling the current broad behavior. Building on what @mimowo and @kshalot discussed, I think we need to control when equivalence skipping applies at requeue time using an allowlist of safe reasons, rather than encoding more into the hash itself. 

Today both `NoFit` and `Preempt-no-candidates` fall through `RequeueReasonGeneric`, so we'd first need explicit requeue reasons for each, then allow hashing only for `NoFit`. That way `Preempt-no-candidates` (transient), `NamespaceMismatch` (workload-specific), and `PreemptionGated` (needs individual processing) all default to no-hash without needing UUID tricks or hash modifications. The broader `queueInadmissibleWorkloads` blind-requeue problem is separate effort and shouldn't block this.
