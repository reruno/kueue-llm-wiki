### Comment by [@kannon92](https://github.com/kannon92) — 2026-03-19T23:58:24Z

This is fixed in #10036.

**Root cause:** The `getWorkloadsOnNode` function in `pkg/controller/tas/node_controller.go:322` iterates over all workloads in the cluster and accesses `wl.Status.Admission.PodSetAssignments` without checking if `Admission` is nil. Workloads that haven't been admitted yet (e.g., pending in the queue due to insufficient resources) have a nil `Admission` field, so when any node is reconciled and an unadmitted workload exists, it triggers a nil pointer dereference panic.

**Fix:** Added a nil check (`wl.Status.Admission != nil`) before accessing `PodSetAssignments`. Unadmitted workloads can't have a TAS assignment on any node, so they safely fall through to the next check.

**Testing:** Added an integration test that creates both an admitted and a pending (unadmitted) workload, then deletes a node. Verified the test fails (panic) without the fix and passes with it.
