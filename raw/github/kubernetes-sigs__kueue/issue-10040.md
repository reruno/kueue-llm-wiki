## Description

## What happened

In a MultiKueue setup with KubeRay, upgrading from v0.13.3 to v0.16.4 caused a storm of Patch conflicts and multikueue reconciler errors, leading to 23 workload evictions during a 2.5-hour window before rollback.

### Root cause

`ShouldSkipLocalExecution` is gated behind `isComposable` ([reconciler.go:1257](https://github.com/kubernetes-sigs/kueue/blob/main/pkg/controller/jobframework/reconciler.go#L1257)). RayJob doesn't implement `ComposableJob`, so the manager always runs `startJob` — even when the workload has a MultiKueue admission check.

Since RayJobs on the manager have `managedBy=kueue.x-k8s.io/multikueue` (KubeRay skips them), the race is 2-way: `startJob` (Patch to unsuspend) vs `SyncJob` (PatchStatus to copy remote status). Both use strict MergePatch with shared `resourceVersion`, so concurrent writes cause 409 Conflicts.

The conflict self-resolves — once startJob succeeds and unsuspends the job, it stops writing, and SyncJob succeeds on the next attempt. But the error volume (133 + 264 errors in 2.5h) prompted controller restarts during debugging. Each restart drops remote cluster connections, and workloads whose AC was already `CheckStateReady` get evicted after `workerLostTimeout` (15 min) with "Reserving remote lost."

**Note:** There is no timeout for ACs stuck in `CheckStatePending` — the `workerLostTimeout` eviction only applies to ACs that already reached `CheckStateReady`. The evictions came from the controller restarts, not directly from the conflicts.

### Why this regressed

The gate was introduced in #8189 as `isComposable || isBatchJobWithoutManagedBy`. When `MultiKueueBatchJobWithManagedBy` went GA, the batch condition was always false and got removed, leaving only `isComposable` — which covers only Pod (the sole ComposableJob implementer). All other 17 job types with MultiKueue adapters were unprotected.

### Production data

**v0.13.3 (stable, 4-hour window):**
- 17 startJob conflicts on worker clusters (2-way: startJob vs KubeRay)
- All resolved via re-reconcile
- Zero evictions

**v0.16.4 (2.5-hour window before rollback):**
- 133 startJob conflicts on the **manager** (shouldn't happen at all)
- 264 multikueue reconciler errors
- 23 evictions in 15-minute waves (from controller restarts + workerLostTimeout)
- 4 controller restarts during debugging amplified the issue

### Proposed fix

Remove the `isComposable` gate so `ShouldSkipLocalExecution` applies to all job types. This eliminates startJob on the manager entirely. PR: #10061

## Environment

- Kueue version: v0.13.3 (production), v0.16.4 (brief upgrade, rolled back)
- Kubernetes version: v1.31
- KubeRay version: v1.5.0
- MultiKueue: enabled (1 manager, 4 workers)

### Comment by [@trilamsr](https://github.com/trilamsr) — 2026-03-20T23:11:27Z

Hey @mimowo, thanks for the pushback on `WithRetryOnConflict`. You were right that re-reconcile handles the conflict correctly — we confirmed that on our v0.13.3 cluster (17 conflicts in 4 hours, all self-healed in 29–152ms). I've closed the fix PR (#10041).

But after digging into our Axiom logs from the brief v0.16.4 upgrade, I found the actual root cause. It's not about retry strategy — it's about `ShouldSkipLocalExecution` being gated behind `isComposable`.

### The bug

In `startJob` ([reconciler.go:1257](https://github.com/kubernetes-sigs/kueue/blob/main/pkg/controller/jobframework/reconciler.go#L1257)):

```go
if isComposable {
    skip, err := admissioncheck.ShouldSkipLocalExecution(ctx, r.client, wl)
```

RayJob doesn't implement `ComposableJob`, so the manager always runs `startJob` for RayJob — even when the workload has a MultiKueue admission check. Combined with the `IsSuspended` guard removal from SyncJob (#8308), this creates a 3-way race on the local RayJob on the manager:

1. **startJob** — Patch to unsuspend
2. **SyncJob** — PatchStatus to copy remote status
3. **KubeRay** — finalizer + state transitions

On v0.13.3 this was 2-way (SyncJob skipped suspended jobs), and re-reconcile handled it fine. On v0.16.4 the 3-way race is persistent enough that SyncJob keeps failing → `syncReservingRemoteState` never runs → AC stays in Pending → after `workerLostTimeout` (15 min) → eviction.

### Production evidence (v0.16.4, 2.5 hours before rollback)

- 133 startJob conflicts (vs 17 on v0.13.3 in 4 hours)
- 264 multikueue workload reconciler errors
- 23 evictions in 15-minute waves matching `workerLostTimeout`

The startJob conflicts are happening on the **manager** (not the worker), which shouldn't be running startJob at all for MultiKueue workloads.

### Suggested fix

Remove the `isComposable` gate so `ShouldSkipLocalExecution` applies to all job types. Happy to send a PR if this direction makes sense.

### Comment by [@trilamsr](https://github.com/trilamsr) — 2026-03-20T23:57:05Z

Corrections after deeper investigation:

**The race is 2-way, not 3-way.** RayJobs on the manager have `managedBy=kueue.x-k8s.io/multikueue`, so KubeRay skips them entirely. The only concurrent writers are `startJob` (spec patch to unsuspend) and `SyncJob` (status patch to copy remote state). They share `resourceVersion`, so both use strict MergePatch and conflict.

**The conflict self-resolves.** Once startJob succeeds and unsuspends the job, it stops — next reconcile sees `IsSuspended() == false` and does nothing. SyncJob then succeeds on its next attempt, `syncReservingRemoteState` runs, AC reaches Ready. There's no persistent loop.

**The evictions came from controller restarts, not from the conflicts directly.** There's no timeout for ACs stuck in `CheckStatePending` — the `workerLostTimeout` eviction only fires when AC is already `CheckStateReady` and the remote connection is lost. The 23 evictions happened because the error storm (133 + 264 errors) prompted 4 controller restarts during debugging. Each restart drops remote connections → workloads that were Ready get evicted after 15 min.

**The `isComposable` gate's history:** It was `isComposable || isBatchJobWithoutManagedBy` in #8189. When `MultiKueueBatchJobWithManagedBy` went GA, the batch condition was removed, leaving only `isComposable` — which covers only Pod. All other 17 job types with MultiKueue adapters were unprotected.

**Removing the gate entirely is correct** — `managedBy` stops external controllers, `ShouldSkipLocalExecution` stops the reconciler. Both should always be active. The check is cheap (one informer-cached `client.Get`).

Updated the PR and issue descriptions in #10061 with these corrections.
