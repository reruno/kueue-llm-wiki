**What would you like to be added**:

Two related improvements to `queueInadmissibleWorkloads`:

1. **Preserve `noFitSchedulingHashes` across requeue cycles**: Currently, [`queueInadmissibleWorkloads`](https://github.com/kubernetes-sigs/kueue/blob/046242d69e66579429d6b6937c4278fce9dec950/pkg/cache/queue/inadmissible_workloads.go#L168-L194) unconditionally clears `noFitSchedulingHashes` (line 174) every time it runs. This forces the scheduler to rediscover which equivalence classes are inadmissible from scratch each cycle, even when conditions for those classes haven't changed. Instead, only hashes associated with resource flavors/conditions that actually changed should be cleared.

2. **Selective requeue based on the triggering event**: `queueInadmissibleWorkloads` should only move inadmissible workloads back to the heap if the specific resource conditions that caused them to be inadmissible have changed. This is tracked as a long-standing TODO in the codebase:

   https://github.com/kubernetes-sigs/kueue/blob/67b14e1e18dede91e7d655d33d0af7783105ad92/pkg/cache/queue/manager.go#L334-L335

   ```go
   // TODO(#8): Selectively move workloads based on the exact event.
   // If any workload becomes admissible or the queue becomes active.
   ```

**Why is this needed**:

In production, we observed priority-based head-of-line blocking with BestEffortFIFO when a resource flavor is stuck at full capacity (e.g., s390x at 100% due to infrastructure issues).

We are aware of the mitigations introduced under the umbrella issue #9715:
- **Batched requeue** (#8095): Completions are coalesced with a 1s batch period via `workqueueRequeuer`
- **`SchedulingEquivalenceHashing`** (#9694): Bulk-moves equivalent NoFit workloads to inadmissible in one pop
- **`SchedulerLongRequeueInterval`** (#9799): Increases batch period to 10s

However, the core problem persists because:

1. **Without `SchedulingEquivalenceHashing` (the default)**: The starvation is severe. Every requeue cycle moves all stuck high-priority workloads back to the top of the priority-sorted heap. The scheduler pops them one-by-one, each failing and going back to inadmissible, but the next requeue moves them right back before the scheduler reaches admissible lower-priority workloads. Resources sit idle while the scheduler cycles through stuck workloads that can never be admitted.

2. **With `SchedulingEquivalenceHashing` enabled**: The starvation is short-lived (~seconds per cycle) but creates **unnecessary churn** — every requeue cycle clears ALL `noFitSchedulingHashes` and moves ALL inadmissible workloads to the heap, even when nothing changed for those workloads. The scheduler must rediscover the same NoFit hashes from scratch each cycle. In large environments (thousands of workloads), this adds up to significant wasted work.

**Scenario:**
- ClusterQueue with multiple resource flavors (x86, s390x, ppc64le)
- s390x/ppc64le stuck at 100% due to infrastructure issues
- High-priority (700) workloads targeting stuck flavors in the queue
- Lower-priority (400, 500) workloads targeting x86 with available quota
- x86 workloads complete at ~0.5-1/sec, each triggering requeue notifications
- Without `SchedulingEquivalenceHashing`: the scheduler never admits the lower-priority x86 workloads despite available resources
- With `SchedulingEquivalenceHashing`: the scheduler reaches x86 workloads each cycle but wastes time rediscovering NoFit hashes

**Related issues:** #8, #9715, #9694, #8095, #9799
