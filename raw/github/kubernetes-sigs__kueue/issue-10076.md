## Description

## Summary

TopologyAwareScheduling (TAS) releases capacity in its internal accounting when a Workload CRD is marked Finished, but the underlying pods may still be in Terminating state and consuming node resources. This causes TAS to assign new pods to nodes that are actually full, resulting in persistent FailedScheduling events.

## Root Cause

TAS tracks capacity at the Workload level, not the pod level:

1. When a Workload's `Finished` condition is set, `deleteWorkload()` subtracts the workload's resource usage from `tasUsage` in `TASFlavorSnapshot`
2. However, the workload's pods may still be Terminating on the node — kubelet hasn't reclaimed the resources yet
3. Terminating TAS pods are also excluded from `nonTasUsageCache` because `filterPod()` in `non_tas_usage_controller.go` skips pods with TAS annotations
4. When a new workload is admitted, `findTopologyAssignment` → `fillInCounts` sees the node as having free capacity and assigns the new pod there via hard `nodeSelector`
5. kube-scheduler cannot place the pod because the Terminating pod still holds the resources

Terminating TAS pods fall into a gap between both tracking systems: no longer in `tasUsage` (Workload is Finished) and excluded from `nonTasUsageCache` (pod has TAS labels).

## Impact

This is particularly impactful for large GPU workloads (e.g., 8x A100 GPUs per pod) because:
- Each pod consumes an entire bare-metal node's resources
- TAS writes hard `nodeSelector` (hostname-level), so the new pod cannot fall back to another node
- If a node has hardware issues (GPU errors, RDMA failures), pods can be stuck Terminating for hours or days
- This creates a cascading effect where multiple new workloads get assigned to the same "phantom free" node and all fail to schedule

In our production environment, we observed:
- A pod stuck Terminating for 43+ hours on a node with GPU PCIe errors (kubelet issued 139+ kill attempts)
- Its Workload CRD was cleaned up, so TAS believed the node had 8 free GPUs
- Multiple new workloads were assigned to that node, all hitting FailedScheduling
- Dozens of FailedScheduling events across different workloads in a single hour, all with the same pattern: `1 Insufficient nvidia.com/gpu, 288 node(s) didn't match Pod's node affinity/selector`

## Environment

- Kueue with TAS enabled
- Topology: 3 levels (network_block_id → rack_id → hostname)
- ResourceFlavor with `topologyName` referencing the topology
- Workloads using `kueue.x-k8s.io/podset-preferred-topology` annotation
- Bare-metal GPU nodes (BM.GPU.A100-v2.8)

## Possible Fixes

Some options to consider:

1. **Don't release `tasUsage` until all pods are actually terminated** — delay subtracting from the TAS capacity cache until pod finalizers/deletion complete, not just when the Workload is marked Finished
2. **Include Terminating TAS pods in `nonTasUsageCache`** — remove the TAS pod exclusion in `filterPod()` for pods that have a `deletionTimestamp` set, so they're counted against node capacity
3. **Add a grace period to TAS capacity release** — after a Workload is Finished, keep its capacity reserved for `terminationGracePeriodSeconds` + buffer before releasing

### Comment by [@ananth1996](https://github.com/ananth1996) — 2026-04-23T08:48:24Z

We're seeing the same gap on our side, though in a different shape from the stuck-Terminating case.

Our PyTorchJob pods have `terminationGracePeriodSeconds` set to ~5min so checkpointing can finish on SIGTERM. That's by design and the node is healthy, but because TAS releases capacity as soon as the Workload is marked Finished, any newly admitted job targeting the same topology domain gets placed immediately and its pods sit Pending for the full grace period waiting for the old pods to actually exit.

In practice we had to bump `waitForPodsReady.timeout` well above the checkpointing grace period to stop these admitted-but-blocked workloads from being evicted and requeued.

@mimowo any rough sense of when the configurable release policy you mentioned (fast on `deletionTimestamp` vs. delayed until pods are terminal, via `IsActive`) might land, or whether there's a design doc / PR we could follow or help with? For the "planned long grace period" case option (b) would be exactly what we need.
