## Description

## What happened

With the following node configuration, CQ/cohort settings, and deployments:

**node**:

|*node*|cpu|gpu|
|---|---|---|
|node-1|8|8|
|node-2|8|8|

**cohort**:

|*cohort*|cpu quota|gpu quota|
|---|---|---|
|cohort|16|16|

**CQ**:

|*CQ*|parent cohort|cpu borrowing limit|gpu borrowing limit|
|---|---|---|---|
|low|cohort|16|16|
|standard|cohort|16|16|

```
preemption:
  reclaimWithinCohort: Any
  withinClusterQueue: LowerPriority
  borrowWithinCohort:
    policy: LowerPriority
```

**deployment**:

|*deployment*|CQ|priority|replicas|requests/cpu|requests/gpu|
|---|---|---|---|---|---|
|low-cpu|low|-1000|2|5|-|
|low-gpu|low|-1000|2|10m|8|
|standard|standard|0|1|5|8|

When a user submits the workloads in the order `low-cpu` -> `low-gpu` -> `standard`, Kueue does not admit `standard`, even though it has higher priority.

This seems to happen only when admitting `standard` would require both quota-level preemption and topology-level preemption. In the above example:
- Quota-level preemption is required since `low-gpu` consumes all of the cohort's nominal gpu quota. 
- Topology-level preemption is required since each node can take at most one pod from `low-cpu` or `standard`.

By contrast, `low-cpu` -> `standard` works as expected. `low-gpu` -> `standard` also works as expected.
In those cases, `standard` is admitted by preempting lower-priority workloads as needed.

## What you expected to happen

The higher-priority workload `standard` is admitted by preempting lower-priority workloads as needed.

### Comment by [@pajakd](https://github.com/pajakd) — 2026-03-27T12:56:52Z

I'm not sure if this is using TAS (Topology Aware Scheduling). I see no explicit annotations so perhaps it is not.

If it is not using TAS then the explanation is simple: the classical preemption without TAS is not aware of the requirement: 

> each node can take at most one pod from low-cpu or standard.

because it is agnostic to nodes etc (it is only working on total available quotas for CQs/cohorts)

### Comment by [@pajakd](https://github.com/pajakd) — 2026-03-27T14:26:42Z

Ok, I think there is a subtle bug in FlavorAssigner with TAS.

We have initial assignment (based on total quotas): `GPU: Preempt, CPU: Fit`. Now with TAS, we **should** update the assignment to `GPU: Preempt, CPU: Preempt`. Below is the code that should do it but if the `RepresentativeMode ` is `Preempt` it only simulates if preempting everything would allow to fit the workload (which is true in this case). But it does not update the initial assignment. 

https://github.com/kubernetes-sigs/kueue/blob/bfa271e407ee09d46298b4823cc66713ea213227/pkg/scheduler/flavorassigner/flavorassigner.go#L685-L712

To confirm/disprove could you please @wafrelka check what would happen if we would change the GPU requirements of both `standard` and `low-gpu` from 8 to 5 GPUs? If the bug is indeed here, the `standard` should preempt both `low-gpu` and `low-cpu` and get admitted.
