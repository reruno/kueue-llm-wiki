## Description

**What would you like to be added**:
Ensure that MK controller places MultiKueue specific labels on manager Pods.
This issue is focused on MK **manager** clusters. For MK worker clusters, see #10083.

**Why is this needed**:
The goal is to make it observable whether a given Kueue controller deployment is using MultiKueue (on the manager side). For this, we'd like to put some specific label (e.g. `kueue.x-k8s.io/multikueue-manager-cluster=true`) on its Pods.

**Design considerations**
There are few approaches on how the label can be placed/removed:
1. Add the label if there is a MK cluster with AC attached, never remove it.
+ adv: the simplest one
2. Add the label if there is a MK cluster with AC attached to a "healthy" CQ ("can admit workloads"). Otherwise remove the label.
3. Add the label if there is a MK cluster with AC and some workloads are actually admitted through it. Otherwise remove the label.
+ adv: most consistent with the worker side

Possible options how to detect if AC is running/changed/not running:
Use MK AC controller.
1. Create and keep the set of admitted workload. Update it based on workload change events in Workload Reconciler. Track this set to update labels on `kueue-controller-manager` pod.
2. Use information about number of Admitted Workloads from scheduler cache. The issue here is that MK controller has no direct access to it. Possible solution is calculating whether there are admitted workloads inside [setupControllers](https://github.com/kubernetes-sigs/kueue/blob/766f0232b868cc1ce204d15fc13a7f89f7ff082c/cmd/kueue/main.go#L426), and then passing this info to Workload Reconciler through [multikueue.SetupControllers](https://github.com/kubernetes-sigs/kueue/blob/766f0232b868cc1ce204d15fc13a7f89f7ff082c/cmd/kueue/main.go#L470).
