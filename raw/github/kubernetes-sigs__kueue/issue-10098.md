**What happened**:
As per  below piece of code length of workload name is allowed to be 253 characters.
https://github.com/kubernetes-sigs/kueue/blob/main/apis/kueue/v1beta2/workload_types.go#L158C2-L158C42

However PrebuiltWorkloadLabel which holders the name of the workload is used as label and 
kubernetes has hard limitation of 63 characters for a label value so in case workload name exceeds this value  using this label causes failure.

In my case MultiKueue ExternalFramework adapter was not able to create resource on  remote cluster when workload named exceeded 63 characters

**What you expected to happen**:
I can think of two possible options here?

1. Limit the length of workload name to 63 characters. 
     This might not be  a feasible solution  here as reducing the length might break the existing functionality.

2. Instead of labels we should be  move to annotation for PrebuiltWorkloadLabel.
    As Annotations are more flexible than labels we can move to annotations for linking the resources to workload.

**How to reproduce it (as minimally and precisely as possible)**:
This is reproducible in multiKueue environment where original job name is very long (63 chars). As job name is already 63 characters so there is no room left for workload to add prefix and postfix.

### Comment by [@mimowo](https://github.com/mimowo) — 2026-03-26T17:46:02Z

@pramodbindal thank you for the issue 👍 We recently got exactly the same feedback from another user, so I opened https://github.com/kubernetes-sigs/kueue/issues/9872, that caused issues with LWS: https://github.com/kubernetes-sigs/kueue/issues/10032

We also consider these two options: 
1. shorten the workload name generation as in https://github.com/kubernetes-sigs/kueue/pull/9973
2. use annotations instead or along with labels

cc @tenzen-y @mbobrovskyi @gabesaba 

 I think we may actually need both approaches. (2.) can fix the issues with LWS, but it still remains issue if someone wants to use workload name for labeling of some in--house resources. 

So I'm leaning towards accepting the 63 PR as alpha so that some users can already mitigate their environments. At the same time we should work on fixing the issue with LWS, MultiKueue and PrebuiltWorkloads even if the gate is disabled.
