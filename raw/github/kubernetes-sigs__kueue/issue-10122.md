## Description

### Background

In MultiKueue, the [workload reconciler](https://github.com/kubernetes-sigs/kueue/blob/ac4afcf77713d48e2faa67e53c522709ecc8c7ed/pkg/controller/admissionchecks/multikueue/workload.go) is set up [to watch changes in MultiKueueConfig objects](https://github.com/kubernetes-sigs/kueue/blob/ac4afcf77713d48e2faa67e53c522709ecc8c7ed/pkg/controller/admissionchecks/multikueue/workload.go#L915).

Thanks to this, if new clusters are added ([here](https://github.com/kubernetes-sigs/kueue/blob/ac4afcf77713d48e2faa67e53c522709ecc8c7ed/apis/kueue/v1beta2/multikueue_types.go#L150)), the watch handler [will notice this](https://github.com/kubernetes-sigs/kueue/blob/ac4afcf77713d48e2faa67e53c522709ecc8c7ed/pkg/controller/admissionchecks/multikueue/workload.go#L857) and [trigger reconciliation](https://github.com/kubernetes-sigs/kueue/blob/ac4afcf77713d48e2faa67e53c522709ecc8c7ed/pkg/controller/admissionchecks/multikueue/workload.go#L860) for all potentially affected workloads.

In particular, some pending workloads may be _immediately_ dispatched and scheduled on the new workers.

### The issue

The issue is that we watch for changes only in *one place* in the following diagram:

```
ClusterQueue object ---> AdmissionCheck object ---> MultiKueueConfig object ---> .Clusters field
```

Specifically:

* We will detect when `.Clusters` inside a single MultiKueueConfig is *changed*.

* We **will not** detect when the whole MultiKueueConfig is **replaced** in the AdmissionCheck.

* We will detect when the whole AdmissionCheck is replaced in the ClusterQueue. \
  (A bit around: replacing the AdmissionCheck will trigger an update of the manager workload in the Kueue core workload controller, and that will trigger a reconcile in the MultiKueue workload controller).

As a result, replacing MultiKueueConfig inside the AdmissionCheck may leave MultiKueue workloads pending, despite new capacity being available.

I guess the intended behavior is to handle this in the same way as for an updated MultiKueueConfig.

### Real demo

[This branch](https://github.com/olekzabl/kueue/tree/repro-5/hack/bug-repro) contains scripts which all test the following scenario:

0. (Not covered by the script) Run our [standard "set up MultiKueue with TAS" script](https://github.com/kubernetes-sigs/kueue/blob/3f0c4b2884fe5577d7a7ae4c3579a49718077be3/site/static/examples/multikueue/dev/setup-kind-multikueue-tas.sh).
1. Set up MultiKueue with manager quota of 3 CPU, and worker quotas of 1 CPU.
2. Initially have just 1 worker.
3. Submit 2 Jobs, 1 CPU each. (Obviously only one will run).
4. *Add* another worker.
5. Wait for the second Job to start.
6. If the second Job is still pending, submit a third one. (To verify that we successfully added a worker).

The key difference is *how* we add the second worker in step 4:

* [`test-update-mkconfig.sh`](https://github.com/olekzabl/kueue/blob/repro-5/hack/bug-repro/test-update-mkconfig.sh) updates the MultiKueueConfig;
* [`test-update-ac.sh`](https://github.com/olekzabl/kueue/blob/repro-5/hack/bug-repro/test-update-ac.sh) replaces the MultiKueueConfig referenced in the AdmissionCheck;
* [`test-update-cq.sh`](https://github.com/olekzabl/kueue/blob/repro-5/hack/bug-repro/test-update-cq.sh) replaces the AdmissionCheck referenced in the ClusterQueue.

The outputs of these scenarios can be viewed in the corresponding `*.out` files. Basically:

* In the "update MKC" and "update CQ" scenarios, the second Job runs immediately.
* In the "update AC" scenario, the second Job does not start while the third Job does.

(Note: we deliberately set a small `workerLostTimeout` and wait longer than that. See "Remarks" below for why that duration might matter).

### Remarks

1. Whenever the MK workload reconciler is triggered for a particular manager workload, it will learn the new MultiKueueConfig due to fetching the whole chain of dependencies ([here](https://github.com/kubernetes-sigs/kueue/blob/ac4afcf77713d48e2faa67e53c522709ecc8c7ed/pkg/controller/admissionchecks/multikueue/workload.go#L298)). 
   So the issue is only about triggering such reconcile.

2. While above we discussed *adding* workers, the story for *removing* workers is a bit different:

   1. It's not clear what the desired outcome is.

      * (A) Should the manager immediately detect workloads left on the removed remotes, and mark them pending from MultiKueue perspective (leading to re-dispatching and re-scheduling, if possible)?

      * (B) Or rather let the running remotes run, as they've already started, and avoid duplication of work on the "still registered" workers?

   2. The *actual* behavior is also a bit subtler.

      * Even if the worker removal happens via a change of `.Clusters` in a single MultiKueueConfig (and hence is detected by existing watches), the effect is that the MK workload reconciler - seeing no running remote among the still-registered workers - [considers](https://github.com/kubernetes-sigs/kueue/blob/ac4afcf77713d48e2faa67e53c522709ecc8c7ed/pkg/controller/admissionchecks/multikueue/workload.go#L506-L507) this as a "lost remote" case, and plans for re-dispatching only after `workerLostTimeout` passes.

      * OTOH, if the worker removal happens via a replacement of MultiKueueConfig ("undetected" by existing watches), the MK workload reconciler is *still likely* to react in the same way after `workerLostTimeout`, due to [this](https://github.com/kubernetes-sigs/kueue/blob/ac4afcf77713d48e2faa67e53c522709ecc8c7ed/pkg/controller/admissionchecks/multikueue/workload.go#L504). In practice, the reaction will just happen *somewhat later* because of the moment when we "started the watch" ([admission time](https://github.com/kubernetes-sigs/kueue/blob/ac4afcf77713d48e2faa67e53c522709ecc8c7ed/pkg/controller/admissionchecks/multikueue/workload.go#L508) for in-MKC updates vs. last reconcile time for MKC replacements).

   Therefore, for workload removals, I could agree that the current behavior is already acceptable.
   (Still, adding the missing watch should not hurt).
