**What happened**:

When using fair sharing with hierarchical cohort, preemption caused by reclamation seems to have several problems:
* Preemption candidate is selected randomly, not respecting priority.
* Preemption is happening more than required, causing unnecessary disruption in the workloads.

**What you expected to happen**:

* Preemption by reclamation should respect priority when cohorts of preemption candidates have equal fair share but candidates have different priorities.
* Preemption should not happen more than necessary (freeing more than what's actually required).

**How to reproduce it (as minimally and precisely as possible)**:

Suppose this setup in Kueue:
```
root-cohort (fair sharing enabled)
├── team-a-cohort (weight: 1)
│   ├── team-a-guaranteed-cq (2 GPU nominal, no borrowing)
│   └── team-a-borrowing-cq (0 GPU nominal, borrowingLimit: nil) → running 4 GPU high-priority workload
├── team-b-cohort (weight: 1)
│   ├── team-b-guaranteed-cq (2 GPU nominal, no borrowing)
│   └── team-b-borrowing-cq (0 GPU nominal, borrowingLimit: nil) → running 4 GPU normal-priority workload
└── team-c-cohort (weight: 1)
    ├── team-c-guaranteed-cq (8 GPU nominal, no borrowing)
    └── team-c-borrowing-cq (0 GPU nominal, borrowingLimit: nil) → running 4 GPU low-priority workload
```

3. Submit a 4-GPU workload to team-c-guaranteed-cq

### Comment by [@gabesaba](https://github.com/gabesaba) — 2026-03-27T09:24:09Z

Both issues are related. if we fix priority issue, we fix non-deterministic preemption for free, and potentially some simplification of the logic.

* **over-preemption:** caused by non-determinism in target selection for Cohorts ([this loop](https://github.com/kubernetes-sigs/kueue/blob/246f87ff89513ac8c4aa23cd84a55d5aa15639eb/pkg/scheduler/preemption/fairsharing/ordering.go#L169)). We added determinism for CQs in https://github.com/kubernetes-sigs/kueue/pull/6764, but don't have something similar for Cohorts. This should be straightforward to add: tiebreak on Cohort name (or UUID)
* **priority not respected:** We are making greedy decision about highest DRS, when searching for preemption target (recursively from root Cohort). We could update the search function to traverse multiple paths when equal DRS, and then we can choose a better (lower-priority in this case) workload. This is the general case of #6764, and can simplify the code, if implemented elegantly
