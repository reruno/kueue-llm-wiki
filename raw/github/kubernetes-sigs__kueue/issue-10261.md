@tenzen-y Thanks for the feedback. I think the situation has changed since the [earlier](https://github.com/kubernetes-sigs/kueue/issues/8243#issuecomment-3676842078) discussion in #8243.

At that point, the DRA-related work was still effectively bundled under a single gate, so it made sense to think about beta for the whole area as one decision. Since then, we split that work into separate gates: `DynamicResourceAllocation`, `DRAExtendedResources`, and `DRAPartitionableDevices`. That lets us evaluate each path on its own merits instead of tying the maturity of the base path to newer work like partitionable devices.

We already follow that pattern for TAS. It went beta in v0.14 and `ElasticJobsViaWorkloadSlicesWithTAS` was introduced as alpha in 0.17. At this point, the base `DynamicResourceAllocation` gate is essentially the `ResourceClaimTemplate` path. That path has been in alpha for four releases, and I’m not aware of issues that would suggest it requires more time to mature for beta. The core Kueue-side integration points there, such as `deviceClassMappings`, `ResourceClaimTemplate` processing, and the DRA preprocessing flow, have also been fairly stable.

So for 0.18, I think it is reasonable to consider: DynamicResourceAllocation -> beta, DRAExtendedResources -> beta
DRAPartitionableDevices -> new alpha. Happy to hear other thoughts.

@tenzen-y Thanks again for being open to the decoupled gates approach. It really helps us look at each path independently.

For MultiKueue + DRA, as @kannon92 mentioned, this is already covered through the E2E tests and the KEP updates. Just to add a bit more context, the Job adapter deep copies the full spec, including resourceClaims, and we expect ResourceClaimTemplate and DeviceClass objects to already exist on worker clusters. The E2E suite exercises this end to end, including the happy path, handling missing RCTs, correct worker routing, and verifying DRA resource usage.

For TAS + DRA, I think the main question is around priority, especially given the recent upstream work on workload aware scheduling. Things have evolved a bit since the original beta criteria were defined. Kubernetes v1.35 introduced some initial building blocks, and KEP-5732 is aiming to bring topology aware workload scheduling into kube scheduler in v1.36 as alpha. That makes me wonder if the long term solution for topology aware DRA placement may live in the scheduler rather than in Kueue TAS.

Given that, it might be worth reconsidering whether TAS + DRA should block the base `DynamicResourceAllocation` beta. It feels like it ties a relatively stable path to an area that is still evolving, both in Kueue and upstream. Maybe we could move TAS + DRA to GA criteria, or treat it as follow up work aligned with the upstream WAS direction. Either way, enabling the gate itself does not change current TAS + DRA behavior for users.

On the K8s 1.33 and API availability concern, that is a valid point since DRA APIs may not exist on older clusters. We've already addressed this downstream in the kueue-operator ([#1578](https://github.com/openshift/kueue-operator/pull/1578)). Also, we can handle this in a similar way to what Kueue already does elsewhere, for example in pkg/controller/jobframework/setup.go, by checking API availability via REST mapping at startup. If `resource.k8s.io/v1` is not available, we can log a warning and skip DRA specific initialization. With that guard in place, enabling the gate by default should be safe for clusters that do not have DRA APIs. I am happy to help add this as part of the graduation work.

Finally, on the Kubernetes API graduation model, I think there is a small distinction. On the Kubernetes side, the concern is about when APIs are served and enabled by default. In Kueue, `DynamicResourceAllocation` does not introduce a new Kubernetes API. It only controls how Kueue accounts for workloads that use existing DRA resources. So the graduation rules do not map exactly 1:1 here. Whether we enable the gate by default at beta is still a Kueue decision, but it may feel more comfortable once the API availability check is in place.

Let me know your thoughts.
