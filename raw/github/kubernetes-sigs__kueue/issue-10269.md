## Description

**What happened**:

On a large cluster (19K pods, 1571 nodes, 277 Kueue Workloads), the kueue-controller-manager holds ~190 MB of heap from per-reconcile zap logger allocations. Top inuse allocators from `go tool pprof`:

| inuse_space | Function |
|-------------|----------|
| 60.56 MB | `go.uber.org/zap/internal/bufferpool.init.NewPool.func1` |
| 44.54 MB | `go.uber.org/zap/zapcore.(*jsonEncoder).clone` |
| 8.50 MB | `go.uber.org/zap.(*Logger).clone` |
| 77.06 MB cum | `github.com/go-logr/zapr.(*zapLogger).WithValues` |

Each reconcile calls `WithValues` which clones the logger, core, and encoder, drawing a new buffer from `bufferpool`. Under sustained throughput (~thousands of reconciles/sec across all controllers), the sync.Pool refills faster than GC drains it. The per-reconcile `reconcileID` prevents caching.

The call chain per reconcile:

```
reconcileHandler -> NewLogConstructor.func1
  -> zapr.WithValues -> zap.Logger.With -> ioCore.With
    -> jsonEncoder.Clone -> bufferpool.Get  (x2 per reconcile)
```

The default `LogConstructor` chains two `WithValues` calls (controller name, then namespace/name/reconcileID), producing two full clone-and-encode cycles per reconcile.

**What you expected to happen**:

Per-reconcile logger allocations should not dominate the heap profile (~190 MB at this scale).

**How to reproduce it (as minimally and precisely as possible)**:

On a cluster with >= 15K pods and Kueue managing >= 200 workloads:

1. Enable pprof (`pprofBindAddress: :8083` in the ConfigMap)
2. At steady state, capture the heap profile:
   ```
   kubectl port-forward -n kueue-system pod/<kueue-pod> 8083:8083
   go tool pprof -inuse_space http://localhost:8083/debug/pprof/heap
   ```
3. Run `top 20`. The top allocators are `bufferpool`, `jsonEncoder.clone`, and `zapr.WithValues`.

**Proposed fix: cache the base logger per controller**

Override `LogConstructor` to cache the controller-level logger at setup and call `WithValues` once per reconcile instead of twice:

```go
controllerLogger := baseLogger.WithValues("controller", name) // once at setup
opts.LogConstructor = func(req *reconcile.Request) logr.Logger {
    if req == nil {
        return controllerLogger
    }
    return controllerLogger.WithValues(
        "namespace", req.Namespace,
        "name", req.Name,
        "reconcileID", uuid.NewUUID(),
    )
}
```

This halves per-reconcile encoder clones and can be implemented entirely within Kueue. For broader improvement, `go-logr/zapr` could expose `WithLazy` ([zap v1.26.0+](https://github.com/uber-go/zap/pull/1319)), and controller-runtime could consolidate its `WithValues` chain ([#1290](https://github.com/kubernetes-sigs/controller-runtime/issues/1290)).

**Anything else we need to know?**:

In our case the highest-throughput controller was `NonTasUsageReconciler` (TAS), reconciling all 19K pods. Disabling `TopologyAwareScheduling` dropped zap-related heap from ~190 MB to ~1 MB. The problem affects any high-throughput controller at scale.

Related upstream issues:
- [uber-go/zap#1348](https://github.com/uber-go/zap/issues/1348) -- proposes mutable cores to avoid clone overhead (open)
- [uber-go/zap#1319](https://github.com/uber-go/zap/pull/1319) -- `WithLazy` merged in v1.26.0; defers field encoding until first write
- [controller-runtime#1290](https://github.com/kubernetes-sigs/controller-runtime/issues/1290) -- `WithValues` bypasses `KubeAwareEncoder` (open)
- [kueue#3818](https://github.com/kubernetes-sigs/kueue/issues/3818) -- high memory at 90K jobs; root cause was cache retention, not loggers

**Environment**:
- Kubernetes version: v1.31.x (GKE)
- Kueue version: v0.16.2
- controller-runtime: v0.22.4
- go.uber.org/zap: v1.27.1
- Cloud provider: GKE (19K pods, 1571 nodes)

### Comment by [@ikchifo](https://github.com/ikchifo) — 2026-04-03T15:12:53Z

Ahhh, `WithValues` snapshots the string, so caching a logger with `replica-role` baked in would go stale after leader election.

The safe narrowing would be to cache only the `WithName` result:

```go
func NewLogConstructor(tracker *RoleTracker, baseName string) func(*reconcile.Request) logr.Logger {
    namedLog := ctrl.Log.WithName(baseName)
    return func(req *reconcile.Request) logr.Logger {
        log := namedLog.WithValues("replica-role", GetRole(tracker))
        if req == nil {
            return log
        }
        return withRequestContext(log, req.Namespace, req.Name)
    }
}
```

This saves one clone per reconcile (3 -> 2) -- modest. The real memory reduction came from disabling `TopologyAwareScheduling` on clusters that don't use it, which eliminated the reconcile volume driving the allocations. Once the reconcile rate is reasonable, logger overhead is secondary.

Bigger wins need changes in `go-logr/zapr` (e.g. exposing `WithLazy` from zap v1.26.0+) or in controller-runtime's `WithValues` chain. Happy to keep this open as a low-priority optimization (as the memory savings won't be as much) or close if you'd rather track it upstream.
