### Comment by [@reruno](https://github.com/reruno) — 2026-04-13T16:56:36Z

I analyzed these test runs, to determine what could be the root cause of this problem. 

Specifically these runs
- bad run: https://prow.k8s.io/view/gs/kubernetes-ci-logs/logs/periodic-kueue-test-e2e-release-0-16-1-33/2041394602723774464
- good run: https://prow.k8s.io/view/gs/kubernetes-ci-logs/logs/periodic-kueue-test-e2e-release-0-16-1-33/2041213407486545920

These runs are performed on the same commit, the only difference between them is when these runs were performed.

**The root cause of the problem:** 
In a bad run kubelet on worker nodes had an issue with updating status phase for the `jax-e2e-wnm7m/jax-simple-worker-1` and `jax-e2e-wnm7m/jax-simple-worker-0` from `Running` to `Succeeded` in a timely manner, that is why JAX integration test failed. Specifically it failed on this check jaxjob_test.go:138 `util.ExpectWorkloadToFinish(ctx, k8sClient, wlLookupKey)` 

**Proof:** 
Exit command for pod was sent by http at 06:16:01. The test failed at 06:16:46.55 because of timeout and by looking at logs we can see that kind-worker2 was noticing the inconsistency with cached status for the pod every 10s until 06:17:20, and the actual patch status for pod came at around 06:17:26

The test failed at 06:16:46.55 because of timeout and by looking at logs we can see that kind-worker2 was noticing the inconsistency with cached status for the pod every 10s until 06:17:20, and the actual patch status for pod came at around 06:17:26

Ok, so in that case waiting the LongTimeout (90s) would help, this timeout was actually introduced for cases like this, where the Pod termination takes very long on a loaded CI, see [here](https://github.com/kubernetes-sigs/kueue/blob/f6cc162d758d5c2f2cf8c584dfd80b8e1e7749d7/test/util/constants.go#L37-L39)

So, I think we still can deflake in Kueue by using the longer timeout.
