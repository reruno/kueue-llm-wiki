**What happened**:

When Kueue is configured with only the `pod` integration, the `batch/v1 Job` mutating webhook (`mjob.kb.io`) still runs for Jobs, and fails to create a job.

```
job "...-launcher-0" creation failed with error: admission webhook "mjob.kb.io" denied the request:
panic: runtime error: invalid memory address or nil pointer dereference
sigs.k8s.io/kueue/pkg/cache/queue.(*Manager).DefaultLocalQueueExist(0x0, ...)
sigs.k8s.io/kueue/pkg/controller/jobframework.ApplyDefaultLocalQueue(...)
sigs.k8s.io/kueue/pkg/controller/jobs/job.(*JobWebhook).Default(...)
...
```

**What you expected to happen**:

When the `batch/job` integration is disabled, the webhook should be skipped.

**How to reproduce it (as minimally and precisely as possible)**:

Disable `batch/job` framework integration, and create a batch/job.

**Anything else we need to know?**:

This looks like a regression introduced during the webhook refactoring. `batch/job` seems to be missing that handling, unlike the other integrations. 
I also checked the discussion in the PR, but I could not find any explicit decision to keep the batch/job webhook active when the integration is disabled.
