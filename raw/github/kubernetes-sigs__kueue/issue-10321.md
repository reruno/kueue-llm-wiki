### Comment by [@mimowo](https://github.com/mimowo) — 2026-04-16T17:28:48Z

I looked at the logs of the recent failure and I think I understand the problem, it is actually showing a possible production issue, that the last deleted may still keep the CQ around, and the CQ may report `pendingWorkloads: 1` blocking the removal of the finalizer, this is what happened in the namespace `preemption-9j92v`.
1. the workload `second-high-prio` is picked fro scheduling "Obtained ClusterQueue heads"
2. the workload  `second-high-prio`  is deleted, "Workload delete event"
3. the CQ is reconciled for the last time "Reconcile ClusterQueue", it still sees the inflight workload, and so it reports `pendingWorkloads: 1`
4. the workload request to schedule fails with "Workload not admitted because it was deleted"
5. NO MORE CQ events and so the finalizer is never removed

Other tests usually first delete the LQ, and that triggers the CQ reconcile, but here we have only two lines:
```golang
			gomega.Expect(util.DeleteWorkloadsInNamespace(ctx, k8sClient, ns)).To(gomega.Succeed())
			util.ExpectObjectToBeDeleted(ctx, k8sClient, cq, true)
```

This scenario is very unlikely for production because it assumes deleting all workloads, but not deleting the LQ, yet trying to delete the CQ. 

So, I'm leaning to say this is very low priority, when an admin attempts to delete a CQ they would normally delete also all LQs first, or point them to other CQs. I'm ok to just plumb the test for now, by removing the LQs to make it stable and open a dedicated issue to track this problem.

Still, I'm wondering if there are some ideas to solve it, IIRC @sohankunkerkar was fixing recently a similar ClusterQueue flake.
