## Description

<!-- Please only use this template for submitting enhancement requests -->

**What would you like to be added**:

I'd like an integration point for DRA and Admission Fair Sharing. Right now, Kueue limits admission fair sharing weights to the resources specified in the requests but if one uses DRA you won't be able to use weights.

**Why is this needed**:

Admission Fair Sharing allows you to specify weights for resources but this will only work with extended Resources or classic resources.

Ideally one could use weights on ResourceClaimTemplates or something like that to enforce weights on resources.

### Comment by [@kannon92](https://github.com/kannon92) — 2026-04-09T12:47:58Z

ResourceClaimWeights would not work as a resourceClaim is a request to a deviceClass (cluster resource). We can't add weights for resourceClaims as they are a namespaced resource and can be ephermeral to a request so naming discovery would be challenging.

So we have a few options:

Option 1:
- Add a weights to the DeviceClass in the DRA configuration for this.
- Pros: weights are in the actual resource that Kueue will manage quota for
- Cons: spreads AFS to an unrelated configuration.

Option 2:
- The resource weights for AFS could also mean a deviceClass weight if its not present on the resource list
- Pros: no much code changes
- Cons: Hard to know if the weights correspond to requests or deviceClass.

Option 3:
- Add `deviceClassWeights` in AFS to make it clear what resources are weighted.
- Pros: clear responsibility of weights
- Cons: new API and a bit of implementation leakage on deviceClasses which are the responsibility of the cluster admin and not the user.

### Comment by [@sohankunkerkar](https://github.com/sohankunkerkar) — 2026-04-09T16:32:06Z

hmm... close to Option 2 in spirit, but not as a separate `deviceClass weight` concept. The key for AFS is the logical quota resource from `deviceClassMappings.name`, not the `DeviceClass` name. So with `name: nvidia.com/gpu`, `deviceClassNames: [gpu.nvidia.com]`, the AFS weight key is `nvidia.com/gpu`. That same name shows up in CQ resources and workload ResourceUsage after admission, so AFS weights it without a new API. If the concern is that this isn't obvious to admins, that sounds like a docs gap.
