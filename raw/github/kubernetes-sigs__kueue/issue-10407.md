## Description

## What happened?

When a pod workload is evicted due to `waitForPodsReady` timeout, Kueue deletes the pod via `Stop()` and requeues the workload with exponential backoff. If the pod is owned by a Deployment, the Deployment controller creates a replacement pod which gets its own new workload. However, the old evicted workload keeps getting requeued despite its owning pod being deleted, causing unbounded stale workload accumulation.

With `backoffLimitCount` unset (unlimited retries), each eviction cycle adds one stale workload referencing a deleted pod. When re-admitted, these stale workloads consume ClusterQueue quota for pods that no longer exist.

### Observed behavior

We observed **52 admitted workloads** for a **2-replica Deployment**. Each PodsReady timeout eviction cycle creates one orphaned workload that continues to be requeued and re-admitted indefinitely.

### Expected behavior

When a workload's owning pod has been deleted (e.g., due to eviction), the workload should be detected as stale and finished rather than re-enqueued.

## How to reproduce

1. Configure `waitForPodsReady` with a short timeout (e.g., 3s)
2. Create a pod owned by a Deployment with a `kueue.x-k8s.io/queue-name` label
3. Let the pod be admitted — do not set it to `Ready` (simulating a pod that never becomes ready)
4. Wait for the `PodsReady` timeout to trigger eviction
5. The pod controller calls `Stop()`, which deletes the pod
6. The Deployment creates a replacement pod (new workload)
7. The old workload is requeued after backoff and re-admitted despite its pod being gone
8. Repeat — stale workloads accumulate with each cycle

## Impact

- **Quota leak**: Stale workloads consume ClusterQueue quota for non-existent pods, preventing legitimate workloads from being admitted
- **Unbounded growth**: Without `backoffLimitCount`, stale workloads accumulate indefinitely
- **Silent failure**: No indication that the admitted workloads reference deleted pods

## Environment

- Kueue version: main branch
- Configuration: `waitForPodsReady` enabled, pods managed by Deployments
