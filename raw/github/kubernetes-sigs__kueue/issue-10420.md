### Comment by [@mimowo](https://github.com/mimowo) — 2026-04-17T14:05:41Z

/assign
I looked at the logs, first we see that the failure was in JustAfterEach which runs the dummy Job to verify the node is recovered.  Then I identified the Pod was `frp-ltlkx/dummy-job-kind-worker-xhmtq`, running on kube-worker. 
And looking at [kubelet logs](https://storage.googleapis.com/kubernetes-ci-logs/logs/periodic-kueue-test-e2e-customconfigs-release-0-17/2042973271896363008/artifacts/run-test-e2e-customconfigs-1.35.0/kind-worker/kubelet.log) I see the Pod lifetime:
1.`14:48:50.718251` created 
2. `14:48:51.029620` creating containers
3. `14:48:53.834531` running
4. `14:49:35.427362` - deleted

So basically the Pod never finished. I think there might be two reasons: (1) as we know the CI machines are overloaded and so for Pod lifecycle we use LongTimeout, (2.) the Pod does not specifiy requests nor limits for CPU, so it might be realistically close to 0 when other tests are running. I think it would be good to give it  200m cpu as other Pods.
