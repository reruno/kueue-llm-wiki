## Context

In a MultiKueue setup, worker clusters are specified via [this field](https://github.com/kubernetes-sigs/kueue/blob/f6cc162d758d5c2f2cf8c584dfd80b8e1e7749d7/apis/kueue/v1beta2/multikueue_types.go#L150) in MultiKueueConfig, referring to [MultiKueueCluster objects](https://github.com/kubernetes-sigs/kueue/blob/f6cc162d758d5c2f2cf8c584dfd80b8e1e7749d7/apis/kueue/v1beta2/multikueue_types.go#L117), which in particular store information on how to connect to the worker cluster.

If a worker is "explicitly" removed from the `.Clusters` list, we have this handled quite nicely. For example, MultiKueue will [find all workloads](https://github.com/kubernetes-sigs/kueue/blob/f6cc162d758d5c2f2cf8c584dfd80b8e1e7749d7/pkg/controller/admissionchecks/multikueue/workload.go#L866) depending on the changed MultiKueueConfig, and trigger reconcile for them.

However, if a worker remains in `.Clusters` but sth goes "broken" in the MultiKueueCluster object, the handling seems - at first glance - less perfect. Specifically, I see 2 cases:

* **Case 1:** The MultiKueueCluster object is deleted (making an entry in `.Clusters` effectively invalid). \
   This is handled [here](https://github.com/kubernetes-sigs/kueue/blob/f6cc162d758d5c2f2cf8c584dfd80b8e1e7749d7/pkg/controller/admissionchecks/multikueue/multikueuecluster.go#L447).
* **Case 2:** The MultiKueueCluster keeps existing but connection becomes broken, by losing a KubeConfig. \
   This is handled [here](https://github.com/kubernetes-sigs/kueue/blob/f6cc162d758d5c2f2cf8c584dfd80b8e1e7749d7/pkg/controller/admissionchecks/multikueue/multikueuecluster.go#L454).

In both cases, we call [`stopAndRemoveCluster`](https://github.com/kubernetes-sigs/kueue/blob/f6cc162d758d5c2f2cf8c584dfd80b8e1e7749d7/pkg/controller/admissionchecks/multikueue/multikueuecluster.go#L395), which **does not seem to trigger** any workload-level or queue-level reconciles. (Unless there are some magic implications of cancelling a Context [here](https://github.com/kubernetes-sigs/kueue/blob/f6cc162d758d5c2f2cf8c584dfd80b8e1e7749d7/pkg/controller/admissionchecks/multikueue/multikueuecluster.go#L399) - seems worth verifying).

## Suspected issue

Imagine MultiKueue with 2 workers, `worker1` and `worker2`, each fitting one instance of some Job. \
Then, consider e.g. the following scenario:

1. Submit a high-priority Job instance, `job1`, suppose it lands on `worker1`.
2. Submit a low-priority, long-living Job instance, `job2`. Then it must land on `worker2`.
3. Wait for `job1` to finish (while `job2` is still running).
4. _Break_ the connection to `worker2`, e.g. by removing the MultiKueueCluster object.

* Desired effect (I guess): MK reschedules `job2` to `worker1`. \
  In particular, I'm guessing that's going to happen if we explicitly removed `worker2` from `.Clusters` in MultiKueueConfig.

* Actual effect (I guess): MK "does not notice". \
   Specifically, it _will_ reschedule `job2` to `worker1` once anything else triggers a reconcile for `job2` (as then [calling `readGroup`](https://github.com/kubernetes-sigs/kueue/blob/f6cc162d758d5c2f2cf8c584dfd80b8e1e7749d7/pkg/controller/admissionchecks/multikueue/workload.go#L224) will "realize" [here](https://github.com/kubernetes-sigs/kueue/blob/f6cc162d758d5c2f2cf8c584dfd80b8e1e7749d7/pkg/controller/admissionchecks/multikueue/workload.go#L259) that `worker2` is no longer usable). \
  But I'm afraid this will need "some other trigger", which in some scenarios could not happen for a long time.

## Thoughts on fixing

When "Case 1" or "Case 2" happen, triggering the desired reconciles may be actually non-trivial. 

For example, to trigger reconciles for all affected Workloads, we'd need to know them, for which the most natural way would be to start with an API call towards the affected worker cluster... which, at least in "Case 2", just stopped being possible!

Then, we'd need:

* either some "smarter way" of finding the workloads in the available data \
  (maybe sufficient knowledge on worker clusters can be found somewhere in the local Workload fields?)
* or some MultiKueue cache retaining the Workloads <-> MultiKueueClusters relations.
