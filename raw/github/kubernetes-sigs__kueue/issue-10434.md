## Description

Hello Kueue Maintainers,

First off all, I want to thank all of the maintainers and contributors for investing their time into this project, ❤️. I have been using Kueue in production for ~2 years, and the breadth of features put out just in that amount of time has been crazy to keep up with.

I have a question about how admins should set `admissionFairSharing.resourceWeights` -- specifically in the controller config. not LocalQueue `fairSharing.weight`.

For example, given a cluster capacity of,
- 1000 CPUs
- 1000 GiB Memory
- 10 Accelerators, e.g. `nvidia.com/gpu.count`

and the a policy intent to make,
- 1 GiB of Memory 4x the weight of a CPU
- 1 GPU to be 50x the weight of a CPU

What should the resource weights be?

### Option A
```yaml
admissionFairSharing:
  usageHalfLifeTime: 120h
  usageSamplingInterval: 5m
  resourceWeights:
    cpu: 1
    memory: 4
    nvidia.com/gpu.count : 50
```

### Option B
Or should weights be scaled / normalized to underlying quantity units (for example bytes for memory), so memory would need a very small value like `m/(2^n)`, where `m` is the intended weight and `n` is the intended scale conversion to bytes. 

```yaml
admissionFairSharing:
  usageHalfLifeTime: 120h
  usageSamplingInterval: 5m
  resourceWeights:
    cpu: 1.0
    # Memory is counted in bytes, so convert "4x per GiB" to "4x per byte":
    # 4 / 2^30 = 3.725290298e-9
    memory: 0.00000000372529
    nvidia.com/gpu.count: 50.0
```

Looking at the source code, it seems memory may be effectively accounted by byte-scale quantity unless admins manually normalize weights, but _I’m not sure if that is the intended model_.

https://github.com/kubernetes-sigs/kueue/blob/b89dae018b31ca32eed282040e19eceee09941f9/pkg/workload/workload.go#L431-L446

### Humble Request
Could you clarify the intended interpretation and provide a concrete docs example for mixed CPU/memory/GPU weights?

### Comment by [@mimowo](https://github.com/mimowo) — 2026-04-14T06:42:17Z

Thank you for the issue. 

Indeed, I think option A is more readable as 1:1 ratio intuitively means 1Cpu to 1Gi. That convention is explicit in [GKE autopilot](https://docs.cloud.google.com/kubernetes-engine/docs/concepts/autopilot-resource-requests), but I saw it also common in other links. I didn't find a standard k8s documentation, but still the industry standard seems to be 1Cpu per 1Gi. 

So, I'm leaning towards calling it a bug, and changing the code, ofc behind a feature gate for bailout. I think we could still do that as the feature is beta. Let me cc @PBundyra @mwielgus @tenzen-y in case folks have another opinion.
