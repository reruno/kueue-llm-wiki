## Description

<!-- Please only use this template for submitting clean up requests -->

**What would you like to be cleaned**:
Currently the type of [admissionFairSharing.resourceWeights](https://github.com/kubernetes-sigs/kueue/blob/b89dae018b31ca32eed282040e19eceee09941f9/apis/config/v1beta2/configuration_types.go#L610) is a `float64`. However, The Kubernetes API convention states that:

> Avoid floating-point values as much as possible, and never use them in spec. Floating-point values cannot be reliably round-tripped (encoded and re-decoded) without changing, and have varying precision and representations across languages and architectures.

https://github.com/kubernetes/community/blob/main/contributors/devel/sig-architecture/api-conventions.md#primitive-types

We should consider changing the type of float64 to other type.

**Why is this needed**:
To follow Kubernetes API conventions.

### Comment by [@shinybrar](https://github.com/shinybrar) — 2026-04-14T16:55:03Z

Bringing in some user perspective, if `resourceWeights` are normalized to some underlying human-expected units, it would help if this rule applied uniformly to any present or future resource dimensions, including arbitrary/custom quantities that might be supported by Kueue in the future.

For example, if admission logic wants to considers network bandwidth-style resources expressed with suffixes like `10M, 50M`, admins should have a clear path to influence this unit-aware normalization similar memory.

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: bandwidth-limited-app
  annotations:
    kubernetes.io/ingress-bandwidth: "10M"
    kubernetes.io/egress-bandwidth: "50M"
```

An option could be to introduce, `resourceWeightUnits` or something similar, rather than silently changing the business logic under the hood,

```yaml
admissionFairSharing:
  usageHalfLifeTime: 120h
  usageSamplingInterval: 5m
  resourceWeights:
    cpu: 1
    memory: 4
    nvidia.com/gpu.count : 50
  resourceWeightUnits:
    memory: "GiB"
```

If we end up having to until the transition away from `v1beta2` for these changes, that should be okay for a future proof decision.
