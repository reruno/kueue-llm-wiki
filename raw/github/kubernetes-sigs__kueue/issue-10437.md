## Description


**What happened**:
In a MultiKueue environment, when a workload is executed on a worker cluster, 
the workload.status.accumulatedPastExecutionTimeSeconds field is not being populated/updated on the Workload object. This field remains unset or at its initial value (0 or nil) even after the workload has completed execution on the worker cluster.

**What you expected to happen**:
The workload.status.accumulatedPastExecutionTimeSeconds field should be updated with the actual execution time when a workload completes on a worker cluster in a MultiKueue setup. This field is critical for tracking workload re-execution and implementing features based on past execution history.

**How to reproduce it (as minimally and precisely as possible)**:
1. Set up a MultiKueue environment with a manager cluster and at least one worker cluster
2. Create a workload that gets admitted to the worker cluster
3. Wait for the workload to complete execution on the worker cluster
4. Check the workload status on the manager cluster:
$ kubectl get workload <workload-name> -n <namespace> -o jsonpath='{.status.accumulatedPastExecutionTimeSeconds}'
5. Observe that the field is not populated (returns empty, 0, or nil)

**Environment**:
- Kubernetes version (use `kubectl version`): v1.31.14
- Kueue version (use `git describe --tags --dirty --always`): v0.16.4
- Cloud provider or hardware configuration: HP A100 GPU Server 
- OS (e.g: `cat /etc/os-release`): Ubuntu 24.04.3
- Kernel (e.g. `uname -a`): Linux 6.8.0-90-generic
- Install tools: Helm
