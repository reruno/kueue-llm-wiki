### Comment by [@sohankunkerkar](https://github.com/sohankunkerkar) — 2026-04-14T12:23:50Z

I think we need the timing bump (8s→16s parallel sleep, 32→60 sequential tasks) which was originally in [#10035](https://github.com/kubernetes-sigs/kueue/pull/10035) specifically to support the pod replacement test step. It was   reverted per [this](https://github.com/kubernetes-sigs/kueue/pull/10272#discussion_r3053683277) review comment. I think this test is flaking due to that as the RayJob completes before the pod replacement can be verified. The bump needs to go back in to give enough headroom for the delete + replacement cycle. See hiboyang's [explanation](https://github.com/kubernetes-sigs/kueue/pull/10035#discussion_r3023207560)

### Comment by [@mimowo](https://github.com/mimowo) — 2026-04-14T12:44:49Z

Oh, I see, so the entire RayJob completes before the replacement, and so it deletes all Pods, and this is why we observe 0 Pods - this makes sense.

I'm just wondering how we could stabilize the test without sacrificing the total runtime too much.

I'm wondering of the alternative to inject the pod-replacement step before scale up starts. iiuc in this case the Workload would not end prematurely, because there would still be a number of tasks which trigger scale up.
