### Comment by [@mimowo](https://github.com/mimowo) — 2026-04-15T06:30:00Z

I agree, so I propose to split this suite into two:
- e2e-sequential-baseline 
- e2e-sequential-extended

The "sequential" in name comes from the MultiKueue convention, and baseline / extended from integration tests. "sequential" better corresponds to reality, because some of the tests aren't really using custom configs, as they change the deployment rather than configMap. The main common denominator is that they need to run sequentially.

wrt the meaning of baseline / extended I can see two natural options:
1. time short / long - as we do for integration tests currently; short it baseline, long is extened
2. no integrations / with integrations - baseline are tests for "core" features, while extended are tests which require integrations (JobSet, Appwrapper, etc)

I'm leaning towards migrating to option (2.) as this can save a ton of time to skip installing all the integrations when we want to test a core functionality.
