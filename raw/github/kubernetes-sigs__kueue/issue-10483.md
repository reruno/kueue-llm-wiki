## Description

# Description

Currently, the SparkApplication documentation mentions enabling the SparkApplicationIntegration feature gate but does not mention that sparkoperator.k8s.io/sparkapplication must also be added to integrations.frameworks in the Kueue configuration: https://kueue.sigs.k8s.io/docs/tasks/run/kubeflow/sparkapplications/

# Steps

## Overview

1. Install Kueue v0.17.0 with the SparkApplicationIntegration feature gate enabled
2. Install Spark Operator v2.4.0
3. Create a ClusterQueue, ResourceFlavor, and LocalQueue
4. Create a SparkApplication with kueue.x-k8s.io/queue-name label

### Expected behavior

Kueue creates a sparkapplication-* Workload and manages admission.

### Actual behavior

- No Workload is created for the SparkApplication
- The SparkApplication runs immediately, bypassing Kueue admission
- Kueue logs show no reconciliation for the SparkApplication
- If suspend: true is set manually, the SparkApplication stays suspended forever

### Root cause

The feature gate enables the webhooks and RBAC but does not register the framework. The framework must also be listed in integrations.frameworks for the reconciler to start.

# Suggestion

The page https://kueue.sigs.k8s.io/docs/tasks/run/kubeflow/sparkapplications/ currently says:

_You can modify kueue configurations from installed releases to include SparkApplication as an allowed workload._

This should explicitly state to add sparkoperator.k8s.io/sparkapplication to integrations.frameworks.

Alternatively it may be activated by default in https://github.com/kubernetes-sigs/kueue/blob/v0.17.0/config/components/manager/controller_manager_config.yaml#L60 .

I'm not sure which of the two options is better, maybe the second one can only be done once the integration is not in Alpha anymore ?
