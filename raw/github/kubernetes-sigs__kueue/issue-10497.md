**What happened**:

With classic TAS and `WithinClusterQueue = LowerPriority`, a higher-priority workload was blocked even though it could fit after preempting lower-priority workloads in the same TAS domain.

The workload is reported as blocked by GPU shortage, for example: `couldn't assign flavors ... insufficient unused quota for nvidia.com/gpu in flavor flavor-hoge, 22 more needed`. However, making the workload fit also requires evicting lower-priority workloads that do not use GPU, but still occupy the same TAS domains.
In this case, Kueue can under-select preemption targets. It preempts the GPU-consuming blocker, but leaves a non-GPU blocker in the same topology domain, so the workload still cannot fit.

**What you expected to happen**:

The higher-priority workload preempts lower-priority workloads within the same TAS domain when it can fit.

**How to reproduce it (as minimally and precisely as possible)**:

Minimal reproducer:

1. Enable TAS with hostname-level topology.
2. Use a single TAS flavor and a single node with:
   - `nvidia.com/gpu=8`
   - `pods=2`
3. Admit two low-priority workloads onto the same hostname:
   - `low-gpu`: 1 pod using all GPU on the node
   - `low-cpu`: 1 non-GPU pod with `topologyAssignment` on the same hostname
4. Submit a higher-priority workload requiring:
   - 2 pods
   - each pod requesting GPU
   - required hostname topology
5. Only `low-gpu` is selected for preemption
   - `low-cpu` remains admitted
   - workload remains pending because the TAS domain is not fully freed

sample test case: https://github.com/Ladicle/kueue/blob/b04c1c1369ed563bc9314d82a45d7d93005d31e9/pkg/scheduler/scheduler_tas_test.go#L4484-L4662

### Comment by [@pajakd](https://github.com/pajakd) — 2026-04-17T09:17:48Z

The reason is that when selecting candidates for preemption we select only the candidates that use some of the resources that are needed by the incoming workload AND are managed by the CQ. So if `Pods` are not managed there is no intersection between the resources requested by `low-cpu` and `high-priority`.

 I tried giving some (large) quota to the CQ `Resource(corev1.ResourcePods, "99999")` and updated the Assignments of the admitted workloads to include the `Pods` resource and the test passed (`high-priority` managed to preempt the `low-cpu` workload).

Or (perhaps a more practical option) would be to give the `high-priority` some `CPU` requests.

In general it looks WAI to me. 

@Ladicle let me know if I'm missing something.
