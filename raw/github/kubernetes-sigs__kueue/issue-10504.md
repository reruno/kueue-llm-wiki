 **What happened**:

In a Kueue + DRA setup, when `deviceClassMappings` is removed from the Kueue configuration (e.g., via the Kueue CR in OpenShift), Kueue loses all awareness of DRA-managed GPU resources. However, it continues to admit workloads that reference DRA `ResourceClaimTemplates` and unsuspends Jobs as if everything is properly configured.

This means:
  - DRA workloads are admitted and Jobs unsuspended without any warning that `deviceClassMappings` is not configured
  - The `ResourceClaim` is allocated directly by the Kubernetes DRA scheduler, bypassing Kueue's quota entirely
  - The ClusterQueue's GPU resource usage remains at `0` despite a physical GPU being consumed
  - Since Kueue always sees DRA resource usage as `0`, it will admit an unlimited number of DRA workloads regardless of the `nominalQuota` value.

**What you expected to happen**:
When a DRA workload references a `DeviceClass` that has no corresponding `deviceClassMapping` configured, Kueue could either:
  - Reject admission with a message like: `"Workload references DeviceClass 'gpu.nvidia.com' but no deviceClassMapping is configured"`
  - Or set a warning condition on the Workload indicating the DRA resource is not being quota-managed

**How to reproduce it (as minimally and precisely as possible)**:

  1. Remove `deviceClassMappings` from the Kueue configuration and wait for reconciliation
  2. Verify that `deviceClassMapping` was removed from the kueue-manager-config ConfigMap
  3. Create Kueue resources with a GPU-related `coveredResource` (ResourceFlavor, ClusterQueue, Namespace, and LocalQueue)
  4. Submit a DRA workload referencing a `ResourceClaimTemplate`
  5. Observe the workload is admitted, the Job is unsuspended, and the GPU is consumed outside Kueue's control via DRA
  6. Check `kubectl describe clusterqueue` — GPU usage shows `0` despite physical GPU being consumed through DRA

**Anything else we need to know?**:

This silently bypasses quota enforcement. Since Kueue never accounts for DRA resource usage, it will admit an unlimited number of DRA workloads regardless of `nominalQuota`. There is no error, warning, or condition indicating the misconfiguration — making it easy to over-commit physical GPUs without any visibility.
