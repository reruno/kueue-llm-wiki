## Description

<!-- Please only use this template for submitting enhancement requests -->

**What would you like to be added**:
I would like to emit a workload metric that tells me which nodes workloads are assigned to so that I can query something like "how many 64 GPU sized slots are there?" from Kueue's perspective per topology domain.

It could look something like:
kueue_tas_domain_usage{flavor="flavor-123", domain_id="host-123", resource="gpu"} 8

**Why is this needed**:
To understand Topology fragmentation, there's 2 tiers: 1.) k8s scheduler view and 2.) Kueue controller view

From the k8s scheduler view, you can sort of derive what Kueue may be seeing by mapping created pods to nodes and construct a query that looks something like: "how many 4 GPU sized slots are there?"

However there's potentially a difference between what Kueue sees and what the k8s scheduler sees. 
- Kueue could get to QuotaReserved state without admitting the workload. 
- People may also be confused from the k8s view where you see that there's slots available, but Kueue is still not admitting workloads because the quota has already been reserved (just the pods arent created yet)

I think this could be complicated to determine these slots when workloads within a flavor may satisfy different hierarchy levels. But I welcome discussion/brainstorming on how to get this overall result.

It does not complain, but some features only work if the lowest recognized topology level is hostname, because only then we have enough resolution to see every Node.

It works, but:
1. you lose some features which require node-granularity, like taints and tolerations support
2. it works just by aggregating basically the numbers from the node level, so we compute the GPU at the rack level, and build assignments, but this number of GPU is not taking into account the taints for example
3. I think NodeHotSwap will not work for TAS

EDIT: you may locate most (but not all tweaks) here: https://github.com/kubernetes-sigs/kueue/blob/df4b1c04af93b23afb16193d6e27cb9060c79560/pkg/cache/scheduler/tas_flavor_snapshot.go#L1561-L1590

> If its not recommended, maybe we should programatically prohibit it to prevent user error.

That is a possibility I'm happy to consider. So far we haven't done this, and the main reasons are:
1. maybe some niche usecases are better of by course grained resolution like optimization of memory used
2. we considered also a virtual node level, even if user specifies the "rack" level that would keep all the information about taints, but this is not done yet

But for (1.) these are speculative use cases where memory plays the role, (2.) we could relax validation ones supported.  So, I know none of them is particularly convincing, but at this point banning that is tricky.
