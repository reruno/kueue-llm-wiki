**What would you like to be added**:
Replace the shell-based e2e Kind cluster lifecycle in `cmd/experimental/kueue-populator/` with a Go implementation.

Concretely, the idea is:

- `make test-e2e` builds the image and renders the manifests
- the test suite reads those artifacts from env vars
- the cluster lifecycle now handled in `run-e2e.sh` (`kind` create/delete, image loading, manifest application, readiness waits) moves to Go

The implementation I have in mind uses [kindkit](https://github.com/IrvingMg/kindkit), a small Go library I built. It wraps `sigs.k8s.io/kind` for common lifecycle operations.

This is limited to `cmd/experimental/kueue-populator/` and does not propose changes to the main repo-root `test/e2e/` suite.

Draft PR: #10513

The draft PR contains the implementation details. This issue is mainly to get feedback on whether the approach is acceptable.

#### Tradeoffs / concerns to evaluate up front

**Dependency size / vendoring cost**

Using kindkit adds an external dependency and its transitive dependencies. This increases the size of `vendor/`, which may be a concern.

**External dependency**

kindkit is an external repository that I maintain, not Kueue-owned code or a `kubernetes-sigs` project. If that is a concern, I’m open to other options, such as donating it, copying a minimal version into Kueue, or restructuring it as needed.

#### Questions for maintainers

1. Is an external dependency acceptable for an experiment under `cmd/experimental/`?
2. Is the added dependency and vendor size acceptable for this kind of test-infrastructure change?
3. If the dependency is the main concern, would you prefer a Kueue-owned copy instead?

**Why is this needed**:

Kueue’s e2e setup uses shell scripts to manage Kind cluster lifecycle. These scripts have become more complex over time as more setup, readiness checks, and cleanup steps were added.

Shell works, but it becomes harder to keep failure handling, cleanup, and debugging consistent as the logic grows.

The goal here is to try moving the cluster lifecycle to Go in a small experimental target and see whether that makes the flow easier to understand and maintain.

`cmd/experimental/kueue-populator/` is a good place to try this first. It is small, self-contained, and already experimental, which makes it a low-risk target for evaluating the approach.

This is only meant to test the direction in a limited scope first. If it works well, it can help decide whether it is worth exploring further.
