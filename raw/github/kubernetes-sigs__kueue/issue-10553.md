**What would you like to be added**:

I would like to revisit the architectural problem we encountered and discussed here: https://github.com/kubernetes-sigs/kueue/pull/10496#discussion_r3087674111

There are some ideas here. Let me add another possiblity worth exploring which is to externalize the visbility server to a dedicated Pod/Deployment (as KueueViz or kueue-populator). This is something that @Nilsachy @dkaluza may need to consider for their use case anyway. This would allow the "core" Kueue to stop using leaderAwareReconciler model.

**Why is this needed**:

To make clarify on the long term cache management for the project.

### Comment by [@olekzabl](https://github.com/olekzabl) — 2026-04-17T10:45:49Z

Basically, both these tasks are about adding some MultiKueue-related functionality to the Visibility Server.
To handle this efficiently, we're both planning to cache _some information_ on the remote Workloads in the MultiKueue Workload Reconciler, and let Visibility use that.
(Where _some information_ should be noticeably less than "full remote Workloads", to save on memory).

If Visibility lands in a separate process, then I think we're in trouble.

Theoretically, we could try either of the following:

1. Duplicate some MK logic inside Visibility Server
2. Move the things we want to expose from Visibility Server to Kueue API (e.g. Workload status)

However:

* Approach # 1 intuitively feels like a suspicious blurring of responsibility. I'd double think before going this way.
* Approach # 2 may be completely infeasible in @Singularity23x0 's task (where, AFAIK, the goal is to make manager-side Visibility expose some info depending on worker-side Visibility output).
  Also in [my KEP](https://github.com/kubernetes-sigs/kueue/pull/10145), it'd bring various disadvantages (which are already discussed [here](https://github.com/olekzabl/kueue/blob/6c004256858686d95b54d20f3b195ba2e7f1c8d0/keps/9988-multikueue-global-quotas/README.md#expose-the-cross-worker-resource-stats-in-the-manager-clusterqueue-status)).

Not necessarily, note that currently visibility server is served by follower replicas on Kueue, so each of them is a separate binary anyway. I imagine the new visiblity server would reuse the cache and the code from the core Kueue. We just need to figure out how to populate the cache. For the cache population we basically need to run just Create/Update/Delete functions of Kueue controllers (workload controller, CQ controller etc) without running Reconcile. 

This could be acheieve in two ways:
1. pass extra param to the controllers so that they know they are "Visbility" mode, and skip Reconcile in them
2. for the visiblility server register new set of controllers, commonizing the functions for Create/Update/Delete handling with the core Kueue
