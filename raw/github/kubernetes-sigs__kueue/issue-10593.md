## Description

**What happened**:

Patching a Kueue Workload via the `v1beta1` API fails with a CEL validation error on clusters where the Workload CRD storage version is `v1beta2`.

The error:
```
The Workload "<name>" is invalid: <nil>: Invalid value: "object": no such key: priorityClassSource evaluating rule: priorityClassSource is immutable while workload quota reserved
```

This affects any PATCH or UPDATE to an admitted workload (one with `QuotaReserved: True`) when accessed through the `v1beta1` API endpoint. The same operation via `v1beta2` succeeds.

**What you expected to happen**:

Patching a workload via the `v1beta1` API (e.g., toggling `/spec/active`) should succeed, even when the workload is stored as `v1beta2`. The CEL validation rules on `v1beta1` should account for the fact that `spec.priorityClassSource` may not exist on objects originally stored as `v1beta2` (where it was replaced by `spec.priorityClassRef`).

**How to reproduce it (as minimally and precisely as possible)**:

1. Deploy a cluster with Kueue 0.16 or 0.17 (v1beta2 storage version)
2. Create a ResourceFlavor, ClusterQueue, and LocalQueue
3. Submit a workload (e.g., a Job) with a `kueue.x-k8s.io/queue-name` label
4. Wait for the workload to be admitted (`QuotaReserved: True`)
5. Attempt to patch the workload via the v1beta1 API:

```bash
kubectl patch workloads.v1beta1.kueue.x-k8s.io <workload-name> -n <namespace> \
  --type='json' \
  -p='[{"op":"replace","path":"/spec/active","value":false}]' \
  --dry-run=server
```

This fails. The same patch via v1beta2 succeeds:

```bash
kubectl patch workloads.v1beta2.kueue.x-k8s.io <workload-name> -n <namespace> \
  --type='json' \
  -p='[{"op":"replace","path":"/spec/active","value":false}]' \
  --dry-run=server
```

**Root cause**:

The v1beta1 `Workload` type has a CEL validation rule at [`apis/kueue/v1beta1/workload_types.go:790`](https://github.com/kubernetes-sigs/kueue/blob/main/apis/kueue/v1beta1/workload_types.go#L790):

```
rule: (has(oldSelf.status) && has(oldSelf.status.conditions) && oldSelf.status.conditions.exists(c, c.type == 'QuotaReserved' && c.status == 'True')) ? (oldSelf.spec.priorityClassSource == self.spec.priorityClassSource) : true
message: priorityClassSource is immutable while workload quota reserved
```

This rule unconditionally accesses `oldSelf.spec.priorityClassSource` and `self.spec.priorityClassSource` when `QuotaReserved` is `True`. However, workloads stored as `v1beta2` don't have `spec.priorityClassSource` (it was replaced by `spec.priorityClassRef`). When the API server converts the stored v1beta2 object to v1beta1 for validation, the `priorityClassSource` field may not be present, causing the CEL evaluation to fail with `no such key`.

The same issue likely affects the adjacent CEL rule at line 791 which references `spec.priorityClassName` (also removed in v1beta2, replaced by `spec.priorityClassRef.name`).

**Suggested fix**:

Guard the field accesses with `has()` checks:

```
rule: (has(oldSelf.status) && has(oldSelf.status.conditions) && oldSelf.status.conditions.exists(c, c.type == 'QuotaReserved' && c.status == 'True') && has(oldSelf.spec.priorityClassSource) && has(self.spec.priorityClassSource)) ? (oldSelf.spec.priorityClassSource == self.spec.priorityClassSource) : true
```

**Environment**:
- Kueue version: 0.16.5 (Red Hat Build of Kueue)
- Kubernetes version: 4.x (OpenShift)
- CRD storage version: v1beta2
