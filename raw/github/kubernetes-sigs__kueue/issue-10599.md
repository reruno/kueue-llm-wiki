Looking at the logs from the latest, the namespace is `frp-mhdsl`, and we see that again the Pod `frp-mhdsl/dummy-job-kind-worker-22ntr"` was evicted by the taint eviction controller in [logs](https://storage.googleapis.com/kubernetes-ci-logs/logs/periodic-kueue-test-e2e-customconfigs-release-0-16/2045744149910523904/artifacts/run-test-e2e-customconfigs-1.35.0/kind-control-plane/pods/kube-system_kube-controller-manager-kind-control-plane_d68f7b49529d521d07dd25ed652ddffa/kube-controller-manager/0.log).

Something I'm surprised by is that the Job didnt' create the replacement Pod as it seems looking at the [kube-scheduler logs](https://storage.googleapis.com/kubernetes-ci-logs/logs/periodic-kueue-test-e2e-customconfigs-release-0-16/2045744149910523904/artifacts/run-test-e2e-customconfigs-1.35.0/kind-control-plane/pods/kube-system_kube-scheduler-kind-control-plane_d70d010b880b30ee0fa449f8c48bcdc5/kube-scheduler/0.log), we only find one pod by `"frp-mhdsl/dummy-job-kind-worker-` as the search prefix.

Checking the [Kubelet logs](https://storage.googleapis.com/kubernetes-ci-logs/logs/periodic-kueue-test-e2e-customconfigs-release-0-16/2045744149910523904/artifacts/run-test-e2e-customconfigs-1.35.0/kind-worker/kubelet.log) for `"Status for pod updated successfully" pod="frp-mhdsl/dummy-job-kind-worker-22ntr"`we see the Pod remained Running for very long, and its status was updated to Succeeded only at `06:18:11.245026` which is already after the test.

Still, the Job controller should create the replacement Pod early as it should classify any Pod with deletionTimestamp as Failed (using the default PodReplacementPolicy=FailedOrTerminating). So I suspect this might be a bug in the Job controller too, but wondering how we could work around that.
