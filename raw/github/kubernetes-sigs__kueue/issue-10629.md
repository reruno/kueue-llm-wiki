## Description

_(I hesitated between "cleanup", "bug" and "feature request" :) Picked the least prominent one)_

MultiKueue controllers (on the manager cluster) fetch K8s objects from workers using remote clients.
Let's focus on Workloads and generic job kinds, as these can grow largest, in terms of QPS or volume.
For those types, MultiKueue does the following data fetches:

* A `.Watch` call [here](https://github.com/kubernetes-sigs/kueue/blob/d97ed7916df34aa9dafd017ca52755d639aa089a/pkg/controller/admissionchecks/multikueue/multikueuecluster.go#L207) (for Workloads when called from [here](https://github.com/kubernetes-sigs/kueue/blob/d97ed7916df34aa9dafd017ca52755d639aa089a/pkg/controller/admissionchecks/multikueue/multikueuecluster.go#L178), and for generic jobs when called from [here](https://github.com/kubernetes-sigs/kueue/blob/d97ed7916df34aa9dafd017ca52755d639aa089a/pkg/controller/admissionchecks/multikueue/multikueuecluster.go#L190)).
  Used to detect any changes, and trigger MK Workload Reconciler on them.
* A `.Get` call [here](https://github.com/kubernetes-sigs/kueue/blob/d97ed7916df34aa9dafd017ca52755d639aa089a/pkg/controller/admissionchecks/multikueue/workload.go#L314) (for Workloads).
  Used to reconcile a MK Workload.
* A `.List` call [here](https://github.com/kubernetes-sigs/kueue/blob/d97ed7916df34aa9dafd017ca52755d639aa089a/pkg/controller/admissionchecks/multikueue/multikueuecluster.go#L292) (for Workloads).
  Used for GC, so not very often (and can be completely [opted out](https://github.com/kubernetes-sigs/kueue/blob/d97ed7916df34aa9dafd017ca52755d639aa089a/pkg/controller/admissionchecks/multikueue/multikueuecluster.go#L667-L670) of via [Kueue Config](https://github.com/kubernetes-sigs/kueue/blob/d97ed7916df34aa9dafd017ca52755d639aa089a/apis/config/v1beta2/configuration_types.go#L305)).

Notably, the `.Get` and the `.List` use a remote client which is **uncached**.
(Evidence: it's built [here](https://github.com/kubernetes-sigs/kueue/blob/d97ed7916df34aa9dafd017ca52755d639aa089a/pkg/controller/admissionchecks/multikueue/multikueuecluster.go#L170), with `Options.Cache` unset, which calls [this](https://github.com/kubernetes-sigs/kueue/blob/d97ed7916df34aa9dafd017ca52755d639aa089a/pkg/controller/admissionchecks/multikueue/multikueuecluster.go#L166) -> [this](https://github.com/kubernetes-sigs/kueue/blob/d97ed7916df34aa9dafd017ca52755d639aa089a/pkg/controller/admissionchecks/multikueue/multikueuecluster.go#L124-L133) -> [this](https://github.com/kubernetes-sigs/controller-runtime/blob/e4a998cc6b09afaf5f2d7f30b6a9b728f21918a3/pkg/client/watch.go#L31) -> [this](https://github.com/kubernetes-sigs/controller-runtime/blob/e4a998cc6b09afaf5f2d7f30b6a9b728f21918a3/pkg/client/client.go#L204) where empty `Cache` is *not* overridden).

IIUC this leads to **potentially wasteful remote fetches**: 
* whenever the `.Watch` notifies about a remote resource change, we're given the full object (e.g. [here](https://github.com/kubernetes-sigs/kueue/blob/d97ed7916df34aa9dafd017ca52755d639aa089a/pkg/controller/admissionchecks/multikueue/multikueuecluster.go#L232)), which we even use in the MK job adapters...
* ... but then, if it's a Workload, we **drop** that full object, just to fetch it **again** in the `.Get` call.

Rough thoughts on how we could address this:

1. Just enabling caching could lead to memory outages for setups with lots of Workloads.
   We'd need a way to give up the cache if it's grown too large.
   Not sure if such a mechanism is easily available in K8s libraries.

2. Even better, given that we typically need `.Get`, not `.List`, I'd intuitively seek for a mature "bounded buffer" cache, e.g. with "Least Recently Used" per-item eviction.
   This could work poorly for `.List` calls, but would still offer some balance between "memory usage" and "not making wasteful API calls" for `.Get`s, and that's what we mostly care for here.
   Though here, it's even less clear if this can be easily done in K8s.

3. The default K8s client cache is based on an internal watch, for which I don't see a way to expose its change notifications to the caller.
  This means that, if we use this (say, with some opt-out on memory pressure, as in # 1), we'd in turn most likely duplicate the "watch" API call.
   Not sure how bad this would be.
   If it would, then one way around could be using `NewSharedInformerFactory` (thus getting access to the underlying informers?) - though I acknowledge this is not used in Kueue so far.
