**What happened**:

When a Node belongs to the multiple ResourceFlavors like the following, Kueue TAS over-subscribes the Node capacities for Workloads while the Quota Reservation is correct.

```yaml
# This Node belongs to the following both ResourceFlavors.
kind: Node
metadata:
  labels:
    example.com/machine: standard
spec:
  taints:
  - effects: NoSchedule
    key: example.com/machine
    value: standard
  # The Node doesn't have example.com/instance-type taints.
  ...
---
kind: ResourceFlavor
...
spec:
  topologyName: flat
  nodeLabels:
    example.com/machine: standard
  nodeTaints:
  - effect: NoSchedule
    key: example.com/machine
    value: standard
  - effect: NoSchedule
    key: example.com/instance-type
    value: partial-reserved
---
kind: ResourceFlavor
...
spec:
  topologyName: flat
  nodeLabels:
    example.com/machine: standard
  nodeTaints:
  - effect: NoSchedule
    key: example.com/machine
    value: standard
```

**What you expected to happen**:

There is not over subscription in TAS.

**How to reproduce it (as minimally and precisely as possible)**:
I reproduced this problem in the following scheduler UT case.
In this scenario, the `partial-reserved-pending` wl should get Node x2 Topology assignment because Node x1 has already been occupied by the `ondemand-admitted-a` and `ondemand-admitted-b` wls.

However, the current `partial-reserved-pending` wl topology assignment will be Node x2 (oversubscription).

https://github.com/kubernetes-sigs/kueue/pull/10657
https://github.com/tenzen-y/kueue/blob/6523d77d8534932e91255d25c7333478103d3254/pkg/scheduler/scheduler_tas_test.go#L2391-L2569

**Anything else we need to know?**:

Again, the Quota Reservation is correct.

> Can you elaborate what is PartialReservation flavor? Are both flavors declared in the same CQ? I think an alternative we would often use is to have a single flavor, Reservation but manage quota infependently but using two different CQs

Sure. In my cluster, some Nodes don't have interconnect, while others do. In that case, all Pods should be scheduled into a single Node when requesting a flavor that belongs to the Nodes without interconnects so that we can avoid performance regressions.

In this situation, some sub-organizations want to partially reserve a Node without interconnect capacities like 6 accelerators.
If I (cluster admin) can assume that reservation capacities are consumed only by the workload with 1 accelerator request, I partially reserve the accelerator quota for dedicated tenants. The cq nominal quotas are the sum of multiple Nodes which means I (cluster admin) can not partially reserve the accelerator quotas for workloads with multiple accelerators (> 1).

Because I (cluster admin) can guarantee that in the worst-case scenario, if there are enough nominal quotas, they can obtain 1 accelerator within a Node.

The following is a full example. In this example, project-alpha cq partially (Node) reserves for 3 example.com/gpu, but both `partially-reserved` and `ondemand` flavors point to the same Node (or Node groups <with the same compute attributions>).

```yaml
apiVersion: kueue.x-k8s.io/v1beta2
kind: Topology
metadata:
  name: flat
spec:
  levels:
  - nodeLabel: "example.com/machine"
  - nodeLabel: "kubernetes.io/hostname"
---
apiVersion: kueue.x-k8s.io/v1beta2
kind: ResourceFlavor
metadata:
  name: "partial-reserved"
  labels:
    example.com/machine: standard-partial-reserved
spec:
  topologyName: flat
  nodeLabels:
    example.com/machine: standard
  nodeTaints:
  - effect: NoSchedule
    key: example.com/machine
    value: standard
  - effect: NoSchedule
    key: example.com/instance-type
    value: partial-reserved
---
apiVersion: kueue.x-k8s.io/v1beta2
kind: ResourceFlavor
metadata:
  name: "ondemand"
spec:
  topologyName: flat
  nodeLabels:
    example.com/machine: standard
  nodeTaints:
  - effect: NoSchedule
    key: example.com/machine
    value: standard
---
apiVersion: kueue.x-k8s.io/v1beta2
kind: ClusterQueue
metadata:
  name: "project-alpha"
spec:
  namespaceSelector: {}
  resourceGroups:
  - coveredResources: ["example.com/gpu"]
    flavors:
    - name: "partial-reserved"
      resources:
      - name: "example.com/gpu"
        nominalQuota: "3"
    - name: "ondemand" # the project-alpha always borrows this flavor quota from the global shared cq or cohort.
      resources:
      - name: "example.com/gpu"
        nominalQuota: "0"
```
