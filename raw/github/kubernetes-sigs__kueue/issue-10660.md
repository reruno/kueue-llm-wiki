## Description

**What happened**:

We have a case when a failing workload with unhealthy nodes is not evicted due to AdmissionCheck=Retry, not PodsReadyTimeout exceeded.

This is the snippet of the workload state:

```yaml
admission_checks:
  - name: "some-ac"
    requeue_after_seconds: 10
    state: "Retry"
conditions:
  - message: "At least one pod has failed, waiting for recovery"
    reason: "WaitForRecovery"
    status: "False"
    type_: "PodsReady"
  - message: "Quota reserved in ClusterQueue cq"
    reason: "QuotaReserved"
    status: "True"
    type_: "QuotaReserved"
  - message: "Previously: Workload was evicted as there was no replacement for unhealthy node(s): xyz"
    reason: "QuotaReserved"
    status: "False"
    type_: "Evicted"
  - message: "The workload is admitted"
    reason: "Admitted"
    status: "True"
    type_: "Admitted"
  - message: "Workload was evicted as there was no replacement for unhealthy node(s): xyz"
    reason: "NodeFailures"
    status: "True"
    type_: "Requeued"
unhealthy_nodes:
  - name: "xyz"
```

This is likely because of this check which short-circuits the additional downstream checks: https://github.com/kubernetes-sigs/kueue/blob/main/pkg/controller/core/workload_controller.go#L263-L266

This kept the workload "admitted" for unnecessarily long.

A quick experiment shows that all unit tests pass if the check is removed, so maybe we can drop the "if" completely - I see no benefit. It was introduced in https://github.com/kubernetes-sigs/kueue/pull/6648

**What you expected to happen**:

The workload with AdmissionCheck = Retry, or exceeding the PodsReadyTimeout is evicted.
