### Comment by [@mimowo](https://github.com/mimowo) — 2026-04-22T05:00:36Z

Actually, I'm confused by the assert: https://github.com/kubernetes-sigs/kueue/blame/main/test/integration/singlecluster/scheduler/fairsharing/dra_fair_sharing_test.go#L173-L198

If we mark all workloads as finished then the usage decays down to 0 from both LQs. So, Eventually makes the assert counter-intuitive, instead of waiting "Eventually" the assert only passes if we are quick enough before the usage decays. Also the title ("Waiting for AFS ConsumedResources to reflect phase-1 usage and invariant to hold") is misleading, the usage is not 'invariant"  - it decays, but with half -life. As @tenzen-y we could make the test pass extending the half-life, but this still makes the test intention unclear.

I think to make the test intuitive I would propose to split into the following steps:

STEP1: ginkgo.By("Waiting for AFS ConsumedResources to reflect the admitted workloads")

STEP2: ginkgo.By("Finishing all initial workloads")

STEP3: ginkgo.By("Verify the usage remains positive after a while since the workloads are finished")
use Consistently with a small interval, like 1s to check the usage > 0

Tbh step3 feels optional as this is testing AFS (which already has many tests) rather than than DRA.
