### Comment by [@mimowo](https://github.com/mimowo) — 2025-10-01T09:12:15Z

Ok, I did a bit of archeology and for context we had this similar issue here: https://github.com/kubernetes-sigs/kueue/pull/3986, and as mitigation steps the proposal was summarized here: https://github.com/kubernetes-sigs/kueue/issues/4040:

```
For the experiments in the context of [Kuberay multikueue integrations tests](https://github.com/kubernetes-sigs/kueue/pull/3986) we learned that:
- increasing resources for the job for integration tests helped a little bit, but didn't fully solve the problem with running Ray (see [comment](https://github.com/kubernetes-sigs/kueue/pull/3986#issuecomment-2609155189))
- we know that setting INTEGRATION_NPROC=3 makes the tests usually pass, but it means lower parallelism enforced on the lightweight tests, which is not ideal
```
However, we have never fully understood why the issue was occurring, just found the above mitigates the issue.

I'm not yet totally sure what is the best way to move forward. I suppose the issue was triggered again by including the TrainerJob CRDs, but I would like to understand what is going on. Notably with decreased parallelism I wasn't able to repro here: https://github.com/kubernetes-sigs/kueue/pull/7085

If the issue continues and we cannot understand still maybe we need to mitigate again by decreasing parallelism (not sure yet, brainstorming).
