## Description


**What happened**:

When CQ quota is specified as 1E to simulate "infinity", then effectively it is handled as 0.

To see this is supported check the tests for resource.Quantity https://github.com/kubernetes/kubernetes/blob/master/staging/src/k8s.io/apimachinery/pkg/api/resource/quantity_test.go#L317

**What you expected to happen**:

We correctly support 1E since allowed by API. If this is really problematic to support, then we should have validation against it.

**How to reproduce it (as minimally and precisely as possible)**:

1. Create a CQ and specify nominalQuota as 1E.
2. Submit a workload with cpu request.
Issue: the workload fails with `couldn't assign flavors to pod set main: insufficient quota for cpu in flavor default-flavor, previously considered podsets requests (0) + current podset request (3) > maximum capacity (0)`

**Anything else we need to know?**:

1P seems to work fine

## AI Analysis (Claude) — Root Cause & Fix Plan

### Root Cause

The bug is in `pkg/resources/requests.go:104-108` — the `ResourceValue` function:

```go
func ResourceValue(name corev1.ResourceName, q resource.Quantity) int64 {
    if name == corev1.ResourceCPU {
        return q.MilliValue()
    }
    return q.Value()
}
```

When `nominalQuota: 1E` is set for **CPU** resources:
- `ResourceValue` calls `q.MilliValue()`
- "1E" = 10^18. Milli-units would be 10^21, which **overflows int64** (max ~9.2×10^18)
- Kubernetes' `MilliValue()` silently returns **0** on overflow

Verified by testing:
```
1E  -> Value()=1000000000000000000  MilliValue()=0        ← BUG (overflow → 0)
1Ei -> Value()=1152921504606846976  MilliValue()=-9223372036854775808  ← overflow to negative
1P  -> Value()=1000000000000000    MilliValue()=1000000000000000000   ← works fine
```

This explains why "1P seems to work fine" (as noted in the issue) — 1P × 1000 = 10^18 which fits in int64, while 1E × 1000 = 10^21 does not.

The error message `maximum capacity (0)` confirms the quota is stored as 0 after the overflow.

### Proposed Fix

**Modify `ResourceValue`** to detect int64 overflow and cap at `math.MaxInt64`:

```go
func ResourceValue(name corev1.ResourceName, q resource.Quantity) int64 {
    if name == corev1.ResourceCPU {
        v := q.MilliValue()
        // MilliValue() can overflow for very large quantities (e.g. "1E").
        // If the quantity is positive but MilliValue() returns <= 0, cap at MaxInt64.
        if v <= 0 && q.Cmp(resource.Quantity{}) > 0 {
            return math.MaxInt64
        }
        return v
    }
    v := q.Value()
    if v <= 0 && q.Cmp(resource.Quantity{}) > 0 {
        return math.MaxInt64
    }
    return v
}
```

Users specifying "1E" for CPU quota clearly want "unlimited", and `MaxInt64` milliCPU (~9.2 exaCPU) achieves that intent.

Optionally, webhook validation in `pkg/webhooks/clusterqueue_webhook.go` could also be added to warn users when a quantity would overflow, but the `ResourceValue` fix alone is backward-compatible and sufficient.

### Files to modify
- `pkg/resources/requests.go` — overflow detection in `ResourceValue`
- `pkg/resources/requests_test.go` — test cases for overflow scenarios

### Comment by [@mimowo](https://github.com/mimowo) — 2026-03-16T08:55:56Z

I don't think this is a good idea, because **it only makes the code fail later in subtle places. In particular computation of the aggregated capacity in cohorts will still overflow**. Scattering the checks against exceeding int64 across the codebase is not maintainable.

To fix it properly I think we need to stop using int64 basically, because it can always overflow, especially for quota management which adds (aggregates quotas). Or, at least have some concept of unlimited.

I'm thinking we need a dedicated struct like `Amount` which would replace int64 for quota computations.

For the implementation I'm ok-ish to make it both precise, and just "unlimited", so either:

1. Precise wrapper:
```golang
type Amount struct {
  q resource.Quantity
}
```
2. Capped wrapper:
```golang
type Amount struct {
  unlimitted bool
  value          int64
}
```
Both have some merit:
(1.) is precise, but performance may slightly suffer
(2.) is enough to reflect the user intention of "infinite" capacity, makes sure we don't overflow, and will have virtually the same "speed" as bare int64

I don't have a strong view, probably leaning slightly towards (1.), but I'm happy to change my mind given feedback from other maintainers, cc @gabesaba @tenzen-y wdyt?

For me the key part of the fix is to have the abstraction "Amount" which will allow to fix the issue. We may later swap the underlying implemenation
