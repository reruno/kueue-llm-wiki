### Comment by [@mimowo](https://github.com/mimowo) — 2026-04-10T16:01:46Z

> I wonder if we can encapsulate these metrics in the hierarchy package, both as it has the best picture of what is going on, and so details don't leak around the code base.

Yeah I think for this specific metric is might be sensible, but I was also thinking about this in the context of other metrics - for the guage for admitted workloads in a cohort subtree: https://github.com/kubernetes-sigs/kueue/pull/10013. 

In that case the hierarchy package does not have the full context so it does not seem like a univesal approach. 

Also, since the package is reused by both "scheduler cache" and "queue cache" we may need to be careful to avoid updating the metrics twice for a single change, this is harmless by itself, but might be wasted work. 

I'm not saying this is bad idea, but just something to consider.
