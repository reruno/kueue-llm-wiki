## Description

#### What type of PR is this?

/kind bug

#### What this PR does / why we need it:

During a RayJob with [autoscaling enabled](https://docs.ray.io/en/latest/cluster/kubernetes/user-guides/configuring-autoscaling.html) running time, if kill a worker pod, a new pod will be started but stuck in scheduling gate status. The new pod does not change to running state due to reconciling never run again.

This PR adds pod level watch for RayJob to trigger reconcile and ungate pods.

RayService also has similar issue.

#### Which issue(s) this PR fixes:

Fixes #9986
Fixes #9879

### Comment by [@mimowo](https://github.com/mimowo) — 2026-03-20T06:08:32Z

I would like to take a step back here. The proposed approach adds new interface which wouldn't be needed if solved properly, so I'm hesitant to merge it.

Let's establish first:
1. why the Reconcile for the RayCluster is not triggered when pods fail, or if triggered, why they don't call StartWorkloadSlicePods
2.does the issue also affect other Job integrations - namely k8s batch Job integration?

To determine (1.) I would really like to see the report from the watch on such a RayCluster, by `kubectl get -w --output-watch-events -oyaml`, while the Pods are being replaced.

Now, as mentioned in another comment I believe the proper way of fixing is a new Pod controller filtering Pods by the scheduling gate for elastic Jobs. When it observes such a Pod it triggers reconcile for the associated WorkloadSlice, and the Reconcile function ungates the Pods if the latest Workload in the slice in admitted.

This is basically the same pattern as for TopologyUngater for TAS.

Now, I understand this is much more work to refactor, and the feature is Alpha, and the bug is important. 

So, if we need a quick fix I would rather add 10s RetryAfter which is independent of the Job type, whenever an ElasticJob is running. However, I would then add a graduation criteria for Beta to refactor the ungating model.
