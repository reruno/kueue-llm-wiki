## What this PR does / why we need it

`startJob()` and `stopJob()` use `clientutil.Patch` with strict resourceVersion but without `WithRetryOnConflict()`. When external controllers (KubeRay, MultiKueue SyncJob) modify the job object concurrently, the patch fails with a conflict error and the job remains stuck permanently.

This is a 2-line fix that adds `clientutil.WithRetryOnConflict()` — already used in 6 other call sites — to both functions.

## Which issue(s) this PR fixes

Fixes #10040

## Does this PR introduce a user-facing change?

```release-note
Fix startJob and stopJob to retry on conflict when external controllers (e.g., KubeRay) concurrently modify the job object, preventing jobs from getting permanently stuck in suspended state.
```

## Additional context

The conflict was made much more likely after #8308 removed the `IsSuspended` guard from CRD MultiKueue adapters (correctly fixing #8162), which allowed SyncJob to write during the admission-to-unsuspend window. In MultiKueue + KubeRay setups, three controllers race on the same object:

1. KubeRay operator (finalizer, state transitions — 4 writes per new RayJob)
2. MultiKueue SyncJob (status sync from manager)
3. Local kueue controller (unsuspend via startJob)

Without retry, `startJob`'s patch is nearly guaranteed to hit a stale resourceVersion.

In v0.13.3, the `IsSuspended` guard incidentally prevented the third writer, making the two-way race narrow enough to rarely trigger. The guard removal was correct (fixing stale status), but exposed this latent fragility.

The fix is orthogonal — it doesn't revert #8308, it makes the unsuspend/suspend path resilient to concurrent writes regardless of the source.

---

**Note:** The companion test PR #10058 adds an integration test that reproduces this bug using a client interceptor. Once this fix lands, the test assertion should be updated from `Consistently(Suspend=true)` to `Eventually(Suspend=false)` to verify prompt unsuspension.
