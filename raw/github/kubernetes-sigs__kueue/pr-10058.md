## What this PR does / why we need it

Adds tests that reproduce the startJob Patch conflict race condition described in #10040. Both tests **fail without the fix**, serving as regression tests.

**Unit test** (`TestStartJobRetriesOnConflict`): Injects a single Patch conflict via interceptor and asserts that `ReconcileGenericJob` retries and succeeds.

**Integration test** (`Job controller startJob conflict retry`): Injects 12 consecutive Patch conflicts for RayJob objects via a manager-level client interceptor. After admitting a workload, asserts the job gets unsuspended (`Eventually(Suspend=false)`) within the test timeout. Without retry-on-conflict, exponential backoff accumulates to ~20s, exceeding the 10s timeout.

## Additional context

Companion to #10041 (which contains the fix). The tests assert the correct behavior — both fail on current main, demonstrating the bug.

The conflict race occurs in MultiKueue + KubeRay deployments where three controllers write to the same job object during the admission-to-unsuspend window, causing `clientutil.Patch` to fail with 409 Conflict. Without retry, each failure triggers controller-runtime exponential backoff (5ms base, doubling), keeping the job suspended for extended periods.

>## What this PR does / why we need it
>
>Adds an integration test that **deterministically reproduces the startJob Patch conflict race condition** described in #10040.
>
>In MultiKueue + KubeRay deployments, three controllers race on the same RayJob object during the admission-to-unsuspend window:
>
>1. **KubeRay operator** — finalizer add, state transitions (New → Initializing → Suspending → Suspended): ~4 writes
>2. **MultiKueue SyncJob** — status sync from manager to local job
>3. **Local kueue controller** — `startJob` Patch to set `Suspend=false`
>
>Each concurrent write bumps `resourceVersion`, causing kueue's `clientutil.Patch` (which includes resourceVersion in the MergePatch) to fail with 409 Conflict. Without retry-on-conflict, each failure triggers a full re-reconcile with controller-runtime's exponential backoff (5ms base, doubling each failure). After 12 conflicts, the cumulative backoff exceeds 20 seconds, during which the job remains suspended.
>
>### Test approach
>
>The test uses a **client interceptor** (via `framework.WithNewClient`) to inject deterministic Patch conflict errors for RayJob objects, controlled by an atomic counter. This avoids timing-dependent goroutine races and makes the reproduction reliable.
>
>The test flow:
>1. Create a RayJob → kueue suspends it → workload is created
>2. Inject 12 Patch conflicts (targeting the `startJob` window specifically)
>3. Admit the workload to trigger `startJob`
>4. Assert the job **remains suspended** during the backoff window (`Consistently`)
>
>The `Consistently(Suspend=true)` assertion documents the current buggy behavior. The fix PR (#10041) will update this to `Eventually(Suspend=false)` to verify prompt unsuspension.
>

The test reproduces this regardless of the conflict source — any concurrent writer that bumps resourceVersion triggers the same backoff cascade.
