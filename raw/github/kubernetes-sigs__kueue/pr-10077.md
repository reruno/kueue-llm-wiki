+---
+title: "Dynamic Resource Allocation"
+date: 2026-03-22
+weight: 7
+description: >
+  Quota management for workloads using Kubernetes Dynamic Resource Allocation (DRA).
+---
+
+## Overview
+
+[Dynamic Resource Allocation (DRA)](https://kubernetes.io/docs/concepts/scheduling-eviction/dynamic-resource-allocation/)
+is a Kubernetes API for requesting and managing hardware devices such as GPUs,
+FPGAs, and network adapters. Kueue can account for DRA devices in quota
+management through two paths:
+
+1. **ResourceClaimTemplates**: Pods explicitly reference a `ResourceClaimTemplate`
+   that specifies a device request. Kueue maps each `DeviceClass` referenced by
+   the claim to a logical resource name using `deviceClassMappings` in the
+   Kueue Configuration.
+
+2. **Extended Resources**: Pods request DRA devices using the traditional
+   `resources.requests` syntax (e.g., `nvidia.com/gpu: 1`). When the
+   Kubernetes `DeviceClass` has an `extendedResourceName` field set
+   ([KEP-5004](https://github.com/kubernetes/enhancements/issues/5004)),
+   the kube-scheduler automatically creates `ResourceClaim` objects from
+   these requests. Kueue detects this and avoids double counting.
+
+**Which path should I use?** If your workloads already use `resources.requests`
+for devices (e.g., `nvidia.com/gpu: 1`), use the extended resource path. If
+your workloads explicitly create `ResourceClaimTemplate` objects, use the
+ResourceClaimTemplate path.
+
+## ResourceClaimTemplate path
+
+{{< feature-state state="alpha" for_version="v0.14" >}}
+{{% alert title="Note" color="primary" %}}
+`DynamicResourceAllocation` is currently an alpha feature and is disabled by default.
+
+You can enable it by editing the `DynamicResourceAllocation` feature gate. Refer to the
+[Installation guide](/docs/installation/#change-the-feature-gates-configuration)
+for instructions on configuring feature gates.
+{{% /alert %}}
+
+When a Pod references a `ResourceClaimTemplate`, Kueue reads the
+`deviceClassName` from the `exactly` field (only `ExactCount` allocation mode
+is supported), looks it up in `deviceClassMappings`, and charges quota using
+the configured logical resource name. CEL selectors and the `All` allocation
+mode are not supported in alpha.
+
+The administrator must configure `deviceClassMappings` in the Kueue
+Configuration for this path.
+
+## Extended resource path
+
+{{< feature-state state="alpha" for_version="v0.17" >}}
+{{% alert title="Note" color="primary" %}}
+`DRAExtendedResources` is currently an alpha feature and is disabled by default.
+It requires the `DynamicResourceAllocation` feature gate to also be enabled.
+
+You can enable both by editing the feature gates. Refer to the
+[Installation guide](/docs/installation/#change-the-feature-gates-configuration)
+for instructions on configuring feature gates.
+{{% /alert %}}
+
+When a Pod requests an extended resource backed by DRA, the kube-scheduler
+auto-creates a `ResourceClaim`. Without `DRAExtendedResources` enabled, Kueue
+would charge quota for both the `resources.requests` entry and the claim,
+double counting the same device.
+
+With the feature gate enabled, Kueue detects the matching `DeviceClass`, uses
+`extendedResourceName` as the quota key, and drops the auto-created claim from
+accounting. No `deviceClassMappings` configuration is needed. The mapping is
+discovered from the `DeviceClass` automatically.
+
+{{% alert title="Note" color="info" %}}
+This requires the Kubernetes `DRAExtendedResource` feature gate on
+kube-apiserver and kube-scheduler. It is alpha (disabled by default) in
+Kubernetes 1.34 and 1.35.
+{{% /alert %}}
+
+### Path separation
+
+The two paths are independent:
+- **ResourceClaimTemplate path**: uses `deviceClassMappings` configuration.
+- **Extended resource path**: uses auto-discovery from `DeviceClass` objects.
+
+Avoid configuring the same `DeviceClass` in both paths for the same workload.
+If overlap occurs, Kueue merges the resources and uses the `deviceClassMappings`
+logical name as the unified quota key.
+
+## Admin-facing configuration
+
+### deviceClassMappings (ResourceClaimTemplate path)
+
+To use the ResourceClaimTemplate path, add `deviceClassMappings` to the Kueue
+Configuration:
+
+```yaml
+apiVersion: config.kueue.x-k8s.io/v1beta2
+kind: Configuration
+resources:
+  deviceClassMappings:
+  - name: example.com/gpu           # Logical resource name for quota
+    deviceClassNames:
+    - gpu.example.com               # DeviceClass name(s)
+```
+
+- `name`: The resource name used in `ClusterQueue` quotas and `Workload` status.
+- `deviceClassNames`: One or more `DeviceClass` names that map to this resource.
+
+Multiple device classes can map to the same logical resource name. For example,
+if you have separate device classes for different GPU models but want a single
+quota pool:
+
+```yaml
+resources:
+  deviceClassMappings:
+  - name: example.com/gpu
+    deviceClassNames:
+    - gpu-a100.example.com
+    - gpu-h100.example.com
+```
+
+### Extended resource path (no configuration needed)
+
+For the extended resource path, no `deviceClassMappings` configuration is
+required. The administrator only needs to ensure:
+
+1. The Kubernetes `DeviceClass` has `spec.extendedResourceName` set.
+2. Both `DynamicResourceAllocation` and `DRAExtendedResources` feature gates
+   are enabled in Kueue.
+
+Kueue automatically discovers the mapping by indexing `DeviceClass` objects.
+
+### ClusterQueue setup
+
+Configure your `ClusterQueue` to track the DRA resource, just like any other
+resource:
+
+```yaml
+apiVersion: kueue.x-k8s.io/v1beta2
+kind: ClusterQueue
+metadata:
+  name: cluster-queue
+spec:
+  resourceGroups:
+  - coveredResources: ["cpu", "memory", "example.com/gpu"]
+    flavors:
+    - name: default-flavor
+      resources:
+      - name: "cpu"
+        nominalQuota: 40
+      - name: "memory"
+        nominalQuota: 200Gi
+      - name: "example.com/gpu"
+        nominalQuota: 8
+```
+
+## User-facing APIs
+
+Users submit workloads using either approach. No Kueue-specific annotations are
+required for DRA workloads.
+
+### Using ResourceClaimTemplates
+
+```yaml
+apiVersion: resource.k8s.io/v1
+kind: ResourceClaimTemplate
+metadata:
+  name: single-gpu
+  namespace: default
+spec:
+  spec:
+    devices:
+      requests:
+      - name: gpu
+        exactly:
+          deviceClassName: gpu.example.com
+---
+apiVersion: batch/v1
+kind: Job
+metadata:
+  generateName: sample-job-
+  namespace: default
+  labels:
+    kueue.x-k8s.io/queue-name: user-queue
+spec:
+  template:
+    spec:
+      containers:
+      - name: dummy-job
+        image: registry.k8s.io/e2e-test-images/agnhost:2.53
+        args: ["pause"]
+        resources:
+          claims:
+          - name: gpu
+          requests:
+            cpu: "1"
+            memory: "200Mi"
+      resourceClaims:
+      - name: gpu
+        resourceClaimTemplateName: single-gpu
+      restartPolicy: Never
+```
+
+Kueue counts this workload as using 1 unit of `example.com/gpu` quota (from
+the `deviceClassMappings` configuration).
+
+### Using extended resources
+
+```yaml
+apiVersion: batch/v1
+kind: Job
+metadata:
+  generateName: sample-job-
+  namespace: default
+  labels:
+    kueue.x-k8s.io/queue-name: user-queue
+spec:
+  template:
+    spec:
+      containers:
+      - name: dummy-job
+        image: registry.k8s.io/e2e-test-images/agnhost:2.53
+        args: ["pause"]
+        resources:
+          requests:
+            cpu: "1"
+            memory: "200Mi"
+            nvidia.com/gpu: "1"
+          limits:
+            nvidia.com/gpu: "1"
+      restartPolicy: Never
+```
+
+When the `DeviceClass` has `extendedResourceName: nvidia.com/gpu`, the
+kube-scheduler automatically creates a `ResourceClaim`. Kueue detects this and
+charges quota only for the `nvidia.com/gpu` extended resource, not for the
+auto-created claim.
+
+## MultiKueue
+
+DRA workloads are supported with [MultiKueue](/docs/concepts/multikueue). The
+`ResourceClaimTemplate` objects are synced to worker clusters alongside the
+workload.
```

  The MultiKueue section says ResourceClaimTemplate objects are synced to worker clusters. Current MultiKueue adapters only sync the workload and the owner job object; ResourceClaimTemplates must be created on the worker cluster separately (as noted in keps/2941-DRA). Please adjust this section to reflect that templates are not automatically synced, or document the exact mechanism if this has changed.
  ```suggestion
  DRA workloads are supported with [MultiKueue](/docs/concepts/multikueue). MultiKueue
  adapters sync the workload and its owning job to worker clusters, but they do not
  automatically sync `ResourceClaimTemplate` objects. Any required
  `ResourceClaimTemplate` must be created on each worker cluster separately (for
  example, by the cluster administrator).
  ```

+Kueue counts this workload as using 1 unit of `example.com/gpu` quota (from

+### Using extended resources
+
+```yaml
+apiVersion: batch/v1

+            memory: "200Mi"

+          limits:

  This doc uses `example.com/gpu` in the ClusterQueue example, but the extended resource Job example later uses `nvidia.com/gpu`. Since the resource name must match `spec.extendedResourceName` and the ClusterQueue's `coveredResources`, mixing names in one page is likely to confuse readers. Consider using a single resource name consistently throughout (or explicitly call out that these are alternative examples and must be aligned in real setups).
