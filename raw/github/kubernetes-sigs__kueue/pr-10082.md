### Comment by [@j-skiba](https://github.com/j-skiba) — 2026-03-27T11:15:07Z

I agree, my solution introduces this problem once again.

I have other high level idea, let me know what do you think about this.

Instead of using queue heap as a lock we could impelment a preemption reservation system in cache and remove stickiness mechanism. Instead of using queue heap as a lock we will move the lock to the cache.

1. Preemption reservations mechanism:
- when a workload triggers preemption, the cache creates a reservation for these resources. When other workloads are evaluated they see less available resources - solves #6929
- when the owner of the reservation is evaluated, it is allowed to use its own reservation

2. Delete stickiness
We revert heap to `Priority > Timestamp` - this solves #7301 partially because there is still [situation](https://github.com/kubernetes-sigs/kueue/issues/7301#issue-3524501313) when `j03` is in the process of preempting and already has some reservations then when `j04` comes it doesn't want to wait for preemption process of `j03` to finish, instead it could transfer those reservations to `j04`. 

3. Reservation transfer
So there would have to be another mechanism of transferring reservations between workloads.
If a strictly higher priority workloads comes while a reservation is active then it should be allowed to transfer the reservation to itself. This allows the high-priority job to fast-track and take the resources freed by the low-priority job immediately.

+ I'm not sure about TAS in this approach

I might be missing something here, but I wonder if this would be doable. Wdyt?
