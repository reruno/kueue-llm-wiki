  Oh indeed, disabling kubelet on a node may require waiting time to re-enable the node. Even if we wait for the node to be Ready in the test code it does not necessarily mean that all informers already consider the Node as ready. So, it is possible that Kueue scheduler may still see the Node notReady, and avoid scheduling here, which may result in some flakes also in other tests.
  
  We solved this problem for node HotSwap by running a dummy workload (which requires the temporarily disabled node specifically) after the re-enablement of the Kubelet on the node. So I propose to extract this code https://github.com/kubernetes-sigs/kueue/blob/main/test/e2e/tas/hotswap_test.go#L93-L120 as a helper and use it here
