#### What this PR does / why we need it:

- If the admission config is not set, the [updateAfsConsumedUsage](https://github.com/kubernetes-sigs/kueue/blob/91cbbe65fca0d99a13ffc3647b785e141e3d3add/pkg/controller/core/workload_controller.go#L1111) wouldn't have a chance to run in the WorkloadReconciler.
- Also, If the resources listed on the localqueue entry penalties have value as zero, the entry penalty can be cleaned up. This helps with ensuring no pending items are listed for the localqueue penalties and local queue reconciler do not think anymore that there are entry penalties to handle.

### Review by [@sohankunkerkar](https://github.com/sohankunkerkar) (commented) — 2026-04-02T03:12:39Z

There's an edge case here: The scheduler adds a penalty to the map when a workload is admitted, and with this fix the workload controller subtracts the same amount, leaving the value at zero. But Sub never removes the key from the map, so `HasPendingFor` still sees it and returns true. This makes the LQ reconciler think there are pending penalties to process and it runs on every event instead of waiting for the sampling interval. We could fix the thrashing by either deleting the key when values hit zero or having `HasPendingFor` check if values are actually positive, but that can be a follow-up.

Also you might need to change the release note to something like `Fixed AFS entry penalties...`
