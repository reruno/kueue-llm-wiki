## Description

## Summary

This PR fixes two related issues with orphaned workloads holding ClusterQueue quota:

### 1. Stale workload accumulation after PodsReady timeout eviction

When a pod workload is evicted due to `waitForPodsReady` timeout, Kueue deletes the pod via `Stop()` and requeues the workload with exponential backoff. If the pod is owned by a Deployment, the Deployment creates a replacement pod with a new workload. The old evicted workload keeps getting requeued despite its owning pod being deleted, causing unbounded stale workload accumulation.

**Fix:** Add a `FinishOrphanedWorkloads` feature gate (Beta in 0.18, default enabled) that detects when a workload's controller owner no longer exists (`isControllerOwnerGone` check in the workload reconcile path) and finishes the workload with reason `OwnerNotFound`.

### 2. Jobs deleted with `--cascade=orphan` hold quota indefinitely (#1789)

When a Job is deleted, the jobframework reconciler's `dropFinalizers` path removes the workload's finalizer but does not finish it. With `--cascade=orphan`, the workload remains admitted and holds quota indefinitely.

**Fix:** Call `workload.Finish` with `OwnerNotFound` in the `dropFinalizers` path before removing the finalizer, so the workload releases quota immediately.

- `pkg/controller/core/workload_controller.go` — Added `isControllerOwnerGone` orphan detection in the general reconcile path
