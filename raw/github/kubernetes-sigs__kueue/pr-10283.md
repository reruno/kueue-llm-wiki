+#### Processing Flow
+
+Partitionable devices reuses the upstream `dracel` compiler from
+[#7298](https://github.com/kubernetes-sigs/kueue/issues/7298) but performs its own
+ResourceSlice listing with pool-aware processing, since the CEL validation path does not
+return matched device objects or do pool grouping:
+
+1. Kueue detects that the RCT's DeviceClass is in `counterMappings`
+2. Lists ResourceSlices for the configured driver, groups them by pool and checks completeness
+3. Compiles and evaluates CEL selectors against devices in complete pools
+4. For matched devices, reads `consumesCounters` entries
+5. Resolves the quota resource name from the matching `counterMappings` entry
+6. Uses the maximum consumption across matched devices as the per-device counter charge
+7. For requests with `count > 1`, multiplies per-device consumption by count
+
+Counter resources are injected through the existing `WithPreprocessedDRAResources` path.
+
+In Alpha, partitionable devices performs its own ResourceSlice listing independently from
+the CEL validation path because the two have different requirements: CEL validation only
+needs a match count, while counter processing needs matched device objects and pool-aware
+grouping. In Beta, these can be consolidated into a shared ResourceSlice listing layer.

+When no devices match or ResourceSlice data is unavailable, the workload is marked
+inadmissible. Workloads are never admitted with zero counter charge when `counterMappings`
+are configured. Kueue only considers ResourceSlices from the pool with the highest

+and the workload is marked inadmissible.

+When CEL selectors match devices with different profiles (e.g., both 1g.10gb and 7g.80gb),
+Kueue charges the maximum consumption. Users should write specific CEL selectors that
+narrow to a single profile for accurate quota.

+slices in the same pool. On K8s 1.34 (where partitionable devices are alpha) this
+validation does not apply and some drivers put both in one slice
+(`resourceSliceCount: 1`). Pool completeness checks `len(slices) == resourceSliceCount`
+so both layouts work.
+
+The example below shows the separate-slice layout. Only the `memory` counter is shown.
+The driver also
+publishes `multiprocessors` and `memory-slice-0` through `memory-slice-7` which the
+kube-scheduler uses for MIG placement but Kueue does not need for quota:
+
+```yaml
+# ResourceSlice 1: SharedCounters (total capacity for this GPU)
+spec:
+  driver: gpu.nvidia.com
+  pool:
+    name: node1-gpu0
+    generation: 1
+    resourceSliceCount: 2
+  sharedCounters:
+  - name: gpu-0-counter-set
+    counters:
+      memory:
+        value: 80Gi
+      # Driver also publishes multiprocessors, memory-slice-0 through
+      # memory-slice-7 - used by kube-scheduler, not by Kueue.
+---
+# ResourceSlice 2: Devices with ConsumesCounters
+spec:
+  driver: gpu.nvidia.com
+  pool:
+    name: node1-gpu0
+    generation: 1
+    resourceSliceCount: 2
+  nodeName: node1
+  devices:
+  - name: gpu-0-mig-1g.10gb-0
+    attributes:
+      gpu.nvidia.com/profile:
+        string: "1g.10gb"
+    consumesCounters:
+    - counterSet: gpu-0-counter-set
+      counters:
+        memory:
+          value: 9856Mi
+  - name: gpu-0-mig-7g.80gb-0
+    attributes:
+      gpu.nvidia.com/profile:
+        string: "7g.80gb"
+    consumesCounters:
+    - counterSet: gpu-0-counter-set
+      counters:
+        memory:
+          value: 80Gi
+```

+#### User Workload
+
+Users request MIG profiles via CEL selectors on ResourceClaimTemplates. Counter tracking
+is transparent to the user:
+
+```yaml
+apiVersion: resource.k8s.io/v1
+kind: ResourceClaimTemplate
+metadata:
+  name: mig-1g-10gb
+spec:
+  spec:
+    devices:
+      requests:
+      - name: gpu
+        exactly:
+          deviceClassName: mig.nvidia.com
+          selectors:
+          - cel:
+              expression: "device.attributes['gpu.nvidia.com'].profile == '1g.10gb'"
+```

+#### Configuration
+
+A new `counterMappings` field in the Configuration API maps driver counters to Kueue quota
+resources. Each counter mapping binds a DeviceClass to a counter-based quota resource. A
+DeviceClass can appear in either `deviceClassMappings` or `counterMappings` but not both.
+This is validated at config load time.
+
+The `Resources` struct is extended with a `CounterMappings` field:
+
+```golang
+type Resources struct {
+    // ...existing fields...
+
+    // CounterMappings defines mappings from DRA driver counters to logical resources
+    // for partitionable device quota management.
+    // +optional
+    CounterMappings []CounterMapping `json:"counterMappings,omitempty"`
+}
+
+// CounterMapping maps a DRA driver counter to a Kueue quota resource for
+// partitionable device support.
+type CounterMapping struct {
+    // Name is the Kueue resource name used in ClusterQueue coveredResources
+    // and Workload resourceUsage.
+    // Must follow the same naming rules as DeviceClassMapping.Name.
+    // +required
+    // +kubebuilder:validation:Required
+    // +kubebuilder:validation:MaxLength=253
+    Name corev1.ResourceName `json:"name"`
+
+    // DeviceClassNames lists the DeviceClasses that use counter-based quota.
+    // A DeviceClass can appear in either deviceClassMappings or counterMappings
+    // but not both. This is validated at config load time.
+    DeviceClassNames []corev1.ResourceName `json:"deviceClassNames"`
+
+    // Driver is the DRA driver name used to filter relevant ResourceSlices.
+    // Must match the spec.driver field on ResourceSlice objects.
+    // +required
+    // +kubebuilder:validation:Required
+    // +kubebuilder:validation:MaxLength=253
+    Driver string `json:"driver"`
+
+    // CounterName is the counter name within the device's consumesCounters
+    // entries to track for quota. Must match a counter name published by
+    // the driver in ResourceSlice devices' consumesCounters field.
+    // Counter set names are per-GPU identifiers (e.g., gpu-0-counter-set,
+    // gpu-1-counter-set) so requiring one mapping per GPU would be impractical.
+    // +required
+    // +kubebuilder:validation:Required
+    // +kubebuilder:validation:MaxLength=63
+    CounterName string `json:"counterName"`
+}
+```
+
+```yaml
+apiVersion: config.kueue.x-k8s.io/v1beta2
+kind: Configuration
+featureGates:
+  DynamicResourceAllocation: true
+  DRAPartitionableDevices: true
+resources:
+  deviceClassMappings:
+  - name: whole-gpu
+    deviceClassNames: [gpu.nvidia.com]
+
+  counterMappings:
+  - name: gpu.memory
+    deviceClassNames: [mig.nvidia.com]
+    driver: gpu.nvidia.com
+    counterName: memory
+```
+
+`gpu.nvidia.com` uses device counting (deviceClassMappings). `mig.nvidia.com` uses
+counter-based quota (counterMappings). A DeviceClass cannot appear in both. This is
+enforced at config load time.
+
+Counter names are driver-specific (NVIDIA uses `memory`, another vendor may use `gpu-mem`),
+so `counterMappings` translates them to Kueue quota resource names. Not all counters are
+quota-relevant. NVIDIA MIG devices have 15+ counter dimensions. The admin chooses which
+counters to track for quota. The scheduler handles the rest.
+
+ClusterQueue uses counter resources like any other resource:
+
+```yaml
+apiVersion: kueue.x-k8s.io/v1beta2
+kind: ClusterQueue
+metadata:
+  name: mig-queue
+spec:
+  resourceGroups:
+  - coveredResources: ["gpu.memory"]
+    flavors:
+    - name: a100-mig-pool
+      resources:
+      - name: gpu.memory
+        nominalQuota: "320Gi"
+```
+
+#### Processing Flow
+
+Partitionable devices reuses the upstream `dracel` compiler from
+[#7298](https://github.com/kubernetes-sigs/kueue/issues/7298) but performs its own
+ResourceSlice listing with pool-aware processing, since the CEL validation path does not
+return matched device objects or do pool grouping:
+
+1. Kueue detects that the RCT's DeviceClass is in `counterMappings`
+2. Lists ResourceSlices for the configured driver, groups them by pool and checks completeness
+3. Compiles and evaluates CEL selectors against devices in complete pools
+4. For matched devices, reads `consumesCounters` entries
+5. Resolves the quota resource name from the matching `counterMappings` entry
+6. Uses the maximum consumption across matched devices as the per-device counter charge
+7. For requests with `count > 1`, multiplies per-device consumption by count
+
+Counter resources are injected through the existing `WithPreprocessedDRAResources` path.
+
+In Alpha, partitionable devices performs its own ResourceSlice listing independently from
+the CEL validation path because the two have different requirements: CEL validation only
+needs a match count, while counter processing needs matched device objects and pool-aware
+grouping. In Beta, these can be consolidated into a shared ResourceSlice listing layer.
+
+When no devices match or ResourceSlice data is unavailable, the workload is marked
+inadmissible. Workloads are never admitted with zero counter charge when `counterMappings`
+are configured. Kueue only considers ResourceSlices from the pool with the highest
+generation. If the slice count is less than `resourceSliceCount`, the pool is incomplete
+and the workload is marked inadmissible.
+
+When CEL selectors match devices with different profiles (e.g., both 1g.10gb and 7g.80gb),
+Kueue charges the maximum consumption. Users should write specific CEL selectors that
+narrow to a single profile for accurate quota.
```

  I added explicit note for the count > 1 case. The alpha behavior is intentionally conservative: max * count rather than top-N summation. In practice, users write profile-specific selectors so all matched devices have the same counter values and max * count equals actual consumption.
