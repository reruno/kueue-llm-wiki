  Note that it avoids the extra allocations related to building the same map twice as in the current branch - once to check `CanFit` (represented by `sum`) and once to do `Add`.  In the current branch, the allocations are saved only in a happy path. In the not so happy path, there is the same amount of allocations, or even more than in the proposal in this comment. 
  
  Still, I think it is a secondary question which proposal we choose out of those listed in comments https://github.com/kubernetes-sigs/kueue/pull/1039#discussion_r1290282730, https://github.com/kubernetes-sigs/kueue/pull/1039#discussion_r1290352459, and this one. What they have in common is to avoid the extra checks. As discussed in the comment https://github.com/kubernetes-sigs/kueue/pull/1039#discussion_r1291003945 we already have allocations and read locks taken in each cycle. 
  
  The cost of the extra checks is code complexity, which means longer dev time in the future, and risk of issues. Finally, it is not even clear which solution is faster. I think I make the argument clear, but I see there might be a trade off, so let me defer the decision to @alculquicondor .
