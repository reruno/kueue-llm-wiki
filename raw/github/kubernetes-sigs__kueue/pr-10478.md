#### What this PR does / why we need it:

When a workload with completed podSets (count=0 after reclaim) is preempted and re-queued, FindTopologyAssignmentsForFlavor produces a non-nil TopologyAssignment with empty Domains for the count=0 podSets. This converts to empty Slices in the API format, which CRD validation rejects with "topologyAssignment.slices: Required value", causing an infinite scheduling loop with no backoff.

#### Which issue(s) this PR fixes:

Skip count=0 podSets in WorkloadsTopologyRequests. PodSets with 0 pods need no topology assignment

N/A

No the count is decremented to 0 when 1 of the replicated jobs completes

> 2. is the preemption / reclamation relevant here? What makes the difference between the first admission and the repeated admission

yea preemption triggers the rescheduling based on the state where some of the jobs have completed already to trigger the bug

> 3. how does the infinite loop break now, does the workload get re admitted with podSets.count=0?

Manual remediation is needed to cleanup the workload 

> Would you like to put together an integration test for the scenario? It would be great

Ive added an integration test for this scenario, the exact case that TAS doesnt handle well today is

1. JobSet is created with 2 replicated jobs (A,B)
2. A completes but B is still running
3. JobSet is preempted and TAS attempts to reschedule
4. Since A has completed, TAS generates an empty topology assignment which fails validation against the API server
5. The reschedule and validation failure loop continues indefinitely until the JobSet is deleted
