#### What this PR does / why we need it:

Adds a new optional field `spec.workloadDefaults.maximumExecutionTimeSeconds` to `LocalQueueSpec` that allows cluster administrators to configure a default maximum execution time for workloads submitted to a LocalQueue. When a job is submitted without the `kueue.x-k8s.io/max-exec-time-seconds` label, the workload automatically inherits the LocalQueue's default.

This builds on the existing maximum execution time feature (KEP-3125).

**Precedence rules:**
1. Job label (`kueue.x-k8s.io/max-exec-time-seconds`) â€” highest priority
2. LocalQueue default (`spec.workloadDefaults.maximumExecutionTimeSeconds`) â€” used when job has no label
3. None â€” no maximum execution time set

**Changes include:**
- KEP-7029 document and kep.yaml
- New `LocalQueueWorkloadDefaults` struct with `maximumExecutionTimeSeconds` field added to both `v1beta1` and `v1beta2` `LocalQueueSpec` (nested under `spec.workloadDefaults`)
- New `localqueue_defaults.go` with `LocalQueueDefaults` internal type and two exported helpers: `GetLocalQueueDefaults` (looks up the LocalQueue and extracts defaults) and `ApplyLocalQueueDefaults` (applies defaults to a workload without overriding existing values)
- Reconciler logic in `pkg/controller/jobframework/reconciler.go` applies LocalQueue defaults both at workload creation (`handleJobWithNoWorkload`) and workload update (`updateWorkloadToMatchJob`)
- Updated `EquivalentToWorkload` to only compare `MaximumExecutionTimeSeconds` when the job explicitly sets the label â€” if the label is absent, the workload's value may have been inherited from the LocalQueue and no mismatch is reported
- Deprecated `ApplyLocalQueueMaxExecutionTime` wrapper for custom reconcilers, pointing to `GetLocalQueueDefaults`/`ApplyLocalQueueDefaults` instead
- LeaderWorkerSet reconciler uses `GetLocalQueueDefaults`/`ApplyLocalQueueDefaults` directly in `createPrebuiltWorkload`
- Generated code: deepcopy, conversion, CRD manifests, client-go apply configurations, API reference docs
- Unit tests for Job, JobSet, and LeaderWorkerSet controllers
- Integration tests for Job and LeaderWorkerSet controllers (including a new LWS integration test suite)

#### Which issue(s) this PR fixes:

Implements https://github.com/kubernetes-sigs/kueue/issues/7029

#### Special notes for your reviewer:

- The default is applied to the Workload, not the Job, to avoid surprising mutations on user objects and to ensure universal coverage across all job types.
- The `workloadDefaults` struct is designed to be extensible â€” future LocalQueue-level defaults (e.g. priority, TAS configuration) can be added here without further API changes.
- The `EquivalentToWorkload` change is important: without it, a workload that inherited its timeout from the LocalQueue would be seen as mismatched vs. the job (which has no label), causing an unnecessary recreate loop.
- The LocalQueue lookup uses the controller-runtime client cache, so the additional GET has negligible performance impact.
- Existing workloads are not retroactively updated when the LocalQueue's timeout is modified or the job moves to a different queue.
