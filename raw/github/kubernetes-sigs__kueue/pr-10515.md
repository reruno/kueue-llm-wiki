## Description

## Summary
Addresses #1789 by making orphan cleanup rely on workload-stored owner metadata instead of Job-side orphan handling.

- record owner GVK, owner name, and owner UID on created and prebuilt workloads
- finish orphaned workloads from the workload controller after orphan propagation removes owner references, including the window where the owner object is already deleting
- update job and pod controller expectations and add focused unit/integration coverage for orphaned and stale workload behavior

### Comment by [@mimowo](https://github.com/mimowo) — 2026-04-17T10:59:08Z

@sebest I think the main complication here is how to distiguish a Workload which lost a parent due to --cascade=orphan from a workload which is prebuilt - like in the context of MultiKueue. 

I think we can safely delete the Workloads which had the Job in the past and lost it. So, we would need some kind of annotation to annotate they are adopted which outlives the ownerReference loss. Maybe something like kueue.x-k8s.io/owner-reference-uid: <>`. This could be populated either when creating a new Workload, or when adopting a worokload (setting owner reference).

Alternative is to use some pre-existing annotation/label to determine the distinction, controllerconsts.JobUIDLabel seems like a good fit actually.
