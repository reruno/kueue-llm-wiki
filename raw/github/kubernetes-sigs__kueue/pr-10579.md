#### What this PR does / why we need it:
Update the SubtreeQuota of all ancestors to reflect the removal of the deleted cohort. 
Without this, stale SubtreeQuota values would cause any subsequent RecordCohortMetrics call for ancestor cohorts to re-publish the pre-deletion metric values.

1. The scenario is:
After `DeleteCohort`, the hierarchy relation is updated, but the ancestor `SubtreeQuota` in cache used to stay stale.
`ClearCohortMetrics `could still publish the correct post-delete metric value, because it simulates the removal.
Later, another `RecordCohortMetrics` call would read the stale cache and publish the old value again.

I also have a minimal repro in mind at the cache/metrics layer, without relying on timing:
clear metrics for the deleted cohort, delete it, then trigger recording from a sibling/child cohort again and check that root goes back to the old quota pre-fix.

2. The metrics test is mostly uncovering `resourceNode` cache consistency issue, not just a metrics-only problem.

3. AFS is not the root cause, it just happens to be around for the integration tests, because we cloned the setup from fair_sharing.
It just increases the chance of reproducing it, because it causes frequent queue rebuild / reconcile activity, and that leads to more RecordCohortMetrics calls against stale cache
