#### Special notes for your reviewer:

The test was added recently in https://github.com/kubernetes-sigs/kueue/pull/1576 to cover the logic modified in https://github.com/kubernetes-sigs/kueue/pull/1399.

The test used to flake as the order of considered workloads in a single scheduling cycle could be (A,B) or (B, A). The intention of the test is to cover scenario when (A,B). Thus adding 2*time.Second to the creation time of B.

Also, make sure all timestamps refer to the same `now`. Otherwise, it would be technically possible that `time.Now().Add(time.Second) > time.Now().Add(2*time.Second)` (though it would be super unlikely).

### Comment by [@mimowo](https://github.com/mimowo) — 2024-01-15T11:24:22Z

> Why does the test flakes when order is (B, A)? It seems B should be just scheduled and A tries to preempt anyway

When we have (B, A), then the workload B is admitted and its usage is added here https://github.com/kubernetes-sigs/kueue/blob/2ee9e0f6eea07e25be6722c314dbb9a7c9a79ca1/pkg/scheduler/scheduler.go#L207. 
Now, when A comes we have `hasCommonFlavorResources` returning True, and so `CanFit` is checked and responds with false (because it does not take preemption into account). Now, we execute `continue` and the admission (with preemption) is skipped for entry A. 

TBH I'm not sure this is WAI or a bug. I guess it can is not a big deal because in the next scheduling cycle we will be able to progress with A (if there are not other workloads in the scheduling cycle).

Still, the intention of the unit test was to test (A, B), because in this case B isn't blocked after https://github.com/kubernetes-sigs/kueue/pull/1399.
