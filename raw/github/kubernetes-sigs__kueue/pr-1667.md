#### What this PR does / why we need it:

Solve two issues with events for pod groups:
1. the event indicating the a pod in pod group was emitted repeatedly, while the intention is to sent it once
2. the event was emitted only for the first pod in group, but because the Load function may return the pods in different order, we could get both pods covered with at least one event

This PR also adds e2e test to ensure the events are emitted per pod in the group.

### Comment by [@mimowo](https://github.com/mimowo) — 2024-02-02T12:14:33Z

Ok, I understand the root cause. We assume with the stop interface that the only stopped Job is represented with the object ([here](https://github.com/kubernetes-sigs/kueue/blob/a8c11f351858cd159a3f823b15860918c0490cbc/pkg/controller/jobframework/reconciler.go#L746-L749)), but this is not the case for composable jobs, where multiple pods are stopped.

Also, I know why it sometimes passes, this is because the `stopJob` routine is called multiple times before the workload finished (around 10 times), and the `Load` function returns the pods in a non-deterministic order. So sometimes we get the event for `pod-1`.

There is another issue, we actually send the `Stopped` now event multiple times for the same pod, because the Stop function returns always true, assuming no errors: https://github.com/kubernetes-sigs/kueue/blob/a8c11f351858cd159a3f823b15860918c0490cbc/pkg/controller/jobs/pod/pod_controller.go#L462.

I think one solution would be to extend the API of Stop by returning a slice of actually stopped Jobs "now", or slice of extra jobs stopped now. WDYT?

EDIT: pushed initial version of the fix. It also prevents sending multiple events for the same object. PTAL.
