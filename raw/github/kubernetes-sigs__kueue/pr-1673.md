  I think it is much better to combine these two PRs. 
  
  The issue with this one merged first is that it introduces a bug, that even a temporary loss of connectivity may result in two jobs running at the same time. This is problematic for users who push results to database, and don't want two jobs pushing results at the same time. 
  
  Still, this situation is acceptable, but the time needs to be controllable by users. Let's have one PR which addresses the issue of permanent connection loss, by requeuing of the workloads, where "permanent" is 15min default, but controlled by users.
