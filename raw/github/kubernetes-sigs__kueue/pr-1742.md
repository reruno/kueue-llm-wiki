#### What this PR does / why we need it:

Allow for existing jobs to be imported into Kueue by disabling suspension/stopping of jobs that do not have admitted Workloads. This allows Kueue to create Workload objects and quota to be reconciled for existing workloads. Intended to facilitate the installation of Kueue into a running cluster.

Example steps for adopting Kueue:

1. Ensure all new jobs in a cluster contain kueue labels.
2. Create all ClusterQueues, etc.
3. Run Kueue in import-mode.
4. Wait for all non-labeled jobs to finish.
5. Validate expected Workload admission statuses exist.
6. Switch Kueue into non-import-mode.

### Comment by [@nstogner](https://github.com/nstogner) — 2024-02-16T14:23:42Z

> which kueue labels concretely, just queue-name or some other too? This entails a separate script I guess which iterates over all jobs and patches them, right?

> How we would have non-labeled jobs since we added the labels in step (1.)

I guess there are multiple paths that could be taken, I was describing one where there is a system that is responsible for submitting new jobs, and that system is first updated to start submitting jobs as if Kueue were in place. All labels for identifying queues, pod group sizes, etc. would be included even though Kueue is not yet in place. You could also run a script to label running workloads, but my description was assuming that you would just allow those workloads to age out before running Kueue. A user could take either or both approaches (age-out / script-label).

> Additional, extra question: do we need to deal / how do we deal with overcommitted resources? Meaning, say someone has running jobs exceeding the quota configured in point (2.).

My original thought was the following which sounds like your (1.):

Allow Kueue to create Workloads for all running jobs during import mode. While suspension/stopping is disabled, some of these Workloads would have an admitted condition and some would not, but all would continue to run. The administrator could then inspect which of the running Workloads are considered admissible under the new Kueue system they are introducing. They could then turn off import-mode and allow the Workloads in violation to be evicted as per the quotas they have defined in the queues - or they could change their queues until all workloads fit in their queues.
