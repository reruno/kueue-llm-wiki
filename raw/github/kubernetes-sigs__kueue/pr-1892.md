<blockquote>
<h2>v1.1.0 release</h2>
<h1>Highlights</h1>
<ul>
<li>
<p><strong>RayJob improvements</strong></p>
<ul>
<li><strong>Gang / Priority scheduling with Kueue:</strong>
<ul>
<li><a href="https://docs.ray.io/en/master/cluster/kubernetes/examples/rayjob-kueue-gang-scheduling.html#kuberay-kueue-gang-scheduling-example">Gang scheduling doc</a></li>
<li><a href="https://docs.ray.io/en/master/cluster/kubernetes/examples/rayjob-kueue-priority-scheduling.html">Priority scheduling doc</a></li>
<li><a href="https://docs.ray.io/en/master/cluster/kubernetes/k8s-ecosystem/kueue.html#kuberay-kueue">Kueue / RayJob - tutorial with toy example that everyone can try it locally</a></li>
</ul>
</li>
<li><strong>ActiveDeadlineSeconds (new field)</strong>: A feature to control the lifecycle of a RayJob. See <a href="https://docs.ray.io/en/master/cluster/kubernetes/getting-started/rayjob-quick-start.html">this doc</a> and <a href="https://redirect.github.com/ray-project/kuberay/issues/1933">#1933</a> for more details.</li>
<li><strong>submissionMode (new field)</strong>: Users can specify “K8sJobMode” or “HTTPMode”. The default value is “K8sJobMode”. In HTTPMode, the submitter K8s Job will not be created. Instead, KubeRay sends a HTTP request to the Ray head Pod to create a Ray job. See <a href="https://docs.ray.io/en/master/cluster/kubernetes/getting-started/rayjob-quick-start.html">this doc</a> and <a href="https://redirect.github.com/ray-project/kuberay/issues/1893">#1893</a>  for more details.</li>
<li>Fix a lot of stability issues.</li>
</ul>
</li>
<li>
<p><strong>Structured logging</strong></p>
<ul>
<li>In KubeRay v1.1.0, we have changed the KubeRay logs to JSON format, and each log message includes context information such as the custom resource’s name and reconcileID. Hence, users can filter out logs associated with a RayCluster, RayJob, or RayService CR by its name.</li>
</ul>
</li>
<li>
<p><strong>RayService improvements</strong></p>
<ul>
<li>Refactor health check mechanism to improve the stability.</li>
<li>Deprecate the <code>deploymentUnhealthySecondThreshold</code> and <code>serviceUnhealthySecondThreshold</code> to avoid unintentional preparation of new RayCluster custom resource.</li>
</ul>
</li>
<li>
<p><strong>TPU multi-host PodSlice support</strong></p>
<ul>
<li>The KubeRay team is actively working with the Google GKE and TPU teams on integration. The required changes in KubeRay have already been completed. The GKE team will complete some tasks on their side this week or next. Then, users should be able to use multi-host TPU PodSlice with a static RayCluster (without autoscaling).</li>
</ul>
</li>
<li>
<p><strong>Stop publishing images on DockerHub; instead, we will only publish on Quay.</strong></p>
<ul>
<li><a href="https://quay.io/repository/kuberay/operator?tab=tags">https://quay.io/repository/kuberay/operator?tab=tags</a></li>
<li>Users should use docker pull <code>quay.io/kuberay/operator:v1.1.0</code> instead of docker pull <code>kuberay/operator:v1.1.0</code>.</li>
</ul>
</li>
</ul>
