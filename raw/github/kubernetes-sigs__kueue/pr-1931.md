### Comment by [@mimowo](https://github.com/mimowo) — 2024-04-11T15:44:44Z

> Relying on kueue's internal metrics in this scenario is problematic because:
> 
> * Adds extra CPU consumption in minimalkueue: Serving the scraping will add extra CPU consumption and make the critical path (scheduling) less relevant in the overall consumption.

Prometheus monitoring of metrics is used for scalability tests for core k8s, and we didn't see the extra CPU cycles to be a problem. Metrics are scraped in prod environments as well. 

> * Are unflexible: No workload class specific measurements can be extracted. In my opinion being able track the time to admission for specific type of workload (having different priorities) is important.

I admit this is a valid point, some things are not easy to observe with metrics. We could observe 99% percentile for the `AdmissionWaitTime`, but this would not tell us its name, to find it in logs.

I talked with scalability folks internally, and most of the asserts (like 90%) are metrics-based. However, indeed, there are things monitored differently, for example [pod startup latency](https://github.com/kubernetes/perf-tests/blob/master/clusterloader2/pkg/measurement/common/slos/pod_startup_latency.go) is based on events.

Most (if not all?) SLOs which are defined (and validated by scalability tests) are metrics-based. For example, for [API server responsiveness](https://github.com/kubernetes/community/blob/master/sig-scalability/slos/api_call_latency.md).

A benefit of using metrics is that they benefit prod environments, so scalability tests for developers give a healthy incentive to monitor the production system as well. If there is no metric for something we want to measure, we should have a preference / incentive to add the metric, rather than hack just for scalability tests.

An interesting question would be if we can load the time series into Prometheus, so that we can evaluate the PromQL queries. PromQL is handy at expressing conditions like this, which we could use then for asserts. Then,we would not have custom code for x-percentiles, average, etc. Similarly to how we do in the k8s scalability tests [see here](https://github.com/kubernetes/perf-tests/blob/fcc4cfd0fab4193837b33b7b82ea15093a7b6fa0/clusterloader2/pkg/measurement/common/slos/api_responsiveness_prometheus.go#L54-L59).

> * Are not very precise:
>   
>   * Scraping at 1s intervals while the workloads we are using run for at most 1s (90% of them way under) can miss important hot spots.

I don't think we use less than 5s for k8s scalability tests, from the [OSS](https://github.com/search?q=repo%3Akubernetes%2Fperf-tests+interval%3A&type=code&p=1)). I proposed 1s "just in case". 

Sure, there could be hot spots, but we are actually more interested in flattened metrics to make a judgment how kueue copes with the spikes over time. For example the SLI for API server responsiveness looks into 5m windows: "Latency of processing non-streaming read-only API calls for every (resource, scope) pair, measured as 99th percentile over last 5 minutes".

>   * With some exceptions, the timings computed for the metrics are based on some k8s api stored time (creation, condition last transition) which have seconds resolution.
> It needs to be mentioned that the runner components that are extracting the measurements have additional attributes, (managing when a workload is declared finished, managing workload's eviction , determine when the scenario execution has ended.)

Right, but this has not proven to be a problem for scalability tests for the core k8s, so I would suggest starting with metrics, and extend if we have some observability gaps.

### Comment by [@mimowo](https://github.com/mimowo) — 2024-04-12T10:36:01Z

I think using PromQL and full Promethueus as part of the CI run might be hard (I don't know how to do it myself either). 

I think metrics as a tool to visualize what is happening during a run is enough for MVP. Actually, manual inspection of metrics and pprof was the main tool to investigate the performance bugs mentioned above.

I got promising results already using metrics based on `run-scalability-in-cluster`. This is what I did:

0. Have an external cluster with pre-installed Kueue, I expose the metrics by port forward `kubectl port-forward pod/kueue-controller-manager-66c8489484-v5zbw -nkueue-system 12345:8080`
2. Run the following `collect.sh` script to scrape the data:
```sh
while true; do
    filename="dump/$(date +%s).prometheus"
    echo "dump to ${filename}"
    curl -v http://localhost:12345/metrics > "${filename}"
    sleep 1
done
```
3. Run the following `import.sh` script to import the data to a local instance of VictoriaMetrics
```sh
for file in ./dump/*.prometheus; do
    if [[ -f $file ]]; then
        filebase=$(basename "$file" .prometheus)
        timestamp="${filebase}000"
        echo "$timestamp"
        curl -vX POST -T "$file" http://localhost:8428/api/v1/import/prometheus?timestamp="$timestamp"
    fi
done
```
4. Once in VictoriaMetrics I could visualize all the metrics with Prometheus queries:

![image](https://github.com/kubernetes-sigs/kueue/assets/10359181/ac740312-32f7-46f4-b27d-bf7ffc73eeb7)

5. Loaded the data to grafana for better visualization capabilities, in particular handling of histograms:

![image](https://github.com/kubernetes-sigs/kueue/assets/10359181/6b035dec-64d2-4065-9690-80d686c538e5)

I think this is encouraging and gives us metrics OOTB for kueue, and controller-runtime. 

So, I think what I would like to see in the PR / Issue:
- code that dumps the metrics in 1s interval (but configurable), analog of `collect.sh`, but part of the runner
- compress the dumps into a tar.gz or zip, which is archived
- a usability script (analog of import.sh) to load the zip into VictoriaMetrics

Then drop the custom code for API, for the detection of the completion of the scenario either use the small portion of the API-based code or base it on metrics (await for 0 pending_workloads and 0 reserving_workloads).

EDIT: I have added the "Collection of metrics and visualization" into the [doc](https://docs.google.com/document/d/1iO8CGO6kgav2JL6RKYnNYvkeQIHpFqB-SQqJ7eUsjPs/edit?tab=t.0#heading=h.e5jhgh56ju14).
