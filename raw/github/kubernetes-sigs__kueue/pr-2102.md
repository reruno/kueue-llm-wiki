#### What this PR does / why we need it:
In the test environment, we observed many logs due to completing the scheduling loop very fast and entering it again immediately. This change adds a backoff if there were no admissions during a scheduling loop

### Comment by [@gabesaba](https://github.com/gabesaba) — 2024-05-07T10:26:08Z

I uploaded a simple implementation using [NewItemExponentialFailureRateLimiter](https://github.com/kubernetes/kubernetes/blob/02365ecec1cb1ddf993cf4ed12407737db950cea/staging/src/k8s.io/client-go/util/workqueue/default_rate_limiters.go#L95). The downside with this approach is that Sleep will not notice an interrupt (ctx.Done). On the other hand, we are only sleeping for up to 100ms.

A slightly more involved approach would be to create a custom [BackoffManager](https://pkg.go.dev/k8s.io/apimachinery/pkg/util/wait#BackoffManager) which implements the above logic. We would then call [BackoffUntil](https://pkg.go.dev/k8s.io/apimachinery/pkg/util/wait#BackoffUntil) rather than [UntilWithContext](https://github.com/kubernetes-sigs/kueue/blob/bd42edb21408a70d8e52c76ddc8713b78aedd966/pkg/scheduler/scheduler.go#L128). BackoffUntil handles the [ctx.Done](https://github.com/kubernetes/apimachinery/blob/v0.30.0/pkg/util/wait/backoff.go#L238-L245) signal properly when backing off.

If you prefer the more involved approach, please let me know.
