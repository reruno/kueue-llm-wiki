### Comment by [@kerthcet](https://github.com/kerthcet) — 2022-05-07T07:45:49Z

This maybe a bug with `ClusterQueueBestEffortFIFO`:

https://github.com/kubernetes-sigs/kueue/blob/20406a45269c12e6f37bfbaff0fb5ad4625a8b9a/test/integration/scheduler/scheduler_test.go#L304-L355

Refer to above spec, here's the possible sequence of events:
1. LowPriorityWorkload and HighPriorityWorkload created
2. Since this is a BestEffortFIFO, unadmitted LowPriorityWorkload will be stored in inadmissibleWorkloads
3. Test passed, will enter cleanup phase.
4. When deleting the namespace, we will delete the queues, but forget to cleanup the LowPriorityWorkload in  inadmissibleWorkloads, this is a bug I think.
5. Deleting workloads, but since queues are already deleted, we will directly return in `deleteWorkloadFromQueueAndClusterQueue`, and the LowPriorityWorkload is still in inadmissibleWorkloads.
6. Somehow triggered the broadcast, drive the workloads from inadmissibleWorkloads and went into the `Heads()` process, since clusterQueues haven't been deleted, but queues are already deleted, panic then. We can find the proofs from the log below: the queues are all deleted, together with the workloads, but clusterQueues haven't been deleted for namespace is still in terminating.
