  I have synced with @mbobrovskyi and the underlying issue is that as soon as the replacement pod is created the pod_controller removes the finalizer for the other Pod (see [here](https://github.com/kubernetes-sigs/kueue/blob/main/pkg/controller/jobs/pod/pod_controller.go#L1170-L1172)).
  
  So, occasionally `g.Expect(k8sClient.Get(ctx, origKey, &p)).To(gomega.Succeed())` may fail for the second pod due to 404.
  
  Splitting into two phases eliminates the problem, because we don't create the replacement until we checked both pods are failed. Alternative approach, would be to check Failed OR 404, but this is an extra complication.
  
  I'm ok to refactor the test to simplify.
