### Comment by [@mimowo](https://github.com/mimowo) — 2024-08-02T07:48:25Z

> We need to find a way to publish the stats as metrics and be able to tract the trends, continuing to just bump the threshold for any flake (most likely due to test infra overloading) will just get us to a pint where a real regression cannot be detected.

That is true, however:
1. I checked the commits, and there was nothing that would indicate there is a regression. 
2. overloaded infra happens, we need to take it into account. Otherwise, everyone will suffer failures on branch builds, and this really makes developer experience not good. 
3. I'm really not sure how overloaded infra would contribute to higher CPU usage as indicated in this failed test - I flagged this early on with @alculquicondor that the metric is hard to make sense, during the review, and we suggested to drop this threshold entirely (for reference: [link](https://github.com/kubernetes-sigs/kueue/pull/2067#discussion_r1579791977)). 

Given the above, I think bumping is justified in this case.

Having said that, I see value in the performance tests, and would support an effort to collect and process historical data quickly.

### Comment by [@mimowo](https://github.com/mimowo) — 2024-08-02T13:27:45Z

> For me this is an indication that the performance expectations should not change.

None of the bumps to the thresholds so far was attributed to a regression. So following this idea would mean accepting continuous failures at the  constant frequency (for correct branches).

After we introduced the first thresholds we had the first flake reported 2 days later (reported as issue, there could have been more), so probably it would mean ~1 random failure every two days. Does not sound good, so I think the decision to bump was justified.

I doubt that if we kept accepting failures at the initial rate then people would diligently study the reason for failures. More likely, people would just keep ignoring, but get annoyed.
