I discussed this with @dgrove-oss offline.
In conclusion, we reached an agreement for the current basic implementation since `.status.resourceRequests` is reset after the admission, and the `.status.admission` has transformed the resource name and computation unit as well.
Additionally, the field is useful even when the transformation is disabled since the field has reflected runtime overhead and lilmitranges as well.

But, we had some discussion points for the API name and feature gate for the `.status.resourceRequests`.

Especially, for the feature gate, we have some option in the following since the `.status.resourceRequests` is not only for this transformation feature, but for the future break, guarding the status field by feature gate would be better:

1. We do not add any gate the field.
2. We add the `ConfigurableResourceTransformations` to the field.
3. We add a dedicated and separated from the `ConfigurableResourceTransformations` gate to the field like `KueueRecognizedResourceRequests`.

TBH, I (@dgrove-oss as well) am leaning toward option 3 since the feature gate could allow us to break specifications (especially, whether or clearing the field after the admission, not, etc...), and the dedicated gate give us flexibility for the graduation. Even when we need to introduce or break the transformation mechanism, we can graduate the `.status.resourceRequests` to the beta or GA.

If we select the option 3, we can prepare the dedicated KEP or just add additional feature gate to the transformation KEP.

@mimowo Let us know what you think.

### Comment by [@mimowo](https://github.com/mimowo) — 2024-10-16T06:54:22Z

Thank you for the summary and driving the discussions. I'm happy of the general conclusion to introduce `Status.ResourceRequests`. My thinking process:
* with this feature (Configurable Resource Transformations) users may not be able to see the actual transformation matrix due to RBAC restrictions, so we need to expose the information somehow.
* the minimal structure to expose it would be `TransformedResourceRequests` (or `DerivedResourceRequests` as they didn't exist before), but it would mean that users investigating scheduling decisions need to look at two places: `status.derivedResourceRequests` and `.spec.podSets[*].template.spec`, and the values would require rescaling (total vs per pod).
* `status.resourceRequests` contains duplicated information, but it improves visibility (single place to check). Size is not an issue, because it anyway needs to be replaced with the structure of the same size on admission.
* the new field is a new feature and setting it should be gated

Regarding the choice of gating the field, I consider (2.) or (3.) as viable options. I don't see a need for a new KEP, because this KEP introduced the motivation for it. We still can have 2 feature gates in a single KEP,  [example](https://github.com/kubernetes/enhancements/blob/master/keps/sig-apps/3329-retriable-and-non-retriable-failures/kep.yaml#L40-L50)) for the core k8s. If we go for a new feature-gate, maybe call it `WorkloadResourceRequestsSummary`.
