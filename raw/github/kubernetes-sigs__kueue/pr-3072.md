### Comment by [@mbobrovskyi](https://github.com/mbobrovskyi) — 2024-10-08T18:21:22Z

> Looking at the requirements for this to work (namely the RBAC complication) I wonder if we can instead do a nslookup?
> 
> The domain name of the first pod can be passed statically.

That's where we started. The issue is that the InitContainer can't get the IP address from the service because the pod hasn't been initialized or started yet. We can check it within the container, but in that case, the user must use an image that includes nslookup.

### Comment by [@mimowo](https://github.com/mimowo) — 2024-10-09T07:38:44Z

Exactly, we started with the approach of using nslookup, but this has cons either:
1. it required the fully qualified Pod name with the cluster domain, so we had to add ".svc.cluster.local" suffix, which could not work for some clusters, see [thread](https://github.com/kubernetes-sigs/kueue/pull/3072#discussion_r1787774660)
2. it needs to run on the user's image, which may not necessarily have the tool if this is just. Some AI/ML oriented images may not have it, creating friction to use the feature. For example I checked that `nvidia/cuda:11.8.0-devel-ubuntu22.04` doesn't have it.

Having said that the need to configure RBAC is not ideal either, so I'm on the fence with the feature. 

Seeing none of the approaches as ideal I proposed to move forward as is, but hide it behind the opt-in option: `--first-node-ip`.  Once we have feedback from users we could revisit the approach. Hopefully most users don't need IP, and could rely on DNS resolution based on names in `SLURM_JOB_NODELIST`.

### Comment by [@mimowo](https://github.com/mimowo) — 2024-10-14T13:01:10Z

Today I have discussed this with @mwielgus and there is one more idea to use the https://kubernetes.io/docs/concepts/workloads/pods/downward-api/. The flow would be:
1. use the downward API to dump the .status.PodIP field into a volume / file as in the [example](https://kubernetes.io/docs/tasks/inject-data-application/downward-api-volume-expose-pod-information/)
2. in the init container the 0-th index pod reads the IP from status, sets the SLURM_JOB_FIRST_NODE_IP, and the 0th-indexed Pod becomes running, and so resolvable behind headless service
3. other pods wait on resolving the IP with nslookup (or another tool that we install on the container image); all the pods start

@mbobrovskyi it would be great if you can check if this works.
