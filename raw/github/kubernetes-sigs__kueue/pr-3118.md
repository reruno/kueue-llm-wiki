### Comment by [@trasc](https://github.com/trasc) — 2024-09-25T11:43:16Z

> Not sure if this is the same issue, or different. If this is a different issue I'm happy to open as a follow up.

It is related but not very easy to solve, the problem being that in a scenario in which we have both partial admission and dynamic reclaim, the lower count in the admission can be the result of either:

- The job has executed in the past, the reclaimable count was incremented,  evicted, and readmitted with the lower count.
- The job was partially admitted.

Currently we don't have a way to distinguish between the two, one way we can do it is to "save" the reclaimable count at the scheduling time in the admission struct, but since we need to update the API, maybe  a follow-up is better. 

EDIT:

As a temporary solution, just to avoid confusion, we can think of disabling the dynamic-reclaim for jobs that are running with partial admission.
