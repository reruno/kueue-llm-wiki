I discussed the necessity for the `requeue: true | false` field with @PBundyra offline.
In conclusion, we decided not to introduce the field and introduce the `KeepQuotaForProvReqRetry` feature gate to the existing Retry mechanism as a deprecated default disabled feature gate, then we will completely remove the `KeepQuotaForProvReqRetry` in the v0.10 or v0.11 so that we can give users a reprieve period. This could be justified since the CA ProvisioningRequest API is still in the alpha stage, and Kueue integration could potentially be broken.

  I discussed this with @PBundyra offine.
  This feature tries to introduce the requeuing Workload (WL) mechanism when the AdmissionCheck (AC) gets the `Rejected` condition due to ProvisioningRequest (PR) getting the `Failed` condition.
  
  The current ProvisioningRequest deactivation mechanism is `Failed (PR) -> Rejected (AC) -> <<RETRY by Reconcile Requeue>> -> Deactivated (WL)`. So, the WL will be deactivated without requeueing.
  Hence, this feature will introduce the requeuing mechanism to `Rejected (AC) -> Deactivated (WL)` phase like `Failed (PR) -> Rejected (AC) -> <<REQUEUE WL>> -> Failed (PR) -> Rejected (AC) -> Deactivated (WL)`.
  
  This allows us the possibility to distinguish between infrastructure and application configuration errors because, after this feature, the WaitForPodsReady indicates only application configuration errors due to capacity checks can be made before admission. But distinguishing between both error reasons could be considered as a separate feature from this feature. 
  
  So, we will implement the distinguishness after investigating the values, and we will focus only on the PR requeuing feature here.
