### Comment by [@mimowo](https://github.com/mimowo) — 2024-11-26T14:58:43Z

@tenzen-y The underlying mechanism for each of the integrations is different - for Deployment we use single pod integration, for STS we use pod groups, and for Jobs we use GenericJob reconciler. So, the e2e tests aren't redundant and it makes sense to cover the new integrations with e2e tests rigorously. 

I think the main reason we have e2e tests for job isn't actually dependence on node (i actually don't know about such test in Kueue), but just testing the interplay between the Job controller and the controllers in Kueue. The same goes for Deployments and StatefulSets - we want to check the interplay between their corresponding k8s controllers, and the controllers in Kueue. I believe this is also the reason we have e2e tests for the PodGroup integration - the overall complexity of the feature, and interplay between an external controller managing the group and Kueue.

### Comment by [@mimowo](https://github.com/mimowo) — 2024-11-26T15:14:04Z

> So, does that mean that this E2E aims to verify the behavior couple of Kueue and kube-controller-manager?

IIUC yes - I called it interplay between k8s controllers and Kueue controllers, but "couple" probably could reflect the same.

> Is it enough the integration testings?

In my opinion there are scenarios we want to cover e2e to gain greater confidence in the features. Technically, you can reach 100% code coverage with just integration tests (or even unit tests), but I believe e2e are important to make sure the integrations work.

> What is the primary reason that we added this E2E, but we did not add E2E for RayCluster?

The integrations for Deployment, StatefulSet, PodGroup and Job use all different integration mechanisms, so I think it make sense to cover them. RayCluster tests we could consider, but they would be mostly redundant with Job, as the underlying integration mechanism is the same. Note that we have e2e tests for JobSet, and I think they also make sense.
