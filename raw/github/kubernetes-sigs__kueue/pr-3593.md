  This is great, but might be problematic on upgrade, because we have in our code places which check, for example:
  - [here](https://github.com/kubernetes-sigs/kueue/blob/b0665c7d41b998d490e1f5f8a15dba278836868e/pkg/workload/workload.go#L771)

  possibly more - we will see them all if you rename the const too, to `WorkloadDeactivated`.
  
  IIUC changing the name will impact the workloads, and we need to make sure the upgrade is smooth, so I would suggest:
  - introduce the new reason, and set it 
  - instead of removing the old reason, deprecate it, but support it if the old reason is set 
  
  We will remove the old reason in one or two releases. WDYT @PBundyra @tenzen-y ?
