### Comment by [@mimowo](https://github.com/mimowo) — 2024-12-04T15:37:25Z

I thought a little bit more about the knob and metrics cardinality:
1. we have the enable/disable knob for CQs, so I think this makes sense for LQs too: https://github.com/kubernetes-sigs/kueue/blob/e5e340da00732ccf61df4b3bbf944783c35e66a8/apis/config/v1beta1/configuration_types.go#L148 (by analogy)
2. the cardinality of namespace is ok-ish based on the k8s metrics, `https://kubernetes.io/docs/reference/instrumentation/metrics/`
3. too much cardinality at some points become a problem surely (too much memory, big outputs from the endpoint), so we need to be cautious 
4. I was not aware of the knob for CQ when was leaving the [comment](https://github.com/kubernetes-sigs/kueue/pull/3673#discussion_r1861643242).

So, my proposal for 0.10 are:
1. keep as currently, just feature gate
2. add the knob just to enable / disable `EnableLocalQueueMetrics`

I think (2.) makes sense regardless if we do it for 0.10, so that we have a tool to control cardinality if it is a problem, and by analogy to the CQ knob - it does not make sense to the the knob for CQ and not for LQ. 

We can try to go ahead with this PR as is (feature gate), and you can follow up with the knob (2.) - it either gets to 0.10 or not. WDYT?

For Beta, we can consider additional API with selectors, but I would refrain from doing so, until we have feedback indicating we really need them.
