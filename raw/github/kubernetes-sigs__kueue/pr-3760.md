### Comment by [@mbobrovskyi](https://github.com/mbobrovskyi) — 2024-12-11T13:22:20Z

OK, I think I understand what’s happening. The rotator updates the certificates in the secret https://github.com/open-policy-agent/cert-controller/blob/master/pkg/rotator/rotator.go#L484, and this secret is mounted as a volume in the pod https://github.com/kubernetes-sigs/kueue/blob/main/config/default/manager_webhook_patch.yaml#L16. The rotator waits for the volume to be updated https://github.com/open-policy-agent/cert-controller/blob/master/pkg/rotator/rotator.go#L856-L857. After the secret is updated, Kubernetes needs some time to update the volume in the container. That’s why we need to wait for 2 minutes. Once the volume is updated, the container runs successfully https://github.com/open-policy-agent/cert-controller/blob/master/pkg/rotator/rotator.go#L893.

### Comment by [@mimowo](https://github.com/mimowo) — 2024-12-16T07:50:41Z

I carried a couple of startup experiments using the script (with tweaking the limits):
```sh
KUEUE_DIR="$HOME/go/src/sigs.k8s.io/kueue"
MANIFESTS="$KUEUE_DIR/artifacts/manifests.yaml"

export CPU_LIMITS="1000m"

yq eval '
  (. | select(.kind == "Deployment") | .spec.template.spec.containers[0].resources.limits.cpu) = strenv(CPU_LIMITS)
' -i $MANIFESTS

TIMING_FILE="timing_branch_$CPU_LIMITS.log"

IMAGE=$(cat $MANIFESTS | yq ".spec.template.spec.containers[0].image | select(.)")

for i in {1..20}
do
  kind create cluster --config "$HOME/kind/kind-single-config.yaml" --name kind --image kindest/node:v1.31.2
  kind load docker-image "${IMAGE}" &>/dev/null
  kubectl apply --server-side -f $MANIFESTS
  start=$(date +%s%3N)
  kubectl wait deploy/kueue-controller-manager -nkueue-system --for=condition=available --timeout=5m
  end=$(date +%s%3N)
  echo "$((end - start))" >> "$TIMING_FILE"
  kind delete cluster &>/dev/null
done
```
Results (after 20 repeats on Kind):
- limits 500m (current value): mean 122089s, std dev: 20455
- limits 1000m: mean 66004s, std dev: 50550
- limits 2000m (as in JobSet): 21123, std dev: 337

The results on GKE seem consistent with that picture (but I didn't do too many repeats).

So basically, I think code-wise this PR is ok, as it implements the same approach as JobSet in https://github.com/kubernetes-sigs/jobset/pull/722. 

And, the increase in startup is sort of expected given kube-rbac-proxy is CPU intensive on startup, and didn't have any limits set.

So the options in Kueue:
1. keep the current 500m - and accept there is a slowdown of startup (prod and tests)
2. bump to 1000m which +/- matches the "before" state, probably on the lower end, maybe 1500 is better
3. bump to 2000m which matches the JobSet config

I'm leaning to 3, setting high limits for CPU should not be a problem as this is a compressible resource. It will give us a littl bit of boost for running tests when the CI machines are not loaded. For production environments the resources (limits and requests) will be adjusted anyway.
