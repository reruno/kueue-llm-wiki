### Comment by [@dgrove-oss](https://github.com/dgrove-oss) — 2024-12-12T02:24:50Z

This code is cleaner and easier to understand.  It also avoids a Get of the `ReplicaSet` parent for Deployments to do a "real" check on the controller ref. 

A downside is that an annotation like this is much easier for a user to fake than a controller ref is, thus allowing a user to bypass quota enforcement.

### Comment by [@mimowo](https://github.com/mimowo) — 2024-12-12T05:58:36Z

> A downside is that an annotation like this is much easier for a user to fake than a controller ref is, thus allowing a user to bypass quota enforcement.

Options I see:
1. use annotation only (as in the PR), but make the annotation immutable
2. lookup via ReplicaSet only, but then of the owner gvk check if there is a framework enabled with that gvk
3. lookup the framework via annotation and then confirm it matches the required gvk (slightly faster than 2 because avoids lookup via ReplicaSet).

Actually, I think the main consideration is to avoid hard-coding known frameworks, so I'm perfectly good with 2.

### Comment by [@mimowo](https://github.com/mimowo) — 2024-12-12T06:14:44Z

Actually, I think the way we lookup the parent (Deployment or StatefulSet) is worth a while as 
-  we would need to do the similar thing for LWS 
- it would be a basis for the ServingWorkload interface

Maybe the annotation (1.) is more flexible as it would allows to directly jump over the "internal" StatefulSets. WDYT @tenzen-y @dgrove-oss ?

### Comment by [@dgrove-oss](https://github.com/dgrove-oss) — 2024-12-12T13:42:59Z

Making the annotation immutable doesn't help prevent a user from bypassing us..... the attack is to put the annotation on a pod that isn't actually owned by the framework.  For example, sticking `kueue.x-k8s.io/parent-framework: "deployment"` on a plain pod. 

If we think we want the future flexibility, option 3 can also work, but I agree we can wait until we know we need it.
