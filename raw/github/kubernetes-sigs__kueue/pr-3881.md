### Comment by [@mykysha](https://github.com/mykysha) — 2025-01-21T15:18:22Z

I believe that the tests are ran independently, one Where-It per metric group, each having its own namespace, as well as the metrics reading pod. "Where"s were introduced as a way to reduce the amount of verbosity with all of the "defer cleanup"s and "By"s. And yes, every "It" can be ran independently, would be way more complicated to design and manage an interconnected test suite

### Comment by [@mimowo](https://github.com/mimowo) — 2025-01-22T13:12:03Z

> It per metric group, each having its own namespace, as well as the metrics reading pod. 

Interesting, so the metrics seen by the metrics reading pod are only from the namespace of the pod? 

I thought metrics in k8s are collected globally irrespective of the namespace, and namespace is just a label for the metrics, like [here](https://github.com/kubernetes-sigs/kueue/blob/main/pkg/metrics/metrics.go#L325). So, I assumed the metrics from the first "It" would show up when reading the metrics from the second "It".

If this is the case, maybe when we assert on the metrics I would suggest to have an extra filter per extra label (like namespace, CQ name or LQ name). This way we could make sure metrics from one "It" don't show up when running another "It".

### Comment by [@mykysha](https://github.com/mykysha) — 2025-01-22T15:14:26Z

I see. From this standpoint it does make sense. Even though these tests can be ran independently, in the e2e environment some metrics stay and are not deleted along with their correlated resources. 

Btw I think that such metric deletion logic should be covered as well to ensure that everything is being cleaned successfully when correlated resources are deleted. 

As for those metrics that stay forever and may indeed appear from the previous "It" cases, I'll check for the way to uniquely identify those between the runs.

Thank you for the PR, testing e2e if metrics are available is useful and already proved to find a bug.
However, we need to think more about the best strategy for the testing. 

Do I understand correctly, that the metrics aren't cleaned between the tests, and so there are dependencies between the test cases? Meaning if I run one test case "It" then it initializes the metrics for other tests? This is not necessarily bad, but I'm wondering if the maintenance might get tricky, when someone adds a metric to assert on and the test passes on CI when all tests run, but not in isolation.

I need to think a little more if there is a better idea for testing the metrics. I'm also thinking about more "minimal" testing, with a very smaller subset of metrics, because most often when one is set then it entails setting another.
