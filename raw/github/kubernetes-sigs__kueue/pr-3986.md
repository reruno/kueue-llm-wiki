### Comment by [@mimowo](https://github.com/mimowo) ŌĆö 2025-01-21T11:18:03Z

I synced with @mszadkow and after reading a bit and it seems the very installation of the CRDs make the framework fail to initialize during "BeforeSuite". This might be explained by API server getting overwhelmed during the initialization phase, and running INTEGRATION_NPROC=4 processes contributes to the problems. 

Running 3 test processes makes the initialization pass typically, but having 3 processes only for other integration tests may be too constraining, so I propose to explore two solutions:
1. check if we could start the framework without the RayCRD (or any CRDs even) and install them in the BeforeSuite explicitly, as we do it here with [InstallCRDs](https://github.com/kubernetes-sigs/kueue/blob/3397e1330ad78dac40cb860ea11c771e5efec506/test/integration/controller/jobframework/setup/setup_controllers_test.go#L106). Then we could control the load with sleep, or retry. This is still not ideal, but could work.
2. delegate the multiKueue integration tests to a dedicated CI job as we do for e2e tests. The very nature of MultiKueue integration tests makes them heavy (as 3 Kueue processes are running there), so it feels justified. If we still hit the issue we could apply (1.) or reduce the parallelism just for this CI job, even down to 1. This would also benefit us by reducing the time for the main integration tests CI job.

Any opinion @tenzen-y ?
