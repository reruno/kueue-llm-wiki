#### What this PR does / why we need it:
Added watchCRDs: Introduced a function to monitor CRDs using the apiextensions/v1 API, triggering waitForAPI only when a CRD becomes established.
Integrated watchCRDs into SetupControllers: Spawned a goroutine to watch CRDs and feed discovered GroupVersionKind (GVK) into the discoveredCRDs channel.
Updated setupControllersFromDiscoveredCRDs  the logic to dynamically match and register GVKs with integrations based on incoming discoveredCRDs.

  It registers controllers and webhooks for these static types, ensuring they’re ready to manage resources (e.g., Jobs) immediately upon system startup.
  
  See unlike dynamic types (CRDs), which can be discovered later, static types require instant availability. Without this setup, the system can’t process them.
  
  See if we remove setupStaticControllers, controllers for static types aren’t registered. For example:
  An E2E test creating a Job expects a Workload resource to be generated.
  Without the controller, the Job isn’t reconciled, no Workload is created, and the test fails (Resource Not Found Err)
  
  
  The separation of static and dynamic type handling is intentional, The static  is handled by setupStaticControllers for immediate .Dynamic Types: Managed via CRD discovery at runtime.
