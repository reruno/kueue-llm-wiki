### Comment by [@tenzen-y](https://github.com/tenzen-y) ŌĆö 2025-02-06T20:00:22Z

I understand your motivation. The consideration is whether or not it aligns with k/k policies.
I'm leaning toward to park during 2 minor versions since if we park during 2 minor versions, we can always obviously say that we follow the k/k policies in the feature stage, API, and so on. Additionally, the Beta feature is enabled by default, which indicates that user can easily introduce it to their cluster.

The only thing is the production code. But I think that the risk is minimal since our release cycle is slightly rapid (2 - 3 months).

### Comment by [@mimowo](https://github.com/mimowo) ŌĆö 2025-02-07T06:58:51Z

> The beta features typically are parked during at least 2 minor versions. So, I'm wondering if we align with k/k feature graduation policy.
> @mimowo Any thoughts?

Is this actually a policy? Searching the "website" repo by [stage: stable](https://github.com/search?q=repo%3Akubernetes%2Fwebsite%20%20stage%3A%20stable&type=code) I can find many feature gates which graduated to stable after one release. For example: [`StrictCostEnforcementForWebhooks`](https://github.com/kubernetes/website/blob/70a8ccdf821c90b7258519b92264740aeb48d02f/content/en/docs/reference/command-line-tools-reference/feature-gates/StrictCostEnforcementForWebhooks.md?plain=1), or [`HPAContainerMetrics`](https://github.com/kubernetes/website/blob/70a8ccdf821c90b7258519b92264740aeb48d02f/content/en/docs/reference/command-line-tools-reference/feature-gates/HPAContainerMetrics.md?plain=1#L18).

Since this happens for k8s (not even so rarely) I think it is ok in Kueue. I think generally Kueue is expected by users to move faster than the core k8s so that we can exercise ideas earlier. 

Still, I prefer to wait 2 releases for graduation of feature-gates protecting complex code, but in this case I find graduation after one release justified.
