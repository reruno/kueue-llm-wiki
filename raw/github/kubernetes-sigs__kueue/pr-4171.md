#### Special notes for your reviewer:

I will follow up enabling preemption across multiple ClusterQueues in a cohort. This is slightly more complicated because we currently allow for multiple workloads to be considered in the same scheduling cycle. For this we need the ability to reserve quota partially. This requires more code, and more tests.

For the ease of reviewing let me annotate the purpose of the main changes:
1. [this code](https://github.com/kubernetes-sigs/kueue/pull/4171/files#diff-441da8393964615cf5b020d0afb9199588e157e50718e83b527788eb679391ecL285-L288) enables the preemption for TAS which was explicitly blocked, CQs with preemption and TAS were deactivated
2. [this code](https://github.com/kubernetes-sigs/kueue/pull/4171/files#diff-3e801025e890fc24825a58862fb7b9132ccc64d37b72453696f1a25960bdc927R83-R94) is used to update the TAS CQ snapshot as the simulation of preemption progresses
3. [this code](https://github.com/kubernetes-sigs/kueue/pull/4171/files#diff-0870ec2c0ea330a7d087a6ef599905bf9b7164a0dd889d1b00ba17c31a1bfd05R406-R433) is to lower the assignment mode, based on quota mode, in case there is not enough of TAS node capacity
4. [this code](https://github.com/kubernetes-sigs/kueue/pull/4171/files#diff-cd44f6e671ae7fdc7882224c13986f4b1b98e1ea45f65bdbcc9ae185d8e91496R586-R587) is the additional condition added to workloadFits to not only check the quota, but also the TAS node capacity.
5. `simulateEmpty` is introduced to check if there is enough capacity if all workloads were removed, this is for the check analogous to [this code](https://github.com/kubernetes-sigs/kueue/blob/248651b0fe8ca4cd7ad7c631ad6eaf4b06b2fb81/pkg/scheduler/flavorassigner/flavorassigner.go#L665), to optimize (3.)

### Comment by [@mimowo](https://github.com/mimowo) — 2025-02-18T07:59:50Z

> @mimowo, To help understand problems in collaborations between Preemption and TAS, would you mind summarizing the problems in the previous TAS mechanism for supporting preemption?

Basically there was not support for preemption for TAS, so there were no problems :) . 

This was explicitly blocked, and is now ublocked in this change: https://github.com/kubernetes-sigs/kueue/pull/4171/files#diff-441da8393964615cf5b020d0afb9199588e157e50718e83b527788eb679391ecL285-L288.

EDIT: Oh, you probably mean why we didn't do it in the first iteration. The main complication is to respect the node capacity constraints. The preemption so far only looked at quota, not at node capacity. Now, we need to keep preempting workloads until:
1. quota is satisfied
2. there is enough node capacity, and (if used) there is enough capacity to satisfy "required" constraint

Both of the checks are now performed within the workloadFits function. This check implements (2.): https://github.com/kubernetes-sigs/kueue/pull/4171/files#diff-cd44f6e671ae7fdc7882224c13986f4b1b98e1ea45f65bdbcc9ae185d8e91496R586-R587. The other changes in flavor assigner are to correctly determine the assignment mode.

EDIT2: This change https://github.com/kubernetes-sigs/kueue/pull/4171/files#diff-5414dcb324187d1d853329b419691cde79632a6d77e1a7c2b503dc1a3688c081R214-R221 (and the places around) is needed to keep track of the usage in TAS snapshot when simulating workload removal / addition. The preemption simulation (as before) keeps updating the snapshot with respect to the set of removed workloads, so we also need to keep track of the TAS usage.

  I believe the amount of code to iterate over all workloads to simulate their usage removal would be more.
  Also, from the performance point-of-view removal of all workloads from snapshot would be worse - O(W*N) - number of workloads times number of domains.
