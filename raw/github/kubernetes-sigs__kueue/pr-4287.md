#### What this PR does / why we need it:
Simplifies logic of WaitForPodsReady countdown. Countdown is now measured since the `Admitted` condition a not since the `PodsReady=False` condition, which I believe should be intended behavior.

### Comment by [@mimowo](https://github.com/mimowo) — 2025-02-18T10:49:40Z

/lgtm
/approve
Let's see the old code would basically take `max(PodsReady.LastTransitionTime when False, Admitted.LastTransitionTime when True)`. Basically we set the PodsReady=False just after workload creation, so it would only make a difference
if PodsReady transitioned from True to False while admitted. However, this is currently prevented by this [check](https://github.com/kubernetes-sigs/kueue/pull/4287/files#diff-60dd240c20adbd6a189d018d1c216c2d296730f446c341d8bf449fa6657964ffR824-R827). So I believe this check is dead code effectively. Except for situations where the conditions were manipulated manually, but we don't support that.

Moreover, the KEP explicitly says (per comment https://github.com/kubernetes-sigs/kueue/pull/4287#issuecomment-2665095866) the time should be measured from admission,  I believe this way it should be implemented day one.

cc @tenzen-y
