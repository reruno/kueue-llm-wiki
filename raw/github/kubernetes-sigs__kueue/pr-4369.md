<summary>Release notes</summary>
<p><em>Sourced from <a href="https://github.com/ray-project/kuberay/releases">github.com/ray-project/kuberay/ray-operator's releases</a>.</em></p>
<blockquote>
<h2>v1.3.0</h2>
<h2>Highlights</h2>
<h3>RayCluster Conditions API</h3>
<p>The RayCluster conditions API is graduating to Beta status in v1.3. The new API provides more details about the RayCluster’s observable state that were not possible to express in the old API. The following conditions are supported for v1.3: <code>AllPodRunningAndReadyFirstTime</code>, <code>RayClusterPodsProvisioning</code>, <code>HeadPodNotFound</code> and <code>HeadPodRunningAndReady</code>. We will be adding more conditions in future releases.</p>
<h3>Ray Kubectl Plugin</h3>
<p>The Ray Kubectl Plugin is graduating to Beta status. The following commands are supported with KubeRay v1.3:</p>
<ul>
<li><code>kubectl ray logs &lt;cluster-name&gt;</code>: download Ray logs to a local directory</li>
<li><code>kubectl ray session &lt;cluster-name&gt;</code>: initiate port-forwarding session to the Ray head</li>
<li><code>kubectl ray create &lt;cluster&gt;</code>: create a Ray cluster</li>
<li><code>kubectl ray job submit</code>: create a RayJob and submit a job using a local working directory</li>
</ul>
<p>See the <a href="https://docs.ray.io/en/latest/cluster/kubernetes/user-guides/kubectl-plugin.html">Ray Kubectl Plugin docs</a> for more details.</p>
<h3>RayJob Stability Improvements</h3>
<p>Several improvements have been made to enhance the stability of long-running RayJobs. In particular, when using <code>submissionMode=K8sJobMode</code>, job submissions will no longer fail due to the submission of duplicate IDs. Now, if a submission ID already exists, the logs of the existing job will be retrieved instead.</p>
<h3>RayService API Improvements</h3>
<p>RayService strives to deliver zero-downtime serving. When changes in the RayService spec cannot be applied in place, it attempts to migrate traffic to a new RayCluster in the background. However, users might not always have sufficient resources for a new RayCluster. Beginning with KubeRay 1.3, users can customize this behavior using the new UpgradeStrategy option within the RayServiceSpec.</p>
<p>Previously, the <code>serviceStatus</code> field in RayService was inconsistent and did not accurately represent the actual state. Starting with KubeRay v1.3.0, we have introduced two conditions, <code>Ready</code> and <code>UpgradeInProgress</code>, to RayService. Following the approach taken with RayCluster, we have decided to deprecate serviceStatus. In the future, serviceStatus will be removed, and conditions will serve as the definitive source of truth. For now, serviceStatus remains available but is limited to two possible values: &quot;Running&quot; or an empty string.</p>
<h3>GCS Fault Tolerance API Improvements</h3>
<p>The new GcsFaultToleranceOptions field in the RayCluster now provides a streamlined way for users to enable GCS Fault Tolerance on a RayCluster. This eliminates the previous need to distribute related settings across Pod annotations, container environment variables, and the RayStartParams. Furthermore, users can now specify their Redis username in the newly introduced field (requires Ray 2.4.1 or later). To see the impact of this change on a YAML configuration, please refer to the <a href="https://github.com/ray-project/kuberay/blob/master/ray-operator/config/samples/ray-cluster.external-redis-uri.yaml#L7-L18">example manifest</a>.</p>
<h2>Breaking Changes</h2>
<h3>RayService API</h3>
<p>Starting from KubeRay v1.3.0, we have removed all possible values of RayService.Status.ServiceStatus except Running, so the only valid values for ServiceStatus are Running and empty. If ServiceStatus is Running, it means that RayService is ready to serve requests. In other words, ServiceStatus is equivalent to the Ready condition. It is strongly recommended to use the Ready condition instead of ServiceStatus going forward.</p>
<h2>Features</h2>
<ul>
<li>RayCluster Conditions API is graduating to Beta status. The feature gate RayClusterStatusConditions is now enabled by default.</li>
<li>New events were added for RayCluster, RayJob and RayService for improved observability</li>
<li>Various improvements to Ray autoscaler v2</li>
<li>Introduce a new API in RayService <code>spec.upgradeStrategy</code>. The upgrade strategy type can be set to <code>NewCluster</code> or <code>None</code> to modify the behavior of zero-downtime upgrades for RayService.</li>
<li>Add RayCluster controller expecatations to mitigate stale informer caches</li>
<li>RayJob now supports submission mode InteractiveMode. Use this submission mode when you want to submit jobs from a local working directory on your laptop.</li>
<li>RayJob now supports <code>spec.deletionPolicy</code> API, this feature requires the <code>RayJobDeletionPolicy</code> feature gate to be enabled. Initial deltion policies are <code>DeleteCluster</code>, <code>DeleteWorkers</code>, <code>DeleteSelf</code> and <code>DeleteNone</code>.</li>
<li>KubeRay now detects TPUs and Neuron Core resources and specifies them as custom resources to ray start parameters</li>
<li>Introduce <code>RayClusterSuspending</code> and <code>RayClusterSuspended</code> conditions</li>
<li>Container CPU requests are now used in Ray <code>–num-cpus</code> if CPU limits is not specified</li>
<li>Various example manifests for using TPU v6 with KubeRay</li>
</ul>
