  Good call @mbobrovskyi !
  
  I think we should be constraining to lower-case since this is how it is constrained here: https://github.com/kubernetes-sigs/kueue/pull/4417/files#diff-9563bbb52ed59c0410b231092d16afda11e47ad3c39c2784719e06fb2dc9ee50L249.
  
  The PodSet name is used in some places for API object names, I remember ProvisioningRequest templates: https://github.com/kubernetes-sigs/kueue/blob/398c74d105743af8a3ad0d660323e5fdbfaf6e30/pkg/controller/admissionchecks/provisioning/provisioning.go#L65-L68, so when you allow upper case it may break for some users as all object names should be lowercased: https://kubernetes.io/docs/concepts/overview/working-with-objects/names/#dns-subdomain-names. We may consider supporting upper case one day if we have such a user request, but it will be some extra work needed, so for now I would continue assuming lowercase in the validation.
  
  To move this PR forward you could maybe leave the validation pattern `+kubebuilder:validation:Pattern` next to the fields, not the types. 
  
  But ultimately I would like to see them all align on lowercase, because PodSet.Name is lowercase anyway.
