### Comment by [@mimowo](https://github.com/mimowo) — 2025-03-10T08:38:57Z

>But, let me add a question. How to notice to users that the workload is scheduled by TAS? I think it is matter since implicitly behavior should inform to users.

Good question. For workloads which are already scheduled (admitted) by TAS this can be inferred easily by the TopologyAssignment as shown in the integration test: https://github.com/kubernetes-sigs/kueue/pull/4519/files#diff-95ef2b4868044a4fbcc96fcda6df858f3b79548d2e4548fafcd9280114928dfcR826-R835

However, it is a valid questions: how users would know that a CQ is using TAS, since the CQ and RF configuration are often admin-only. However, this is not a new problem for TAS, as even currently the only way to know is to inspect the RFs, to know that TAS annotations should be added. Actually, this PR makes it a bit easier to the end users, as they no longer need to add any annotations.

I think ideally, we extend the LocalQueue status with the flavor information, which will allow end-users, who only see LQs, that TAS is used. In particular, we could easily extend [`LocalQueueFlavorStatus`](https://github.com/kubernetes-sigs/kueue/blob/833adb1955e225be2af843c6f3fffd7999d9e831/apis/kueue/v1beta1/localqueue_types.go#L51) to add a field `TopologyName`. I think this would be a nice enhancement regardless. I opened: https://github.com/kubernetes-sigs/kueue/issues/4534
