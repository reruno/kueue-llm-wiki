### Comment by [@mwielgus](https://github.com/mwielgus) — 2025-03-12T15:53:46Z

Why is this a problem? For implicit topology (equivalent to user saying "i don't care") no rank-ordering should be expected.

### Comment by [@mimowo](https://github.com/mimowo) — 2025-03-12T16:01:19Z

> For implicit topology (equivalent to user saying "i don't care") no rank-ordering should be expected.

I interpret the implicit differently. Implicit means currently in code "preferred: lowest level", check [KEP](https://github.com/kubernetes-sigs/kueue/tree/main/keps/2724-topology-aware-scheduling#implicit-defaulting-of-tas-annotations).

By supporting rank-based ordering we could reduce friction for users who currently use "preferred: hostname", but need to put it explicitly, and who care about rank-based ordering.

### Comment by [@mwielgus](https://github.com/mwielgus) — 2025-03-12T16:24:30Z

Yes, implicit topology is set to `prefer=hostname` however this is not the intention of the user, just implementation detail on the Kueue side. I imagine that this default will soon change to unconstrained (with graduation TAS to beta or so). And in unconstrained, ranking doesn't make much sense.

### Comment by [@mimowo](https://github.com/mimowo) — 2025-03-12T16:36:54Z

Actually, rank based ordering isn't something a user requests currently in any way. It is a property of the integration. Basically we try to order the pods by ranks regardless of algo, so in a way it is technical detail and users would probably expect it works regardless. 

At least this would be my interpretation of the least surprise principle in this context. 

Regarding the implicit default to unconstrained. We actually make the feature gate as deprecated and to be replaced by config. It seems to me that most users who want to migrate to TAS would prefer bin packing over spreading for the default behavior.

### Comment by [@mimowo](https://github.com/mimowo) — 2025-03-13T12:53:04Z

I synced with @mwielgus and we decided to change the "implied" default to unconstrained from "preferred", so that we don't make too high expectations when the explicit TAS annotations aren't specified. We may revisit the defaulting in the future releases. So, in case of implicit "unconstrained" annotation there will be no rank-based ordering.

So, we drop the `TASImplicitDefaultUnconstrained` feature gate.
