#### Special notes for your reviewer:

The key changes required to make this work:
- adding the "topology" scheduling gate unconditionally when creating the Pod, so that we have a choice of using TAS later during scheduling. Note that adding scheduling gates during lifetime of a pod is prohibited by k8s validation.
Thus the [change](https://github.com/kubernetes-sigs/kueue/pull/4591/files#diff-cc4cf0e755f602e9e5b4ba20ec474969cc21383d147d0e54fb48dbc0ad8f5666R253) in tests as now we observe 2 scheduling gates. However, checking the list is not empty probably better reflects the intention than checking the count and names.
- the TASPod Label is added to pods on scheduling, not on pod creation as before
- the e2e test does not assert rank-based ordering as it does not work. We decided not to support rank-based ordering for 0.11 for the "unconstrained" annotation, see https://github.com/kubernetes-sigs/kueue/pull/4574#issuecomment-2721156510
- I add e2e test, because this functionality for Pods highly depends on integration of various components - Pod controller, Pod webhook, and Topology Ungater. I think in this case e2e is the most reliable test.

No release note as defaulting hasn't been yet released to users.

  Actually, I reverted the last commit locally to investigate more why it was passing, and I think I understand, the reason is that we were calling (confirmed with extra logging)
  
  ```golang
  utilpod.Ungate(pod, podconstants.SchedulingGateName) // 1
  if _, found := pod.Labels[kueuealpha.PodSetLabel]; !found {
    utilpod.Ungate(pod, kueuealpha.TopologySchedulingGate)  // 2
  }
  podset.Merge(&pod.ObjectMeta, &pod.Spec, podSetsInfo[podSetIndex]) // 3
  ```
  Now, pod is before merge, then it does not contain  PodSetLabel. As a result we remove both gates (1. and 2. executed).
  
  Then, during "Merge"  (step 3) however, the "topology" scheduling back was put back, depending on admission (podSetInfo). 
  
  However, this flow is very confusing, because in the inerim state which is not observable to user both gates are removed.  I think it might be buggy in case of other scheduling gates, as this process could re-order them, which is also not allowed by k8s validation.
  
  So I prefer the new structuring of the code, where we make decision based on the user-observable state.
