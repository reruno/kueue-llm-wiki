  I find a strange thing here. According to my understanding，the `createdJob3` here should be `unsuspend` because of the `toleration` for `spot resource`. But in fact, the `createdJob3` can't be scheduled successfully. I've tested it many times locally and it works like this. After that, I find it is because we just pick the head of each queue.
  https://github.com/kubernetes-sigs/kueue/blob/da1c239b0e4f47fdd797fa69ae626dc7dd06b129/pkg/queue/manager.go#L227-L239
  
  Then job2 will always block job3. I don't mean to say that this is a bug. But we need to do something to prevent this. There are two solutions I might think of 1. Add a `schedulerAttemps` to record the failure counts. 2. Add a backoff mechanism mentioned here #8  Just personal opinion here.

  Indeed this is supposed to test that the job starts as the "By" log indicates. I fixed that and now job3 doesn't schedule unless I comment out job2.
  
  We do need a fix for requeueing that prevents this starvation problem, which could happen not only because of taints, but also if job2 resource requests don't fit the capacity but job3's resource requests do. 
  
  Before discussing solutions (kube-scheduler provides us with example solutions including backoff), we should discuss requirements and what semantics we would like to have. For example, to maintain FIFO scheduling, we need to make sure that the mechanism we come up with doesn't starve older jobs. Lets start discussing that at https://github.com/kubernetes-sigs/kueue/issues/8
