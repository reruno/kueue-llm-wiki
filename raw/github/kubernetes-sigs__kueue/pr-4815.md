#### What this PR does / why we need it:

Updates Kueue to support auto-scaling RayCluster objects.

#### Which issue(s) this PR fixes:

Related to #77 but follows the methodology suggested in https://github.com/kubernetes-sigs/kueue/pull/1852#issuecomment-2675074264

#### Special notes for your reviewer:

This PR is incomplete - I've only made changes related to RayCluster and I would like some help on how to proceed. I believe the missing bit is the `kueue.x-k8s.io/standalone` label mentioned in https://github.com/kubernetes-sigs/kueue/pull/1852#issuecomment-2675074264 - I couldn't find how to use it.

When spec.enableInTreeAutoscaling is false

- The commit, maintains the current behaviour for RayCluster

When spec.enableInTreeAutoscaling is true
- The RayCluster object is associated with a Workload object that accounts only for the resources of the Head node pod
- Pods that the Ray Auto Scaler creates are annotated with the same queue as the head node
- This should have triggered Kueue to create 1 workload per worker pod - assumes "plain pod" feature flag is on.

What is missing:
- Apply the same logic to RayJob objects

```
### c. Limitations

If you configure Kueue to manage [plain Pods](/docs/tasks/run/plain_pods/) then you have the option to switch on In-Tree Autoscaling by setting `spec.rayClusterSpec.enableInTreeAutoscaling` to `true`. If In-Tree Autoscaling is disabled then you should also limit the number of worker groups in your RayCluster object. Because a Kueue workload can have a maximum of 8 PodSets, the maximum number of `spec.workerGroupSpecs` is 7

Else, if the In-Tree Autoscaling is enabled then there are no limitations to the number of worker groups in your RayCluster object because Kueue will create 1 Workload object per worker pod that KubeRay creates on behalf of your RayCluster object.
```

  ```yaml
  spec.rayClusterSpec.enableInTreeAutoscaling: false
  ```
