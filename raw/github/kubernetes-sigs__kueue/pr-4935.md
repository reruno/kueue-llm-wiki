### Comment by [@mimowo](https://github.com/mimowo) — 2025-04-14T15:11:11Z

Ok, I looked a bit more (but haven't got time to confirm anything with an actual experiment). I believe that indeed we have a bug in Kueue, and there is a way to repro with integration tests.

First of all, IIUC, there are never two owners of a field in SSA, we can have only one owner - either Kueue or the external controller. So, generally the state of admission checks is under ownership of external controllers, but sometimes Kueue takes them over. When the external admission checks are taken over by Kueue I think Kueue should keep sending them, otherwise they would be deleted by API server (which I think is what happens here). I think API server deletes all fields owned by Kueue which haven't been sent. Otherwise, the intent of the actor as Kueue would not be clear - is not sending the owned field "I don't want to change it" or "I want to delete it".

So, I believe the fix in the current form is correct, but it would be great to have a repro. To repro I see that Kueue takes over the admission checks in the preemption path, ie. [here](https://github.com/kubernetes-sigs/kueue/blob/ebe37a32cd69f1e2d4215562716f35751ac03a4c/pkg/scheduler/preemption/preemption.go#L262). Not sure this is exactly your case. Then, we need to somehow make Kueue step into [here](https://github.com/kubernetes-sigs/kueue/blob/ebe37a32cd69f1e2d4215562716f35751ac03a4c/pkg/scheduler/scheduler.go#L654). This can be done if the workload is "skipped". It could be considered skipped if there was another workload in the cohort admitted before it, which took all the quota, see [here](https://github.com/kubernetes-sigs/kueue/blob/ebe37a32cd69f1e2d4215562716f35751ac03a4c/pkg/scheduler/scheduler.go#L254). ### Comment by [@mimowo](https://github.com/mimowo) — 2025-04-22T09:11:16Z

I looked this morning into the integration test. Let's consider the following scenario:
0. setup: we have 2 CQs A and B in cohort "cohort", and A has an admission check (both have X nominal and no borrowing limit, so 2X capacity total)
1. we have workload wl1 (size 2X, priority 1) and QuotaReserve and set admission check to Ready, wait for Admitted condition by workload_controller
2. create wl2 (size 2X, priority 2) in CQ B which stays pending
3. create wl3 (size X, priority 3). wl3 preempts wl1, and both wl1 and wl2 are pending. wl1 has [admission checks reset](https://github.com/kubernetes-sigs/kueue/blob/3279d9c05817e465229fac6bdc64250c890ea7dd/pkg/scheduler/preemption/preemption.go#L262)
4. finish wl3, releasing quota so that wl2 and wl1 are considered in the same scheduling cycle, and both are marked "fit". Once wl2 (as higher priority) is admitted, wl1 stays pending and is "skipped". This should make the wl1 stepping into this [problematic code](https://github.com/kubernetes-sigs/kueue/blob/3279d9c05817e465229fac6bdc64250c890ea7dd/pkg/scheduler/scheduler.go#L659). So, in test we could await for the specific reason and message. 

If the problem is with inadmissibleWorkloads queue, I think we could use StrictFIFO queues which retry scheduling on every cycle.

Let me know if I'm missing something (haven't actually ran any experiments). I think if using BestEffortFIFO the issue might be indeed tricky to repro.

Another approach to the PR I would be ok accepting is to just wrap the [3 lines](https://github.com/kubernetes-sigs/kueue/blob/3279d9c05817e465229fac6bdc64250c890ea7dd/pkg/workload/workload.go#L712-L714):
