### Comment by [@mimowo](https://github.com/mimowo) — 2023-01-05T14:17:03Z

Today, while testing e2e I discovered that there is an issue with the current implementation - once a workload's admission is canceled due to timeout, the condition `Admitted` remains `True`. What's more it is not updated on a new scheduling, its LastTransitionTime is not bumped, so the newly re-admitted workload is exceeding the timeout immediately, thus gets re-admitted in a loop. This can be easily reproduced in an integration test like this (it gets re-admitted >1000 times in 30s):

```golang
		ginkgo.It("Should re-admit a timed out workload only once", func() {
			ginkgo.By("create the 'prod' workload")
			prodWl := testing.MakeWorkload("prod", ns.Name).Queue(prodQueue.Name).Request(corev1.ResourceCPU, "2").Obj()
			gomega.Expect(k8sClient.Create(ctx, prodWl)).Should(gomega.Succeed())
			ginkgo.By("checking the 'prod' workload is admitted")
			util.ExpectWorkloadsToBeAdmitted(ctx, k8sClient, prodClusterQ.Name, prodWl)
			util.ExpectAdmittedWorkloadsTotalMetric(prodClusterQ, 1)
			ginkgo.By("exceed the timeout for the 'prod' workload")
			time.Sleep(podsReadyTimeout + time.Second)
			ginkgo.By("verify the 'prod' workload gets re-admitted once")
			util.ExpectWorkloadsToBeAdmitted(ctx, k8sClient, prodClusterQ.Name, prodWl)
			util.ExpectAdmittedWorkloadsTotalMetric(prodClusterQ, 2)
		})
```

 I'm wondering how to fix this.  As a simple proposal I’ve pushed a proposed fix that sets `Admitted=Unknown` condition before setting Spec.Admission=nil. It seems to provide correct behavior in the scenarios I could think of. 

However, I’m not convinced I don't forget about something and it also seems to increase complexity as we need to think about order of setting the condition and the field. 

For alternatives I was also thinking about a new interim condition type, `ExceededTimeout`, or extending the `Admission` structure to have the `Reason` and `Field` when canceling. Then, we could add the `Admitted=False` condition based on the values in the structure. After the condition is set, the Admission could be nil’ed back. However, the alternative approaches seem more involving than what is done, so not sure I should pursue them. WDYT @alculquicondor @ahg-g?

### Comment by [@mimowo](https://github.com/mimowo) — 2023-01-05T18:59:55Z

> > And would be consider a Workload with PodsReady=Unknown to be past the timeout?
> 
> Why do we want that?

My intention was to count towards the timeout only when Admitted=True (in order to treat it as unset). Pushed a small commit to fix that. Each admitted workload has the condition `Admitted` set to `True` soon after admission by workload_controller.

I think that the interim `PodsReadyTimeoutExceeded` condition might be tricky to do right. We need to be careful of race conditions, for example to clear it so that a re-admitted workload does not get admission cancelled.

Long term I would lean towards moving `Admission` to `Status`. It seems risky to give it as a tool to admins, to setup flavor usage by hand. It would remain possible by modifying status as a backdoor. Instead, we could consider having a dedicated field in `spec` for admins to give hints to scheduler, for example `rank` to be used for ordering entries for scheduling. This way an admin could say "admit this next", but without the need to specify flavor usage by hand.
