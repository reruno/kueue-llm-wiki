+  workload to extend the "TAS" assignment, but respecting the previously
+  selected resource flavor.

  > Can we tell CA by ProvReq what topology do we need to schedule the Pods?
  
  Currently OSS CA does not have such an API, and is not aware of topology. A closed source ProvReq implementation may support it, but there is no standardized API for now.
  
  > I can imagine the situation where Pods will be stacked when CA schedules a node with enough capacity, but the node does not satisfy the required topology by TAS.
  
  Do you mean when the user specifies "required: rack", but the provisioned machines are scattered across multiple racks? Indeed this may happen. Currently,  CA will add the BookingExpired condition, and the workload will be re-queued by Kueue. 
  
  I think this is not an issue if the user specifies the "preferred" or "unconstrained". One idea is to block "required" entirely, but since the set of possible classes is open (GKE or other cloud providers may support compact provisioning of nodes) we may allow it, but just warn about using required with ProvReq, wdyt?
