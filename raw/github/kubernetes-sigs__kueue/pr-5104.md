#### What this PR does / why we need it:

Currently, `tasCache` for `ResourceFlavor` is set after existing `clusterqueue`s are added.
When the controller is restarted, existing `clusterqueue`s are added (existing `workload`s are also added together) but since `tasCache` is empty, the TAS controller thinks all nodes are free to assign any `workload`s.

This PR makes sure that the `tasCache` is populated before the controller adds existing `clusterqueue`s and `workload`s to keep track of previously admitted running workloads even though the controller is restarted.

### Comment by [@mimowo](https://github.com/mimowo) — 2025-04-25T07:06:23Z

> Yes, events are delivered, but if tasCache is nil, they are not updated.

I would suggest to start with a repro integration test as mentioned in https://github.com/kubernetes-sigs/kueue/pull/5104#issuecomment-2828520053, then I see options:
1. trigger re-accounting for all workloads after the Topology is observed (and so tasCache is present)
2. block startup (as currently)

I would probably suggest to start with (1.) as more similar to regular quota accounting. The issue with (1.) with a window where workloads are not accounted exists, but can be mitigated by HA mode, still, maybe we want to also solve it for a single replica of Kueue, but I'm not sure at this moment, and this is a bigger change to do right.

Technically, for (1.) once there is an event adding Topology (and thus setting [tasCache](https://github.com/kubernetes-sigs/kueue/blob/d7c6b977186d2e1b61f768ec967d4e87afd05196/pkg/controller/tas/topology_controller.go#L156), then we can trigger CQ update by [NotifyTopologyUpdateWatchers](https://github.com/kubernetes-sigs/kueue/blob/d7c6b977186d2e1b61f768ec967d4e87afd05196/pkg/controller/tas/topology_controller.go#L146C17-L146C45) and re-consider all workloads. However, this is just an idea - you may discover some complications when working on it.
