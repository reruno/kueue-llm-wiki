## Description

This change prevents a potential nil pointer panic in the scheduler when a ClusterQueue is deleted between the time a workload is selected for scheduling and when its ClusterQueue is retrieved from the snapshot.

  This is actually motivated by my recent PR on TAS + ProvReq, where we do the second pass without re-checking the CQ (I need to yet think if we should re-check or not). For example, in case of other AdmissionChecks, as soon as all checks are green we promote the workload from QuotaReserved to Admitted without re-checking the CQs.
  
  Still, it might be possible (but harder)  to repro the panic in the scenario:
  1. CQ exists, `headWorkloads := s.queues.Heads(ctx) `
  2. CQ deleted, then it is not returned by `s.cache.Snapshot(ctx)`
  3. the nomination steps [here](https://github.com/kubernetes-sigs/kueue/blob/890d82e2018ec191c3e380c386a28e768465cabf/pkg/scheduler/scheduler.go#L359), and thus the entry is NotFit
  4. The entry steps into [here](https://github.com/kubernetes-sigs/kueue/blob/890d82e2018ec191c3e380c386a28e768465cabf/pkg/scheduler/scheduler.go#L216) and panics, as cq=nil
  
  Indeed, this might not be reproducible on the current codebase by restart, but by deleting the CQ, but would require tricky timing to make the test reliable. 
  
  I would appreciate a manual test by injecting time.Sleep(5s) between `s.queues.Heads(ctx)` and `s.cache.Snapshot(ctx)` to allow deleting the CQ in the meanwhile manually. This would confirm the scenario above is real (I have not done the experiment myself yet, just static code analysis).
