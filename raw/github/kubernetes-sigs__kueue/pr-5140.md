### Comment by [@mimowo](https://github.com/mimowo) — 2025-05-05T06:46:04Z

Exposing this information to LQ users (teams) might be leaking too much. For example, the quota for other teams.

So far, we didn't allow users to view any cluster-scoped resources. To be able to see some parts of them we introduced this KEP: https://github.com/kubernetes-sigs/kueue/tree/main/keps/3122-expose-flavors-in-localqueue-status. If we expose CQs to users soon there will be asks about RF and Topology.

So, I would like to first understand what is missing the the pre-existing KEP aiming very similar information to be exposed.
/hold

### Comment by [@mimowo](https://github.com/mimowo) — 2025-05-06T06:31:11Z

>  However, currently LQ status looks like tool box. 

Indeed, we may consider refactoring it, some ideas:
1. a dedicated UserView subresource inside LQ instead of status, 
2. visibility API with the UserView
3. dedicated CRD with the user view

I'm currently leaning towards (2.) as cleanest.

Still, exposing the global resources (CQ, Topolog, RF) to all users may be leaking information which some admins of Kueue prefer to keep secretive to teams (such as team quotas).
