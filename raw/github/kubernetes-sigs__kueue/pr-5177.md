  > We need to be careful about this possible scenario
  > 
  > 1. Kueue deactivates using DeactivationTarget
  > 2. User re-activates and the workload is re-scheduled
  > 3. User de-activates manually
  > 
  > In this case I think we should preserve the workload (not delete) because last deactivation was manual. @mbobrovskyi please check this will work ok using the new scheme, otherwise I'm ok to use the previous method.
  
  I think another problem by the preserved DeactivationTarget condition is inconsistency activation-deactivation mechanism. 
  
  Previously, we establish the solid approach in each way:
  - `.spec.active`: Presistended activated / deactivated state on Workload
  - `DeactivationTarget`: Temporary activated / deactivated state on Workload
  
  However, in https://github.com/kubernetes-sigs/kueue/pull/5214, we break the strategy.
  This inconsistent activation mechanism will potentially cause bugs and unhandled controller logics between controllers. So, I would keep the current mechanism for this feature, and we will work on the preserved DeactivationTarget condition as separately, and then every activation mechanism should rely on the DeactivationTarget condition mechanism and `.spec.active` field will be used only by non-kueue action.
  
  In other words, I would work on the following mechanism refining as a separately:
  
  - `.spec.active`: This will be used by non kueue controller activation features including manual
  - `DeactivationTarget`: This will be used by all kueue controller specific activation features
