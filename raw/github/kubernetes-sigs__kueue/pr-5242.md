  > My intuition tells me that we are better off with a single `hack/internal/tools` so that we keep the versions consistent, and there are fewer dependabot updates.
  
  I've checked and the only tools I see `kueviz` is using is `CompileDaemon`. I can add it to the tools `go.mod` and treat it in the same way.  I agree it'd be nice to have a shared toolbox. 
  
  > I was imagining removing hack/internal/tools and then managing everything with a single go module.
  
  @tenzen-y yeah, but mixing tools and application dependencies can be problematic: one might update a shared dependency and break the other, hence the use of separate `go.mod` files. I assume the same reasoning led to the creation of the `tools` package. But apart from that, the new directive allows us to do some other useful things, let me explain further.
  
  Now we can use `go get -tool` to add a tool as a `tool` entry in the `go.mod` file. From that point on, we can run `go tool <tool>`, and Go will ensure the tool is installed or updated before executing it. This is great because it allows us to remove all the Makefile code related to tool installation and updates (which is a good chunk of the [Makefile-deps.mk](http://makefile-deps.mk/) file). We could simply run `go tool <tool>`, and Go would handle everything for us.
  
  However, if a separate `go.mod` file is used for tools (as proposed in this PR), we need to specify its location. We have two options:
  
  1. `cd` into the `./hack/internal/tools` directory and run `go tool <tool>`. However, with this approach, the tool won’t find the project files unless we explicitly provide their paths.
  2. Use the `-modfile=<path to go.mod>` argument to specify the path to the `go.mod` file. Unfortunately, Go tries to match the contents of the `vendor/` directory and fails with an `inconsistent vendoring` error (because they don’t match). For other `go` commands, this issue is usually resolved with the `-mod=mod` or `-mod=readonly` flags, but unfortunately, `go tool` doesn't support them 🤷🏻. It seems this problem could be resolved by using an independent cache, but I haven’t tested that yet.
  
  As you can see, both approaches would require additional changes to the code. That’s why, in this PR, I opted to use the tool tracking feature (`go get -tool`), keep the existing tool installation/update mechanism, and discuss with you all whether it's worth pursuing any of the alternatives mentioned above
