### Comment by [@mimowo](https://github.com/mimowo) — 2025-05-22T13:07:56Z

Let me summarize more. 

(1.) PR: https://github.com/kubernetes-sigs/kueue/pull/5276
First, I tested this PR hard with >400 restarts under stress and no flakes. Without the fix the new test fails almost consistently. Yes, the PR is tricky, but I don't see any issue with it.

(2.) PR: https://github.com/kubernetes-sigs/kueue/pull/5309
Second, I tried to investigate if we can defer accounting for workload usage until the TAS cache is available. One result of this investigation is https://github.com/kubernetes-sigs/kueue/pull/5309. In this PR I move the registration of topology and resource flavors in cache independently. When both are provided, then we instantiate the cache. This allows us to move all the initialization logic into event handlers for Topology & ResourceFlavors. However, it does not solve the bootup issue fully (just makes it very unlikely), because still the event handlers for Workloads and ClusterQueues may get executed before event handlers for Topology. So, the new test flakes 1/20 builds or so.

(3.) https://github.com/kubernetes-sigs/kueue/pull/5311
Third, I tried to ensure the event handlers for Topology & ResourceFlavors are processed before Workload & ClusterQueue event handlers in https://github.com/kubernetes-sigs/kueue/pull/5311. However, it does not really guarantee that the event handlers for Topology are processed before ClusterQueue. It may still happen (and I confirmed experimentally), that the BeforeEach triggers event handlers for CQ before event handlers for Topology are synced. It only ensures that scheduling is delayed until event handlers are synced. With the (2.) and (3.) PR the new test flakes 1/100, but still the issue exists. Note that (1.) + (2.) + (3.) fixes the bug, and fixes the issue with https://github.com/kubernetes-sigs/kueue/pull/5276#discussion_r2099603212.

As discussed with Gabe probably the best way to fix this is to move the logic in ClusterQueue and Workload controller which calls cache.AddOrUpdateWorkload into the Reconcile function. However, this requires refactoring existing code which is outside of TAS, and so I wouldn’t do it just before release. One needs to be careful also, because the event handlers use the information about prev & new state of Workload.

So, my recommendation would be:
1. Merge (1.) PR https://github.com/kubernetes-sigs/kueue/pull/5276 and cherry-pick, this is the smallest fix I could come up with
2. Merge (2.) PR https://github.com/kubernetes-sigs/kueue/pull/5309 (preferred before 0.12.0)  and I vote for cherry-picking, because it makes the reasoning about TAS cache initialization and access much simpler
3. Merge (3.) (might be post 0.12.0, no preference), but don’t cherry-pick unless requested by users. In the meanwhile work on https://github.com/kubernetes-sigs/kueue/issues/5310 and potentially have a more generic fix. If that happens, then revert (1.) and (3.) on the main branch. (2.) stays merged regardless.
