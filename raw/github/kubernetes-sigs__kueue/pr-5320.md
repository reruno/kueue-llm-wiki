### Comment by [@mykysha](https://github.com/mykysha) — 2025-05-23T10:53:39Z

> could you maybe explain why this helps, and if you confirmed this locally?

As I've mentioned, I've ran the test 50 times with no failures, which is way more than it would be possible to without the changes.

As for "why?", I believe that having the pod to be in the same namespace as Kueue was resulting in it being sometimes taken away with the Kueue restarts defined in the tests. Moving the creation after the Kueue manager has been restarted seems to resolve the issue, as we create it when we have a running Kueue instance

### Comment by [@mimowo](https://github.com/mimowo) — 2025-05-23T11:13:08Z

> As I've mentioned, I've ran the test 50 times with no failures, which is way more than it would be possible to without the changes.

Awesome 👍 

> As for "why?", I believe that having the pod to be in the same namespace as Kueue was resulting in it being sometimes taken away with the Kueue restarts defined in the tests. Moving the creation after the Kueue manager has been restarted seems to resolve the issue, as we create it when we have a running Kueue instance

I think namespaces generally should not matter for scheduling. 

One another possibility based on the symptomes, is that during the rolling restart of Kueue there are 2x more pods running for Kueue at the same time, and so the kube-scheduler may need to preempt the lower-priority pod. We could probably check that by inspecting the state of the curl-metrics pod at the beginning of each test.

The fact that you reproduced it locally, and run >50 times gives me enough evidence to go with the fix.
