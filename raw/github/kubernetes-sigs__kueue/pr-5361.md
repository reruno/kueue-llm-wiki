### Comment by [@IrvingMg](https://github.com/IrvingMg) — 2025-06-09T15:15:24Z

This is because it checks for a provisioning request to delay TAS, which causes failures in the manager cluster, resulting in the assignment being marked as inadmissible.

## Task Summary (as of August 20)

### Goal
Enable MultiKueue (MK), Topology Aware Scheduling (TAS), and ProvisioningRequest to work correctly together on GKE with Dynamic Workload Shaping (DWS) enabled.

---

### What’s Been Done

1. Removed the restriction that explicitly blocked the use of MultiKueue with TAS in [`clusterqueue.go`](https://github.com/kubernetes-sigs/kueue/blob/8def1eef499bd463eb8a3a6113a44059bf3b1aa0/pkg/cache/clusterqueue.go#L281-L284).

2. After this change, encountered the error:  
   ```
   couldn't assign flavors to pod set main: no topology domains at level: kubernetes.io/hostname
   ```

3. Added integration tests in `test/integration/multikueue/tas/suite_test.go` for:
   - MK + TAS
   - MK + TAS + ProvisioningRequest

   These tests passed, but they only verify that the workload is admitted to the correct worker cluster, which also happens correctly in real GKE testing.

4. Identified an issue in MK setups where topology requests were not delayed as expected when DWS is enabled.

   In DWS-enabled scenarios, the scheduler is expected to delay topology assignment until the ProvisioningRequest is fulfilled (e.g., when a new node becomes available). However, in MK setups, the scheduler tried to assign topology too early, before the needed resources were ready. 

   This was caused by a condition in [`tas_flavorassigner.go`](https://github.com/kubernetes-sigs/kueue/blob/8def1eef499bd463eb8a3a6113a44059bf3b1aa0/pkg/scheduler/flavorassigner/tas_flavorassigner.go#L87), which required the ClusterQueue to have a `ProvisioningRequest` admission check and the workload to not have a `QuotaReservation`. 

   That condition is never satisfied in the manager cluster, since the scheduler runs there and its ClusterQueues do not include ProvisioningRequest admission checks. As a result, the topology assignment was not delayed, causing workloads to be marked inadmissible with the error above.

5. Fixed the condition by changing the logic to check whether the ClusterQueue has the MultiKueue admission check instead.

   The updated logic assumes that if a ClusterQueue has a ProvisioningRequest admission check and does not have the MultiKueue one, it is in a worker cluster, and topology assignment should be delayed there.


---

### Current Behavior (MK + TAS + ProvReq)
- When a job is created, the workload is `QuotaReserved` in both the manager and worker clusters.  
- The worker cluster creates a ProvisioningRequest and waits for resources to be provisioned.  
- Once the resources are available, the workload is admitted and runs (my test job executes a simple sleep command, so I assume it is running).
- After the workload completes, both the workload and the job are cleaned up in the worker cluster.  
- In the manager cluster, the workload is marked as `Finished`.  
- However, the job in the manager cluster remains stuck in the `Running` state.  
- This behavior is the same as observed with MK + TAS alone, suggesting a general issue with job status synchronization in MultiKueue setups.
- In addition to the sync issue, even though the workload appears to run correctly in the worker cluster, events in the workload object on the manager cluster show repeated attempts to admit it there, each failing with the known error:  `couldn't assign flavors to pod set main: no topology domains at level: kubernetes.io/hostname`.
 
  This should not happen in the manager cluster. Topology Aware Scheduling should be handled in the worker cluster. These repeated TAS failures in the manager suggest that TAS may not be fully aware of MultiKueue, and could be interfering with proper job status synchronization from worker to manager.


---

### Environment & Testing
Testing was done on a real GKE cluster using Terraform scripts and YAML files (all included in the current PR). The setup covers MultiKueue, TAS, and ProvisioningRequest, and also includes files for testing TAS + ProvisioningRequest in a single-cluster environment, which was verified to work correctly on GKE.

A README file is also included to help understand and reproduce the cluster setup.

The code changes described above are also part of this current PR.

### Comment by [@IrvingMg](https://github.com/IrvingMg) — 2025-08-22T07:42:57Z

After testing different scenarios with TAS disabled in the manager cluster, I believe the problem described above was a configuration issue, not a code issue. The integration of MultiKueue + TAS + ProvisioningRequest was initially tested under the assumption that TAS needed to be enabled in both the manager and worker clusters.
 
However, enabling TAS in the manager cluster makes it try to schedule based on its own local topology, which isn’t the intended behavior, since workloads are meant to execute in the worker clusters, not in the manager. Because the manager lacks access to topology data from the workers, this leads to scheduling failures.
 
Disabling TAS in the manager correctly delegates Topology Aware Scheduling to the worker clusters, where workloads are actually scheduled and run.
 
I also plan to revert the previous code changes made in this PR, which were intended to support TAS in the manager with MultiKueue. That approach added unnecessary complexity, which isn’t needed with the correct configuration in place.
 
I'll also add e2e tests to validate the correct behavior with this setup.
