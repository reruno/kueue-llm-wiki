### Comment by [@pajakd](https://github.com/pajakd) — 2025-06-09T08:42:53Z

I analyzed the test failures. What happens is that a single metric `kueue_admission_cycle_preemption_skips` is missing after we introduce the change. What the test does is as follows:

1. We have CQ1: 2CPU, CQ2: 3CPU
2. We create two low-priority jobs `lower-job-1` and `lower-job-2` that both require 1CPU **both** in CQ1!
3. Create a high-priority `blocker` job using up all the remaining quota.
4. Create two high-priority jobs `high-large-1` and `high-large-2` that both require 4CPU and are submitted to CQ1 and CQ2 respectively.
5. Deactivate `blocker` job.
6. Expect that one of the `lower-job` jobs will be preempted and one of the `high-large` jobs will be admitted.
7. Expect that both `high-large` jobs will be assigned `preempt` mode by the flavorassigner and then in scheduling one will be skipped (here is where the metric should be initialized).

With the proposed change, 7. does not happen because `high-large-2` cannot preempt any of the `lower-jobs` (they are both below nominal quota). After the, change the flavorassigner figures out that the list of preemption targets is empty and the job is given NoFit mode (without the change it was getting Preempt).

I will need to change the test slightly to fix this. I claim that 2. is not correct and the low priority jobs were supposed to be submitted to different queues (with this both high priority jobs will have a possibility to preempt).

I have synced on this with @gabesaba and there is actually a potential regression in this PR in a situation when `preemption.WithinClusterQueue` is disabled.

So, the idea for the PR is to let the best effort workload come in when they could schedule while the "hero" Job is waiting. However, when `preemption.WithinClusterQueue` is disabled, then the hero Job can easily get starved. So, we will follow up with a fix which will make this conditional. This is conceptually very similar to PR https://github.com/kubernetes-sigs/kueue/pull/4813  where we allow for best effort workloads knowing that we are able to kick them off when needed. We want to have a similar guarantee in this case.
