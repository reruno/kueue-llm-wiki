#### Special notes for your reviewer:
Originally, we updated the Kueue Configuration after creating the `ResourceFlavor`, `ClusterQueue`, and `LocalQueue` resources in `BeforeEach` block. However, this can lead to race conditions – especially when resources are being deleted and created, and the controller manager is restarted during that time. This may interrupt operations.

To avoid this, we should update the Kueue Configuration first, and only then create the resources.

Since `updateKueueConfiguration` is a slow operation, I’ve split each configuration update into a separate Describe block and moved the logic into the BeforeAll block. This approach will make it easier to extend and add new test cases in the future.

### Comment by [@mbobrovskyi](https://github.com/mbobrovskyi) — 2025-06-20T06:17:09Z

> Does this PR categorize the test cases by dedicated ginkgo.Describe instead of ginkdo.When so that we can avoid the race condition due to Kueue Config updating?

Yes. The main issue is that we create the `Namespace`, `LocalQueue`, `ClusterQueue`, and `ResourceFlavor` while also updating the Kueue configuration before restarting the `kueue-controller-manager`. This sequence can lead to race conditions, potentially causing data loss during the restart.

To address this, I propose creating all resources after restarting the `kueue-controller-manager`.

There are two possible solutions:

1. Use `updateKueueConfiguration` in a `Describe's` `BeforeEach` block—before creating `LocalQueue`, `ClusterQueue`, and `ResourceFlavor`.
2. Split the tests into separate `Describe` blocks and use `updateKueueConfiguration` in `BeforeAll`.

I believe the second solution is better. Since `updateKueueConfiguration` is a slow operation, running it in `BeforeAll` allows us to reuse the same configuration across multiple test cases. In the future, this will make it easier to extend tests without needing to restart the `kueue-controller-manager` unnecessarily, as long as the configuration remains the same.
