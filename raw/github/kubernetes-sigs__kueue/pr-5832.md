```diff
 		if err := workload.ApplyAdmissionStatus(ctx, r.client, wl, true, r.clock); err != nil {
-			return false, err
+			return false, client.IgnoreNotFound(err)
```

  @tenzen-y  is right — `IgnoreNotFound` should be used inside `ApplyAdmissionStatus` after its internal logic completes, rather than applying `IgnoreNotFound` uniformly from the outside.
  
  If `IgnoreNotFound` is handled externally, it could lead to issues like this:
  
  When `ApplyAdmissionStatus` internally returns a `NotFound` error, and the outer logic checks `err != nil` 
  ```
  if evictionTriggered, err := r.reconcileCheckBasedEviction(ctx, &wl); evictionTriggered || err != nil {
  	return ctrl.Result{}, client.IgnoreNotFound(err)
  }
  ```
  applying `client.IgnoreNotFound(err)` will cause the `reconcile` to exit early. But the original reconcile logic is actually supposed to continue executing.
  
  But there's a question here: if `ApplyAdmissionStatus` has already returned a `NotFound` error, should we still continue executing the following logic?
