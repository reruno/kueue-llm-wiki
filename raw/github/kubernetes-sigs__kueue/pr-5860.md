### Comment by [@mbobrovskyi](https://github.com/mbobrovskyi) — 2025-07-03T08:08:01Z

> @mbobrovskyi What about the performance? Alpine does not have glibc, which means it sometimes has performance degradation.

I don’t think we’re seeing significant issues in our case. Our React application uses a lightweight setup with the command `exec serve -s build -l 8080` to serve static files from the `build` directory, which is primarily an I/O-bound workload. For such tasks, the performance difference between `musl libc` and `glibc` is typically negligible, as serving static files involves minimal CPU-intensive operations and no complex native dependencies.

But I’m okay with changing it to slim.

However, I still want to recommend debian slim image since Alpine is a high-maintenance cost image. There have been many problems in the past years. The most famous problem is DNS resolution problem: https://kubernetes.io/docs/tasks/administer-cluster/dns-debugging-resolution/#known-issues

I think maintaining Alpine based image is very hard since we occatially need to debug the image / OS specific problem due to musl libs.
