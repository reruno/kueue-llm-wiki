Adds a FailFast mode to TAS on node failure. After Kueue tries to find a replacement for a failed node and fails it evicts the Workload

### Review by [@mimowo](https://github.com/mimowo) (commented) — 2025-07-04T08:14:10Z

LGTM overall as for alpha, but my worry is that we are waiting for 2 sequential requests in the scheduler cycle. Scheduling cycles are super important to be performant.

Two optimizations I see:
1. 1 request per workload when the annotation is inside status (https://github.com/kubernetes-sigs/kueue/issues/5560)
2. we issue the eviction requests to all affected workloads in parallel (similarly to IssuePreemptions)
3. (alternatively) scheduler could only mark the workload in cache as already tried. The actual eviction would happen by another controller, for example node failure controller.

I'm leaning to say that optimizing the scheduler main loop is a blocker for graduation of the TASFailedNodeReplacementFailFast feature gate to beta. 

@PBundyra please add a note to re-evaluate the optimization to the scheduler loop before graduating TASFailedNodeReplacementFailFast to beta.
