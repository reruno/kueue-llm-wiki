### Comment by [@mimowo](https://github.com/mimowo) — 2025-07-09T06:14:58Z

> I feel that this enhancement introduces the PodSet slice group instead of PodSet group since the PodSet slices are leader / worker. And this feature try grouping those PodSet slices which mean in the example, the leader podset slice and worker podset slice a single group of podset slice.

The work in TAS for 2-level admission (JobSet or LWS) does not change the notion of PodSet. Leaders and Workers in LWS are represented by PodSets, which can be seen in the example Workload status for LWS group:

k get wl/leaderworkerset-leaderworkerset-multi-template-0-1a622 -oyaml
```yaml
...
status:
  admission:
    clusterQueue: cluster-queue
    podSetAssignments:
    - count: 1
      flavors:
        cpu: default-flavor
      name: leader
      resourceUsage:
        cpu: 50m
    - count: 6
      flavors:
        cpu: default-flavor
      name: worker
      resourceUsage:
        cpu: 300m
```
- PodSetSlice - is a PodSet "chunk" -  main use case is to represent scheduling of individual Jobs in JobSet
- PodSetGroup - represents a set of PodSets to be co-scheduled - main use case is to co-schedule the leader and worker PodSets

So, 2-level scheduling in case of JobSet means (PodSet and PodSetSlice), but in case of LWS means (PodSetGroup and PodSet). We don't yet have a case of 3-level scheduling involving all layers.

  I thought about dedicated annotation to simplify that setup for user. In general we need to know which PodSets to include in the group and in which topology domain level to look for placement. To co-locate leader and workers user would have to make sure the group name is the same in both PodSets and the required-topology is the same in both PodTemplates. With dedicated annotation they just add one annotation and we will make it work by assigning topology and group names to PodSets.
  
  The feature of finding common flavor is quite flexible, so we could leverage from allowing users to manipulate those groups and forcing common flavor for arbitrary number of PodSets (I do not know if there is a use-case, but it is possible).
  
  However, finding topology domain for PodSets is limited. My experiment showed that it would be best to limit that to two PodSets only with one of them having 1 (or at most - relatively small) number of Pods, so with that many constraints I thought how to make it simpler for the user to just request what they need.
  
  WDYT?
