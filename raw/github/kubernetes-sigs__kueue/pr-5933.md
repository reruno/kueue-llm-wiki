I'm concerned the implementation is racy, and there could be a time window before we set the penalty when the other workloads could slip in. 

I suspect that to avoid race conditions we should use a mechanism similar to "assuming" workload into cache. Similar as we account for quota before full admission of the workload, see [AssumeWorkload](https://github.com/kubernetes-sigs/kueue/blob/edd0a329e1b90c66c4d567775a883ab1e590baa6/pkg/cache/cache.go#L642) I was thinking to be used to lock the penalty usage.  Then penalty usage could be released at the next cycle.

In particular, I suspect it is possible to admit a workload before the AFS is fully initialized for the LQ. The scenario is, a user creates LQ and puts the workloads into the LQ quickly, exploiting this [line](https://github.com/kubernetes-sigs/kueue/pull/5910/files#diff-a7490e7f5efa2a59ab508f592b67dd6e7530b9d2e1b19a1b6a7438e6c2386905R310-R313).

### Comment by [@PBundyra](https://github.com/PBundyra) — 2025-07-21T13:43:10Z

Hi @IrvingMg, thank you for working on that. There are few things that caught my attention:

1. I believe this implementation leads to race condition.

Assuming the scenario:
1. Workload gets admitted
2. Penalty is calculated
3. LocalQueue controller is notified

What can happen next is that the next scheduling loop will happen faster then the update coming from the LocalQueue controller which leads to race condition. We should ensure that the update will happen before next scheduling loop

2. I think the current approach disrupts the feature as we delete the logic for calculating the usage that takes into account current usage. We shouldn't do that, and we should only add the penalty on top of the existing algorithm.

3. (minor, stylistic) Setting half-time as 0 is not an error, and shouldn't be treated as such. It's a value that results in reseting the LQ's usage but this is WAI.

### Comment by [@PBundyra](https://github.com/PBundyra) — 2025-07-21T15:01:16Z

>  We should ensure that the update will happen before next scheduling loop

A way to accomplish that would be to have a shared penalty map between scheduler and LQ's controller. At the beginning of each scheduling cycle we rebuild the heap with respect to LQ's usage in status + penalty from the map.

If scheduler admits a Workload it informs LQ about it and delegates bumping the usage with an API call (similarly to current implementation). The LQ controller pops the penalty from the shared map and sends an API request. Until the map is not emptied, scheduler sums the usage from status and the penalty and rebuilds the heap at the beginning of each scheduling cycle.

  I'd rather have it in queue manager as it conceptually fits there better. In order to include it during Snapshoting let's introduce a function that return the map and pass it to the `Snapshot()` function.
  
  Either way we'll run into race condition between scheduler and LQ controller, resulting with incorrect results. E.g. if we keep in manager than between heapifying and snapshoting, LQ controller can modify map. If we keep in cache than when heapifying LQ controller can modify the map.
  
  As a solution to that we need to introduce a new mutex that is part of the penalty map structure
  
  Something along the lines:
  
  ```
  type PenaltyMap{
      mutex
      penaltyMap map[...]...
  } 
  ```
  
  Scheduler should take the mutex before heapifying and release it after snapshoting. LQ controller should take and release similarly to the current cache's mutex
