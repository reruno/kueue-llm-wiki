#### What this PR does / why we need it:
Kueue doesn't natively support non-preemptible workloads. This is a useful feature because sometimes specific workloads should never be preempted, even when preemption is generally permitted in the cohort or queue.

This PR allows users to annotate underlying objects with `kueue.x-k8s.io/non-preemptible` to indicate that it should not be preempted.

This change is backwards compatible because currently all workloads are tacitly preemptible.



#### How this was tested
I updated the unit tests. I also manually tested it since I'm using this fork in my service.

### Comment by [@mimowo](https://github.com/mimowo) — 2025-07-21T07:27:37Z

> So this could be a dangerous (as much as jobs being preempted can be considered dangerous) change to many people. If someone is submitting to a CQ that enables prememption, there isn't really any reason why someone can't just add this annotation and disable preemption for all their workloads.
> And then this will circument the admin setting up preemption.

Totally, adding this as an option directly on Workload may bypass admin's policy.

I would like to propose an alternative design by extending the [WorkloadPriorityClass](https://kueue.sigs.k8s.io/docs/concepts/workload_priority_class/) with the `preemptionPolicy` field analogous to [preemptionPolicy](https://github.com/kubernetes/kubernetes/blob/c44bf18b9b26d4526670d77ea60329f89a5bec2c/staging/src/k8s.io/api/scheduling/v1/types.go#L59 in the core k8s for Pods. This will allow admins to decide first if they want to support such a policy in their clusters or not. The options could be like `Always` (current default), `Never` (non-preemptible strictly), but maybe in the future could be `OnlyWhenFairSharing`, `OnceCheckpointed`, or whatever.

Indeed, assuming we follow the WorkloadPrioriryClass pattern, I would like to see this [KEP](https://github.com/kubernetes-sigs/kueue/blob/8dbda73b549d0e5295423a5f30c716ee7e10be61/keps/973-workload-priority/README.md) extended.
