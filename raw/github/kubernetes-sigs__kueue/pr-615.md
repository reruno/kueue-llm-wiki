  Indeed the field management can be done at a controller level, the problem is , as you mentioned , the need to extract the other field of interest every time a patch operation is done. 
  
  From an execution point of view, having a dedicated manager for every option gives us the opportunity to modify the options concurrently. With the current approach the one think we are not "guarded" against is the concurrent update "PodsReady" and    "Finished" by the job controller.
  
  What the core types use:
  1. Code generated by `applyconfiguration-gen` which is not trivial to integrate.
  2. `k8s.io/apimachinery/pkg/util/managedfields.ExtractInto` used to extract the values of a fieldManager from an existing option.
  
  `managedfields.ExtractInto` could by used without any code generators (by using  `typed.DeducedParseableType` ) , however the conditions slice for our scenario is treated as `atomic` hence no concurrency in updates.
