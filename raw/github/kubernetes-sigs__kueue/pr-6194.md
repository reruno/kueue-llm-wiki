#### What this PR does / why we need it:

#### "Old" Workload slice status update.

The current implementation of the "old" workload slice status update—specifically, the activation of the `Finished` condition—does **not** perform the update on a fresh copy of the workload. Instead, it **mutates the cached object** stored within the ClusterQueue cache.

This leads to a potential **race condition**, where both the `scheduler` and the `workload-controller` may concurrently update the same workload object. As a result, the associated ClusterQueue state—such as admitted workload count and flavor resource utilization—can become **inconsistent** or **invalid**, depending on the update timing.

#### Workload sort order
This PR introduces changes to improve sorting of workload slices in scenarios where multiple workloads may end up with the **same creation timestamp**. This can occur when workloads are scaled up in quick succession against an etcd/API server configured with **second-level time precision**.

In such cases, two workloads may have identical timestamps, which can lead to incorrect "old" vs. "new" slice ordering. To address this, the PR adds a **fallback mechanism** that uses **resource version** comparison when timestamps are equal.
