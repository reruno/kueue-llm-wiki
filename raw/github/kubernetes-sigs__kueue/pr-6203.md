### Comment by [@ichekrygin](https://github.com/ichekrygin) — 2025-07-28T16:20:33Z

To clarify (after the PR merged): the use of `DeepCopy` is necessary, but it is **not related** to the workload slice `creationTimestamp` collision mentioned in the PR title.

The timestamp issue was a **separate edge-case**, observed during integration, where both the initial workload and its scaled-up successor were created within the same second (due to second-level precision). This led to incorrect sorting between the "old" and "new" slices.

The **reason for using `DeepCopy`** is to avoid a **race condition** between the scheduler and the workload controller, both of which may modify the **shared** workload object stored in the cache:

```go
clusterWorkloads[key] *workload.Info → workload.Info.Obj → *kueue.Workload
```

Without deep copying, both components could end up mutating the same underlying object, leading to inconsistent state and hard-to-debug side effects.
