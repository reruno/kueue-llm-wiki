#### What this PR does / why we need it:
Based off documentation and existing integration tests, the focus of topological aware scheduling seems moreso around data center rack and block topology. These tests explicitly demonstrate that TAS can be used for a simpler scenario.

Smaller scenario:
- Kueue knows that there are 8 GPUs in a cohort available
- However it is 1 GPU per 8 nodes, and you need a full single 8 GPU node to run the workload
- So Kueue sorta flaps around / preemption churns because it thinks it can schedule something, but it actually can't

Bigger version of above scenario:
- You submit a workload that needs 100 nodes & it gets admitted
- 99 nodes get scheduled
- the last 8 GPU worth of quota flaps around bc the capacity isn't on one node
- Meanwhile, this results in 99 nodes worth of wastage until it hits timeout

With TAS turned on, you can reject these unscheduable workloads at admission time.

#### Special notes for your reviewer:
These these are moreso to demonstrate use cases rather than testing any untested areas of code. (As a user I often reference tests to see how a feature can be used.)
