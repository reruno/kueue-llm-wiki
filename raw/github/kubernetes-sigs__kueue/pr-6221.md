#### What this PR does / why we need it:

**This PR addresses a regression introduced in [#6203](https://github.com/kubernetes-sigs/kueue/pull/6203)**

In the process of handling workload slice `creationTimestamp` collisions, the behavior of `FindNotFinishedWorkloads` was inadvertently altered. Instead of returning a **sorted list of active workload slices** (oldest first), the updated version returned **a single, and potentially incorrect, "old" workload slice**.

As a result, the regression extended beyond workloads with timestamp collisions and began affecting **any job reconciled in a transitional state**, where both an old and a pending new workload slice are present, **regardless of `creationTimestamp` values**.

This PR corrects the regression by restoring the original behavior of returning a sorted list of active workload slices (excluding those marked `Finished`), while also introducing a **proper tiebreaker** for workloads with identical `creationTimestamp` values. This tiebreaker is based on `resourceVersion`, using the Kubernetes `storage.APIObjectVersioner` for safe and forward-compatible comparison.

For further details, see the "Special notes for your reviewer" section below.

#### Special notes for your reviewer:

##### 📦 Summary

This PR improves the sorting logic for workload slices in `FindNotFinishedWorkloads` by handling cases where two workloads share an identical `creationTimestamp`. This scenario is rare in production but can occur in integration or e2e tests due to Kubernetes’ second-level timestamp precision.

---

###### 🔍 Problem

Kubernetes stores object `creationTimestamp` values with **RFC3339 second-level precision**, which can result in **identical timestamps** for multiple objects created in rapid succession. When this happens, sorting by `creationTimestamp` alone is insufficient and may yield non-deterministic ordering of workload slices, impacting logic that depends on chronological order, such as selecting the “old” workload slice.

---

###### ✅ Solution

This PR introduces a **deterministic tiebreaker** by using `resourceVersion` comparison via `storage.APIObjectVersioner`, which provides version-aware ordering compliant with Kubernetes API semantics.

###### Key Changes:

* Updated `FindNotFinishedWorkloads` to sort workloads:

  * First by `creationTimestamp` (oldest first).
  * Then, if equal, by resourceVersion using `storage.APIObjectVersioner.CompareResourceVersion`.
* Introduced a `versionHelper` instance at package level with appropriate explanatory comments.
* The comment clarifies that direct numeric comparison is avoided, per Kubernetes guidelines, and future changes to resourceVersion format will be surfaced via type safety.

---

##### ⚠️ Notes

* The use of `resourceVersion` is guarded and abstracted to ensure forward compatibility.
* This change is intended to improve correctness in edge cases without affecting existing workloads or admission behavior.

### Comment by [@ichekrygin](https://github.com/ichekrygin) — 2025-07-29T15:43:53Z

> @ichekrygin it would be great to add an integration test which (1) demonstrates that the main version is broken, and (2) demonstrates that the fix works. Is it feasible?

Hi @mimowo, thank you for taking a look.

First things first, the function is **broken** because it no longer aligns with the intended behavior as described in the design and captured explicitly in the GoDoc:

```go
// FindNotFinishedWorkloads returns a sorted list of workloads "owned by" the provided job object/gvk combination and
// without "Finished" condition with status = "True".
```

The phrase *“returns a sorted list of workloads”* is an **operative requirement**.

Additionally, the intended sort order is explicitly defined in the original implementation:

```go
// Sort workloads by creation timestamp, oldest first.
sort.Slice(list.Items, func(i, j int) bool {
	return list.Items[i].CreationTimestamp.Before(&list.Items[j].CreationTimestamp)
})
```

The **pre-regression** code was functioning correctly. The timestamp collision issue was uncovered during integration testing, where a workload and a follow-up scale-up request were created in rapid succession. These collisions were rare, roughly *3 failures in 100 runs*. That said, this scenario **does not reflect common production usage**, and if anything, modifying the test to tolerate such rare collisions would have been preferable to altering the core logic in a way that introduced broader breakage.

To make matters worse, a unit test introduced in the regression unintentionally **codified a misunderstanding** of the original purpose of `FindNotFinishedWorkloads`.

Regarding the integration test proposal: I’m not convinced that adding an integration test specifically to reproduce timestamp collisions is a productive or sustainable approach to improving test stability. These collisions are inherently nondeterministic and environment-sensitive.

In contrast, the **regression itself is fully unit-testable**. This PR already includes a correct and aligned unit test that validates the intended behavior.

Please let me know if this reasoning makes sense, or if you'd like to discuss further.
