#### What this PR does / why we need it:
This PR allows to patch existing pods with pod group metadata. 

Following scenario was failing:
- Kueue is running with pod integration enabled
- Pod exists with  kueue admission gate and pod-suspeding-parent annotation
- Pod is patched with pod group metadata 
- Workload is created and admitted
- Pod spec changed - kueue added node selectors
- Workload is looking for matching pods  -> 0 pods are found
- Workload has no matching pods -> workload is deleted

Fix from PR:

- Kueue is running with pod integration enabled
- Pod exists with kueue admission gate and pod-suspeding-parent annotation
- Pod is patched with pod group metadata -> roleHash annotation is saved
- Workload is created and admitted - with podset_name_0 = roleHash
- Pod spec changed - kueue added node selectors -> roleHash annotations on pod exists and is not changed
- Workload is looking for matching pods. It checks RoleHash annotation -> pods are found
-  workload executes

Exact scenario:

pods node-0,node-1, node-2 created as below:
```
apiVersion: v1
kind: Pod
metadata:
 name: node-0
 namespace: foo
 annotations:
  kueue.x-k8s.io/pod-suspending-parent: Controller
spec:
 schedulingGates:
 - name: kueue.x-k8s.io/admission
 nodeSelector:
   cloud.google.com/gke-nodepool: dws-flex-queued-a2
   cloud.google.com/gke-accelerator: nvidia-tesla-a100
 restartPolicy: Never
 containers:
 - name: sleep
   image: busybox
   command: ["sh", "-c", 'echo "hello world from the leader pod" && sleep 3']
   resources:
     requests:
       nvidia.com/gpu: 1
     limits:
       nvidia.com/gpu: 1
```
Kueue configuration:
```
apiVersion: kueue.x-k8s.io/v1beta1
kind: ResourceFlavor
metadata:
  name: "default-flavor"
---
apiVersion: kueue.x-k8s.io/v1beta1
kind: AdmissionCheck
metadata:
  name: dws-prov
spec:
  controllerName: kueue.x-k8s.io/provisioning-request
  parameters:
    apiGroup: kueue.x-k8s.io
    kind: ProvisioningRequestConfig
    name: dws-config
---
apiVersion: kueue.x-k8s.io/v1beta1
kind: ProvisioningRequestConfig
metadata:
  name: dws-config
spec:
  provisioningClassName: queued-provisioning.gke.io
  managedResources:
  - nvidia.com/gpu
---
apiVersion: kueue.x-k8s.io/v1beta1
kind: ClusterQueue
metadata:
  name: "dws-cluster-queue"
spec:
  namespaceSelector: {}
  resourceGroups:
  - coveredResources: ["cpu", "memory", "nvidia.com/gpu", "ephemeral-storage"]
    flavors:
    - name: "default-flavor"
      resources:
      - name: "cpu"
        nominalQuota: 1000000000    # "Infinite" quota
      - name: "memory"
        nominalQuota: 1000000000Gi  # "Infinite" quota
      - name: "nvidia.com/gpu"
        nominalQuota: 1000000000    # "Infinite" quota
      - name: "ephemeral-storage"
        nominalQuota: 1000000000Ti  # "Infinite" quota
  admissionChecks:
  - dws-prov
---
apiVersion: kueue.x-k8s.io/v1beta1
kind: LocalQueue
metadata:
  namespace: "foo"
  name: "dws-local-queue"
spec:
  clusterQueue: "dws-cluster-queue"
---
```
Patch applied:
```
kubectl patch pod node-0 -n foo --patch '{
  "metadata": {
    "annotations": {
      "kueue.x-k8s.io/pod-group-total-count": "1",
      "provreq.kueue.x-k8s.io/maxRunDurationSeconds": "600"
    },
    "labels": {
      "kueue.x-k8s.io/pod-group-name": "job1",
      "kueue.x-k8s.io/queue-name": "dws-local-queue",
      "kueue.x-k8s.io/managed": "true"
    }
  }
}'

```

Workload is deleted after is admitted.

```release-note
Fix support for PodGroup integration used by external controllers, which determine the 
the target LocalQueue and the group size only later. In that case the hash would not be 
computed resulting in downstream issues for ProvisioningRequest.

Now such an external controller can indicate the control over the PodGroup by adding
the `kueue.x-k8s.io/pod-suspending-parent` annotation, and later patch the Pods by setting
other metadata, like the kueue.x-k8s.io/queue-name label to initiate scheduling of the PodGroup.
```
