#### What this PR does / why we need it:
- When we build manifests from /tests/e2e/config/certmanager, the certmanager resources don't have the kueue- prefix.
- After deleting internalcert, the kueue- prefix is removed from secretName, we should add it manually.
- Add the kueue- prefix using a patch for the certificate commonName.

### Comment by [@mbobrovskyi](https://github.com/mbobrovskyi) — 2025-08-01T05:22:51Z

> What about the secret name in the generated manifest.yaml (the release artifact)?

We don't provide release manifests for cert-manager in the artifacts — it's considered an additional configuration that users need to set up manually. For built-in certs, we rely on an internal secret, and everything works fine if that secret exists.

However, when using cert-manager, users need to delete the internal secret and let cert-manager generate a new one using the name specified in secretName. If the internal secret is deleted, Kustomize doesn’t apply the kueue- prefix, which leads to cert-manager creating `webhook-server-cert` and `metrics-server-cert` without the prefix.

This behavior is inconsistent with both the internal cert setup and the Helm configuration, where the kueue- prefix is used. We should ensure the prefix is applied in the Kustomize setup as well to maintain consistency.
