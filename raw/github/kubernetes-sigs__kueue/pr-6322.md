
  This makes sense. 
  
  The root cause is probably that the ProvReq logic used wlPatch to reason about the retries, but it was not initialized, so it was behaving as it was first try.
  
  Now, this may not be the only issue with the previous code, suppose:
  1. ProvisioningRequest is set to Retry and RequeueState is initialized - ProvReq takes ownership of the field
  2. on the second attempt the ProvisioningRequest wants to transition Pending -> Ready, and does not need to bump RequeueState as it is successful.  
  Issue: Now, because it owns the field (step 1), and does not update it, it could clear the field on setting the AC to Ready. This may not be desired as other ACs may still be pending.
  
  I'm not totally sure this could happen (just static code analysis). We could exercise this in a follow up integration test, if you support there was a bug indeed, so that we could prevent regression in the future.
  
  Still, since we use SSA for ProvReq, we need to send each time every field the controller may own, so this initialization makes perfect sense to me.
