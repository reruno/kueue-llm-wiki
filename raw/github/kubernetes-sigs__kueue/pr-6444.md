## Description

This PR is not meant to be merged, but as an example for a potential way to improve preemption performance as discussed in [this issue](https://github.com/kubernetes-sigs/kueue/issues/6143)

- Introduced a new entry status 'preempting' to indicate when a workload is in the process of preempting others.
- Updated the scheduler to skip workloads that are already in the 'preempting' state during scheduling.
- Added checks in the preemptor to ignore candidates that have been evicted, ensuring only valid workloads are considered for preemption.

  Similar disclaimer, I’m not fully confident that this change has any effect on preemption target collisions.
  
  As I understand it, once a preemption is issued, we *intentionally skip* processing the candidate that triggered it. That candidate is then re-evaluated in the next scheduling cycle. Ideally, by that time, the targeted workload has been de-admitted, freeing up capacity in the ClusterQueue so the candidate can be admitted.
  
  If not, i.e., if there’s still insufficient capacity, the scheduler will attempt to identify a new preemption target, which is often different from the one preempted in the previous cycle.
  
  It’s worth noting that `s.preemptor.IssuePreemptions(ctx, &e.Info, preemptionTargets)` is an *imperative* call, not a declarative one. In other words, it either succeeds and the target workload is preempted, or it fails, in which case the candidate is marked for requeueing and will be retried in a future cycle.
  
  Given that, I’m not sure how this change would influence preemption collision handling. Could you clarify the intended benefit?
