### Comment by [@ichekrygin](https://github.com/ichekrygin) — 2026-04-01T22:03:44Z

A quick recap on **why ReplicaSet and not Deployment**

ReplicaSet is the natural integration point for ElasticWorkloads in Kueue because it aligns with the core requirement of immutability after admission.

ReplicaSets are effectively immutable in practice, with the exception of scaling via `replicas`. While the API allows updates (including to the pod template), such changes do not affect existing Pods and are not how higher-level controllers like Deployment manage updates. Instead, they create new ReplicaSets, making each ReplicaSet a stable snapshot of desired state. This provides a stable contract for Kueue, where a Workload represents an admitted unit of resources that should not be arbitrarily mutated afterward. Scaling via `replicas` maps cleanly to elastic behavior without breaking admission guarantees or requiring re-evaluation of placement decisions.

In contrast, Deployments are highly mutable, covering changes to pod template, labels, annotations, and rollout strategy. Supporting ElasticWorkloads at the Deployment level introduces ambiguity around what constitutes a stable admitted workload versus a mutation that should trigger re-admission or re-scheduling. This becomes especially problematic for post-admission changes, where Kueue does not currently have a well-defined model for handling mutability.

From a MultiKueue perspective, the key requirement is the ability to consistently correlate pods and capacity decisions across clusters. This does not fundamentally depend on whether slicing is modeled at the Deployment or ReplicaSet level. In both cases, we rely on stable identifiers (for example, the workload-slice-name annotation) to associate pods with a logical workload. ReplicaSet-level integration preserves this property while keeping the admission unit aligned with the actual execution boundary.

Additionally, ReplicaSet sits directly at the level that maps to actual pod lifecycle and scheduling, which is what Kueue ultimately manages. Deployment support then follows naturally as a higher-level abstraction, similar to how CronJob builds on Job, without requiring dedicated integration.

Finally, ReplicaSet-based integration provides a more predictable model during rolling updates and when handling variations such as different node selectors, since each ReplicaSet represents a concrete, stable slice of desired state.

Overall, the choice of ReplicaSet is not a shortcut, but a way to align ElasticWorkloads with Kubernetes execution semantics, while keeping the admission model simple and consistent across both single-cluster and MultiKueue scenarios.

### On Reusability

I’m not entirely sure what was meant by “Reusability” in the original table. As far as I can tell, there isn’t meaningful reuse across Kueue integrations at the semantic level. 

Pod-based integration may appear “reusable” because it operates on a common low-level primitive (Pods), but that is only mechanical reuse. In practice, each higher-level workload (e.g., LWS, StatefulSet) still requires custom logic to reconstruct grouping, lifecycle, and admission semantics on top of Pods. This suggests that Pod-level integration is not a reusable abstraction, but rather a shared low-level mechanism, with the actual workload semantics reintroduced separately in each integration.

### On Multikueue

It appears there is a strong emphasis on MultiKueue support. I already mentioned earlier that MultiKueue support is TBD and not part of this PR. I suspect this is a forward-looking perspective, which I appreciate.

However, speaking strictly in current terms, while Deployment + Pod-integration is technically supported in MultiKueue, that support operates at the Pod level and does not provide consistent workload-level placement or dispatching guarantees. As a result, Pods belonging to the same Deployment may be distributed across worker clusters, which can lead to fragmentation and makes the behavior harder to reason about and operate in multi-cluster environments.

Additionally, the Workload-per-Pod model inherently increases control plane load, as it requires syncing and reconciling a large number of individual Pod-derived workloads and their status updates across cluster boundaries, rather than operating on a single workload unit.

Given that, it is hard for me to treat MultiKueue support as a meaningful differentiator here, because neither option provides solid workload-level MultiKueue semantics today. The key difference is that for ReplicaSet + Deployment such support is explicitly TBD and can be designed around a proper workload boundary, whereas Pod-based integration already reflects the limitations of operating without that boundary.

### Summary

The core difference comes down to whether Kueue operates on Pods or on workloads. 

* Pod-level integration treats every Pod independently, which leads to best-effort grouping and inconsistent behavior for Deployments. 
* ReplicaSet-level integration introduces a stable workload boundary, allowing Kueue to make consistent admission, placement, and preemption decisions. 

For a system that is fundamentally workload-oriented, this distinction is critical.

  Thanks, @mimowo, for raising this, I agree with the underlying concern. Tracking suspended state purely in memory is less than ideal, especially given that the state is lost on Kueue restart, so it’s reasonable to look for alternatives that are more persistent and observable.
  
  I looked more closely at **PodGroups** as a possible model, and I think parts of it could apply, but with an important caveat. PodGroup is an artificial construct that relies heavily on finalizers on the parent objects (Pods). It is also currently the only integration that implements the `Composable` interface. While that makes it an interesting reference point, there are some misalignments if we try to apply the same pattern to ReplicaSet or Job-like controllers:
  
  * It would require introducing a finalizer on the ReplicaSet object to follow the `Composable` paradigm.
  * It conceptually represents the Job ↔ Workload relationship as “composed,” whereas for ReplicaSet the Workload is not composed and does not have `ChildrenWorkloads`.
  
  I do like the idea of examining the **actual Pods** directly, rather than relying on in-memory state. Where I struggle a bit is that the current `ComposableJob` interface feels fairly narrow and largely scoped to PodGroup-style use cases.
  
  An ideal outcome for ReplicaSet, in my view, would be to refactor the interface slightly so we can reuse the parts we need without adopting the full composable model. For example, if we could separate the “load” aspect into its own interface, something along these lines:
  
  ```go
  type JobWithCustomLoad interface {
    Load(...)
  }
  
  type ComposableJob interface {
    JobWithCustomLoad
    ...
  }
  ```
  
  With this split, a ReplicaSet integration could implement `LoadableJob` to load the ReplicaSet and its child Pods, and then continue through the “regular” job processing path, without having to opt into finalizers or a composed workload model.
  
  Curious what others think about this direction, does separating the loading concern from composability seem reasonable here?
