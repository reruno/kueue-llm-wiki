#### What this PR does / why we need it:
We need to be able to measure how long eviction of a running workload takes. There's various production issues that are the result of long evictions.

If `PodsReady=True`, this measure the time between `Evicted=True` and:
- `QuotaReserved=False` OR
- `Finished=True` OR
- Evicted reason `Deactivated`

### Comment by [@mimowo](https://github.com/mimowo) — 2025-10-30T14:00:50Z

I would like to consider a slight simplification to the process to always set QuotaReserved=False whenever we mark workload Finished, or Deactivated. Then the "reason" in the metric could be always just the reason of the "QuotaReserved" condition. I think this will also make the API cleaner to users, I already got asked a couple of times ~"why the quota is freed if my workload shows QuotaReserved=true, is it a bug?". Note that it wouldn't require new API requests, we could just tweak the QuotaReserved condition any time we transition the Finished or Deactivated conditions. 

This will also be closer to how we populate the reason for the "Eviction" metrics (like PodsReadyToEvictedTimeSeconds), directly from the reason of the "Evicted" condition. I think it simplifies the reasoning. 

Another small thing is that the metric should always be updated "after" the request succeeds  to avoid double-counting in case of an error. Take a look at how we account for the eviction metrics, only when err=nil, [here](https://github.com/kubernetes-sigs/kueue/blob/main/pkg/workload/workload.go#L1294-L1299).
