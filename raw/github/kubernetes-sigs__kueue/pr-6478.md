#### What this PR does / why we need it:

This PR adds an integration test to cover the workload-slice scenario where the new slice has not yet been admitted, while the old slice has already been evicted by the scheduler.

#### Special notes for your reviewer:

**Additional Change** 
This PR replaces the use of `resourceVersion` as a tiebreaker for workload slice collisions (i.e., when two workloads have the same creation timestamp) with the more deterministic `WorkloadSliceReplacementFor` annotation.

As noted in the comment, this issue surfaces in testing when the original and scaled-up workloads are created in rapid succession. However, in the same scenarios, workloads can be updated just as rapidly, making `resourceVersion` an unreliable tiebreaker. Using an explicit annotation provides a more robust and deterministic solution.

  As I understand it, the parent group (`Describe "Job controller"`) does not run with the Kueue scheduler enabled. That’s why I added this integration test there, because it’s the **only reliable way I know** to reproduce the issue.
  
  Just to recap, this issue occurs when:
  
  * A new workload slice is created **and**
  * At the same time (not before the slice is created, and not after it's evaluated by the scheduler), the old slice is preempted by the scheduler.
  
  This exact timing makes the scenario difficult to reproduce in a regular scheduling flow. That’s why I opted for low-level condition manipulation within this test, it provides deterministic control over the setup.
