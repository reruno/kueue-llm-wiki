Summary:
1. LocalQueue doesn't exist, we add Admitted=false
2. LocalQueue created
3. Scheduler starts scheduling the workload once
4. Workload controller receives update event for the condition, adds the workload to the queue. This succeedes because the workload is not currently in the queue, as it's being scheduled.
5. First scheduling attempt succeeds, sets Admitted=true
6. Second scheduling attempt starts (because it's still in the queue) for the same workload, but the workload doesn't fit.
7. We add Admitted=false. Because it uses SSA, it succeeds.
8. The workload is now unadmitted, so it enters admission again.

### Comment by [@alculquicondor](https://github.com/alculquicondor) — 2023-03-28T16:12:26Z

As for solutions:

a. One option is to give up on SSA for admission and use a regular Update. But then we would increase chances of conflict when a workload is first inadmissible and then admissible: the scheduling loop might have the workload without the first condition added, thus reaching a conflict and wasting a scheduling cycle.

b. A better middle ground solution could be to migrate one of the condition changes back to Update (or add resourceVersion to the patch).
For example, marking a Pod as Admitted=false could use Update, as failing to verify that the workload is not already admitted leads to workload interruption.
But admitting could continue to use SSA. This might be fine because:
1. I don't think it's possible that we admit the same workload twice, as `Assume` would fail to assume an already assumed or admitted workload.
2. It isn't that bad if we assume a workload twice (other than checks in integration test). It would just increase the metric value, but it wouldn't change anything in the cache or affect running workloads.

c. The best long term solution would be to continue to use SSA if it offers some kind of fine grained preconditions. For example, we could verify that the condition is false when we try to transition it to true and viceversa.

Le'ts proceed with b.
