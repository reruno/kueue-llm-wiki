#### What this PR does / why we need it:
Attempts to find new assignments if the original preemption targets overlap with previous preemption targets.

More detailed explanation of changes:

#### The overlapping preemption logic
- Within `nominate`, workload heads still gets a first pass of preemption targets. Within the iterator step, we now track preemption targets by previous workloads and pass them into `excludeWorkloads` parameter to skip them in subsequent calls of `getAssignments` when there's a preemption target overlap. 
- adds `excludeWorkloads` parameter throughout preemption code flow. For both fairshare and classical preemption, if the workload is excluded, it's simply skipped in the `findCandidates` step. So that we can consider other candidates that don't overlap.
- `fits()` is updated to deduplicate double subtraction of resources. We want to safeguard the same preemption target from being added to both `preemptedWorkloads` and `newTargets`

#### Allowing 1 preemption target to be claimed by multiple preemptors
Unfortunately also now part of the scope of this PR is allowing 1 preemption target to be claimed by multiple preemptors. 

Here's the core problem. Consider this scenario:
* Borrowing Workloads: 2CPU, 1CPU
* Preempting workloads: 1CPU, 1CPU

Consider both preemptors target the 2CPU workload. When the second getAssignment is called when there are overlapping preemption targets, we need to make sure the snapshot includes:
1. addition of usage from 1st admission candidate (1CPU)
2. removal of admission from preemption candidate (2CPU)
Today, usage is only calculated as a unit of a full workload (ie usage from the entire workload is either removed or added). Given that, if we remove the full usage from the preemption candidate from consideration, the second admission calculation for the second admission workload would incorrectly result in Fit mode.
What is now increasing the scope is this: [#6581](https://github.com/kubernetes-sigs/kueue/issues/6581) Allow 2 preemptors point to the same preemption target in Preempt mode. This involves tracking what portion of the preemption target gets claimed by a preemptor (ie how much remains for future preemptors). So in the above scenario, both preemptors would point to the 2 CPU workload as its preemption target and set the preemptor in Preempt mode. We also can't have the second preemptor point to the 1CPU borrowing. This is because the snapshot removes the first 2CPU borrowing workload, sees that there's enough room, but no preemption targets available.

This still allows for the following scenario to work while also preventing the above buggy scenario from occurring:
* Borrowing Workloads: 2CPU, 2CPU
* Preempting workloads: 2CPU, 2CPU
* Each preempting workload would end up finding a different target on getAssignment

##### 1 preemption target to be claimed by multiple preemptors code changes
* PreemptedWorkloadsV2 now tracks usage claimed by preemptors
* A workload can be considered excluded in subsequent assignment retry if all of its usage is claimed
* To assuage any concerns, there's no concept of "partial preemption" mixed into this

### Comment by [@amy](https://github.com/amy) — 2025-08-21T19:13:41Z

Notes for why allowing multiple preemptors claim the same preemption target is now within scope:
1. Unable to remove workload usage from snapshot during cycle. It results in fit mode seen here: https://github.com/kubernetes-sigs/kueue/pull/6541#discussion_r2273264645
2. When you don't remove workload usage from the snapshot, fit check fails because we admit without removing preemption target from snapshot. (Preemption usage removal only reflects in the next schedule() invocation on successful preemption. Existing RemoveWorkload calls are only done in relation to simulation to get NEW preemption targets then reverted.)
3. So fit needs to check fitness if we remove preemption targets from the snapshot. But we only want to remove the claimed usage, so that future preemptors can correctly result in Preempt mode. And if needed, select a preemption target that another preemptor also targeted.

Will update PR description with this when I'm done fixing the tests/code. Just want to put this context somewhere.

----

I probably have an issue with where I add the claimed usage from previous preemption targets back into the snapshot. I think I need to add them around where `runFirstFsStrategy` is called instead of deep within `workloadFits` function. (and in a similar bubbled up spot for classical preemption) The simulation where we add preemptor usage and remove target usage is only relative to the preemptor's current preemption targets. This worked when there were no overlapping preemption targets, but for overlapping preemption targets, it needs to be adjusted.
