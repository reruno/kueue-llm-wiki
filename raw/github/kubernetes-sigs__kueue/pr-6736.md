### Comment by [@ichekrygin](https://github.com/ichekrygin) — 2025-09-22T15:32:04Z

> So, if I understand correctly, we shouldn't automatically enable pod integration for Deployment in this PR? If that's the case, then I can just remove the `ImplicitlyEnables` field in the Deployment controller, and I think that should be enough to keep the PRs independent of each other.

To clarify, the PR for Kueue-native Deployment **adds** support rather than replacing the existing Deployment pod-integration. At least initially, users can operate in a dual mode:

* Deployment via pod-integration, and
* Deployment as a Kueue-native integration.

Because both paths may coexist, it’s hard to define a fixed dependency graph. I understand the “dual mode” integration framework is somewhat of an edge case, and we could always remove “pod” from the `ImplicitlyEnables` list for Deployments to support it.

My main concern is that this may create a confusing experience for users around the implicit enablement of pod-integration:

* today: not implicitly set,
* tomorrow (with this PR): implicitly set,
* the day after (with Kueue-native Deployment): not implicitly set again.

That said, I don’t see this as a blocker. We can work around it in a follow-up PR if needed.
