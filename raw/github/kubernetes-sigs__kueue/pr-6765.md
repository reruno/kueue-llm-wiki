We can create another version of PatchStatus:
```go
func PatchStatus(ctx context.Context, c client.Client, obj client.Object, update func() (client.Object, bool, error), options ...PatchOption) error {
	return patchCommon(obj, update, func(objOrig client.Object, patch client.Patch) error {
		return c.Status().Patch(ctx, obj, patch)
	}, options...)
}


func updateAndPatch(obj client.Object, options PatchOptions, update func() (client.Object, bool, error), patchFn func(client.Object, client.Patch) error) error {
	objOriginal := getOriginalObject(obj, options)
	objPatched, updated, err := update()
	if err != nil || !updated {
		return err
	}
	patch, err := createPatch(objOriginal, objPatched)
	if err != nil {
		return err
	}
	return patchFn(objOriginal, patch)
}
```
Right now we rely on the PatchStatus which returns the object after update, with this adjustment we use object that we already copied for the patch.
I believe we can reduce number of deep copies in scheduler with that, bc most of the extra copies are used to avoid race conditions.

  Hey folks, just a quick note about `EvictOption(s)`, I’ve seen this pattern used in a few places, so take this suggestion with a grain of salt.
  
  Personally, I find the current setup a bit confusing for a couple of reasons.
  
  ### 1. Option for an option?
  
  We usually use options to tweak some concrete object. But here, `EvictOption` is configuring `EvictionOptions`, which is itself just another layer of options. So it's kind of like an option-for-an-option. Then downstream we check things like:
  
  ```go
  if opts.CustomPrepare != nil {
  ```
  
  That feels pretty specific for something that’s supposed to be generic.
  
  ### 2. One option does everything
  
  This one:
  
  ```go
  func WithCustomPrepare(customPrepare func() (*kueue.Workload, error)) EvictOption
  ```
  
  …basically lets you customize *anything* about the workload. Which is powerful, sure, but also makes every other option kinda redundant. It’s so flexible that it breaks the whole point of having structured options in the first place.
  
  ---
  
  ### What I'm thinking
  
  We might be able to simplify things and make it a bit more intuitive if we skip the whole “option configuring another option” approach, and just define options that directly modify the `Workload`.
  
  Something like:
  
  ```go
  type EvictOption func(*kueue.Workload)
  
  func WithPreemptedConditionEvictOption(reason, message string) EvictOption {
  	return func(wl *kueue.Workload) {
  		SetPreemptedCondition(wl, reason, message)
  	}
  }
  ```
  
  Then the `Evict()` function just takes those directly:
  
  ```go
  func Evict(..., options ...EvictOption) error {
  	wl := wlOrig.DeepCopy()
  	for _, opt := range options {
  		opt(wl)
  	}
  	...
  }
  ```
  
  ---
  
  ### Why this might be better
  
  * **Simpler call sites**,
    Instead of:
  
    ```go
    workload.Evict(..., workload.WithCustomPrepare(func() (*kueue.Workload, error) {
        return &wl, nil // does nothing, just needed to match the API
    }))
    ```
  
    You could just call:
  
    ```go
    workload.Evict(...)
    ```
  
    Or if you *do* want to customize:
  
    ```go
    workload.Evict(..., workload.WithPreemptedConditionEvictOption(reason, message))
    ```
  
  * **Clearer intent**,
    Each option does one specific thing. Easier to reason about.
  
  * **No pointless boilerplate**,
    We don’t need to write a custom function just to return the same workload.
  
  ---
  
  I understand, sometimes `option -> option` paradigm is necessary and unavoidable, but I don' think it is needed in this specific case. 
  
  Anyway, I think this would make the code a little easier to follow, and the option API a bit more grounded. Curious what others think, does this feel like a worthwhile improvement?
