## Description

#### What type of PR is this?

/kind bug

#### What this PR does / why we need it:

When using MultiKueue, it's possible for a workload to be preempted in the worker cluster but not the manager cluster. For example, this can happen if there is 5 CPU available in Cluster 1 and 5 in Cluster 2, and someone schedules a workload requesting 6 CPU. The manager cluster would see no issue with this, but one of the worker clusters would need to preempt to admit the new workload.

Currently in this case the manager cluster assumes it's lost connection to the workers. It waits for a timeout and then submits the workload to the worker clusters again.

This PR updates the behavior to instead sync the worker cluster's status back to the manager cluster, and then delete the workload in the worker cluster. This has the nice property that the preempted workload can be readmitted on any worker cluster (not just the one it was previously running on), potentially allowing it to restart sooner.

#### Special notes for your reviewer:
I didn't add unit tests yet because I'm not very confident in this approach. I just wanted to check if this is the right approach and then I'll add unit tests

Reflect workload eviction statuses in the MultiKueue manager cluster.

The behavior I see is that the workload is evicted by the worker, the manager deletes the workload on the worker cluster (tbh I'm not sure why this is, I wasn't able to figure it out), and then I start seeing [this message](https://github.com/kubernetes-sigs/kueue/blob/a3887653ef8e71ef892afd327e9c0b35fc2471f9/pkg/controller/admissionchecks/multikueue/workload.go#L433) in the logs. One other thing that I wasn't certain of was when/what we should sync from the worker cluster status to the manager cluster. And once the workload has been assigned to a worker cluster, is it ok for the workload in the manager cluster to constantly sync from the one in the worker cluster (i.e. not just on eviction)?

### Comment by [@mimowo](https://github.com/mimowo) — 2025-09-11T06:54:56Z

That is a good question. Got me thinking.

For now, we have the design that we book quota on the management cluster while we are trying to schedule on the workers. The question is what we should do when the worker loses quota (is evicted either due to WaitForPodsReady or Preemption). IIUC the current code in Kueue is broken, because we don't evict on the manager - and so clusterName is not reset, yet we don't respect it, just rely on firstReserving.

Some ways I think we could handle that:
1. stick to the clusterName while set, when worker evicts then wait until re-admitted on the same worker
2. stick to the clusterName while set, when worker evicts propagate eviction to the manager so that the manager can reset clusterName
3. stick to the clusterName while set, when worker evicts don't evict fully on manager, just clear clusterName

I'm wondering if you can repro the issue before introducing clusterName (0.12, this would indicate a regression).

I think (3.) is essentially what we had before introducing clusterName (as another worker would be picked by the firstReserving). With the introduction of the clusterName we are in a weird state and I'm not yet sure of all consequences.

Going forward all the options seem like useful, but if we confirm this is a recent regression, then I would consider (3.) as a quick fix.
