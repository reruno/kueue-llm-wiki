## Description

#### What type of PR is this?
Add a guage metric to track the current number of workloads with `PodsReady` state. Two metrics have been added for ClusterQueue `kueue_running_workloads` and LocalQueue `kueue_local_queue_running_workloads`

/kind feature

#### What this PR does / why we need it:

To inform users how many workloads are actually making progress, since having a metric for "admitted" workloads is not necessarily reflecting that.

#### Which issue(s) this PR fixes:

Fixes #6791 

#### Special notes for your reviewer:

#### Does this PR introduce a user-facing change?

```release-note
Add new metrics for running workloads, `kueue_running_workloads` and `kueue_local_queue_running_workloads`
```

### Review by [@PBundyra](https://github.com/PBundyra) (changes_requested) — 2025-09-22T10:11:39Z

- **pkg/cache/queue/cluster_queue.go:1** by [@PBundyra](https://github.com/PBundyra) — 2025-09-22T09:57:03Z
  I believe this doesn't work in production, as when Kueue admits a Workload it pops it from the heap. In other words, there is no admitted Workloads in the heap. So we shouldn't just iterate over the heap but rather update the metrics based on events:
  - bump it by one when the `PodsReady` transitions `false` -> `true`
  - decrease it by one when the `PodsReady` transitions `true` -> `false`
  - decrease it by one when a Workload stops running (finished or evicted)
  
  The code responsible for changing `PodsReady` condition sits here: https://github.com/kubernetes-sigs/kueue/blob/main/pkg/controller/jobframework/reconciler.go#L498
  
  Please note that any changes regarding this metrics should be guarded by the check if WaitForPodsReady feature is enabled: https://github.com/kubernetes-sigs/kueue/blob/main/apis/config/v1beta1/configuration_types.go#L227
  
  Just for the completeness of  @mimowo suggestion about mutex. With the current proposal the order of magnitude of number of locks taken is the same, as with taking it separately for both pending and running workloads. The significant difference between pending and running Workloads is that for pending Workloads we just fetch the number stored in the cache. For running Workloads the proposal iterates over the whole heap - for large clusters it can be O(100k) Workloads, so this is costly.
