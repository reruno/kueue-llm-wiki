## Description

#### What type of PR is this?
/kind feature

#### What this PR does / why we need it:
This adds new `PodSet` grouping validation to `JobSet`, `MPIJob` and further validation `LeaderWorkerSet`, preventing the user from creating an invalid TAS grouping of the defined pod sets.

#### Which issue(s) this PR fixes:
Fixes #6242

#### Special notes for your reviewer:
The paths of the [group size](https://github.com/kubernetes-sigs/kueue/pull/7061/files#diff-19d79fc036ae0d9c40bec0358333338d2dec81eaba98207200ae3ff67e1888deR135) and [replica size](https://github.com/kubernetes-sigs/kueue/pull/7061/files#diff-19d79fc036ae0d9c40bec0358333338d2dec81eaba98207200ae3ff67e1888deR149) (>=1 pod set with 1 pod required) errors are not perfect, but there aren't any much better alternatives. What I also considered:
1. Report the group size error as a `spec` error. This
2. The replica size error (>=1 pod set with 1 pod required) cannot be reported on a specific field, because the size of the `PodSet` is sometimes calculated using multiple fields (for example parallelism, completions, replicas etc. for `JobSet`).

The separate commits for each job type can be made into separate pull requests if necessary, but I kept it in a single PR for now so it's easier to see the whole picture and how the validation methods generalizes to job types other than LWS.

#### Does this PR introduce a user-facing change?
```release-note
TAS: Introduce missing validation against using incompatible `PodSet` grouping configuration in `JobSet, `MPIJob`, `LeaderWorkerSet`, `RayJob` and `RayCluster`. 

Now, only groups of two `PodSet`s can be defined and one of the grouped `PodSet`s has to have only a single `Pod`.
The `PodSet`s within a group must specify the same topology request via one of the `kueue.x-k8s.io/podset-required-topology` and `kueue.x-k8s.io/podset-preferred-topology` annotations.
```
