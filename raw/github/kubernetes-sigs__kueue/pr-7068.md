  Actually I would like to refactor this and not put any WorkloadSlicing code into adapters, but in the generic code outside SyncJob. The issue with buildling code inside adapters is two fold:
  1. it warrants duplication per Job type
  2. it is not compatible with the generic JSON adapter we have https://github.com/kubernetes-sigs/kueue/blob/main/pkg/controller/admissionchecks/multikueue/externalframeworks/adapter.go for external Jobs. 
  
  So, while I understand it might be easier / simpler to hack inside the adapter, I think we should strive to design the solution in such a way that the code can be eliminated. 
  I think thinking of a better design is time well spent here.
  
  cc @tenzen-y
