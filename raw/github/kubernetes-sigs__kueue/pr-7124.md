  I'm not sure we want to drop durations for Config API, because:
  1. the config API is just dumped and parsed as text and mounted to the Pod as a file. So, it is not really editable by external go client libraries, even if we move to v1beta2.
  2. it is much easier for admins to express timeouts like "15m" rather than in seconds
  3. in integration tests we use values like 10ms, which makes the test much faster than if we had to use "1 second" as minimum value.
  
  Certainly, if we move the ConfigAPI to CRD then we should migrate, but this is not prioritzed yet. I would prefer to couple to that effort rather than v1beta2.
