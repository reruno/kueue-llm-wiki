## Description

#### What type of PR is this?
/kind bug

#### What this PR does / why we need it:
When a workload, in a queue with BestEffortFIFO enabled, has preemption targets, we keep it at the head of the queue until it schedules (or returns inadmissible).

See original proposal here: https://github.com/kubernetes-sigs/kueue/issues/6929#issuecomment-3360978483

#### Which issue(s) this PR fixes:
Fixes #7101, #6929

#### Special notes for your reviewer:
Let's discuss whether we want to put this behind a feature gate.

#### Does this PR introduce a user-facing change?
```release-note
Scheduling: With BestEffortFIFO enabled, we will keep attempting to schedule a workload as long as
it is waiting for preemption targets to complete. This fixes a bugs where an inadmissible
workload went back to head of queue, in front of the preempting workload, allowing
preempted workloads to reschedule
```

@gabesaba Thanks for the fix here! Do we want to add a max requeue limit for a sticky workload? so we can avoid dead loop in case some other issue happens, e.g. after hitting max requeue limit, can we mark the workload fail to not try again, or do not treat it as sticky anymore?

### Comment by [@amy](https://github.com/amy) — 2025-10-03T20:24:25Z

> @gabesaba Thanks for the fix here! Do we want to add a max requeue limit for a sticky workload? so we can avoid dead loop in case some other issue happens, e.g. after hitting max requeue limit, can we mark the workload fail to not try again, or do not treat it as sticky anymore?

Still need to parse how inadmissible workloads get reevaluated. But I notice this: https://github.com/kubernetes-sigs/kueue/blob/main/pkg/controller/core/workload_controller.go#L826 and 
```
// The events listed below could make workloads in the same cohort admissible.
// Then requeueWorkloadsCQ need to be invoked.
// 1. delete events for any admitted workload in the cohort.
// 2. add events of any cluster queue in the cohort.
// 3. update events of any cluster queue in the cohort.
// 4. update of cohort.
```

I like the idea of marking a workload as failed, and similar to the k8s scheduler, moving it to a more permanent queue for auditing or different retry logic. I think the k8s scheduler has 3 queues: active, backoff, and unscheduable. Unscheduable is only reevaluated if for instance you add a new node to the cluster.

Kueue seems to conceptually have 2 categories (admissible, inadmissible) rather than 3. The current inadmissible workload map seems like it acts like a backoff queue. Where it retries when workload quota conditions change, ie the snapshot changes. Separately, I think it also mixes in if clusterqueue/flavors configuration changes. 

Perhaps we should have a unscheduable queue where across snapshots, this particular workload fails scheduling. And split up things like clusterQueue/flavor config changes into this unscheduable queue. 

---

Regardless, I think this suggestion should be made in a followup issue/PR. And for us to prioritize merging this more immediate fix. 🙏 WDYT?

### Comment by [@gabesaba](https://github.com/gabesaba) — 2025-10-07T10:30:28Z

> @gabesaba Thanks for the fix here! Do we want to add a max requeue limit for a sticky workload? so we can avoid dead loop in case some other issue happens, e.g. after hitting max requeue limit, can we mark the workload fail to not try again, or do not treat it as sticky anymore?

I like this idea of a mechanism, though I don't see how to implement it accurately, as scheduling cycles are very fast and preemptions may take a long time (especially with graceful termination). How might we configure the limit in this case?

The only case where we consider the workload sticky is when it is waiting for preemptions to complete. Therefore, I'm not worried about an unschedulable workload getting stuck - as soon as it fails to schedule for a different reason, we no longer requeue it as sticky.
