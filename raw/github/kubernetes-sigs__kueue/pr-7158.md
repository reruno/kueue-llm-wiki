> > What about decoupling the tools go.mod into separate one, then enabling indirect updating only for that.

> But why not use pinversion? I think it's much easier.

The reason is Pinverion.go way is just a hack, and "tool" directive is the recommended way.
Additinally, AFAIK, the pinverion.go requires us executing "go install xyz" each for modules, but "tool" directive allows us to use via "go tool xyz" (w/o installing).

### Comment by [@mbobrovskyi](https://github.com/mbobrovskyi) — 2025-10-03T14:07:27Z

> The reason is Pinverion.go way is just a hack, and "tool" directive is the recommended way.

We’re still using Go tools; we just moved packages to direct dependencies. We don’t need to install these packages. Anyway, this is a temporary solution until https://github.com/dependabot/dependabot-core/issues/12050 is applied. I’m not sure if creating a separate module is a better solution.
