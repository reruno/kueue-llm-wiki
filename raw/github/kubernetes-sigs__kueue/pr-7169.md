> EDIT: asking becuase there is no mention of noPreemptionCandidates in the pre-existing code.
> 
> I like the code cleanups as well, but it seems the change is more involving that just shuffling the checks.

Well spotted - but it's still no change in behavior.

`preemptionMode` is an enum of 5 values:

https://github.com/kubernetes-sigs/kueue/blob/4872aeab6d47f6cdba01ed1ea78831e227befe7e/pkg/scheduler/flavorassigner/flavorassigner.go#L360-L370

The original code had `fit` and `.isPreemptMode()` (which covers `preempt` and `reclaim`). \
In my code, I use the _remaining_ values (`nofit` and `noPreemptionCandidates`) - because I found it simpler. \
But the logic differentiating the 3 former from the 2 latter remains as it was.
