```diff
+#### Story 3
+This KEP would enable Kueue to calculate and derive new resources for quota purposes based on the logical relationship between two existing resources when handling workload resource requests. For instance, if a workload requests 2 vGPU resources `nvidia.com/gpu`, with each vGPU allocated 1GB of gpu memory resource `nvidia.com/gpumem`, the total impact on gpu memory resources in this case should amount to 2GB. A new resource—total GPU memory `nvidia.com/total-gpumem` can be defined within ClusterQueues, and its value is calculated by multiplying the number of vGPUs by the memory size allocated to each individual vGPU.
```

  This story makes sense to me; this is a useful expansion of the resource expansion functionality.

  I'm not convinced adding the Operation at this level is the best choice.  `Transformations` is a map, so we can only have at most one `ResourceTransformation` struct for each input resource. 
  
  I think it would be better to instead go for a generalization of `Outputs` that would cleanly combine the new capabilities with the existing ones.  Unfortunately, I didn't have the foresight to make `Outputs` an array of structs, so we can't just add an `OperationType` and optional `OperationTarget` to it.   So my suggestion would be to define a structure that would capture everything (output resource name, operation type, and the other operand (resource name or resource quantity).  Then we add an array of these to `ResourceTransformation` (perhaps called `OutputSpecifications` ?) and eventually deprecate `Outputs` in favor of `OutputSpecifications`.
