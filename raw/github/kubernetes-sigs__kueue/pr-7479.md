#### What this PR does / why we need it:

The loose ReadyReplicas < 3 check allowed scaling up before workload deletion completed, causing a race condition. This fix ensures the scale-down is completely finished before attempting scale-up.

```diff
-			ginkgo.By("Wait for ReadyReplicas < 3", func() {
+			ginkgo.By("Wait for ReadyReplicas to be zero", func() {
```

  So I think what heppens is:
  1. we trigger scale down of StatefulSet from 3 to 0
  2. StatefulSet has no longer any pods, but the Workload still exists (unlikely by possible)
  3. before the workload is deleted we scale up the StatefulSet, and it creates the first pod
  4. the pod is attached to the old Workload
  5. the old Workload is deleted, thus and thus the Pod, no new Workload is created (somehow)
  
  And essentially the scale up is stuck. This condition is very unlikely because in (2.) the workload would usually be already deleted.
