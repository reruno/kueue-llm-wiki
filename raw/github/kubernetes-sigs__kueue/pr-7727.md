Nice test! but unfortunately it doesn't prove that the update works properly, mainly because the `TrainJob` API changed from Trainer v2.0.1 to v2.1.0. Let me explain:

**Trainer v2.0.1** uses the `podSpecOverrides` API to allow the user to inject job level config to the generated JobSet. Kueue uses it to add `NodeSelectors`, `SchedulingGates` etc in case there's any. 

As you can see in your `Before` example, Kueue adds an empty entry because there's nothing to add

>   podSpecOverrides:
>   - targetJobs:
>     - name: node

You can also see that there are no Kueue labels set in that object. This is because `podSpecOverrides` doesn't support injecting neither labels nor annotations. This is why they moved to `PodTemplateOverrides` in Trainer v2.1.0.

What kind of puzzles me is not seeing any `podTemplateOverride` entry in the `After` example. There should be at least one for the `node` job with the Kueue `podset` label... I guess this is because the TrainJob was already running and therefore the controller wasn't triggered...
