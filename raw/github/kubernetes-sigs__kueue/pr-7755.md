### Comment by [@kaisoz](https://github.com/kaisoz) — 2025-11-19T14:23:30Z

> @kaisoz What would you suggest as a good workflow to verify that the upgrade works?

I think a safe test would be to delete and reapply the TrainJob. You should see Kueue adding a new **podTemplateOverride** with, at least, the `podset` label. 

To be honest I'm no completely sure what happens with both the TrainJob and the Workload after the update... I guess Trainer v2.1.0 drops the deprecated `podSpecOverrides` entries. Probably, a suspend/unsuspend action is be enough to trigger a reconciliation in the Kueue TrainJob controller. The corresponding workload should be reachable since the object UID and definition look like didn't change.. could you try that?

However, as I said, I think the safest is delete and reapply the TrainJob, and then ensure that corresponding `podTemplateOverride` entry is there

### Comment by [@mimowo](https://github.com/mimowo) — 2025-11-19T15:00:24Z

> However, as I said, I think the safest is delete and reapply the TrainJob, and then ensure that corresponding podTemplateOverride entry is there

Yeah, this will work because the new binary of Kueue will add it. 

The main potentially problematic case is when we have a pre-existing TrainJob which was created and started with the old version, then the `PodSpecOverrides` gets lost during upgrade. This phenomenon is visible here: https://github.com/kubernetes-sigs/kueue/pull/7727#issuecomment-3548361711 - as the running TrainJob lost 
```
  podSpecOverrides:
  - targetJobs:
    - name: node
```
during the upgrade process.

However, it does not seem to have any catastrophic impact. The only I can see is when the TrainJob was half-started, then the first half of Pods before upgrade would have the overrides, and the second half not. However, it does not sound like a big issue for the integration in early versions.
