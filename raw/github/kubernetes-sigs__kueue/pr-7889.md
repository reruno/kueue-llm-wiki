#### What this PR does / why we need it:

When a `dependsOn` is defined on a JobSet and the `waitForPodsReady`
is configured in Kueue, the PodsReady condition should be udpated
only for the Jobs that are started, which is not the case currently,
all the jobs, including the ones with dependsOn that are not started
are being considered when checking if the respective Pods are ready.
This commit fixes the issue by ensuring that whenever there is some
updated in the job, it means the job has started to be managed and
it should be taken into account when defining if all the Pods are
ready.

### Comment by [@mimowo](https://github.com/mimowo) — 2025-12-17T05:59:31Z

Actually, yes, the PR still has some issues I think so it is better to revert it for now. Sorry for not catching it earlier.

I now see two problems:
1. recoveryTimeout will be used for the following Jobs rather than the "startupTimeout". RecoveryTimeout is rather meant to be small for recreating individual Pods. So I'm worried by this change we will force admins to configure high recoveryTimeout.
2. I think this change is intrinsically flaky user experience, because we would say PodsReady=True in a very short period of time. If the update to set PodsReady=True fails, then on the next Reconcile the Kueue's controller may already observe the next Job starting, and so it will never actually set PodsReady=True. 

So I propose to:
1. revert the PR
2. open KEP update for PodsReady to discuss how to approach Jobset with dependsOn
3. once we have the KEP update merged we will merge the adjusted implementation

As for "how to fix" given the problems above I'm thinking now we should rather just update PodsReady=False with a new message saying "Jobs job-1 is ready/completed, but still not all pods are ready" (or along the lines). This will update the LastTransitionTime for the PodsReady=False condition, and so the Job will not be evicted. This is my current thinking, but let's discuss more.
