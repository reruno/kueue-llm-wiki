  Copied from https://github.com/kubernetes-sigs/kueue/pull/7769/files#r2588885609:
  
  from @mimowo : This has some issues:
  
  the request for the submitter Job is not represented by Kueue Workload, so bypassing quota reservation, this may lead to subtle bugs for TAS
  it seems incompatible with LocalQueueDefaulting, because then Workload will be created for the submitter Job anyway, consequences are hard to predict
  it is incompatible with the configuration setting for manageJobsWithoutQueue: true, then it will just suspend the submitter Job
  I think it will be better to manager the submitter Job as another topLevelJob, and create a Workload for it.
  
  To achieve that we can implement the IsTopLevel for Job to lookup if parent it RayJob with autoscaling enabled.
  
  Long term I think integration for RayJob could work this way by default - to create two workloads, one for submitter Job, one for RayCluster.
  
  from @yaroslava-serdiuk :
  
  I think 2) and 3) comes only if RayJob integration is disabled, so the Submitter Job doesn't have ancestor.
  However the 1) is still valid, since the RayCluster doesn't include submitter Job requests.
  
  When RayJob integration is enabled, in order for isTopLevel() works, the Submitter Job need to have kueue label. This could be added by webhook.
