  Ok I understand the use case to prefer avoiding some nodes in a soft manner. And I understand some admins want to annotate some nodes as "to be avoided if possible".
  
  However, k8s already has API which could serve this purpose: `nodeAffinity.preferredDuringSchedulingIgnoredDuringExecution`. 
  
  Here I imagine the admin rathen than using pre-defined node label can use any like `example.com/node-to-be-avoided`, then then use `nodeAffinity.preferredDuringSchedulingIgnoredDuringExecution` to repel from such nodes.
  
  
  The benefits are:
  1. standard k8s API admins are familiar with
  2. flexibility within the workload, and between workloads
  3. it allows to get repelled from the nodes only to a subset of TAS workloads (high prioirity/ big workloads), allowing less important workloads to try running on the avoided nodes.
  
  So, I would like to have some summary in the alternatives sections if this alternative was considered, and what are the pros / cons.
  
  I imagine one possible con is the implementation complexity to fully support the API. However, I'm not totally clear this is more complex. We could just check which nodes match the selector defined by the `nodeAffinity.preferredDuringSchedulingIgnoredDuringExecution` prioritize them. I don't see much of a difference alorithmic-wise. 
  
  However, I would like to see some deeper comparison of the two options before we commit to the proposal here.
