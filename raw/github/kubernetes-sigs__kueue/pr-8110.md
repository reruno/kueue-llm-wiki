### Comment by [@TapanManu](https://github.com/TapanManu) — 2025-12-11T17:02:24Z

> I don't think this VAP would work.
> 
> You need VAP to enforce these policies in the workloads that users submit (Job, Deployment , etc)

Thanks @kannon92 for pointing that out! I originally targeted Workload to ensure we caught all integration types (like RayJob or MPIJob) with a single rule.

However, I agree that targeting the source objects (Job, Deployment, etc.) provides much better immediate feedback to the user.

I've updated the example to target the standard Kubernetes controllers (Job, Deployment, StatefulSet) which share the spec.template structure. This covers the most common use cases while keeping the CEL expression simple.

### Local Verification Report 🛠️

I verified the updated VAP policy locally using a **k3s v1.30** cluster. The policy correctly handles `requests`, `limits`, and blocks invalid workloads as expected.

**Test Environment:**
* Kubernetes: `v1.30.0+k3s1` (VAP enabled by default)
* Policy Target: `Job` (batch/v1) and `Deployment` (apps/v1)

**Results:**

<details>
<summary><b>✅ Test 1: Job with 'requests' (Allowed)</b></summary>

*Input:*
```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: job-with-requests
  namespace: vap-demo
spec:
  template:
    spec:
      restartPolicy: Never
      containers:
      - name: test-container
        image: busybox
        command: ["sleep", "5"]
        resources:
          requests:
            cpu: "100m"
            memory: "64Mi"
```
</details>
<details>
<summary><b>✅ Test 2: Job with 'limits' (Allowed)</b></summary>

*Input:*
```yaml
apiVersion: batch/v1
kind: Job

  name: job-with-requests
  namespace: vap-demo
spec:
  template:
    spec:
      restartPolicy: Never
      containers:
      - name: test-container
        image: busybox
        command: ["sleep", "5"]
        resources:
```
