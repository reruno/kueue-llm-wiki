### Comment by [@mimowo](https://github.com/mimowo) — 2025-12-12T08:35:51Z

Ok, so first of all I tested locally the fix and the finished workloads are now skipped on restart, which means the main bug is fixed. Regarding the extra check in `AddFromLocalQueue` it operates on `q.items`, and the only function adding those is [AddOrUpdate](https://github.com/kubernetes-sigs/kueue/blob/39c6fdd4fc87f51dd4181319b6051c35db96d5c2/pkg/cache/queue/local_queue.go#L46) which is called from the following places:
1. [AddLocalQueue](https://github.com/kubernetes-sigs/kueue/blob/39c6fdd4fc87f51dd4181319b6051c35db96d5c2/pkg/cache/queue/manager.go#L324-L326) where it is already guarded by `workload.IsAdmissible(&w)` check (added in this PR)
2. [AddOrUpdateWorkloadWithoutLock](https://github.com/kubernetes-sigs/kueue/blob/39c6fdd4fc87f51dd4181319b6051c35db96d5c2/pkg/cache/queue/manager.go#L448-L450), also already guarded
3. [RequeueWorkload(https://github.com/kubernetes-sigs/kueue/blob/39c6fdd4fc87f51dd4181319b6051c35db96d5c2/pkg/cache/queue/manager.go#L493) where it is not guarded explicitly, but RequeueWorkload should be only called by scheduler on fail attempt, so already in the heap. 

Still, there is [this check already in place](https://github.com/kubernetes-sigs/kueue/blob/39c6fdd4fc87f51dd4181319b6051c35db96d5c2/pkg/cache/queue/manager.go#L484C43-L484C62) which seems to serve similar purpose. I think we should adjust it for completeness. 

With all the above I don't see a gap currently, but indeed the code is error prone, so I'm considering adding the check. I'm just a bit hesitant about adding dead code, so probably throwing error might be good idea, similarly as we do in AddOrUpdateWorkloadWithoutLock.
