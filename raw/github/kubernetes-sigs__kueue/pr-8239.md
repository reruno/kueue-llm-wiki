<summary>Release notes</summary>

<blockquote>
<h2>v0.15.1</h2>
<p>Changes since <code>v0.15.0</code>:</p>
<h2>Changes by Kind</h2>
<h3>Feature</h3>
<ul>
<li>TAS: extend the information in condition messages and events about nodes excluded from calculating the
assignment due to various recognized reasons like: taints, node affinity, node resource constraints. (<a href="https://redirect.github.com/kubernetes-sigs/kueue/issues/8132">#8132</a>, <a href="https://github.com/sohankunkerkar"><code>@​sohankunkerkar</code></a>)</li>
</ul>
<h3>Bug or Regression</h3>
<ul>
<li>Fix <code>TrainJob</code> controller not correctly setting the <code>PodSet</code> count value based on <code>numNodes</code> for the expected number of training nodes. (<a href="https://redirect.github.com/kubernetes-sigs/kueue/issues/8145">#8145</a>, <a href="https://github.com/kaisoz"><code>@​kaisoz</code></a>)</li>
<li>Fix a performance bug as some &quot;read-only&quot; functions would be taking unnecessary &quot;write&quot; lock. (<a href="https://redirect.github.com/kubernetes-sigs/kueue/issues/8183">#8183</a>, <a href="https://github.com/ErikJiang"><code>@​ErikJiang</code></a>)</li>
<li>Fix the race condition bug where the kueue_pending_workloads metric may not be updated to 0 after the last
workload is admitted and there are no new workloads incoming. (<a href="https://redirect.github.com/kubernetes-sigs/kueue/issues/8049">#8049</a>, <a href="https://github.com/Singularity23x0"><code>@​Singularity23x0</code></a>)</li>
<li>Fixed a bug that Kueue's scheduler would re-evaluate and update already finished workloads, significantly
impacting overall scheduling throughput. This re-evaluation of a finished workload would be triggered when:
<ol>
<li>Kueue is restarted</li>
<li>There is any event related to LimitRange or RuntimeClass instances referenced by the workload (<a href="https://redirect.github.com/kubernetes-sigs/kueue/issues/8198">#8198</a>, <a href="https://github.com/mimowo"><code>@​mimowo</code></a>)</li>
</ol>
</li>
<li>Fixed the following bugs for the StatefulSet integration by ensuring the Workload object
has the ownerReference to the StatefulSet:
<ol>
<li>Kueue doesn't keep the StatefulSet as deactivated</li>
<li>Kueue marks the Workload as Finished if all StatefulSet's Pods are deleted</li>
<li>changing the &quot;queue-name&quot; label could occasionally result in the StatefulSet getting stuck (<a href="https://redirect.github.com/kubernetes-sigs/kueue/issues/8105">#8105</a>, <a href="https://github.com/mbobrovskyi"><code>@​mbobrovskyi</code></a>)</li>
</ol>
</li>
<li>MultiKueue via ClusterProfile: Fix the panic if the configuration for ClusterProfiles wasn't not provided in the configMap. (<a href="https://redirect.github.com/kubernetes-sigs/kueue/issues/8097">#8097</a>, <a href="https://github.com/mszadkow"><code>@​mszadkow</code></a>)</li>
<li>TAS: Fix handling of admission for workloads using the LeastFreeCapacity algorithm when the  &quot;unconstrained&quot;
mode is used. In that case scheduling would fail if there is at least one node in the cluster which does not have
enough capacity to accommodate at least one Pod. (<a href="https://redirect.github.com/kubernetes-sigs/kueue/issues/8172">#8172</a>, <a href="https://github.com/PBundyra"><code>@​PBundyra</code></a>)</li>
<li>TAS: fix bug that when TopologyAwareScheduling is disabled, but there is a ResourceFlavor configured with topologyName, then preemptions fail with &quot;workload requires Topology, but there is no TAS cache information&quot;. (<a href="https://redirect.github.com/kubernetes-sigs/kueue/issues/8195">#8195</a>, <a href="https://github.com/zhifei92"><code>@​zhifei92</code></a>)</li>
</ul>
<h3>Other (Cleanup or Flake)</h3>
<ul>
<li>Fix: Removed outdated comments incorrectly stating that deployment, statefulset, and leaderworkerset integrations require pod integration to be enabled. (<a href="https://redirect.github.com/kubernetes-sigs/kueue/issues/8054">#8054</a>, <a href="https://github.com/IrvingMg"><code>@​IrvingMg</code></a>)</li>
</ul>
<h2>v0.15.0</h2>
<p>Changes since <code>v0.14.0</code>:</p>
<h2>Urgent Upgrade Notes</h2>
<h3>(No, really, you MUST read this before you upgrade)</h3>
<ul>
<li>
<p>MultiKueue: validate remote client kubeconfigs and reject insecure kubeconfigs by default; add feature gate MultiKueueAllowInsecureKubeconfigs to temporarily allow insecure kubeconfigs until v0.17.0.</p>
<p>if you are using MultiKueue kubeconfigs which are not passing the new validation please
enable the <code>MultiKueueAllowInsecureKubeconfigs</code> feature gate and let us know so that we can re-consider
the deprecation plans for the feature gate. (<a href="https://redirect.github.com/kubernetes-sigs/kueue/issues/7439">#7439</a>, <a href="https://github.com/mszadkow"><code>@​mszadkow</code></a>)</p>
</li>
<li>
<p>The .status.flavors in LocalQueue is deprecated, which will be removed in the future release.</p>
</li>
</ul>
<p>You can consider migrating from the field usage to VisibilityOnDemand. (<a href="https://redirect.github.com/kubernetes-sigs/kueue/issues/7337">#7337</a>, <a href="https://github.com/iomarsayed"><code>@​iomarsayed</code></a>)</p>
