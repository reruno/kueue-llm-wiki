### Comment by [@trasc](https://github.com/trasc) — 2023-06-15T20:14:15Z

> Can you drop the commit for finalizers? Or put it in a separate PR that we can consider for the next release?
> 
> I'm concerned that we don't have exhaustive testing around finalizers and that we are expecting the workload controllers to handle them. When dealing with finalizers, we have to consider that controllers could fail at any point, and there should be a mechanism to retry deleting the finalizers. In general, for kueue we have two scenarios: when removing jobs and when suspending jobs.
> 
> We are getting close to our target deadline for the release 0.4 and I think this PR is adding too much risk.

The finalizes are also helpful in  recovering from a "mid stop" error, essentially if a workload has the finalizer set  and isSuspended, then the restore of PodInfos and finalizer removal are done again.

If a job gets deleted, we select all of it's child workloads and remove the finalizers. This is tested in test/integration/controller/job/job_controller_test.go L1876.

But if you insist I can try to extract this in another PR

  So today there are 3 steps: flip suspend, reset start time, change node selector. The reset start time is rather a hack from kueue, because the Job API doesn't actually support mutating node selectors when a job has already started.
  
  The Job API does this to prevent a situation where you quickly flip suspend, change selectors and unsuspend. Then you could end up with running pods that have different templates.

There are possibilities for races here, but I think they are solvable. Perhaps we can pursue this in the next k8s release (1.29).
