<summary>Release notes</summary>
<p><em>Sourced from <a href="https://github.com/kubernetes-sigs/kind/releases">sigs.k8s.io/kind's releases</a>.</em></p>
<blockquote>
<h2>v0.31.0</h2>
<p>This release contains dependency updates and defaults to Kubernetes 1.35.0.</p>
<p>Please take note of the breaking changes from Kubernetes 1.35, and how to prepare for <strong>future</strong> changes to move off of the deprecated kubeam v1beta3 in favor of v1beta4. We will include updated reminders for both again in subsequent releases.</p>
<!-- raw HTML omitted -->
<p>The default node image is now <code>kindest/node:v1.35.0@sha256:452d707d4862f52530247495d180205e029056831160e22870e37e3f6c1ac31f</code></p>
<!-- raw HTML omitted -->
<p>Kubernetes <a href="https://kubernetes.io/blog/2025/12/17/kubernetes-v1-35-release/#removal-of-cgroup-v1-support">will be removing cgroup v1 support</a>, and therefore kind node images at those versions will also be dropping support.</p>
<p>You can read more about this change in the Kubernetes release blog: <a href="https://kubernetes.io/blog/2025/12/17/kubernetes-v1-35-release/#removal-of-cgroup-v1-support">https://kubernetes.io/blog/2025/12/17/kubernetes-v1-35-release/#removal-of-cgroup-v1-support</a></p>
<p>If you must use kind on cgroup v1, we recommend using an older Kubernetes release for the immediate future, but we also strongly recommend migrating to cgroup v2.</p>
<p>In the near future as Kubernetes support dwindles, KIND will also clean up cgroup v1 workarounds and drop support in future kind releases and images, regardless of Kubernetes version.</p>
<p>Most stable linux distros should be on cgroupv2 out of the box.</p>
<p>This is a reminder to use pinned images by digest, see the note below about images for this release.</p>
<!-- raw HTML omitted -->
<p><strong>WARNING</strong>: Future kind releases will <a href="https://redirect.github.com/kubernetes-sigs/kind/issues/3847">adopt kubeadm v1beta4</a> configuration, <a href="https://github.com/kubernetes/kubeadm">kubeadm</a> v1beta4 has a breaking change to <code>extraArgs</code>: <a href="https://kubernetes.io/blog/2024/08/23/kubernetes-1-31-kubeadm-v1beta4/">https://kubernetes.io/blog/2024/08/23/kubernetes-1-31-kubeadm-v1beta4/</a>.</p>
<p>If you use the <code>kubeadmConfigPatches</code> feature then you may need to prepare for this change.
We recommend that you use versioned config patches that explicitly match the version required.</p>
<p>KIND uses kubeadm v1beta3 for Kubernetes 1.23+, and will likely use v1beta4 for Kubernetes 1.36+
The exact version is TBD pending work to fix this but expected to be 1.36.
It will definitely be an as-of-yet-unreleased Kubernetes version to avoid surprises, and it will not be on a patch-release boundary.</p>
<p>KIND <em>may</em> still work with older Kubernetes versions at v1beta2, but we no longer test or actively support these as Kubernetes only supports 1.32+ currently: <a href="https://kubernetes.io/releases/">https://kubernetes.io/releases/</a></p>
<p>You likely only need v1beta3 + v1beta4 patches, you can take your existing patches that work with v1beta3, explicitly set <code>apiVersion: kubeadm.k8s.io/v1beta3</code> in the patch at the top level, and make another copy for v1beta4. The v1beta4 patch will need to move <code>extraArgs</code> from a map to a list, for examples see: <a href="https://kubernetes.io/docs/reference/config-api/kubeadm-config.v1beta4/">https://kubernetes.io/docs/reference/config-api/kubeadm-config.v1beta4/</a></p>
<p>For a concrete example of kind config with kubeadm config patch targeting both v1beta3 and v1beta4, consider this simple kind config that sets verbosity of the apiserver logs:</p>
<pre lang="yaml"><code>kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
kubeadmConfigPatches:
# patch for v1beta3 (1.23 ...)
- |
  kind: ClusterConfiguration
  apiVersion: kubeadm.k8s.io/v1beta3
  apiServer:
    extraArgs:
&lt;/tr&gt;&lt;/table&gt; 
</code></pre>
