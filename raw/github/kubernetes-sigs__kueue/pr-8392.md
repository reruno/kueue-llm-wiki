  As of right now we have 2 distinct cases of the usage of "DeleteWorkload":
  1) in the update, where we remove the workload from the queue/cq, but it still exists in the client and may be one day required to trigger a QueueAssociatedInadmissibleWorkloadsAfter. If we delete assignment caching, we risk not having the mapping when that happens and would have to rely on using the wl info to get the associated queue.
  2) when deleting the workload - that is when we can safely forget the workload all together, including the assignment mapping. Side note: we also trigger QueueAssociatedInadmissibleWorkloadsAfter here - if  (1) had occurred before and removed the assignment caching, this is where the issue would pop up.
  
  Instead of doing a delete vs forget split, I could alternatively do a QueueAssociatedInadmissibleWorkloadsAfter split using either the wl or only it's key, former would be used for update and latter for proper delete.
  
  Mind you that once the refactor of wl controller commences the separation of Delete and Forget will lilely become obsolete as the Update method will undergo major reworks.
