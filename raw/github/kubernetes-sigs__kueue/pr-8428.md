### Review by [@mbobrovskyi](https://github.com/mbobrovskyi) (commented) — 2026-01-07T18:41:48Z

- **hack/migrate-to-v1beta2.sh:1** by [@mbobrovskyi](https://github.com/mbobrovskyi) — 2026-01-07T18:41:48Z
  Not exactly. Kubernetes stores each object in etcd using only one version – the storage version that was active at the time it was written. Conversions are available to translate between versions on read or write operations. This allows users to retrieve objects in multiple versions if needed, with the API server handling the conversion on the fly.
  However, by default, Kubernetes does not automatically convert existing objects when the storage version changes. If a resource was created in an older version like v1beta1, it remains stored as v1beta1 in etcd until explicitly updated.
  The only way to migrate it is to update the object, which rewrites it in the current storage version. For example, if you update a field in the object, Kubernetes should automatically handle the conversion during the write operation.
  
  Here is official instruction how to upgrade existing objects to a new stored version:
  https://kubernetes.io/docs/tasks/extend-kubernetes/custom-resources/custom-resource-definition-versioning/#upgrade-existing-objects-to-a-new-stored-version
