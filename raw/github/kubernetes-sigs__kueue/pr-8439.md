## Description

The performance test creates ~15,000 workloads (5 cohorts × 6 queues × 500 workloads). Under heavy reconciliation, the default kube-apiserver limits (max-requests-inflight=400, max-mutating-requests-inflight=200) cause the envtest API server to become overwhelmed, resulting in "http2: client connection lost" errors when pending requests queue up
and connections time out.

## Pull request overview

This PR fixes a flaky performance test by correcting how the recorder handles context cancellation. Previously, the recorder treated all context cancellations as errors, including normal test teardown, which caused failures when context cancellation occurred during shutdown while API requests were queued.

- Distinguishes between actual recording timeout (context.DeadlineExceeded) and normal shutdown (context.Canceled)
- Returns an error only when the recording exceeds its configured maximum duration
- Returns nil for context cancellations from normal test teardown


  Thanks for the PR!
  
  I agree with the intent here – we want to treat deadline exceedance as a recorder failure while letting client disconnects succeed gracefully.
  
  However, the current check makes everything except `DeadlineExceeded` return `nil`. Since `ctx.Err()` today is exhaustive (only `Canceled` or `DeadlineExceeded`), it works, but it's a bit fragile: if in the future someone wraps the error or a new cancellation reason appears, it would silently return `nil` even in failure cases.
  
  I think it's safer to invert the logic and explicitly allow only `context.Canceled`.
  
  ```suggestion
  			if !errors.Is(ctx.Err(), context.Canceled) {
  				return ctx.Err()
  			}
  			return nil
  ```
  
  WDYT?
