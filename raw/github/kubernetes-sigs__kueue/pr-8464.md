### Comment by [@mbobrovskyi](https://github.com/mbobrovskyi) — 2026-01-13T08:25:22Z

> **Important**: it requires that the worker clusters don't use the WorkloadPriorityClass instances.

There is no difference whether we use WPC or not on the worker when copying the priority from the manager. We copy the values from the manager’s Workload to the worker’s Workload, so there is no need to fetch the WPC or resolve its value on the worker.

If we were copying the label to the Job, then we would need to do that. In that case, if the user does not have the WPC on the worker, they would encounter an error when trying to update the Job.

The difference between the manager and the worker is that on the worker we use a prebuilt Workload, and after creating it we copy the Job. On the manager, we create the Job first and then the Workload.

Because of this, the manager should be the source of truth, not the worker. The worker’s responsibility is just to run the Job.

We have the same situation with elastic jobs: we only copy data on the Workload side. We do not copy it to the Job.

That's why I think that this comment on release-note doesn't need.

### Comment by [@mimowo](https://github.com/mimowo) — 2026-01-13T11:44:33Z

> There is no difference whether we use WPC or not on the worker when copying the priority from the manager. 

Have you tested that? I'm not saying that the copying wouldn't work, my concern is that when a user specifies the WPC on the worker, then the local Job reconciler will override the manager is currently doing in this PR.

So I'm thinking about the scenario:
1. user creates a Job on the manager with priority 100 via WPC
2. Kueue admits the Job to the worker1 and the job is running
3. user updates on the manager the priority to 200
4. the new code updates the priority on the worker to 200
5. the Job reconciler is triggered on the worker (say a Pod failed and need to be replaced)
6. the triggered Job reconciler overrides the priority back to 100 <- I think this is problematic in the current PR

This is just thought experiment, I have not tested.

Please check on a running cluster how this scenario behaves, so that we can guide the future work.

### Review by [@mimowo](https://github.com/mimowo) (commented) — 2026-01-13T08:02:38Z

This PR looks like the right direction, so we can merge, but it does not seem to address the issue fully. 

The reason is that it creates inconsistency where the Job on the management cluster has a different priority than the Job on the worker cluster.
I think it only works relying on the fact that the worker clusters don't specify the WPC instances, then the local Job reconciler ignores the workload. Otherwise I think there will be a constant back-and-forth between the updates from the two controllers - the workload reconciler on the manager, and the Job reconciler on the worker.

So, I would propose to remove `Fixes #7429` and;
- re-evaluate if this is actually a problem
- if so then consider fixing, some idea: make the Job reconciler on the worker update the Job rather than the workload if it detects the priority was changed by MultiKueue,
