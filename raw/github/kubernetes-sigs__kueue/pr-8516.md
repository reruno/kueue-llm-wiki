### Comment by [@sohankunkerkar](https://github.com/sohankunkerkar) — 2026-01-12T07:15:16Z

>Thank you for describing that. Basically, it would be better to avoid breaking existing NON-alpha features (LWS integration is not alpha) even if it is an enhancement.
So, I would like to investigate the root cause of the reason why only 1.35 cluster handles this as unexpected behavior, then based on the investigation, I would like to take the following either approach:

I’m leaning toward option 2, as the test should be removed from release-0.14 until proper priority update support is added to the LWS reconciler. This is definitely something that was introduced in 1.35 and is causing this regression. I suspect Kubernetes enables the `MaxUnavailableStatefulSet` feature beta-on-by-default in 1.35. This feature changes StatefulSet rolling update behavior to avoid keeping pods in a Terminating state unnecessarily.

On Kubernetes 1.32–1.34, when the LWS label was updated, the StatefulSet rolling update created transient Terminating pods, which triggered the Pod controller’s `updateWorkloadToMatchJob` path and updated the workload priority as a side effect. On Kubernetes 1.35, the optimized rolling update no longer creates this transient state, so the priority update path is never triggered.
