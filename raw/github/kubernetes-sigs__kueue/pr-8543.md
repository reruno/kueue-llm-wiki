## Description

The preemption integration test was flaky because pending workloads assumed by the scheduler were not being cleaned from the cache on deletion, preventing ClusterQueue finalizer removal. This change fixes the code where cache.DeleteWorkload on deletion. The operation is idempotent. If the workload was never in the cache, it's a no-op.

## Pull request overview

This PR fixes a flaky integration test where ClusterQueue deletion was getting stuck. The root cause was that pending workloads that had been "assumed" by the scheduler (optimistically reserved quota but not yet persisted) were not being cleaned from the cache when deleted, preventing the ClusterQueue finalizer from being removed.

**Changes:**
- Modified workload deletion handler to unconditionally attempt cache cleanup instead of only cleaning up finished or quota-reserved workloads
- Updated error logging logic to suppress errors for pending assumed workloads where cache deletion is idempotent

  While this change improves the situation by attempting to delete all workloads from cache unconditionally, there's a potential robustness issue in the cache implementation. The cache.DeleteWorkload method in pkg/cache/scheduler/cache.go returns early with ErrCqNotFound before calling cleanupAssumedState if the ClusterQueue is not found. This means that in edge cases where the CQ lookup fails unexpectedly, assumed workloads would remain in the assumedWorkloads map.
  
  Consider modifying cache.DeleteWorkload to call cleanupAssumedState before checking if the CQ exists, ensuring assumed workload references are always cleaned up regardless of CQ state. This would make the cleanup more robust and prevent potential leaks in the assumedWorkloads map.
