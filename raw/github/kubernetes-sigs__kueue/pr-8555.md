## Description

**What type of PR is this?**
/kind cleanup

What this PR does / why we need it:
This PR adds a Go-based website link verification tool and integrates it into the existing `make verify ` workflow..

It introduces hack/verify-website-links.go, which:
  - Runs make site-server to start the Hugo dev server.
  - Waits for http://localhost:$PORT/ to become reachable.
  - Invokes linkchecker against the local site with --no-warnings, configurable --threads and --timeout, and ignored mailto: / tel: URLs.
  - Uses a temporary directory for Hugo caches and resources so make verify does not produce untracked files.
  - Gracefully shuts down the Hugo server on completion or error.

The PR also adds a `verify-website-links` target to the Makefile and includes it in the `verify` aggregate target, allowing existing Prow presubmits to run the check without introducing a new CI workflow.

### Comment by [@yashnib](https://github.com/yashnib) — 2026-01-26T23:31:47Z

> Maybe we can simply set [`disablePathToLower = true`](https://gohugo.io/configuration/all/#disablepathtolower) in the `site/hugo.toml` and avoid these `CN` to `cn` modifications?

@alexandear You've raised a good point here. I checked that `zh-CN` is the canonical BCP 47 locale code format; `zh-cn` is commonly used but isn’t the recommended (canonical) form.

The `disablePathToLower = true` setting in Hugo is designed to prevent Hugo from lowercasing URL paths derived from content file paths. However, it doesn't apply to language path prefixes in multilingual sites.

In Hugo's multilingual configuration:
```toml
[languages.zh-CN]
    contentDir = "content/zh-CN"
```

The language key `zh-CN` determines the URL prefix for that language. Hugo processes this language key separately from regular content paths and always lowercases it to generate URLs like `/zh-cn/docs/...`, regardless of the `disablePathToLower` setting.

This is essentially a Hugo limitation/design decision - the `disablePathToLower` option only affects:
- Regular content paths (e.g., `/docs/My-Page/` stays as-is instead of becoming `/docs/my-page/`)
- Section names
- Taxonomy terms

But it does **not** affect the language prefix derived from the language configuration key.

I verified this by building the site locally with `disablePathToLower = true` - the output directory was still `public/zh-cn/` (lowercase), not `public/zh-CN/`.                                             
                                                                      

I've added a lightweight reverse proxy between the link checker and Hugo server that rewrites /zh-CN/ URLs to /zh-cn/ before forwarding requests. This lets links use either case while Hugo serves everything at lowercase paths.

  Checking the live site daily is useful for catching external links that break over time (e.g., a third-party site goes down or changes their URLs). We could definitely add that as a scheduled job.
  However, I don't think we should use that approach in the linkchecker for the following reasons:
  
  - Broken links would reach production first. If we only check the live site, a PR with broken internal links gets merged and deployed before we detect the problem. The current approach catches these in the PR itself, so they never go live.
  - Slower feedback loop. With daily checks, a broken link could be live for up to 24 hours before we even know about it. With this approach, the author knows within minutes and can fix it before merging.
  - Harder to fix. When a daily scan finds a broken link, someone has to figure out which commit introduced it, then open a new PR to fix it. With the current approach, the person who introduced it is already in context and can fix it immediately. 
  
  Checking the live website is definitely easier as it avoids making sure all the dependencies for hosting the server are installed, but we will only catch issues after they're deployed which undermines the point of CI.
  That said, if we want to catch external links that go stale over time (sites we link to that later break), a scheduled check against the live site would be a good addition—just not a replacement for the CI check.

+// extractTarGz extracts a .tar.gz archive to the destination directory
+func extractTarGz(r io.Reader, destDir string) error {
+	gzr, err := gzip.NewReader(r)
+	if err != nil {
+		return err
+	}
+	defer gzr.Close()
+
+	tr := tar.NewReader(gzr)
+	for {
+		header, err := tr.Next()
+		if err == io.EOF {
+			break
+		}
+		if err != nil {
+			return err
+		}
+
+		// Sanitize the path to prevent path traversal attacks
+		cleanName := filepath.Clean(header.Name)
+		if strings.HasPrefix(cleanName, "..") || filepath.IsAbs(cleanName) {
+			return fmt.Errorf("tar entry %q attempts path traversal", header.Name)
+		}
+		target := filepath.Join(destDir, cleanName)
+		// Double-check that the resolved path is within destDir
+		if !strings.HasPrefix(target, filepath.Clean(destDir)+string(os.PathSeparator)) {
+			return fmt.Errorf("tar entry %q escapes destination directory", header.Name)
+		}
+
+		switch header.Typeflag {
+		case tar.TypeDir:
+			if err := os.MkdirAll(target, 0755); err != nil {
+				return err
+			}
+		case tar.TypeReg:
+			if err := os.MkdirAll(filepath.Dir(target), 0755); err != nil {
+				return err
+			}
+			f, err := os.OpenFile(target, os.O_CREATE|os.O_RDWR|os.O_TRUNC, os.FileMode(header.Mode))
+			if err != nil {
+				return err
+			}
+			if _, err := io.Copy(f, tr); err != nil {
+				f.Close()
+				return err
+			}
+			f.Close()
+		case tar.TypeSymlink:
+			if err := os.MkdirAll(filepath.Dir(target), 0755); err != nil {
+				return err
+			}
+			// Remove existing symlink if it exists
+			os.Remove(target)
+			if err := os.Symlink(header.Linkname, target); err != nil {
+				return err
+			}
+		}
+	}
+	return nil
