#### What this PR does / why we need it:
See the issue https://github.com/kubernetes-sigs/kueue/issues/8443 .

TL;DR
**Goal**: Get Ray Serve autoscaling to work with Kueue.

**Configuration**:
  - KubeRay 1.4 and Kueue 0.14.3 (with ElasticJobsViaWorkloadSlices feature gate).
  - The Ray Service CR (and the Ray Cluster CR it creates) have spec.enableInTreeAutoscaling: true and the annotation kueue.x-k8s.io/elastic-job: "true"

**Behavior**:
  - When the Ray Serve workload terminates (and the associated Ray Cluster is terminated), KubeRay launches a redis-cleanup batch job, which runs a pod to clean up the redis-cleanup config used by the Ray Cluster (RC), which was launched by Ray Serve.
  - With Kueue, the redis-cleanup pod remains schedule-gated and in Pending state. Some logic in KubeRay eventually kills it but the cleanup never happens.
 - This is because Kueue does not remove the pod scheduling gate in the redis-cleanup pod.

**Analysis**:
Kueue does not to handle autoscaling for the redis-cleanup pod; so, it need not be treated as an elastic job.

**Solution**:
Modify the batch job webhook in Kueue to remove the scheduling gate from the job if (a) it is owned by a ray.io/v1/RayCluster, AND (b) has the label "ray.io/node-type = redis-cleanup"

Fixes #8443 (https://github.com/kubernetes-sigs/kueue/issues/8443)

Recap:
 * The logic of the fix has been reviewed. The e2e test is the only one that is open. 
 * The newly added e2e test aims to verify that, when a Ray Cluster (such as one spawned by a Ray Service) is terminated, the redis-cleanup job that is created does not have Kueue's scheduling gates. 

KubeRay creates a redis-cleanup job for a Ray Cluster under the following conditions:
* KubeRay was deployed with the `ENABLE_GCS_FT_REDIS_CLEANUP` env var.
* The Ray Cluster has the annotation `ray.io/ft-enabled: true` and the finalizer `ray.io/gcs-ft-redis-cleanup-finalizer`. 
* The Ray Cluster's head pod has the env vars relating to REDIS. The redis-cleanup job inherits from the head pod.

 * By running the e2e tests locally, I've ensured that the above conditions are met.
 * This PR's fix has been applied internally and validated as well.
