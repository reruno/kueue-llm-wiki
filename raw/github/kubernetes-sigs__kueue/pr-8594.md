## Description

The provisioning retry test was checking transient state that could be missed if workload_controller reconciles quickly:

1. RequeueState - cleared during eviction, not stable
2. state=Retry - reset to Pending by resetChecksOnEviction

The retry info (RequeueAfterSeconds, RetryCount) is preserved across eviction by resetChecksOnEviction, so these assertions are stable.

## Pull request overview

This PR fixes an optimistic concurrency conflict in the provisioning controller by wrapping PodTemplate update operations in a retry mechanism. The issue occurred when a PodTemplate was modified between Get and Update operations, causing flaky test failures.

**Changes:**
- Wrapped PodTemplate update logic in `retry.RetryOnConflict` to handle concurrent modifications
- Moved PodTemplate object allocation inside the retry loop to ensure fresh state on each retry attempt
- Simplified error handling by returning errors directly from the retry function

  @mimowo The divergence isn't within a single reconcile. It's different reconciles seeing different snapshots of the workload status. And importantly, the race is between different workload_controller reconciles, not between provisioning_controller and workload_controller.                                                                  
                                                                                                                                                                               
  Snapshot A (typical, no flake): workload_controller reads status before eviction happened:                                                                                 
    - AdmissionChecks[i].State = Retry                                                                                                                                         
    - RequeueAfterSeconds = X                                                                                                                                                          
                                                                                                                                                                               `NeedsRequeueAtUpdate() sees Retry → GetMaxRetryTime() returns a valid time → RequeueState.RequeueAt gets set`.                                                              
                                                                                                                                                                               
    Snapshot B (flake): a prior reconcile already ran eviction and reset the check:                                                                                            
    - AdmissionChecks[i].State = Pending                                                                                                                                       
    - RequeueAfterSeconds = X (preserved by resetChecksOnEviction)                                                                                                             
    - RetryCount > 0 (incremented by resetChecksOnEviction)                                                                                                                    
                                                                                                                                                                               
  Now `NeedsRequeueAtUpdate()` calls `GetMaxRetryTime()`, which skips Pending checks entirely and returns zero, and `RequeueState` never gets set. Test times out waiting for it.                                                                                                                                                            
                                                                                                                                                                               
  The provisioning_controller does the same thing in both cases. The difference is whether one workload_controller reconcile evicts before another one gets to set `RequeueState`. That's why treating "Pending + RetryCount > 0" as retry-eligible in GetMaxRetryTime() fixes this. It respects the retry info that `resetChecksOnEviction` intentionally preserves.
  
  One thing worth noting: `resetChecksOnEviction` also resets `LastTransitionTime` to eviction time. So after eviction, `GetMaxRetryTime` would calculate retry time as `eviction_time + RequeueAfterSeconds` rather than preserving the original deadline. I think this is correct semantically as eviction is a new event and we should wait before retrying but wanted to call it out in case you see it differently.
