**1. Repeated `HasExtendedResourcesBackedByDRA` calls**

`HasExtendedResourcesBackedByDRA` makes indexed list queries to the API server for every extended resource in every container. It's called in multiple places per workload lifecycle: `Reconcile()` (line 261), `Create()` predicate (line 968), `Update()` predicate (line 1035), `queueReconcileForPending()` (line 1330), and `AddLocalQueue()` in manager.go (line 420).

In `Reconcile()`, `HasExtendedResourcesBackedByDRA` is called first, then `GetResourceRequestsFromExtendedResources` essentially repeats the same DeviceClass lookups. Consider caching the result or combining these into a single pass.

https://github.com/kubernetes-sigs/kueue/blob/2f9299bb8fdb717197cc462bd0172246b94711d8/pkg/controller/core/workload_controller.go#L258-L266

---

**2. Unnecessary allocation in `HasExtendedResourcesBackedByDRA`**

```go
for _, container := range append(ps.Template.Spec.InitContainers, ps.Template.Spec.Containers...) {
```

`append` here creates a new slice allocation on every call. Since this is in hot paths (predicates), consider iterating separately:
```go
for _, containers := range [][]corev1.Container{ps.Template.Spec.InitContainers, ps.Template.Spec.Containers} {
    for _, container := range containers { ... }
}
```

https://github.com/kubernetes-sigs/kueue/blob/2f9299bb8fdb717197cc462bd0172246b94711d8/pkg/dra/extended_resources.go#L150

---

**3. `context.Background()` in event handlers**

The `Create`, `Update`, and `queueReconcileForPending` handlers call `HasExtendedResourcesBackedByDRA` with `context.Background()` instead of a cancellable context. If the controller is shutting down, these calls won't be cancelled. This is consistent with the existing DRA pattern in the file but worth noting.

https://github.com/kubernetes-sigs/kueue/blob/2f9299bb8fdb717197cc462bd0172246b94711d8/pkg/controller/core/workload_controller.go#L968-L970

---

**4. Missing `DRAExtendedResources` feature gate guard**

`GetResourceRequestsFromExtendedResources` at line 307 is called unconditionally within the DRA block, but there's a `features.Enabled(features.DRAExtendedResources)` check for `HasExtendedResourcesBackedByDRA` at line 259. If `DRAExtendedResources` is disabled but `DynamicResourceAllocation` is enabled and the workload has DRA templates, this will still process extended resources (making unnecessary API calls, though it won't cause bugs since no matching DeviceClasses will be found).

Consider guarding it:
```go
if features.Enabled(features.DRAExtendedResources) {
    extendedResources, replacedExtendedResources, extFieldErrs = dra.GetResourceRequestsFromExtendedResources(...)
}
```

https://github.com/kubernetes-sigs/kueue/blob/2f9299bb8fdb717197cc462bd0172246b94711d8/pkg/controller/core/workload_controller.go#L306-L322

## Pull request overview

This pull request adds support for Extended Resources backed by Dynamic Resource Allocation (DRA) to Kueue. This enhancement allows workloads to request DRA devices using standard `resources.requests` (e.g., `example.com/gpu: 1`) instead of explicit ResourceClaimTemplates, when DeviceClasses have the `extendedResourceName` field set.

**Changes:**
- Implements DeviceClass cache to map extended resource names to DeviceClass names
- Adds automatic conversion of extended resources to DRA logical resources
- Introduces workload re-queuing when DeviceClasses are created or updated
- Extends e2e tests to cover extended resource scenarios
