#### What this PR does / why we need it:

I introduced the `kueue.x-k8s.io/pod-index-offset` annotation for TAS, which is used only by MPIJob integration currently.
The MPIJob integration will inject the annotation to the MPIJob Pod template only when MPIJob is run in LauncherAsWorker mode and doesn't have `kueue.x-k8s.io/podset-group-name` annotation because the offset across multiple replicas is handled by the PodSet Group mechanism in that case.

### Comment by [@tenzen-y](https://github.com/tenzen-y) — 2026-01-16T19:25:52Z

> In particular, I'm wondering what is the difference in user observable behavior between this bug and the follow uo issue you opened.

Basically, both the first issue (^1)  and the second issue (^2) are scenarios where MPIJob uses TAS and runLauncherAsWorker, but only the second issue (^2) is for the PodSet Group mechanism (`kueue.x-k8s.io/podset-group-name`).

The primary differences between the first issue (^1) and the second issue (^2) are that the first issue (^1) bug doesn't work with rank-based ordering within Worker, which doesn't care about PodSet Group rank-ordering scenarios.

OTOH, the second issue (^2) indicates that Kueue ungater doesn't handle correctly in the scenarios where the rank-based ordering across PodSets (Launcher & Worker) (a.k.a. TAS PodSet Group).

^1: first issue: https://github.com/kubernetes-sigs/kueue/issues/8471 (this PR target)
^2: second issue: https://github.com/kubernetes-sigs/kueue/issues/8635
