#### What this PR does / why we need it:
Adds MultiKueue support for LeaderWorkerSet workloads.

- LeaderWorkerSet creates one workload per replica (e.g., an LWS with replicas: 3 creates 3 workloads). Workloads that share the same LWS owner form a workload group. All workload group members must be dispatched to the same worker cluster atomically. To achieve this:

  1. Wait for all workload group members to exist on the manager cluster.
  2. The primary workload (index 0) nominates the target worker cluster and creates the remote workload there.
  3. Once the remote workload is admitted on the worker, the primary’s ClusterName is set.
  4. All other workload group members observe the ClusterName and follow to the same worker cluster.
  5. The LWS is created on the worker cluster only after a cluster has been selected.

  The `WorkloadGroupIndexAnnotation` stores the numeric index for each workload, ensuring correct primary selection.

- The `MultiKueueOriginOwnerUIDAnnotation` annotation has been introduced because LWS creates workloads with UID-based names. Since copying the LWS to the remote generates a new UID for the remote LWS, we need to know the UID of the origin LWS (the one in the manager) in order to properly synchronize workloads from the remote with the manager. This allows the remote LWS reconciler to calculate workload names that match the prebuilt workloads from the manager.

  The `PrebuiltWorkloadLabel` is not used in this case because LWS creates one workload per replica. As a result, the LWS reconciler would be able to find only a single workload from the manager, but not the workloads corresponding to individual replicas. Using `PrebuiltWorkloadLabel` would require extending its design to support multiple replicas, which in turn would require including information about all replicas in the LWS labels.
- The `JobOwnerGVKAnnotation` and `JobOwnerNameAnnotation` are needed to ensure correct adapter lookup when the manager LWS is deleted. LWS workloads have multiple owners (LWS + Pods) because LWS uses SetOwnerReference instead of SetControllerReference to support the pod group pattern. When the manager LWS is deleted, Kubernetes GC removes only the LWS owner reference—the workload survives because the Pod owners still exist.

  The MultiKueue `adapter` function then falls back to refs[0], which is now a Pod, returning the Pod adapter instead of the LWS adapter. The Pod adapter’s SyncJob() cannot delete the remote LWS, leaving it orphaned on the worker cluster.
  
  The annotations preserve the original job owner GVK and name directly on the workload, ensuring that the correct adapter is returned regardless of changes to the owner references.

  Under normal circumstances, getting a workload index of -1 is not expected. The function returns -1 when one of the following conditions is met:
  
  1. wl == nil
  2. wl.Annotations == nil
  3. The annotation component-workload-index is not set
