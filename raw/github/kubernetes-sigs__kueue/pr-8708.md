## Description

#### What type of PR is this?
/kind bug

#### What this PR does / why we need it:
Users are flooded with reconciler errors that cannot be easily hidden with configuration. This PR overrides the log level of reconciler error and sets it to warning by using a custom zap.Core wrapper. 

This PR also introduces logs capturing to integration test logger, so they can be easily used in assertions. These capturing is used in newly proposed integration test checking that the issue is resolved.

#### Which issue(s) this PR fixes:
Fixes #8559

#### Special notes for your reviewer:
The fact that reconciler errors consistently are thrown in the proposed integration test might be a coincidence. A better test candidate idea is welcomed (I still lack knowledge about the system to judge that, but I guess it will not be a problem to move it when it becomes flaky). 

Maybe it is also worth it to disable stack trace for those errors?

#### Does this PR introduce a user-facing change?
```release-note
Observability: do not use "error" for logging when an update request fails because the underlying object was modified
concurrently. Now we use V3 logging level for such cases and don't print stracktrace. 

This is an example log message: "Operation cannot be fulfilled on podtemplates \"job\": the object has been modified;
please apply your changes to the latest version and try again"
```

### Comment by [@dkaluza](https://github.com/dkaluza) — 2026-01-28T09:33:44Z

@mimowo JFYI I have looked into the proposed LogSink based solution. Indeed it is an equivalent of the used zap core. But I think it would be less clear, as we would have to basically re-implement some of the zapr functionalities as the implementation of LogSink there is private ([source](: https://github.com/go-logr/zapr/blob/master/zapr.go#L71)). Probably one can implement this with logr logger in a wrapper manner, or by retrieving sink from the already built logger and creating a copy with a new sink. 

However I'm not convinced it is worth it - sure in this way we can replace zap with some other logger easier, but at the same time implementation will be a little more complex and further extensions will be harder - zapcore gives us more capabilities as a specific implementation. At the same time replacement with other implementation should be straight forward, as this is just the setup - we are still using logr interface for logging.  So my recommendation would be to go with zapcore based solution.
