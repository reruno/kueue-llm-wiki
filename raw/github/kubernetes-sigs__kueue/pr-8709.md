### Comment by [@gabesaba](https://github.com/gabesaba) — 2026-01-21T13:17:33Z

This solution works, but has the same issues @mimowo raised in this thread: https://github.com/kubernetes-sigs/kueue/pull/8484#discussion_r2698841024

We need to have some combination of the suggestions @mimowo made here: https://github.com/kubernetes-sigs/kueue/pull/8484#discussion_r2698873926


> * filter for non-TAS pods (TAS pods are already handled by TAS Workloads)
> * bulk the requests in time, like 1min or so.
> * probably only requeue for ClusterQueues which are affected by the change (using Flavors matching the affected nodes)

1 is already accomplished via event filters. I think 3 is quite tricky.

My recommendation would be to go with option 2. You may be able to use libraries from client-go's workqueue: https://pkg.go.dev/k8s.io/client-go/util/workqueue#pkg-overview, add a dummy item (since, without implementing 1, we are requeueing everything anyway), and then requeue everything (in a batched manner) when the dummy item is picked up for work

Ofc you could parametrize the workqueue (or entrire controller) by the batch time, and for testing use smaller, say 5s, while on prod we would use something like 1min.

## Pull request overview

This PR fixes a bug where inadmissible workloads were not automatically requeued when non-TAS pods terminated or were deleted, potentially causing workload starvation. The fix adds queue manager access to the NonTasUsageReconciler and calls QueueInadmissibleWorkloads after the cache is updated, following the same pattern used in the TAS ResourceFlavor controller.

**Changes:**
- Added queue manager to NonTasUsageReconciler to enable requeuing of inadmissible workloads
- Implemented automatic requeue when non-TAS pods terminate or are deleted, freeing capacity
- Removed workaround code from integration tests that manually triggered requeuing

  > cc @mimowo for thoughts on this pattern, vs using workqueue directly
  
  I think this is quite clever actually, saves the boilerplate code and reduces the chances of mistakes.
  
  My only hesitation was that "we are doing something weird, why no one else is doing that", but I think we already do something like this in the [`BootstrapCerts`](https://github.com/kubernetes-sigs/kueue/blob/0757a82bb2c31f79433fda3c49aab90cce19e7af/pkg/util/cert/cert.go#L46C6-L46C20) where we instantiate the certs rotator which underneath is also a reconciler, see [here](https://github.com/open-policy-agent/cert-controller/blob/b8e7de3b81fc02e124f891d989dea520f596113b/pkg/rotator/rotator.go#L113-L187).
  
  So while the pattern is not that common it happens. 
  
  As a side bonus maybe we can use then controller-runtime metrics to gain visibility into the load of the "reconciler".
  
  One potential complication is to make sure there are no surprises in the HA mode, if we enable leader election (it is probably done by default), then it may not maintain the "heap" on the fallower replicas. I think we should make sure the "heap" is maintained on the fallback replicas of Kueue. This should not be hard, but something to remember about as a potential "gotcha" due to the reconciler model.
  
  In any case +5 from me.
