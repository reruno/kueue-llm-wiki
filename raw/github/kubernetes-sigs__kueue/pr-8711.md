## Pull request overview

This PR sets default QPS (100) and Burst (200) values for the performance scheduler runner to address flaky tests caused by client rate limiter errors. The issue occurred when SSA conflicts between the runner and minimalKueue exhausted retry capacity with the client-go defaults (5 QPS, 10 burst), leading to "context canceled" errors during conflict storms.

**Changes:**
- Added default QPS value of 100.0 when the `--qps` flag is not provided or is 0
- Added default Burst value of 200 when the `--burst` flag is not provided or is 0

  This is not about API server or APF mechanics, it's internal to the perf test recorder. When the workload informer fires, we record state changes to a buffered channel. If the recorder goroutine can't consume events fast enough and the channel fills up, we drop events to avoid blocking controller reconciles.
  
  Dropped events here indicate the test harness infrastructure can't keep up with event volume. This affects test accuracy (we may miss state transitions and get incorrect timing data), so we fail the test. It's a test infrastructure issue, not a Kueue or API server issue.
  
  IMO, with a 1000-event buffer and fast processing, drops should never happen in normal operation. If they do, it signals something is wrong with the test environment.

IMO, for workload events, the buffer increase + drop tracking is appropriate since each workload transition is unique and we want to capture them all
