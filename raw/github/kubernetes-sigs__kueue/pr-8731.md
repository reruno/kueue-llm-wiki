### Comment by [@mbobrovskyi](https://github.com/mbobrovskyi) — 2026-01-28T11:14:09Z

Overall, this PR looks good to me. However, it is quite complex to understand for contributors who are not familiar with how parallelization works in Make. In the future, when adding new generation targets or similar changes, we might encounter flaky issues due to concurrency if we forget to add dependencies to prevent parallel execution.

We discussed this with @vladikkuzn, and I like the approach. The Make setup is very smart, and it allows for significant optimization. As you can see, verify currently takes 13m57s, which is a huge improvement compared to the previous 20+ minutes.

However, we need to add a contribution guide or at least some comments explaining the logic. 

Including a dependency graph could also help clarify the current structure. We also discussed with @vladikkuzn the possibility of creating a tool to generate this graph automatically, which I think is really awesome.

Otherwise, I would be hesitant to merge this PR, as extending generation targets later could cause significant trouble.

### Comment by [@mimowo](https://github.com/mimowo) — 2026-01-28T12:35:26Z

I understand the concern, this is all valid, but my thinking is:
1. using parallel execution allows us to be aware of the dependencies, otherwise we will continue to incorrectly declare dependencies, and thus it will be more work in the future
2. if we encounter flakes in the verify we can always open PR to fallback to single thread, and fix in separate PR, then re-enable

Sure, some tool to visualize the target depenencies might be nice, but I don't consider it a blocker. We can develop a python script when the issues proven to be causing as issues.
