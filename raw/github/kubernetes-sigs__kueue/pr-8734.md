## Pull request overview

This WIP PR adds documentation for extended resources design to support DRA (Dynamic Resource Allocation) integration in Kueue. The enhancement allows workloads to request DRA devices using standard resource requests (e.g., `example.com/gpu: 1`) instead of explicitly referencing ResourceClaimTemplates, leveraging the Kubernetes `DRAExtendedResource` feature gate (alpha in k8s 1.35).

**Changes:**
- Adds documentation for Extended Resources as an alternative approach to ResourceClaimTemplates for DRA integration
- Documents DeviceClass cache implementation and extended resource translation flow
- Updates RBAC permissions to include deviceclasses resources
- Adds test coverage entries and integration test descriptions for extended resources functionality

  Direct `ResourceClaim` references (`resourceClaimName`) are intentionally excluded from quota accounting. This behavior is correct because direct claims refer to pre-existing resource allocations that were already accounted for at the time of creation. In addition, a single claim may be shared across multiple pods, and counting it per pod would result in incorrect double-counting. `ResourceClaim` templates represent new quota requests, whereas direct claims represent the usage of resources that have already been allocated.

  >Has there been any discussion on what we do if these extended resources match in excludeResourcePrefixes?   
  
  The extended resource translation reads directly from the workload spec, not the filtered requests. So if `nvidia.com/` is in excludeResourcePrefixes:                       
    1. The original extended resource (`nvidia.com/gpu`) is dropped from quota requests (prevents double-counting)                                                               
    2. The translated DRA logical resource (e.g., gpu) is still added and quota-enforced
   
  >if someone is using DRA fields on quota that they don't manage for Kueue is there any way to exclude?
  
  Yes, you can either exclude the DeviceClass in deviceClassMappings  and the workload will be marked inadmissible with a clear error explaining the DeviceClass isn't configured or exclude the logical resource name. But the former is safer IMO.
  
  > Its probably not a big deal for GPU right now but since `excludeResourcePrefixes` looks at requests fields I am curious about the overlap.
  
  The processing order ensures no overlap issues:                                                                                                                            
    1. DRA translation runs first (reads original spec)                                                                                                                        
    2. `excludeResourcePrefixes` filters the pod's resources.requests                                                                                                            
    3. Original extended resource is removed from workload (via `replacedExtendedResources`)                                                                                     
    4. Translated logical resource is added (via `preprocessedDRAResources`)

+2. Looks up DeviceClass by `extendedResourceName` in the ExtendedResourceCache
+3. If no matching DeviceClass is found in cache, the extended resource is left unchanged and the

+mechanism.

+
+1. Kueue detects extended resources in `resources.requests` using `IsExtendedResourceName`
+2. Looks up DeviceClass by `extendedResourceName` using a controller-runtime field indexer
+3. If no matching DeviceClass is found, the extended resource is not DRA-backed and is
+   left unchanged (treated as a normal extended resource)
+4. If a matching DeviceClass is found, uses the `extendedResourceName` as the quota key
+5. Removes original extended resource from workload via `replacedExtendedResources` to avoid
+   double-counting
+6. Admits workload against the quota for the `extendedResourceName`

+`excludeResourcePrefixes` filtering is applied. The processing order:
+1. Extended resource translation runs first, reading the original spec
+2. `excludeResourcePrefixes` filters the pod's `resources.requests`
+3. Original extended resource is removed from workload through `replacedExtendedResources`
+4. Translated resource is added through `preprocessedDRAResources`

  With the current path separation, these do not conflict in the resolution logic. If a workload requests an extended resource in `resources.requests`, Kueue uses the `extendedResourceName` as the quota key. If a workload uses a `ResourceClaimTemplate`, Kueue uses the logical resource name from `deviceClassMappings`. A single `DeviceClass` may participate in both paths.
  
  This is not a code-level ambiguity, but it does affect quota semantics. If the two paths resolve to different resource names, they create two independent quota pools over the same physical hardware. If the admin wants both paths to share one quota pool, the `deviceClassMappings` entry must use the same resource name as the `extendedResourceName`.
  
  For Alpha, we don't add validation restrictions on naming collisions between `deviceClassMappings` and `extendedResourceName`. The admin controls both configurations. For Beta, we can evaluate adding a warning when a `deviceClassMappings` logical name matches an `extendedResourceName` from a different DeviceClass, since that could indicate unintended quota sharing across different hardware.

  Kueue accounts quota using the requested `extendedResourceName` itself, not a selected DeviceClass name. The field indexer lookup only confirms the resource is DRA-backed. Multiple matching DeviceClasses are still problematic because kube-scheduler may resolve a specific DeviceClass while Kueue only tracks the logical extended resource, creating potential divergence between quota accounting and physical allocation. In practice, admins should configure one DeviceClass per extended resource name.
  
  >Shouldn't we enforce pre-creating matched DeviceClass before they set up deviceClassMappings in the Kueue Configuration? I believe that the strict ordering limitations probably avoid complicated mechanisms as much as possible.
  
  The extended-resources path does not use deviceClassMappings at all; it auto-discovers matching DeviceClasses by extendedResourceName via the field indexer. deviceClassMappings are only for the ResourceClaimTemplate path. For Alpha, Kueue does not validate that mapped DeviceClasses already exist in the cluster. Admins should create DeviceClasses before submitting workloads, but strict ordering is not enforced by configuration validation.
  
  >What if DeviceClass with extendedResourceName is removed or extendedResourceName is changed during Job queueing (not yet admission)?
  
  In Alpha, Kueue does not reprocess workloads on DeviceClass changes, by design, to keep the implementation simple. As a result, a queued workload may temporarily retain stale preprocessing until another reconcile event occurs. If reprocessed later and no DeviceClass matches the requested extendedResourceName, the request falls back to a normal extended resource path. For post-admission failures caused by DeviceClass changes, enabling waitForPodsReady is recommended as the safety net until event-driven requeue is added as part of the Beta criteria.

+### Extended Resources
+
+This section is gated behind the `DRAExtendedResources` Kueue feature gate.
+
+Kueue also supports workloads requesting DRA devices via `resources.requests` (e.g., `example.com/gpu: 1`).
+When a DeviceClass has `spec.extendedResourceName` set, kube-scheduler automatically creates ResourceClaims.
+This requires the Kubernetes `DRAExtendedResource` feature gate (alpha in k8s 1.35).
+
+An extended resource can be identified by verifying that qualified resource names containing `/` are not in the `kubernetes.io/` or `requests.` namespaces and are not standard resources like `cpu`, `memory`, `ephemeral-storage`, or `hugepages-*`.
+
+#### Configuration
+
+The extended resources path does not require `deviceClassMappings`. Kueue auto-discovers
+DeviceClasses via a field indexer on `spec.extendedResourceName` and uses the
+`extendedResourceName` directly as the quota key in ClusterQueue.
+
+DeviceClass with `extendedResourceName` (DeviceClass API is v1/GA, but the `extendedResourceName`
+field requires the Kubernetes `DRAExtendedResource` feature gate):
+```yaml
+apiVersion: resource.k8s.io/v1
+kind: DeviceClass
+metadata:
+  name: gpu.example.com
+spec:
+  extendedResourceName: example.com/gpu
+```
+
+ClusterQueue uses the `extendedResourceName` directly as the quota resource:
+```yaml
+apiVersion: kueue.x-k8s.io/v1beta2
+kind: ClusterQueue
+metadata:
+  name: gpu-queue
+spec:
+  resourceGroups:
+  - coveredResources: ["example.com/gpu"]
+    flavors:
+    - name: default
+      resources:
+      - name: example.com/gpu
+        nominalQuota: 8
+```
+
+No Kueue configuration changes are needed. No `deviceClassMappings` entry is required for
+extended resources. This is a clean separation from the ResourceClaimTemplate path, which
+continues to use `deviceClassMappings`.
+
+#### Path Separation
+
+The two DRA paths have independent quota resolution:
+1. **Extended resources** (`DRAExtendedResources` gate): auto-discovers DeviceClass via
+   field indexer, uses `extendedResourceName` as quota key. No `deviceClassMappings` needed.
+2. **ResourceClaimTemplates** (`DynamicResourceAllocation` gate): uses `deviceClassMappings`
+   to map DeviceClass names to logical resource names. No auto-discovery.
+
+There is no precedence or fallback between the two paths. Each path has exactly one resolution
+mechanism.
+
+#### Processing Flow
+
+1. Kueue detects extended resources in `resources.requests`
+2. Looks up DeviceClass by `extendedResourceName` by field indexer
+3. If no matching DeviceClass is found, the resource is not DRA-backed and Kueue
+   processes it through the standard resource quota path (counted via `node.Status.Allocatable`)
+4. If a matching DeviceClass is found, uses the `extendedResourceName` as the quota key
+5. Removes original extended resource from the workload's effective resource requests
+   (tracked internally per PodSet) to avoid double-counting
+6. Admits workload against the quota for the `extendedResourceName`
+
+The extended resource translation reads directly from the workload spec before
+`excludeResourcePrefixes` filtering is applied. The processing order:
+1. Extended resource translation runs first, reading the original spec
+2. `excludeResourcePrefixes` filters the pod's `resources.requests`
+3. Original extended resource is removed from the workload's effective resource requests
+4. Translated resource is added through `preprocessedDRAResources`
+
+This ensures no overlap or double-counting between the two mechanisms.
+
+#### Same Hardware with Both Paths
+
+When the same hardware needs to serve both ResourceClaimTemplate users and extended resource
+users, admins configure separate flavors under the same ClusterQueue. Assuming a cluster
+with 1 node and 8 GPU devices available:
+
+```yaml
+# DeviceClass
+apiVersion: resource.k8s.io/v1
+kind: DeviceClass
+metadata:
+  name: gpu.example.com
+spec:
+  extendedResourceName: example.com/gpu
+---
+# Kueue config: deviceClassMappings only needed for ResourceClaimTemplate path
+apiVersion: config.kueue.x-k8s.io/v1beta2
+kind: Configuration
+resources:
+  deviceClassMappings:
+  - name: gpu-claims
+    deviceClassNames:
+    - gpu.example.com
+---
+# ClusterQueue with quota for both paths
+apiVersion: kueue.x-k8s.io/v1beta2
+kind: ClusterQueue
+metadata:
+  name: gpu-queue
+spec:
+  resourceGroups:
+  - coveredResources: ["example.com/gpu", "gpu-claims"]
+    flavors:
+    - name: default
+      resources:
+      - name: example.com/gpu    # for extended resource users
+        nominalQuota: 4
+      - name: gpu-claims          # for ResourceClaimTemplate users
+        nominalQuota: 4
+```
+
+Both quota buckets draw from the same physical hardware. The admin controls how capacity is
+split between the two user populations. Since these are different resource names, the split
+is fixed at configuration time.
+
+#### DeviceClass Resolution via Field Indexer
+
+Kueue resolves `extendedResourceName` to DeviceClasses using a controller-runtime field indexer
+on `DeviceClass` by `spec.extendedResourceName`. This provides fast lookups without adding
+dependencies on non-staging k8s repos.
+
+Even when multiple DeviceClasses share the same `extendedResourceName` (which K8s
+[permits with deterministic tiebreaking](https://github.com/kubernetes/kubernetes/blob/v1.35.0/staging/src/k8s.io/api/resource/v1/types.go#L1816-L1820)),
+Kueue still treats the resource as DRA-backed. The quota key is the `extendedResourceName`
+itself, not the DeviceClass name, so multiple matching DeviceClasses do not affect quota accounting.
```
