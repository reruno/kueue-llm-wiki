### Comment by [@olekzabl](https://github.com/olekzabl) — 2026-01-30T13:55:35Z

Converting it to draft, as it turns out hard to get this PR right.

The problem in a nutshell is this:

1. There's a divergence of flag-consuming systems between Kueue core code & `k8s.io/apiserver` (on which the Kueue visibility server relies):
   - Kueue core code uses Go standard `flag` package
   - `apiserver` uses `pflag` (`github.com/spf13/pflag`).

2. There's no natural way of telling `apiserver` "respect these cmd-line flags: {some specific set} and ignore all others".
   `pflag` throws if cmd-line args contain unrecognized flags.

3. One way to avoid errors is to register in `pflag` _all_ the flags that have been registered in `flag`.
   This is what the following line does:
   ```golang
   pflag.CommandLine.AddGoFlagSet(flag.CommandLine)
   ```
   However, as noted in the comments, this may cause behavioral changes which we don't yet understand.
   E.g. if `apiserver` relies in any way on `--zap-log-level` (declared & used by Kueue core), this PR may change the logging verbosity of visiblity server, just as a side effect.
   And similarly for any other flags "shared" by both codebases.
   If we want to go this path, it feels safest to hide this PR behind a feature gate.

4. An alternative approach would be restrictive: make sure we change only the handling of Kubeconfig.
   Because it's not doable just via flag management (see item # 2 above), we'd need to give up `pflag.Parse()`, and instead do some manual plugging into options like
   ```golang
   kubeconfigPath := flag.Lookup("kubeconfig").Value
   o.CoreAPI.CoreAPIKubeconfigPath = kubeconfigPath
   // ...
   ```
   (plus of course a guard if the flag was set).

   This is safer but "uglier" in that it's arbitrarily scoped down to `--kubeconfig`.

So basically we need to choose whether the approach from # 3 or # 4 is nicer.
