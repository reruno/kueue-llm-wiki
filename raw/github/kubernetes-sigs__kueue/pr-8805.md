  ### Can `kubectl create ... || true` be used?
  
  **It can, but it’s weaker and a bit riskier** than the current `--dry-run | apply` pattern.
  
  - **What it does well**: if the ConfigMap is missing, it creates it; if it already exists, `kubectl create` fails with `AlreadyExists` and `|| true` suppresses that, so the script continues.
  - **Main downsides**:
    - **It masks real failures** too (RBAC denial, wrong kubeconfig/context, API timeout, etc.), not just `AlreadyExists`. You’ll only notice later when the `patch` fails (or worse, something silently doesn’t get set).
    - **It won’t “repair” an existing ConfigMap** to ensure `data.initialized=true` is present. The current `apply` form will enforce that field.
  
  If you wanted the “create only if missing” behavior without masking errors, you’d typically do a conditional check (`get` then `create`), but the current `apply` approach is the usual clean idempotent way.
  
  ### Does the current version override the ConfigMap on the second run?
  
  **No, it doesn’t “replace” the whole ConfigMap.**
  
  - `kubectl create ... --dry-run=client -o yaml` **does not touch the cluster**; it only generates YAML.
  - Piping into `kubectl apply --server-side -f -` will **create if missing** or **merge** the declared fields if it exists.
  - In your case, that manifest only declares **one key**: `data.initialized=true`. On subsequent runs it will keep that key set, and **it will not wipe other `data` keys** that were added later (like `data.${key}=installed` from the `patch`), because server-side apply tracks field ownership at the map-key level.
