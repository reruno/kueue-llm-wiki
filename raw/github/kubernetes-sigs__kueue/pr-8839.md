## Description

When a ClusterQueue is being deleted, the finalizer removal could be delayed if the cache wasn't empty at reconcile time. The controller returned without requeuing, relying on external events to trigger the next reconcile. This caused flaky test timeouts when the workload deletion notification happened before the cache was updated.

> Is this due to the fact that we're doing reconciles in event handlers? Would this be resolved by #5310?

The core issue is that the CQ deletion reconcile returns without requeue while the cache is still non‑empty. When the cache later becomes empty (inside `DeleteWorkload`), there’s no API event to trigger another reconcile. Even if we moved away from event‑handler reconciles per #5310, we’d still need a signal for the cache transition. So #5310 might reduce exposure, but it wouldn’t eliminate the missing “cache became empty” event on its own.

## Pull request overview

Addresses a race during ClusterQueue deletion where finalizer removal could stall if the cache still had workloads at reconcile time, causing flaky timeouts.

**Changes:**
- Register the ClusterQueue controller as a watcher for “ClusterQueue became empty” events from the scheduler cache.
- Emit a generic reconcile trigger when a terminating ClusterQueue transitions to empty during workload deletion.
- Add cache plumbing (watcher interface + registration) to notify when the last workload is removed from a terminating ClusterQueue.
