<blockquote>
<h2>v0.16.0</h2>
<p>Changes since <code>v0.15.0</code>:</p>
<h2>Urgent Upgrade Notes</h2>
<h3>(No, really, you MUST read this before you upgrade)</h3>
<ul>
<li>
<p>Removed FlavorFungibilityImplicitPreferenceDefault feature gate.</p>
<p>Configure flavor selection preference using the ClusterQueue field <code>spec.flavorFungibility.preference</code> instead. (<a href="https://redirect.github.com/kubernetes-sigs/kueue/issues/8134">#8134</a>, <a href="https://github.com/mbobrovskyi"><code>@​mbobrovskyi</code></a>)</p>
</li>
<li>
<p>The short name &quot;wl&quot; for workloads has been removed to avoid potential conflicts with the in-tree workload object coming into Kubernetes.</p>
<p>If you rely on &quot;wl&quot; in your &quot;kubectl&quot; command, you need to migrate to other short names (&quot;kwl&quot;, &quot;kueueworkload&quot;) or a full resource name (&quot;workloads.kueue.x-k8s.io&quot;). (<a href="https://redirect.github.com/kubernetes-sigs/kueue/issues/8472">#8472</a>, <a href="https://github.com/kannon92"><code>@​kannon92</code></a>)</p>
</li>
</ul>
<h2>Changes by Kind</h2>
<h3>API Change</h3>
<ul>
<li>
<p>Add field multiplyBy for ResourceTransformation (<a href="https://redirect.github.com/kubernetes-sigs/kueue/issues/7599">#7599</a>, <a href="https://github.com/calvin0327"><code>@​calvin0327</code></a>)</p>
</li>
<li>
<p>Kueue v0.16 starts using <code>v1beta2</code> API version for storage. The new API brings an optimization for the internal representation of TopologyAssignment (in WorkloadStatus) which allows using TAS for larger workloads (under the assumptions described in issue <a href="https://redirect.github.com/kubernetes-sigs/kueue/issues/7220">#7220</a>, it allows to increase the maximal workload size from approx. 20k to approx. 60k nodes).</p>
<p>All new Kueue objects created after the upgrade will be stored using <code>v1beta2</code>.</p>
<p>However, existing objects are only auto-converted to the new storage version by Kubernetes during a write request. This means that Kueue API objects that rarely receive updates - such as Topologies, ResourceFlavors, or long-running Workloads - may remain in the older <code>v1beta1</code> format indefinitely.</p>
<p>Ensuring all objects are migrated to <code>v1beta2</code> is essential for compatibility with future Kueue upgrades. We tentatively plan to discontinue support for <code>v1beta1</code> in version 0.18.</p>
<p>To ensure your environment is consistent, we recommend running the following migration script after installing Kueue v0.16 and verifying cluster stability: <a href="https://raw.githubusercontent.com/kubernetes-sigs/kueue/main/hack/migrate-to-v1beta2.sh">https://raw.githubusercontent.com/kubernetes-sigs/kueue/main/hack/migrate-to-v1beta2.sh</a>. The script triggers a &quot;no-op&quot; update for all existing Kueue objects, forcing the API server to pass them through conversion webhooks and save them in the <code>v1beta2</code> version.
Migration instructions (including the official script): <a href="https://redirect.github.com/kubernetes-sigs/kueue/issues/8018#issuecomment-3788710736">kubernetes-sigs/kueue#8018</a>. (<a href="https://redirect.github.com/kubernetes-sigs/kueue/issues/8020">#8020</a>, <a href="https://github.com/mbobrovskyi"><code>@​mbobrovskyi</code></a>)</p>
</li>
<li>
<p>MultiKueue: Allow up to 20 clusters per MultiKueueConfig. (<a href="https://redirect.github.com/kubernetes-sigs/kueue/issues/8614">#8614</a>, <a href="https://github.com/IrvingMg"><code>@​IrvingMg</code></a>)</p>
</li>
</ul>
<h3>Feature</h3>
<ul>
<li>CLI: Support &quot;kwl&quot; and &quot;kueueworkload&quot; as a shortname for Kueue Workloads. (<a href="https://redirect.github.com/kubernetes-sigs/kueue/issues/8379">#8379</a>, <a href="https://github.com/kannon92"><code>@​kannon92</code></a>)</li>
<li>ElasticJobs: Support RayJob InTreeAutoscaling by using the ElasticJobsViaWorkloadSlices feature. (<a href="https://redirect.github.com/kubernetes-sigs/kueue/issues/8082">#8082</a>, <a href="https://github.com/hiboyang"><code>@​hiboyang</code></a>)</li>
<li>Enable Pod-based integrations by default (<a href="https://redirect.github.com/kubernetes-sigs/kueue/issues/8096">#8096</a>, <a href="https://github.com/sohankunkerkar"><code>@​sohankunkerkar</code></a>)</li>
<li>Logs now include <code>replica-role</code> field to identify Kueue instance roles (leader/follower/standalone). (<a href="https://redirect.github.com/kubernetes-sigs/kueue/issues/8107">#8107</a>, <a href="https://github.com/IrvingMg"><code>@​IrvingMg</code></a>)</li>
<li>MultiKueue: Add support for StatefulSet workloads (<a href="https://redirect.github.com/kubernetes-sigs/kueue/issues/8611">#8611</a>, <a href="https://github.com/IrvingMg"><code>@​IrvingMg</code></a>)</li>
<li>MultiKueue: ClusterQueues with both MultiKueue and ProvisioningRequest admission checks are marked as inactive with reason &quot;MultiKueueWithProvisioningRequest&quot;, as this configuration is invalid on manager clusters. (<a href="https://redirect.github.com/kubernetes-sigs/kueue/issues/8451">#8451</a>, <a href="https://github.com/IrvingMg"><code>@​IrvingMg</code></a>)</li>
<li>MultiKueue: trigger workload eviction on the management cluster when the corresponding workload is evicted
on the remote worker cluster. In particular this is fixing the issue with workloads using ProvisioningRequests,
which could get stuck in a worker cluster which does not have enough capacity to ever admit the workloads. (<a href="https://redirect.github.com/kubernetes-sigs/kueue/issues/8477">#8477</a>, <a href="https://github.com/mszadkow"><code>@​mszadkow</code></a>)</li>
<li>Observability: Add more details (the preemptionMode) to the QuotaReserved condition message,
and the related event, about the skipped flavors which were considered for preemption.
Before: &quot;Quota reserved in ClusterQueue preempt-attempts-cq, wait time since queued was 9223372037s; Flavors considered: main: on-demand(Preempt;insufficient unused quota for cpu in flavor on-demand, 1 more needed)&quot;
After: &quot;Quota reserved in ClusterQueue preempt-attempts-cq, wait time since queued was 9223372037s; Flavors considered: main: on-demand(preemptionMode=Preempt;insufficient unused quota for cpu in flavor on-demand, 1 more needed)&quot; (<a href="https://redirect.github.com/kubernetes-sigs/kueue/issues/8024">#8024</a>, <a href="https://github.com/mykysha"><code>@​mykysha</code></a>)</li>
<li>Observability: Introduce the counter metrics for finished workloads: kueue_finished_workloads_total and kueue_local_queue_finished_workloads_total. (<a href="https://redirect.github.com/kubernetes-sigs/kueue/issues/8694">#8694</a>, <a href="https://github.com/mbobrovskyi"><code>@​mbobrovskyi</code></a>)</li>
<li>Observability: Introduce the gauge metrics for finished workloads: kueue_finished_workloads and kueue_local_queue_finished_workloads. (<a href="https://redirect.github.com/kubernetes-sigs/kueue/issues/8724">#8724</a>, <a href="https://github.com/mbobrovskyi"><code>@​mbobrovskyi</code></a>)</li>
<li>Security: Support customization (TLSMinVersion and CipherSuites) for TLS used by the Kueue's webhooks server,
and the visibility server. (<a href="https://redirect.github.com/kubernetes-sigs/kueue/issues/8563">#8563</a>, <a href="https://github.com/kannon92"><code>@​kannon92</code></a>)</li>
</ul>
