#### What this PR does / why we need it:

Multi-kueue currently only considers quota reservations but not admission checks when selecting a worker cluster.
This is undesirable for instance when using ProvisioningRequest in the worker clusters to try to obtain GPU resources as one typically sets to quotas to very large values, only relying on the ProvisioningRequest to determine admission. This means all clusters will immediately reserve a quota, one of them is selected "randomly by race condition" but this one might not have been the one granting the ProvisioningRequest first.

https://github.com/kubernetes-sigs/kueue/pull/8465 proposes an alternative (optional) behaviour that waits for admission checks in the worker clusters to be ready before selecting a worker in order to take the ProvisioningRequest into account.




#### Special notes for your reviewer:

Feedback on https://github.com/kubernetes-sigs/kueue/pull/8465 by @mimowo was that the api should first be discussed in the multi kueue KEP (see [here](https://github.com/kubernetes-sigs/kueue/pull/8465#issuecomment-3723329185)).
