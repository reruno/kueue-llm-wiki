#### What this PR does / why we need it:

1. Correctness

If podSetTopologyRequest fails (e.g. "more than one flavor assigned", or missing TAS cache for the assigned flavor), we call psAssignment.error(err) and do not add that PodSet to tasRequests. As a result, FindTopologyAssignmentsForWorkload never runs for it and TopologyAssignment stays nil. The workload should not be admitted in this case, but without the fix it can be, because RepresentativeMode() may still return Fit.

2. User-visible behavior

Admitting a workload whose PodSet failed TAS request construction means that PodSet has no TopologyAssignment. TopologyUngater will never inject topology node selectors for it, so the job may schedule in a way that violates the user’s TAS expectations or cluster policy, without a clear admission failure. Failing admission when Status.IsError() makes the failure explicit and avoids this inconsistent state.

**Root cause**
`WorkloadsTopologyRequests` correctly records failures with `psAssignment.error(err)` and does not add the failed PodSet to tasRequests.

`PodSetAssignment.RepresentativeMode()` only used `Status.IsFit()` and `len(psa.Flavors) == 0` to decide Fit vs NoFit, and did not consider `Status.IsError()`.

So when `Status.err != nil` but `len(psa.Flavors) > 0` (e.g. onlyFlavor failure), the PodSet’s mode could still be Fit, and the workload could be admitted with a PodSet that has no `TopologyAssignment`.
