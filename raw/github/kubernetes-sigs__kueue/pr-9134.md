#### What this PR does / why we need it:

Fixes the flaky `Kuberay Should run a rayjob with InTreeAutoscaling` E2E test.

**Root cause:** Analysis of the autoscaler logs from a failed CI run showed the autoscaler reported "no resource demands" for ~5 minutes. The `@ray.remote` decorator without explicit `num_cpus` allows Ray to detect high logical CPU count from the host, making it think one worker can handle all tasks.

**Fix:**
1. Set `@ray.remote(num_cpus=1)` to ensure deterministic resource demand
2. Increase parallel task sleep from 4s to 8s to give the autoscaler time to react
