## Pull request overview

Cherry-picks upstream fix to make `ClusterQueue.Snapshot()` ordering deterministic by aligning snapshot sorting with the scheduler’s queue ordering and adding a UID tie-breaker when timestamps are equal. This prevents non-deterministic workload ordering (and associated flakes) when workloads share the same queue-order timestamp.

**Changes:**
- Replace boolean “less” ordering with a 3-way comparator and use it for both heap ordering and snapshot sorting (with UID final tie-break).
- Add a unit test to assert `Snapshot()` returns a stable order across repeated calls (including with inadmissible workloads).
- Adjust an integration test setup/timing to avoid UID-based ordering affecting expected admission patterns.

### Reviewed changes

Copilot reviewed 3 out of 3 changed files in this pull request and generated no comments.

| File | Description |
| ---- | ----------- |
| `pkg/cache/queue/cluster_queue.go` | Introduces deterministic comparator (priority → timestamp → UID) and uses it for `Snapshot()` sorting. |
| `pkg/cache/queue/cluster_queue_test.go` | Adds `TestSnapshotDeterministicOrder` to detect order instability across repeated `Snapshot()` calls. |
| `test/integration/singlecluster/scheduler/podsready/scheduler_test.go` | Updates timing/creation sequencing to avoid flakiness introduced by UID tie-breaking when timestamps collide at second precision. |
