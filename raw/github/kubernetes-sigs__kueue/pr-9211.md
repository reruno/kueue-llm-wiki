### Comment by [@j-skiba](https://github.com/j-skiba) — 2026-02-16T08:31:34Z

> 1. How was the placement incorrect?

Given the scenario from the [issue](https://github.com/kubernetes-sigs/kueue/issues/9210).
The initial pod assignments would be [`node1`, `node2`]. After `node1` failure and node replacement the topology assignment would be [`node2`, `node3`] where the pod on `node2` would be running undisrupted so the replacement pod should be scheduled on `node3`, but the terminating pod was deleted quickly enough so it was assigned nodeSelectors for `node2` because of its job labels (index 0).
Usually that doesn't happen because the terminating pod exists and has a conflicting job label and because of that there is fallback to the greedy assignment.

So in case where there were 2 pods that use 40 cpus each while there were nodes with 50 cpus then in the described scenario we would end up with 2 pods scheduled on `node2` where only one pod can run because there is no space for 2 pods that use 40 cpus each.

> 2. How does it happen that the placement is incorrect? Should we rather fix the node hot swap to prevent that in the first place, rather than fix it later?

I don't think we should do anything with the hotswap. I think that node replacement is the right thing to do because the node and the pod are nonfunctional. I would say the problem is with the placement of the pods that was working based on the conflicting job labels and it luckily resulted in sensible placement of the pods.

> Also note that, as long as the "old pod" is terminating we don't need to change the assignment strategy - it will be gone "soon", and the new Pod with the same index will be able to take its place.

I think that's even a worse scenario because it would trigger the bug.
Also when there are problems with kubelet on the node it might take some more time to mark the pod as terminated.
