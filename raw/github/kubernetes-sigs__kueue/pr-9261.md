## Description

## Summary

Fixes #8908.

Adds retry logic with exponential backoff to `cluster_cleanup` in `hack/testing/e2e-common.sh` to work around a Docker daemon race condition that causes `kind delete cluster` to fail intermittently in CI.

## Root Cause

The Prow e2e jobs run Docker-in-Docker (Docker CE 28.x) inside privileged pods on Linux with significant IO contention. Under these conditions, Docker's internal container kill path hits a **hardcoded 2-second timeout** (`daemon/kill.go:200-202`) waiting for the container to acknowledge a SIGKILL. When the container process is in uninterruptible IO sleep (D-state), this timeout expires before the exit event propagates through containerd, and Docker returns:

`could not kill container: tried to kill container, but did not receive an exit event`

This is a confirmed, unfixed upstream Docker bug:
- [moby/moby#51028](https://github.com/moby/moby/issues/51028) — Root cause analysis (Docker 28.4.0, status: Parked)
- [moby/moby#51845](https://github.com/moby/moby/issues/51845) — Same bug with kind clusters specifically (Docker 29.1.4, status: Triage)

Both upstream reports confirm that **containers do exit, just 3-5 seconds late**. Docker then ignores the late exit event because the delete operation already failed. This means a retry after a short delay succeeds reliably.

## Testing

- Created and deleted a kind cluster through the modified `cluster_cleanup` — happy path returns immediately on first success
