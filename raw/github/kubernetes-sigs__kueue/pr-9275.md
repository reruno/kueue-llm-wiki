I traced the [failure logs](https://prow.k8s.io/view/gs/kubernetes-ci-logs/logs/periodic-kueue-test-e2e-multikueue-release-0-15/2022668151992356864) and the old pod was still alive 649ms after all readiness checks passed.

### Comment by [@mimowo](https://github.com/mimowo) — 2026-02-17T07:57:34Z

I'm wondering how endpoints work in API server - is removal of the API objects before or after removal from internal caches. If the endpoints objects are removed, but the internal endpoints in cache are removed reacting to events then it could explain why there is traffic to old endpoints for a short period of time.

### Comment by [@sohankunkerkar](https://github.com/sohankunkerkar) — 2026-02-17T11:15:37Z

> I'm wondering how endpoints work in API server - is removal of the API objects before or after removal from internal caches. If the endpoints objects are removed, but the internal endpoints in cache are removed reacting to events then it could explain why there is traffic to old endpoints for a short period of time.

The API server watches EndpointSlice and updates its internal cache reactively. There's propagation delay between the object update and the cache update. Additionally, even after the cache updates, existing HTTP connections to the old pod remain in http.Transport's connection pool until they error out and POST requests won't be retried on stale connections. We need to handle this at client side.

### Comment by [@mimowo](https://github.com/mimowo) — 2026-02-17T11:32:47Z

Interesting, so this could be it. 

Actually in our code we track "Endpoints", and I'm wondering if we should rather check EndpointSlieces too. Do you know what is the order of propagation when deleting a Pod, is it  Pod deleted -> Update Endpoints -> update EndpointSlices -> update internal cache?

but this is probably not a problem in our case, because the failing requests are issued already after the rolling restart. So most likely just the "cache" propagation issue.

### Comment by [@sohankunkerkar](https://github.com/sohankunkerkar) — 2026-02-18T05:07:08Z

>Actually in our code we track "Endpoints", and I'm wondering if we should rather check EndpointSlieces too

We already check EndpointSlices in `waitForKueueControllerReadyWithWebhookEndpoints` where we verify that EndpointSlice IPs match the ready pod IPs before proceeding.

>Do you know what is the order of propagation when deleting a Pod?

Pod terminating → Endpoints controller updates Endpoints → EndpointSlice controller updates EndpointSlice (both watch Pods, happen nearly simultaneously) → API server's internal webhook routing cache reacts to the EndpointSlice watch event.

>but this is probably not a problem in our case, because the failing requests are issued already after the rolling restart

Agreed. Since requests are issued after the new pod is up, there are no pre-existing stale connections in the pool. The issue is pure cache propagation delay
