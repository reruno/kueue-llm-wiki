  > How about sth like: git tag | grep -v '-' | sort -V | tail -n 1
  
  This is reasonable, but may give unexpected results on historical release branches. For example may end up building 0.15.5 release using 0.16.3 dependency, resulting in release failure. So we need to be more careful I think.
  
  > I'm not sure what's the alternative. Make this part of the release checklist? Manually bump from time to time when we feel like it?
  
  Maybe the alternative is to bump it during the release process, for that we have the Makefile target: `prepare-release-branch`. Then, on release branches we will keep updating to the lastest patch. On the main branch we will update to the "0" patch, next will be 0.17.0.  On top of that we could manually bump to the latest patch on-demand. 
  
  Still, to avoid surprises on releases, I would rather use what is put in plain text, rather than magic like `git tag | grep -v '-' | sort -V | tail -n 1`, but no strong view here - fine as long as we make it reliable.
  
  > Btw. doesn't this mean that there is a version skew between the manifests version and the code that the populator tests are running? I don't see anything overriding KUEUE_VERSION here or in test-infra. cc @mimowo
  
  Yeah, it is probably not ideal. This is the same we do for KueueViz. OTOH, handling version skew is "normal" - users cannot bump both components atomically anyway. So using the latest full release bumped by `prepare-release-branch` seems reasonable to me.
