## Description

#### What type of PR is this?

/kind bug

#### What this PR does / why we need it:


TL;DR: Found that quota was counted twice for unchanged elements of a podSet when:
- using [Elastic Workloads](https://kueue.sigs.k8s.io/docs/concepts/elastic_workload/) (via RayJob integration in my case)
- having a new workload with unchanged quota for one element of the podSet and an upscale for the other

Full context:
When there are multiple podSetAssignments in a workload slice and only one podSet changes (scales up), the unchanged podSet's quota was being double-counted, causing incorrect quota calculations.

For example, with two podSets:
- podSet0 (worker): old=2, new=2 (unchanged)
- podSet1 (coordinator): old=1, new=3 (scaled up)

The old code would:
1. Process podSet0: add 2 CPUs to assignmentUsage (incorrect!)
2. Process podSet1: check quota with assumedUsage=2 (wrong baseline!)

The fix calculates the delta (new - old) for workload slicing and only adds the delta to Usage.Quota. This ensures unchanged podSets don't consume additional quota in the assignment calculation.

After the fix:
1. Process podSet0: delta=0, skip adding to assignmentUsage
2. Process podSet1: delta=2, add 2 CPUs to assignmentUsage (correct!)
