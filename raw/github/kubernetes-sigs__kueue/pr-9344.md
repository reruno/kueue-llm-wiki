#### What this PR does / why we need it:

Currently, the pod framework creates the workload with the pod as the controller owner (and without the `JobOwnerGVKAnnotation`), causing the MK workload controller's `adapter()` function to select the pod adapter instead of the STS adapter.

This PR addresses the issue by having the STS reconciler create the workload (following the LWS pattern) with `JobOwnerGVKAnnotation` set at creation time, ensuring that MK always uses the STS adapter.

### Comment by [@IrvingMg](https://github.com/IrvingMg) — 2026-02-26T17:22:22Z

> 2. Could you check experimentally what will happen with STS workloads created and running, if Kueue is upgraded. will they continue smoothly after the upgrade, or may get disrupted? This question is both for MultiKueue and regular Kueue.

I tested this scenario, and after the upgrade, the StatefulSets should continue running normally.

The important point is that existing STS do not have the `JobOwnerGVKAnnotation`, while new ones created after the upgrade will. The injection should not occur for existing STS because they already have an owner reference. As a result, the injection logic is skipped:

https://github.com/epam/kubernetes-kueue/blob/0e0de702b3e1a912aba6ffa6ddd6166c6817a002/pkg/controller/jobs/statefulset/statefulset_reconciler.go#L217

Since existing STS do not have the `JobOwnerGVKAnnotation`, the multikueue adapter used for them remains the Pod adapter. This means they will maintain the same behavior after the upgrade, continuing to work with the Pod adapter and without creating STS in the worker clusters.

To make existing STS switch to the STS adapter, scaling them down to zero and then scale up should work, as this will recreate the workload with the correct annotation and then create the remote STS.
