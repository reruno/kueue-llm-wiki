## Description

#### What type of PR is this?
/kind feature

#### What this PR does / why we need it:
Adds new labels to pods created or managed by Kueue indicating from which local queue and cluster queue the pod originates. 
Those labels(in combination with other tools)  enable monitoring pods linked to single queue without expensive workload or job based filtering.

#### Which issue(s) this PR fixes:
Fixes #9156

None


### Comment by [@dkaluza](https://github.com/dkaluza) — 2026-02-18T20:41:58Z

>Do you actually need labels or annotations are enough?

From [k8 annotations documentation](https://kubernetes.io/docs/concepts/overview/working-with-objects/annotations/#attaching-metadata-to-objects)

> You can use either labels or annotations to attach metadata to Kubernetes objects. **Labels can be used to select objects and to find collections of objects that satisfy certain conditions. In contrast, annotations are not used to identify and select objects.** The metadata in an annotation can be small or large, structured or unstructured, and can include characters not permitted by labels. It is possible to use labels as well as annotations in the metadata of the same object.

So we from my point of view we definitely need labels. I can verify in various tools if some of them support selection by annotations, but I expect that if we do not use labels we will lose majority of the benefits of this feature.

> check if cq/lq is valid label before attempting to set it

Aren't we already using LQ names as label values but on the job/workload level?
https://kueue.sigs.k8s.io/docs/reference/labels-and-annotations/#kueuex-k8sioqueue-name

It is true for CQ names though I think there might be currently not used as labels. We can add appropriate check before assignment, but I dislike a little bit that they will be appearing conditionally.
