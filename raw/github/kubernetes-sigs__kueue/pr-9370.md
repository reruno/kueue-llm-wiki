### Comment by [@mimowo](https://github.com/mimowo) — 2026-02-20T10:13:11Z

> Glancing at the [list of FGs](https://kueue.sigs.k8s.io/docs/installation/#feature-gates-for-alpha-and-beta-features), it seems that "disabled by default for Alpha, enabled by default for Beta" is our universal convention.

Yes, Beta FGs are enabled by default (except maybe some really rare exceptions in k8s core, like ConcurrentWatchObjectDecode, unrelated here, just in case you are interested: https://github.com/kubernetes/kubernetes/issues/136950).

However, in case of metrics for LocalQueues the KEP proposes another safeguard which is [API-level switch]( https://github.com/kubernetes-sigs/kueue/pull/9370/changes#diff-a1e20d0bfdfa9783f3e4c88cd1bb9f3cb7f3395eeb47834ac525950c93ec17aaR99-R100), and my comment was purely referring to this switch.

This switch was introduced to safe-guard against a potentially high cardinality coming from the LQ metrics.

@olekzabl 
In Kubernetes (NOT Kueue), we can enable the Beta feature by default, which doesn't introduce new Resources.
In other words, if it introduces a new Resource, even if the feature is promoted to Beta, we can not enable it by default.

But these rules do not enforce enabling features by default. As @mimowo mentioned, even if the feaure satisfy the default enabling criteria, sigs / wgs can postpone the decision to the GA.

In the Kueue repository, we roughly follow the criteria.

### Comment by [@tenzen-y](https://github.com/tenzen-y) — 2026-02-20T18:45:49Z

> I'm wondering if we should make them enabled by default or disabled, because we already have some users using LocalQueues and so far no significant issues reported.

@mimowo This feature is disabled by default. So, my concern is that big cluster users might get surprised by scheduling performance and Prometheus server performance issues if we enable this by default. The problem is that if we decide to enable this by default now, we can not change the default (enabled -> disabled) in the future version because such default behavior change break users environment.

So, my question i,s have you heard that your customers with a large cluster (and large number of namespace) enable this feauture and haven't faced any performance issue?

### Comment by [@mimowo](https://github.com/mimowo) — 2026-02-23T12:18:25Z

> So, my question i,s have you heard that your customers with a large cluster (and large number of namespace) enable this feauture and haven't faced any performance issue?

I haven't seen any reports wrt performance issues due to enabling the feature. However "large" deployment is tricky to define. For example some big deployments (in terms of large workloads / many workloads) may not have many LQs, and so they are not exposed. I have not yet seen deployments where the number of LQs would be very high, like >1k. 

Actually, I think it makes sense to keep the `enable` field in this case, for this feature regardless if we enable by default or disable, because disabling by using localQueueSelector will not be intuitive to users. 

I don't have a strong view here, but I'm leaning to:
1. bring back the "enable" field      (to make disabling more intuitive)
2. make "enable=true" by default  (to make the feature easier to use)

Wdyt @tenzen-y @gabesaba ?
