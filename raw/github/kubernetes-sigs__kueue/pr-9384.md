  >Interesting, is my understanding correct this flake happens when we get event about Workloads after events for the LocalQueue, and in this case the metric update for LQ  does not yet see all the Workloads.
    
  Close but not quite. The event ordering is handled correctly by AddLocalQueue/AddClusterQueue which cross-reference each other. The root cause is that after an in-process restart, prometheus gauges retain stale values from the old manager, so Eventually(gauge == 1) passes instantly without waiting for controllers to start. The delete then races ahead of `m.finishedWorkloads` being populated.
  
  >If this is the case, then I agree this is a low priority issue, because any LQ update will fix the inaccurate metric, and I'm ok with the pragmatic fix for the tests.
  
  Agreed. There's no long-term consistency issue. The gauges converge after both event handlers complete. There's a possible transient startup window where the gauge may be briefly inaccurate, but no corruption or permanent drift.
  
  > verify on some experiment PR it works if looped 200 times
  
  I usually put a up PR after this testing. I will add a short note next time when I create a similar PR.
  
  > open a follow up issue to "ensure the LQ metrics are accurate after Kueue restart"
  
  IIUC, in production, process restart zeroes prometheus, so this race can't happen. The transient startup inconsistency resolves within milliseconds as controllers initialize. I don't think a follow-up is needed.
  
  > Also, have you got a chance to investigate what it would entail to fix properly?
  
  Yes, I didn't dig deeper but it would mean either resetting finished workload gauges in NewManager or re-reporting them after all sources sync. Both are narrow but touch metric lifecycle for a transient window with no real-world impact. IMO, the test-side fix with the probe workload barrier is the right scope for this issue.
