#### What this PR does / why we need it:

In a multi-flavor setup, with fair sharing enabled, we noticed that borrowing resources on one flavor affects a CQ's ability to use its nominal quota on a different flavor -- in both admission ordering and preemption.

I've described the issue in more detail in #9406 . This PR tries to handle it the same way that classical preemption does. Ensures that workloads fitting within their CQ's nominal quota are never penalized by borrowing on other flavors.

**Changes:**

- Admission ordering (fair_sharing_iterator.go): Prefer workloads that don't require borrowing before comparing DRS, so nominal-entitled workloads are scheduled first.
- Preemption (preemption.go): Add `preemptorWithinNominalForContestedResources` check before the preemption simulation. When the preemptor CQ (including the incoming workload) stays within nominal quota for the contested flavor-resources, skip the DRS strategy gate in runFirstFsStrategy and preempt unconditionally.
- Refactor: Extract preemptCandidate helper to deduplicate the repeated pop/remove/append/check-fits sequence across four different places.

Hi @mimowo, thanks for your response. I'm still learning the codebase, so appreciate the patience.

TLDR:

FlavorFungibility.Preference (PreemptionOverBorrowing) is only allowed when both WhenCanBorrow and WhenCanPreempt are set to TryNextFlavor (enforced by CEL validation on the API). We use MayStopSearch, so Preference is not applicable to our setup. I've updated the test to [explicitly set MayStopSearch](https://github.com/kubernetes-sigs/kueue/pull/9407/changes#diff-aee60c971eca1e23b63bed32e1e54b3c67c4fee9f35f1ca54e5a2bb879c7683bR919) to reflect this. 

#9462 fixes the flavor assignment step, this PR fixes the fair sharing preemption step. They are independent — this PR is needed with or without #9462, whenever fair sharing is enabled.

---

They address different code paths:
When fair sharing is disabled, flavor assignment picks the right flavor (via MayStopSearch or PreemptionOverBorrowing), sets Mode=Preempt, and getTargets() calls classicalPreemptions() (preemption.go L143), which has its own nominal-aware logic and allows the preemption. #9462 is about a bug in this flavor assignment step.

When fair sharing is enabled, the flavor assignment step works the same way — it still correctly picks the flavor and sets Mode=Preempt. But getTargets() calls fairPreemptions() instead (preemption.go L141). The DRS strategy gates can then block the preemption because the preemptor's DRS is inflated by borrowing on a different flavor, even though it's within nominal quota on the contested flavor. If no targets pass the strategy gate, GetTargets() returns nil and the workload is not admitted (scheduler.go L267-281).

This PR fixes that: when the preemptor is within nominal for the contested resources, bypass the DRS strategy gate so the preemption proceeds. Without this fix, flavor assignment (whether via MayStopSearch or PreemptionOverBorrowing) picks the right flavor, but fair sharing blocks the preemption it chose.

> so in this case the correct flavor and mode is selected. The preemption is possible, but it does not happen in practice, right?

That's exactly right. The correct flavor and mode are selected, but preemption is blocked by the fair sharing DRS strategy gate. The workload ends up [hitting this path](https://github.com/kubernetes-sigs/kueue/blob/main/pkg/scheduler/scheduler.go#L268): `"Workload requires preemption, but there are no candidate workloads allowed for preemption"` and is skipped. This PR solves for that.

This PR also handles [admission ordering](https://github.com/kubernetes-sigs/kueue/pull/9407/files#diff-8716acbdd5bd256fcf6e05caf9b5573b819e7480aca9b6154dc91c8942ae73efR173). The classical iterator already [prefers workloads that don't require borrowing](https://github.com/kubernetes-sigs/kueue/blob/main/pkg/scheduler/scheduler.go#L733-L738), but the fair sharing iterator only uses DRS. This means a workload that fits within its CQ's nominal quota on the contested flavor can be deprioritized solely because the CQ's aggregate DRS is high from borrowing on other flavors. The PR adds the same nominal-first ordering to the fair sharing iterator
