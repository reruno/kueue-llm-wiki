## Description

Use `expectations.Store` (same pattern as pod controller) to track workloads with in-flight preemptions. The expectation is set before calling Evict and rolled back on failure. It is cleared by the workload controller when the eviction (or deletion) is observed.

## Pull request overview

This pull request addresses a bug where the scheduler could issue multiple preemption requests and emit multiple events for the same workload across different scheduling cycles. This occurs when the informer cache hasn't yet reflected a previous eviction.

**Changes:**
- Introduced a `pendingPreemptions` sync.Map field in the Preemptor struct to track workload UIDs with in-flight preemptions
- Modified the preemption logic to check for and skip duplicate preemption attempts
- Added cleanup logic to remove entries when workloads are confirmed evicted or deleted
- Added a test to verify duplicate preemptions are prevented

  The pendingPreemptions entry is stored after the Evict call succeeds. This creates a narrow race condition window between when the eviction is issued and when the entry is stored. If another scheduling cycle starts in this window with a stale snapshot, it could see the same target and attempt to evict it again before the entry is recorded.
