## What this PR does / why we need it

Adds MultiKueue support for RayService, enabling RayService workloads to be dispatched across multiple Kueue-managed clusters.

This PR depends on:
- #9885 (refactoring: shared generic Ray MultiKueue adapter)
- #9886 (dependency: kuberay v1.6.0-rc.0 bump)

### Changes in this PR (RayService-specific only):
- `rayservice_multikueue_adapter.go` — RayService MultiKueue adapter using the shared `ray.NewMKAdapter()` (~51 lines)
- `rayservice_multikueue_adapter_test.go` — 7 test cases covering all adapter operations
- `rayservice_controller.go` — implements `JobWithManagedBy` interface (`CanDefaultManagedBy`, `ManagedBy`, `SetManagedBy`) + registers MultiKueue adapter in `init()`
- `wrappers.go` — test helpers (`ManagedBy()`, `StatusConditions()`)

### Key design decisions:
- RayService is **long-running** — `Finished()` always returns `false`, so the MultiKueue adapter continuously syncs status
- Suspend is via `Spec.RayClusterSpec.Suspend` (embedded), not a top-level field
- `ManagedBy` is at the top level of `RayServiceSpec` (like RayJob)

E2E tests will be in a separate follow-up PR (agreed with @hiboyang).
