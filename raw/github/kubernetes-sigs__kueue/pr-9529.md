### Comment by [@PannagaRao](https://github.com/PannagaRao) — 2026-02-27T05:26:04Z

@mimowo I added diagnostic logging to the test's Eventually block since reconciler-side logs were getting rotated out before artifact collection with 200 iterations. Here's what I observed:                                               
                                                                                                                                                                                                                                              
The updatedReplicas < spec.Replicas check in isRollingUpdateWithSurge (added in #9135) causes a race condition with the LWS controller's rolling update.                                                                                    
   
When updatedReplicas reaches spec.Replicas, Kueue treats the rolling update as complete and deletes the surge workloads. But the LWS controller hasn't finished its own surge cleanup yet — status.Replicas is still greater than spec.Replicas. Kueue deleting the workloads causes the surge pods to terminate, which disrupts the LWS controller's in-progress cleanup and updatedReplicas regresses, creating an oscillation

```
  updatedReplicas=3, status=6 → all 6 workloads admitted    (isRollingUpdateWithSurge=true, surge kept)
  updatedReplicas=4, status=6 → workloads 4,5 deleted       (isRollingUpdateWithSurge=false, surge deleted)
                               → pods for groups 4,5 start terminating
  updatedReplicas=3, status=5 → workload 4 re-created       (regressed, 1 surge group cleaned up by LWS)
  updatedReplicas=4, status=5 → workload 4 deleted          (cycle repeats)
  updatedReplicas=3, status=4 → no surge groups remain      (2nd surge group cleaned up by LWS)
  updatedReplicas=4, status=4, ready=4 → done
```

Prow logs confirming the chain (workload deleted → pod deleting=true → LWS recreates pod → updatedReplicas regresses): https://prow.k8s.io/view/gs/kubernetes-ci-logs/pr-logs/pull/kubernetes-sigs_kueue/9529/pull-kueue-test-e2e-main-1-33/2027106281306722304

This oscillation mostly converges (each cycle the LWS controller cleans up one surge group), but convergence time is non-deterministic and can exceed the 5-min timeout under load.

Simply removing the updatedReplicas check would risk the scale-down flake that #9135 fixed. One approach I'm considering: keep the existing isRollingUpdateWithSurge logic, , but in filterWorkloads, rescue surplus workloads from deletion if their groups still have running pods .

WDYT? Open to other directions I might be missing.
