## Pull request overview

This PR fixes a race condition bug where a workload that has been popped by the scheduler (inflight) can be concurrently pushed back onto the heap by the workload controller's `PushOrUpdate` call. When the scheduler then places the workload in the inadmissible set via `requeueIfNotPresent`, it results in the workload appearing in both the heap and inadmissible set simultaneously, causing spurious extra scheduling cycles. The fix is directly addressing the flaky test reported in issue #9591.

**Changes:**
- Guard `PushOrUpdate` in `ClusterQueue` to skip workloads that are currently inflight (being processed by the scheduler), since `RequeueWorkload` will re-add the workload with the latest version when the scheduling attempt completes.
- Add a unit test that verifies the skipping behavior for inflight workloads in `PushOrUpdate`.

  I don't think so. Actually, before #9232, requeueing was synchronous. There was no `AddAfter` delay, so when a CQ update triggered a requeue, the inadmissible workloads were moved to the heap immediately within the same `UpdateClusterQueue` call, under the manager lock. The scheduler couldn't be processing (inflight) a workload while the requeue happened because both operations were serialized by the manager lock.            
  
  With async requeueing (#9232), the requeue goes through a `workqueue.AddAfter` with a batch delay. The requeue fires asynchronously in a separate goroutine, which means the scheduler can be actively processing a workload (inflight) when the workload controller's PushOrUpdate is called that's the window where the race occurs.
