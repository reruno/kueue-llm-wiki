#### What this PR does / why we need it:

When using test logger in an unsynchronized goroutine, we will meet a race condition such as: https://prow.k8s.io/view/gs/kubernetes-jenkins/pr-logs/pull/kubernetes-sigs_kueue/93/pull-kueue-test-unit-main/1499496533626720256 (in https://github.com/kubernetes-sigs/kueue/pull/93)

We have to synchronize the goroutines in the unit test to solve the problem.

hi @alculquicondor !

First, I tried to reproduce the issue with `go test ./pkg/scheduler/... -race -count 1000`. Then I found that sometimes I can see the "Log in goroutine after..." message when a race condition is generated.

We can found that this panic is generated in https://github.com/golang/go/blob/63bd6f68e6cbb237b46a99775103758afaee370a/src/testing/testing.go#L887 and with the help of [related comments](https://github.com/golang/go/blob/63bd6f68e6cbb237b46a99775103758afaee370a/src/testing/testing.go#L1424) we can conclude that **it will be illegal to use `test.logger` in a test case that has already completed**.

That's to say, before executing the next test case, we have to make sure that no test.logger is still being used anywhere.

By adding a channel to synchronize the goroutines, we can ensure that test.logger will only be used in each individual testcase *(no two testcases are using test.logger at the same time, especially if one of them is considered complete **even though it still has a goroutine running**)*.

We found the cause of the problem, and we can solve it directly through coroutine synchronization. I verified the above through multiple test commands as well.
