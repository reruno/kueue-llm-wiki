## Description

#### What type of PR is this?

Add a guage metric to track the current number of workloads with PodsReady state.

| Metric | Scope | Labels |
|--------|-------|--------|
| `kueue_pods_ready_workloads` | ClusterQueue | `cluster_queue`, `replica_role` |
| `kueue_local_queue_pods_ready_workloads` | LocalQueue | `name`, `namespace`, `replica_role` |

/kind feature

#### What this PR does / why we need it:

To inform users how many pods are making progress (Ready), since the metric for Admitted pods may not reflect being ready; such as waiting for `ImagePull`.

#### Which issue(s) this PR fixes:

Fixes #6791

#### Special notes for your reviewer:

The current approach is edge-based (intentionally based on [feedback](https://github.com/kubernetes-sigs/kueue/pull/6905#discussion_r2367372297) from [#6905](https://github.com/kubernetes-sigs/kueue/pull/6905)), which may lead to inconsistency if any count operation fails. It also doesn't account for the controller restarting (count reset) while workloads are running. I'm not sure there's an alternative given the constraints.

---

#### Testing:

I wrote a [script](https://gist.github.com/jimangel/9976fac4fb02e473a2df1fdaf5995512#file-gen-with-lq-cq-sh) that submits 1,000 jobs randomly between a Cluster Queue and a Local Queue. The metrics are exported to my local Prometheus / Grafana stack.

Here's my first run:

<img width="1448" height="952" alt="image" src="https://github.com/user-attachments/assets/c4459998-30b0-4a77-86f3-9b8a7b4a10b8" />

You'll see a spike in Pods Pendings (green) and a compounding backoff of Pods Ready (red) before catching back up with the queue. I had a bunch of runs to tune this but ultimately after:

- Increasing the CQ nominal quota (from 9 -> 56 CPUs)
- Increasing the KinD max pods per node (from 110 to 250)
- Increasing the KinD nodes (from 1 to 3)
- Updating the script to use less CPU (random 10-250m to random 10-100m)
- Updating the script to use less memory (random 10-150Mi to static 10Mi)

I now get way better, theoretical, utilization (and no pending):

<img width="1430" height="746" alt="image" src="https://github.com/user-attachments/assets/03f6a425-3dac-4562-9c25-7dc81e86ae21" />

The ~750 ready max matches my 3 nodes * 250 pods per node. I would assume adding 2 more nodes would fix the small peak Reserving.

**Note:** I also validated the absence of metrics with LocalQueue featuregate off and with WaitForPodsReady disabled

Special thanks @helayoty for starting this convo / work!

  I think jobframework is not the best place to put the code because:
  1. all the information only depends on the Workload's object state (no Job needed)
  2. JobFramework reconcilers are meant lean because some users may need to maintain their own implementations for their in-house Job CRDs
  3. it will not work OOTB for some internal workloads, like StatefulSet or LeaderWorkerSet which currently don't use jobframework reconciler (IIRC)
  
  So, I would propose moving the code to the workload_controller, and react in the Create/Update/Delete event handlers. This will also allow to account for already running Workloads on restart, because every workload on restart goes via the Create event.
