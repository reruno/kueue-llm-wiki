### Comment by [@Nilsachy](https://github.com/Nilsachy) — 2026-03-09T13:29:05Z

> Since one of the flags is a path (--authentication-kubeconfig=/path/to/kubeconfig), there is a non-zero chance that it will contain whitespace (which is a valid path). This would lead to something like this: https://go.dev/play/p/pwK9lhVL8Fv, so it wouldn't parse.

One limitation we still need to address is how to pass complex, space-containing flags (like file paths) to the embedded visibility server without breaking the main Kueue flag parser. Here is a breakdown of the available options:

**1. Repeated String Slice Flag (Current Approach)**

We implement a custom `flag.Value` slice, allowing the user to specify the flag multiple times. Each invocation passes exactly one argument to the embedded server.

```Bash
--visibility-server "--secure-port=9443" \
--visibility-server "--authentication-kubeconfig=/path/with spaces/to/kubeconfig"
```

Pros: Very safe. It perfectly preserves spaces and quotation marks. This is also a recognized, idiomatic pattern in the Kubernetes ecosystem (e.g., passing multiple --set flags in Helm).

Cons: Verbose. Depending on how many custom configurations are needed, repeating the `--visibility-server` prefix can feel duplicative.

**2. Delimiter Flags (Start/End Markers)**

We introduce "marker" flags that tell the main parser to treat everything in between as a pass-through.

```Bash
--visibility-server-start --secure-port=9443 --authentication-kubeconfig=/path/to/kubeconfig --visibility-server-end
```

Pros: Feels closer to a standard command-line experience for the inner flags.

Cons: Non-standard and to prevent main.go's standard flag.Parse() from failing on "unknown" intermediate flags or worse, apply the flags to the general config, we would have to write custom pre-parsing logic to slice them out of os.Args before the main standard parser runs. This introduces some extra complexity and requires thorough testing to avoid edge-case panics.

**3. Single String with Whitespace Restrictions**

We pass all arguments inside a single string and simply split them by spaces in the code (the original behavior before addressing the space-in-path bug).

```Bash
--visibility-server-flags "--secure-port=9443 --authentication-kubeconfig=/path/to/kubeconfig"
```

Pros: Very concise and easy to implement code-wise (strings.Fields).

Cons: We would have to explicitly document that paths with whitespace are strictly forbidden. If a user provides a valid path containing a space, the naive string splitting will break the arguments and crash the server.

**4. Use a Lexer Library to Parse a Single String**

We keep the single-string approach but replace standard string splitting with a shell lexer library like [github.com/google/shlex](https://github.com/google/shlex). This allows the parser to respect quotes around paths.

```Bash
--visibility-server-flags="--secure-port=9443 --authentication-kubeconfig='/path/with spaces/to/kubeconfig'"
```

Pros: Maintains a concise, single-flag user experience while safely handling whitespace, provided the user wraps the problematic paths in quotes.

Cons: Introduces a new third-party dependency ([github.com/google/shlex](https://github.com/google/shlex)) to the project. It seems the library is out of support and little tested..
