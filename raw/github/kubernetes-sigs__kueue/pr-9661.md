#### Special notes for your reviewer:
When a workload's spec changes while the scheduler is evaluating it, `RequeueWorkload.info.Update` refreshes the inadmissible entry to the latest version, causing `PushOrUpdate`'s spec comparison to see no change and leaving the workload stuck in inadmissible. Track the evaluated generation at Pop() time so `PushOrUpdate` detects spec changes under the CQ lock regardless of caller freshness.

I ran the failed test with this change in a loop (200 times) and it works fine.

## Pull request overview

Fixes a race where workloads can remain stuck in the inadmissible queue after a priority/spec update by tracking the generation that was last evaluated at `Pop()` time.

**Changes:**
- Track `Generation` at `Pop()` in `lastEvaluatedGeneration` so `PushOrUpdate` can detect spec changes masked by `info.Update`.
- Clear `lastEvaluatedGeneration` entries when workloads are deleted or moved back to the active heap.
- Add a regression test covering the generation-change scenario for inadmissible workloads.
