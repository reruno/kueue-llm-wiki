## Pull request overview

Fixes a scheduler race where a workload whose priority/spec was updated during evaluation could remain stuck in the inadmissible queue, delaying rescheduling.

**Changes:**
- Add `LastEvaluatedGeneration` to `workload.Info` to track the generation observed at `Pop()` evaluation time.
- Update `ClusterQueue.PushOrUpdate` to treat a generation change since last evaluation as a spec change (so the workload is reconsidered for admission).
- Add a unit test covering the “generation changed while inadmissible” behavior, including backoff handling.

### Reviewed changes

Copilot reviewed 3 out of 3 changed files in this pull request and generated no comments.

| File | Description |
| ---- | ----------- |
| pkg/workload/workload.go | Adds `LastEvaluatedGeneration` field to `workload.Info` for tracking evaluation-time generation. |
| pkg/cache/queue/cluster_queue.go | Records generation at `Pop()` and uses it in `PushOrUpdate` to avoid masking spec changes after `RequeueWorkload` updates. |
| pkg/cache/queue/cluster_queue_test.go | Adds a regression test ensuring generation changes move workloads out of inadmissible (unless backoff prevents it). |
