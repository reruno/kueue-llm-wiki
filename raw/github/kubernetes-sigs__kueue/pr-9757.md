  Root Cause
  The flake occurs in the AfterEach cleanup of the LeaderWorkerSet e2e test. The current cleanup flow is:
  ```
  ginkgo.AfterEach(func() {
      gomega.Expect(util.DeleteAllLeaderWorkerSetsInNamespace(ctx, k8sClient, ns)).To(gomega.Succeed())
      util.ExpectAllPodsInNamespaceDeleted(ctx, k8sClient, ns)  // <-- times out here
      gomega.Expect(util.DeleteNamespace(ctx, k8sClient, ns)).To(gomega.Succeed())
      util.ExpectObjectToBeDeleted(ctx, k8sClient, cq, true)
      util.ExpectObjectToBeDeleted(ctx, k8sClient, rf, true)
  })
  ```
  
  The problem: After deleting the LWS objects, the test calls ExpectAllPodsInNamespaceDeleted which passively waits (up to 45s / LongTimeout) for all pods to disappear. But the cleanup chain is too long and unreliable, especially for the RecreateGroupOnPodRestart test case:
  1. DeleteAllLeaderWorkerSetsInNamespace uses DeletePropagationBackground — the LWS API object is removed, but cascading deletion of owned resources (StatefulSets, Pods) depends on the Kubernetes garbage collector.
  2. ExpectAllPodsInNamespaceDeleted (test/util/util.go:252-259) simply polls pods.Items to be empty. It does not actively delete pods or strip finalizers.
  3. The RecreateGroupOnPodRestart policy creates a race condition: as the LWS controller is cleaning up pods, the restart policy may trigger recreation of the pod group before the LWS deletion is fully acknowledged, causing new pods to appear that also need to be cleaned up.
  4. No workload finalizer stripping: Workloads owned by the LWS have ResourceInUseFinalizerName which blocks their deletion until the Kueue reconciler processes them. This further delays the cascade.
  In contrast, every other test (statefulset_test.go, deployment_test.go, pod_test.go, jobset_test.go, etc.) follows this pattern:
  // Pattern A (most tests): DeleteNamespace first, which internally strips all finalizers
  gomega.Expect(util.DeleteNamespace(ctx, k8sClient, ns)).To(gomega.Succeed())
  util.ExpectAllPodsInNamespaceDeleted(ctx, k8sClient, ns)
  // Pattern B (pod test inner AfterEach): explicitly delete pods with finalizer stripping
  gomega.Expect(util.DeleteAllPodsInNamespace(ctx, k8sClient, ns)).To(gomega.Succeed())
  util.ExpectAllPodsInNamespaceDeleted(ctx, k8sClient, ns)
  DeleteNamespace (test/util/util.go:132-177) internally calls both deleteAllPodsInNamespace (which strips PodFinalizer) and deleteWorkloadsInNamespace (which strips ResourceInUseFinalizerName), ensuring all blocking finalizers are actively removed.
  The LWS test is the only test that calls ExpectAllPodsInNamespaceDeleted before DeleteNamespace, and without any active finalizer/pod cleanup first.
  
  Proposed Fix
  Reorder the AfterEach to call DeleteNamespace first (which handles active pod/workload cleanup), then check that pods are gone:
  ```
  ginkgo.AfterEach(func() {
      gomega.Expect(util.DeleteAllLeaderWorkerSetsInNamespace(ctx, k8sClient, ns)).To(gomega.Succeed())
      gomega.Expect(util.DeleteNamespace(ctx, k8sClient, ns)).To(gomega.Succeed())
      util.ExpectAllPodsInNamespaceDeleted(ctx, k8sClient, ns)
      util.ExpectObjectToBeDeleted(ctx, k8sClient, cq, true)
      util.ExpectObjectToBeDeleted(ctx, k8sClient, rf, true)
  })
  ```
  This matches the pattern used by all other e2e tests. DeleteNamespace will:
  1. Delete all pods in the namespace and actively strip PodFinalizer from any that have it
  2. Delete all workloads and strip ResourceInUseFinalizerName
  3. Delete the namespace itself
