## Pull request overview

This PR is a cherry-pick of #9731 onto the `release-0.15` branch. It refactors the `workload.Info` scheduling hash computation so that production callers can provide a contextual logger (rather than using the global fallback `log.Log`) when computing the scheduling hash via a new `UpdateSchedulingHash(log)` method. The `NewInfo` signature is unchanged.

**Changes:**
- Adds a new `UpdateSchedulingHash(log logr.Logger)` method to `workload.Info`, and removes the hash computation from `NewInfo` — callers must now explicitly call `UpdateSchedulingHash` after construction.
- Updates all production call sites (`addOrUpdateWorkload` in `clusterqueue.go` and three call sites in `manager.go`) to call `UpdateSchedulingHash` with the appropriate logger.
- Adds a new `TestSchedulingHash` test in `workload_test.go` validating hash equality and inequality across different workload configurations.

  The assertion on line 1977 checks that `info1.SchedulingHash == ""` (empty string), but `computeSchedulingHash` never returns an empty string — it returns either a 16-character hex hash or `SchedulingHashUnknown` ("unknown"). When the `SchedulingEquivalenceHashing` feature gate is enabled (the default), the hash will be a hex string, and when disabled, it will be `"unknown"`. The empty-string check will therefore always pass regardless of the feature gate state, making it a vacuous assertion that doesn't actually validate hash computation.
