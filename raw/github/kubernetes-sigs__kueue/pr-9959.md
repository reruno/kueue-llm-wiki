```diff
+# Priority Boost Controller
+
+The `priority-boost-controller` is an experimental controller that implements
+time-sharing fairness for Kueue Workloads by temporarily boosting the effective
+priority of admitted workloads, preventing them from being preempted by
+same-priority peers for a configurable minimum duration.
+
+## Purpose
+
+In a ClusterQueue with a fixed quota, multiple workloads of the same priority
+and unknown execution duration can lead to starvation: the first admitted
+workload monopolizes resources while others wait indefinitely.
+
+This controller addresses that by:
+
+1. Setting `kueue.x-k8s.io/priority-boost` to a large value (`boostValue`)
+   the moment a workload is admitted, making it appear higher-priority than
+   any same-base-priority pending workload.
+2. Keeping the boost active for `minAdmitDuration` (e.g., 30 minutes),
+   guaranteeing the workload cannot be preempted by same-priority peers during
+   that window.
+3. Removing the boost once the window expires, restoring normal preemption
+   eligibility — enabling round-robin style fairness at the admission level.
+
+This component demonstrates an external policy controller for KEP-7990
+(priority boost signal). It is not part of the main Kueue binary and is
+intended to be built and deployed independently.
+
+**Requires**: the `PriorityBoost` feature gate must be enabled in Kueue, and
+the ClusterQueue must have `withinClusterQueue: LowerOrNewerEqualPriority` (or
+`Any`) preemption configured.
+
+## How It Works
```

  Yeah, current implementation only solve the problem of some workload being not preempted during minDuration. But that doesn't mean controller cannot choose it to stay more. To have some fairness here(and round robin between workloads), I can propose the following:
  controller manages three states for each admitted workload via the `kueue.x-k8s.io/priority-boost` annotation:
  
  | Phase | Condition | Boost Value | Effect |
  |-------|-----------|-------------|--------|
  | **Protected** | Admitted < minAdmitDuration | `+boostValue` (e.g. +100000) | Cannot be preempted by same-priority peers (`effectivePriority = base + boost`) |
  | **Preemptable** | Admitted >= minAdmitDuration | `-boostValue` (e.g. -100000) | Lower effective priority than pending peers; CAN be preempted via `LowerPriority` policy |
  | **Neutral** | Not admitted (pending/evicted) | annotation removed | Normal priority in queue ordering |
  
  ### Why Three Phases Are Necessary
  
  A naive two-phase approach (boost during window, remove after) fails because:
  
  1. **`LowerOrNewerEqualPriority`** uses `CreationTimestamp` for ordering (`workload.go:1263`). A workload created before its peers can never be preempted by them via this policy, regardless of runtime.
  2. **`Any`** policy bypasses priority entirely (`preemption_policy.go:49`), so positive boost has no protective effect during the window.
  3. **`LowerPriority`** requires strictly higher effective priority — if boost is simply removed (both at 0), same-priority workloads can't preempt each other.
  
  The **negative boost** after window expiry solves this: it makes the workload appear as `effectivePriority = base - boostValue`, which is strictly lower than any same-base-priority pending workload (`effectivePriority = base`), enabling preemption via `LowerPriority`.
  
  ### Prerequisite ClusterQueue Configuration
  
  ```yaml
  apiVersion: kueue.x-k8s.io/v1beta2
  kind: ClusterQueue
  spec:
    preemption:
      withinClusterQueue: LowerPriority  # required for negative boost to trigger preemption
  ```
  
  The `PriorityBoost` feature gate must be enabled in Kueue.
  
  ### End-to-End Round-Robin Trace
  
  Setup: CQ with 8 GPU quota, `withinClusterQueue: LowerPriority`, `basePriority = 0` for all workloads.
  
  1. **T=0**: A created, admitted. Controller sets `boost = +100000` → `effectivePriority = 100000`
  2. **T=1**: B created, queued. No annotation → `effectivePriority = 0`
  3. **T=2**: C created, queued. No annotation → `effectivePriority = 0`
  4. **T=30m**: Window expires. Controller sets `boost = -100000` on A → `effectivePriority = -100000`
  5. **Scheduler**: B (priority 0) wants to preempt A (priority -100000). `SatisfiesPreemptionPolicy(B, A, LowerPriority)`: `0 > -100000 = true` → **B preempts A**
  6. **A re-queued**: Controller clears boost (not admitted). A returns to queue at `effectivePriority = 0`.
  7. **T=31m**: B admitted, controller sets `boost = +100000` on B.
  8. **T=61m**: B's window expires → `boost = -100000`. C preempts B.
  9. **Cycle continues**: A → B → C → A → ...

  Kueue only lets a workload preempt another if the preemptor’s effective priority is strictly higher than the victim’s. If they’re equal, preemption doesn’t happen. So while our workload is still at base priority (no annotation) and waiting peers are also at base, they can’t preempt it under that policy — including workloads that don’t use the priority-boost controller at all. They’re just tied at the same priority.
  
  What we really need for the handoff after minAdmitDuration is to make the running workload strictly lower than same-priority waiters. That’s what the negative boost does. The positive boost isn’t doing extra work for that “same priority can’t preempt same priority” rule under LowerPriority.
  
  A few things to keep in mind if we drop the positive phase:
  
  This reasoning is really about LowerPriority. Other preemption modes behave differently, so we shouldn’t assume the same story everywhere.
  Effective priority is also used in a few ordering paths (not only this preemption check). A big positive boost changes those tie-breaks; no positive boost avoids that side effect — worth a quick pass to make sure we still like the behavior.
  Waiters with higher priority could always preempt anyway; that doesn’t depend on us adding a positive boost.
  So I’m aligned: we can likely get the same behavior with no boost during the window, negative after it expires, clear when not admitted — and it might even be simpler (no race to slap on a positive annotation).
