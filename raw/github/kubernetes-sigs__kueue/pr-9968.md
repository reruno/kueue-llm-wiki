## Description

## What this PR does

Enables `AllowWatchBookmarks` on MultiKueue remote client watch connections and adds explicit handling for bookmark events.

## Why this is needed

The MultiKueue remote client uses raw `client.Watch()` calls to watch resources (Jobs, RayJobs, Workloads, etc.) on worker clusters. Unlike informers and reflectors — which set `AllowWatchBookmarks: true` by default — these raw watch calls do not request bookmark events from the API server.

When the network path between the manager and worker cluster includes an HTTP reverse proxy with an idle connection timeout (e.g., Cloudflare Tunnel with a default ~120s `proxy_read_timeout`), watch connections for infrequently-changing resources are terminated because no data flows within the timeout window. This causes:

- Repeated `Unable to start the watcher` errors with HTTP 524 responses
- `MultiKueueCluster` condition flipping to `Active=False` / `ClientConnectionFailed`
- All watches being torn down and restarted in a ~2-minute failure loop

With `AllowWatchBookmarks: true`, the API server sends periodic `BOOKMARK` events (~60s interval) that keep the connection alive through proxies with idle timeouts.

## Why this hasn't been reported before

Most MultiKueue deployments connect clusters on the same private network (same VPC, VPN, or cloud-provider peering), where connections are L4 TCP with no HTTP-level idle timeouts. The missing bookmarks only become a problem when the manager-to-worker path traverses an L7 HTTP reverse proxy that enforces idle connection timeouts — an uncommon but valid topology for multi-cloud or cross-region setups.

## Changes

1. **`startWatcher()`**: Added `AllowWatchBookmarks: true` via `client.ListOptions{Raw: &metav1.ListOptions{AllowWatchBookmarks: true}}` to the `Watch()` call
2. **Watch event loop**: Added explicit `case watch.Bookmark:` to handle bookmark events gracefully (log at V(5) and continue) instead of falling into the `default` case which attempts workload key extraction

## How this was verified

Tested with a MultiKueue setup where the manager cluster connects to a worker cluster through a Cloudflare Tunnel:

- **Without bookmarks**: Watch connections die after exactly ~128s with Cloudflare 524 errors
- **With `allowWatchBookmarks=true`**: Watch connections survive indefinitely, with bookmark events arriving every ~60s

All existing unit tests pass (`go test ./pkg/controller/admissionchecks/multikueue/...` and `go test ./pkg/controller/...`).

## Context

The fix aligns the MultiKueue watch behavior with how client-go's `Reflector` and controller-runtime's cache layer already handle watches — both enable `AllowWatchBookmarks` by default.


```release-note
MultiKueue: Enable AllowWatchBookmarks for remote client watches to prevent idle watch connections from being terminated by HTTP proxies with idle timeouts (e.g., Cloudflare 524 errors).
```

### Comment by [@trilamsr](https://github.com/trilamsr) — 2026-03-18T07:02:48Z

Thanks for the quick review @mimowo! Great question.

You're right that the watch gets interrupted every ~2min and the reconnection mechanism kicks in. But the cluster does end up marked `Active=False` immediately — here's what we observed:

When a watch dies, the reconciler tries to re-establish it by calling `setConfig` → `startWatcher`. The `client.Watch()` call at line 207 is synchronous — it blocks until the server returns the initial response. Through the proxy, this hangs for ~120s before the proxy kills it and returns a 524 error page as the HTTP response body. That error bubbles up to `Reconcile` (line 468), which sets `Active=False` / `ClientConnectionFailed` right away.

So in practice the cluster flaps in a loop: watch starts → runs ~120s → proxy kills it → reconcile tries to reconnect → `Watch()` blocks ~120s → 524 → `Active=False` → retry with backoff → repeat.

Regarding `WorkerLostTimeout` — I think that's working as intended. It's a grace period for workloads that already have reservations on a remote cluster, so they don't get bounced immediately when a worker hiccups. But it doesn't (and probably shouldn't) prevent the cluster-level condition from reporting `ClientConnectionFailed` when the connection genuinely fails.

The reason `batch/v1 Job` was the one consistently failing is that our cluster only had 6 jobs that rarely change — so the watch sits completely idle and the proxy kills it. Busier resources like Workloads sometimes survived because events flowed often enough to reset the proxy's idle timer, but not reliably.

The bookmark fix prevents all of this by ensuring the API server sends a heartbeat every ~60s, well within the proxy's 120s timeout.

### Comment by [@trilamsr](https://github.com/trilamsr) — 2026-03-18T17:30:00Z

You're right — existing jobs survive the flapping thanks to `WorkerLostTimeout` (default 5m). Running workloads with reservations on the worker don't get evicted during the ~2min reconnection cycles, so no data loss or job failures.

The pain is more operational — while the cluster is marked Active=False, new workloads can't be dispatched to that worker. And the constant tear-down/rebuild of all watches adds unnecessary load on the worker's API server. It doesn't break things, but it makes the cluster unreliable for scheduling and noisy to monitor.

### Comment by [@trilamsr](https://github.com/trilamsr) — 2026-03-18T17:55:44Z

Yeah absolutely! Here are the pointers:

client-go's Reflector hardcodes it in both watch paths:
- `tools/cache/reflector.go:534` — `AllowWatchBookmarks: true`
- `tools/cache/reflector.go:764` — `AllowWatchBookmarks: true`

controller-runtime's cache defaults it to true when not explicitly set:
- `pkg/cache/cache.go:521` — `EnableWatchBookmarks: ptr.Deref(config.EnableWatchBookmarks, true)`

And that gets propagated to every informer watch call in `pkg/cache/internal/informers.go` (lines 389, 474, 519, 563).

So every informer-based watch in Kueue already requests bookmarks — this PR just brings the raw `client.Watch()` calls in MultiKueue in line with that.
